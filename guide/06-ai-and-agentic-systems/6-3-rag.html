6.3

RAG — Retrieval-Augmented Generation

Why LLMs need retrieval to work in enterprise, how the chunk-embed-store-retrieve pipeline works, when to choose RAG over fine-tuning, and how to pick a vector database for your Azure stack

✓

When you have completed this section, you should be able to:

  1. Explain why LLMs cannot answer questions about private or recent data without external retrieval
  2. Describe each stage of the RAG pipeline: chunking, embedding, indexing, retrieval, augmentation, generation
  3. Compare chunking strategies and explain the tradeoffs of chunk size for retrieval quality
  4. Distinguish between dense (semantic) search, keyword search, and hybrid search — and explain why hybrid often wins in production
  5. Select the appropriate vector database for a given scenario, and justify Azure AI Search for enterprise Azure workloads

6.3.1

The Problem RAG Solves

An LLM is trained on a snapshot of the internet up to a specific date. It has no knowledge of your internal documentation, your data pipelines, your business rules, your product catalog, or the incident that happened last Thursday. Ask it about any of these things and it will either decline to answer, guess plausibly, or — most dangerously — confabulate confidently.

This is not a temporary limitation that better models will resolve. It is structural. Training is expensive and slow. Even if a model is updated quarterly, it will always lag. And it will never have access to proprietary organizational knowledge that was never published on the internet.

Retrieval-Augmented Generation (RAG) sidesteps the training gap entirely. Instead of asking "what did the model memorize about X?", RAG asks "what do our documents say about X?" — retrieves those documents at query time, and feeds them to the model as context. The model does not need to know the answer in advance; it needs to read and synthesize the answer from what you give it.

The insight is elegant: LLMs are extraordinarily good at reading, understanding, and synthesizing text that is put in front of them. RAG exploits that capability by doing the retrieval work separately, then handing the result to the model. The model contributes comprehension and synthesis; a search system contributes relevance and recency.

This architecture has three important implications for data engineers. First, the quality of your retrieval system directly determines the quality of your answers — garbage in, garbage out applies to RAG at least as much as it does to pipelines. Second, the documents in your vector store must be maintained and kept current, which is a data engineering problem, not an AI problem. Third, the chunking, embedding, and indexing pipeline that feeds the vector store is a data pipeline — it needs the same care (idempotency, schema management, monitoring) as any other pipeline you build.

6.3.2

The RAG Pipeline

RAG consists of two distinct phases: an offline ingestion phase that runs once (or on schedule), and an online query phase that runs at request time.

Phase 1 — Ingestion

During ingestion you prepare your documents for retrieval. The steps are: load the source documents (PDFs, Word files, web pages, database records, Confluence pages), chunk them into smaller segments, embed each chunk into a dense vector using an embedding model, and index those vectors in a vector database. This phase typically runs as a scheduled pipeline — daily or on document change events.

Phase 2 — Query

At query time: take the user's question, embed it using the same embedding model used during ingestion, search the vector database for the top-k most similar document chunks, assemble those chunks into a context block, augment the original prompt with that context, and pass the whole thing to the LLM for generation. The LLM reads the retrieved context and generates a grounded answer.

Figure 6.5The RAG pipeline in two phases. Ingestion (top) is an offline batch process: source documents are chunked, embedded into dense vectors, and stored in a vector database. This runs once per document and is updated when documents change. The query phase (bottom) is the real-time path: the user's question is embedded using the same model, the nearest matching chunks are retrieved, and the resulting context is appended to the prompt before the LLM generates a grounded response.
6.3.3

Chunking Strategies

Chunking is deceptively important. The quality of your RAG system is strongly determined by your chunking strategy, yet it is the step teams most often do carelessly. The fundamental tension is this: small chunks are precise but may lack the context needed to answer a question; large chunks carry more context but dilute the signal when used as retrieved context, consuming expensive tokens and potentially drowning the relevant detail.

The Main Strategies

Fixed-size chunking splits text into segments of N tokens (typically 256–512) with an optional overlap of 10–15% to prevent cutting sentences mid-thought. It is the simplest approach and works well for uniformly structured documents. The weakness: it can split a heading from its content, or a question from its answer.

Sliding window chunking generates overlapping chunks — e.g., chunks of 512 tokens with 100 tokens of overlap between consecutive chunks. This ensures that context spanning a chunk boundary is captured by at least one chunk. The tradeoff is redundancy: the same sentence appears in multiple chunks, which increases storage and may cause duplicated context in retrieval.

Semantic chunking splits on natural document boundaries — paragraphs, sections, or sentence groups with high semantic coherence. It requires more preprocessing (parsing document structure) but produces chunks that are complete, coherent thoughts. For internal documentation, policy documents, and technical specifications, semantic chunking reliably outperforms fixed-size.

Recursive / hierarchical chunking uses a hierarchy of splitters: try to split on double newlines first; if the chunk is still too large, split on single newlines; if still too large, split on sentences; if still too large, split on tokens. This is the approach used by LangChain's RecursiveCharacterTextSplitter and is a solid default for mixed document types.

Figure 6.6Three chunking strategies applied to the same source document. Fixed-size chunking (top) is simple but may split concepts mid-sentence. Sliding window (middle) adds overlap to prevent boundary misses, at the cost of redundancy. Semantic chunking (bottom) respects the document's own structure — each chunk is a complete section. The right choice depends on the document type: semantic chunking wins for structured documents; sliding window is a safer default for heterogeneous corpora.
6.3.4

Embeddings and Vector Search

An embedding is the mathematical engine behind RAG retrieval. When text is passed to an embedding model, it returns a dense vector — typically 1,536 or 3,072 floating-point numbers for current OpenAI models — that encodes the semantic meaning of the text. The key property: texts with similar meanings produce vectors that are numerically close in high-dimensional space.

Similarity between two vectors is measured using cosine similarity — the cosine of the angle between them. Two identical texts have a cosine similarity of 1.0. Two completely unrelated texts might have a similarity near 0. In practice, a score above 0.85 typically indicates strong semantic relevance for retrieval purposes.

The critical rule: the embedding model used during ingestion must be the same model used at query time. Changing embedding models means re-embedding every document in your vector store — there is no way to compare vectors from different models.

Hybrid Search

Pure semantic search (dense vector retrieval) has a weakness: it can miss documents that contain the exact keywords in the query, because keyword matching is not what it optimizes for. A query for "GDPR Article 17 right to erasure" may retrieve documents about data privacy in general rather than the specific article.

Hybrid search combines dense vector search with sparse keyword search (BM25, the algorithm behind traditional full-text search). The two scores are merged — typically using Reciprocal Rank Fusion (RRF) — before returning results. Hybrid search consistently outperforms either method alone in production benchmarks. Azure AI Search implements hybrid search natively, making it the most pragmatic choice for enterprise Azure RAG implementations.

Go DeeperEmbedding models compared — ada-002 vs text-embedding-3-large vs Azure OpenAI

text-embedding-ada-002 was OpenAI's standard embedding model for two years. It produces 1,536-dimensional vectors and performs well for general-purpose retrieval. Cost: $0.0001 per 1,000 tokens.

text-embedding-3-small (2024) outperforms ada-002 on most benchmarks while costing 5x less ($0.00002/1K tokens). It is the recommended default for cost-sensitive applications.

text-embedding-3-large produces 3,072-dimensional vectors and achieves the best retrieval quality for complex documents. Cost: $0.00013/1K tokens — about 30% more than ada-002 but with materially better multilingual and technical document performance.

Azure OpenAI embedding endpoints expose the same models (ada-002 and text-embedding-3) but within your Azure subscription — meaning the same RBAC, private endpoints, and compliance posture as your other Azure services. For enterprise workloads where data sovereignty matters, this is the required choice over the public OpenAI API.

Open-source alternatives: BAAI/bge-large-en-v1.5 and sentence-transformers/all-MiniLM-L6-v2 are strong self-hostable options for organizations that cannot send data to external APIs. bge-large consistently ranks near the top of MTEB (Massive Text Embedding Benchmark) leaderboards at zero API cost.

6.3.5

The vector database is where your embedded chunks live. The choice of vector store affects latency, cost, operational complexity, and — critically for enterprise Azure workloads — compliance, access control, and network security. Go DeeperWhy Azure AI Search wins for enterprise Azure workloads — and how to set it up

Azure AI Search (formerly Azure Cognitive Search) is not just a vector database — it is a full enterprise search platform that added vector capabilities. This heritage matters: it already has mature implementations of RBAC, private endpoints, customer-managed keys, and Azure Active Directory integration. When you are building RAG in an Azure environment where the compliance team requires data residency and audit logging, AI Search removes a significant number of compliance conversations that you would otherwise have to have about a third-party vector service.

The setup path for RAG: create an Azure AI Search resource → define an index with both vector (HNSW) and keyword (BM25) fields → use the Azure OpenAI embedding endpoint to generate vectors → push documents via the REST API or SDK → at query time, use the Retrieval Augmented Generation pattern in Azure AI Foundry or direct API calls. Microsoft's own reference architecture for RAG with Azure OpenAI uses AI Search as the retrieval layer.

Pricing: the Basic tier (~$75/month) supports up to 15GB of index storage and 3 replicas — sufficient for most internal documentation RAG applications. The Standard S1 tier scales to 160GB with higher query throughput. For very large corpora (>1M documents), the Standard S3 HD tier supports higher density at lower per-document cost.

One gotcha: the vector dimensions must be specified at index creation time and cannot be changed without deleting and recreating the index. If you switch from ada-002 (1,536 dims) to text-embedding-3-large (3,072 dims), you must re-index everything. Define your embedding model before creating the index.

6.3.6

RAG vs Fine-Tuning vs Few-Shot

Every team building with LLMs eventually faces the same question: "Should we fine-tune the model on our data?" The answer is almost always no — and understanding why requires being clear about what each approach actually does.

Approach What It Does When It Wins Cost Maintenance
Few-shot prompting Includes examples directly in the prompt to demonstrate desired output format or style Format/style calibration; classification tasks; when examples fit in the context window Very low — no additional infra Low — update examples in prompt text
RAG Retrieves relevant documents at query time and injects them as context Factual grounding on private/recent data; knowledge-intensive Q&A; document search Moderate — vector store + embedding API costs Medium — keep document store current; monitor retrieval quality
Fine-tuning Continues training the model's weights on domain-specific data Specialized reasoning style; rare vocabulary; when latency and context cost must be minimized High — GPU training + model hosting High — re-train when data or requirements change

The key insight: fine-tuning teaches style; RAG teaches facts. If your goal is for the model to answer questions accurately about your internal data, fine-tuning will not achieve that — fine-tuned models still hallucinate facts just as readily, because training embeds statistical patterns, not factual lookup tables. For factual grounding, RAG is the correct architecture. Fine-tuning is appropriate when the model must adopt a specific response style, perform a specialized task type (e.g., medical record coding), or operate in a latency-constrained environment where a large context window cannot be used.

+
Go DeeperHow to evaluate a RAG system — RAGAS, precision, recall, and what production monitoring looks like

RAG systems have two independent failure modes: bad retrieval (the right chunks are not returned) and bad generation (the LLM ignores or misinterprets the retrieved chunks). Evaluation must address both.

Retrieval metrics: Precision@K measures what fraction of the top-K returned chunks are actually relevant. Recall@K measures what fraction of all relevant chunks are returned in the top-K. Mean Reciprocal Rank (MRR) rewards systems that put the most relevant chunk first. These require a labeled evaluation set (questions + ground-truth relevant documents) — which is often the most expensive part of RAG evaluation to build.

Generation metrics: RAGAS (an open-source framework) automates evaluation using an LLM-as-judge approach. It measures: Faithfulness (does the answer stay within the retrieved context?), Answer Relevance (does the answer address the question?), Context Precision (are the retrieved chunks actually used?), and Context Recall (does the retrieved context contain everything needed to answer?). Faithfulness is the most important metric for production — an answer that strays beyond the context is a hallucination risk.

Production monitoring: At minimum, log every query, the top-5 retrieved chunks, and the generated answer. Set up periodic manual review (sample 50 queries per week and grade them). Track user thumbs-down feedback rates if you have a UI. Sudden drops in average retrieval score often indicate that the source documents have changed in structure — a schema drift equivalent for your knowledge base.

🏢Field Note
A Professional Services Firm  ·  Internal Knowledge Base RAG

A global consulting firm had 11 years of internal delivery methodology documents, client templates, and post-project retrospectives — approximately 40,000 documents — stored in a SharePoint that nobody used because it was too slow to search and nobody knew what was in it. New project teams routinely recreated templates that already existed and ignored hard-won lessons from past engagements because they could not find them.

The firm built a RAG system using Azure AI Search as the vector store, Azure OpenAI text-embedding-3-large for embeddings, and GPT-4o for generation. The ingestion pipeline ran nightly, chunking new and updated SharePoint documents semantically (splitting on heading boundaries), generating embeddings, and upserting into AI Search. The query interface was a Teams bot that answered questions like "Do we have a template for a data governance framework?" or "What did we learn from our SAP migrations?"

The critical design decision was hybrid search: pure semantic search missed documents that used different terminology from the query. A query for "data quality framework" failed to retrieve a high-quality document titled "Information Asset Governance Standards" — they were semantically related but not synonymous. Hybrid search (RRF combining BM25 keyword match with vector similarity) reliably retrieved it. Enabling hybrid search improved precision@5 from 61% to 84% on the evaluation set.

The Lesson

Semantic search alone is not enough when domain vocabulary is inconsistent. Hybrid search is the production default. The ingestion pipeline is a real data engineering problem — it needs idempotency, monitoring, and freshness guarantees just like any other pipeline you build.

🏢Field Note
A Financial Services Team  ·  Regulatory Document Q&A

A risk team needed to answer questions against a corpus of 200+ regulatory documents — Basel III frameworks, FCA guidance papers, internal risk policies — that changed quarterly. A compliance analyst could spend half a day finding the specific clause that governed a particular transaction type. The ask was: can we ask a question in plain English and get the right regulatory citation in under 30 seconds?

The team built RAG with a critical constraint: every answer had to include the exact document name, section number, and page number it was sourced from. Pure RAG without citation enforcement failed this — the LLM would synthesize an answer from multiple chunks and lose the provenance. The solution was to include chunk metadata (document name, section, page) in every embedded chunk, and to instruct the LLM via system prompt to always cite sources using the metadata format. Answers that could not cite a source were flagged as "insufficient evidence" rather than generating an unsourced response.

The chunking strategy was deliberately conservative: 200-token fixed-size chunks with no overlap, to ensure that retrieved chunks mapped cleanly to specific paragraphs. Larger chunks made citations ambiguous (which part of the chunk was the source?). Precision over recall was the right tradeoff for a regulatory context.

The Lesson

In regulated contexts, citation and provenance are not optional features — they are the product. Design your RAG chunking strategy and system prompt to enforce citation from the start, not as an afterthought. Smaller chunks improve citation precision at the cost of recall; know which direction to err in your use case.

Question 1 of 5

An LLM is trained on data up to October 2024. A user asks about a regulatory change published in March 2025. Which approach is most appropriate to ensure an accurate, grounded answer?

A
Fine-tune the model on the March 2025 regulatory document
B
Implement RAG: embed the regulatory document, store it in a vector database, retrieve the relevant chunks at query time, and augment the prompt
C
Increase the model's temperature to encourage more creative reasoning about recent changes
D
Include a note in the system prompt stating that the model should reason about regulatory changes after its training cutoff

B is correct. RAG is specifically designed for this scenario: grounding LLM responses in external documents that post-date its training. Fine-tuning (A) would update the model's weights but is prohibitively expensive for a single document, and fine-tuned models still hallucinate rather than lookup facts. Temperature (C) controls randomness, not knowledge access — increasing it would produce more varied but not more accurate responses. A system prompt instruction (D) cannot give the model access to information it was not trained on; it can only encourage the model to acknowledge uncertainty.

Question 2 of 5

A team builds a RAG system and finds that retrieval quality is poor for queries using different terminology than the source documents (e.g., "leave entitlement" vs "annual vacation policy"). What change would most directly address this?

A
Reduce chunk size to 128 tokens to improve semantic precision
B
Switch to a larger embedding model with more dimensions
C
Enable hybrid search (semantic vector search combined with BM25 keyword search via RRF)
D
Add a query expansion step that rephrases the user's question before embedding

C is correct. The problem described is a classic vocabulary mismatch — semantic search alone handles paraphrases well but struggles when domain-specific terms diverge (leave vs vacation). Hybrid search combines dense vector similarity with BM25 keyword matching; BM25 will score highly on exact keyword matches even when the semantic similarity is lower. This consistently improves retrieval recall for specialized domains. Reducing chunk size (A) would not help with vocabulary mismatch. A larger embedding model (B) might marginally improve semantic similarity but would not resolve a fundamental terminology gap. Query expansion (D) is a valid supplementary technique but adds latency and complexity, and the simpler hybrid search approach should be tried first.

Question 3 of 5

A team embeds their document corpus using text-embedding-3-large (3,072 dimensions) and stores the vectors in Azure AI Search. Six months later, they decide to switch to a more cost-effective model, text-embedding-3-small (1,536 dimensions). What must happen?

A
Update the query-time embedding only — the existing index remains valid at a reduced dimension count
B
Add a mapping layer that converts 3,072-dimensional vectors to 1,536 at query time
C
Delete and recreate the index with the new dimension count, then re-embed and re-index all documents with the new model
D
Nothing — Azure AI Search automatically handles dimension mismatches between models

C is correct. Vector similarity search requires all vectors in the index to have the same dimensionality as the query vector — comparing a 3,072-dimensional query vector to 1,536-dimensional document vectors produces no valid result. Azure AI Search enforces the vector dimension at index definition time and will not accept documents with different dimensions. You must delete the index, create a new one specifying 1,536 dimensions, and re-embed and re-index every document. This is the "embedding model migration tax" — it's why you should make the embedding model choice carefully before creating the index, as the Deep Dive noted. Dimension-reducing mappings (B) exist academically (PCA, Matryoshka), but Azure AI Search does not apply them automatically and using them would require custom preprocessing.

Question 5 of 5

A team is building a RAG system and debating between fine-tuning GPT-4o on their legal document corpus vs using RAG with AI Search. The legal documents change quarterly and require citation-level traceability. Which approach is correct and for what reason?

A
Fine-tune, because a fine-tuned model will memorize the legal documents and retrieve specific citations accurately
B
RAG, because fine-tuning embeds statistical patterns not factual lookup — it still hallucinates, cannot cite sources, requires re-training when documents change, and is far more expensive
C
Fine-tune once now, then switch to RAG when the documents update quarterly
D
Fine-tune for the legal vocabulary, then add RAG on top for citations

B is correct. This question tests the core mental model for when each approach applies. Fine-tuning does not make a model able to retrieve specific citations — it trains the model to produce outputs that statistically resemble the training data. A fine-tuned model presented with a legal question will not "look up" the answer; it will generate the most plausible-sounding response, which may be wrong. When documents change quarterly, re-fine-tuning quarterly costs tens of thousands of dollars in GPU compute and requires labeled training data. RAG, by contrast, requires only re-running the ingestion pipeline (chunk → embed → index) when documents change — a scheduled data engineering job. Source metadata in chunks enables citation out of the box. The combination of factual accuracy, traceability, low maintenance cost, and update frequency all point to RAG.

RAG in One Sentence

Retrieve relevant chunks from your own documents → inject into the prompt → let the LLM synthesize an answer. The model provides comprehension; your search system provides relevance and recency.

Same Embedding Model

The embedding model used at ingestion must be identical to the model used at query time. Different models produce incomparable vector spaces. Changing models means re-embedding everything.

Fine-tune ≠ Factual Recall

Fine-tuning teaches style, not facts. A fine-tuned model still hallucinates answers to factual questions — it produces plausible-sounding completions, not verified lookups. Use RAG for factual grounding.
