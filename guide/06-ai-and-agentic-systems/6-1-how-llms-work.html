6.1

How LLMs Work

Tokens, context windows, embeddings, hallucination, temperature, and the foundational mechanics that every data engineer deploying AI in a production pipeline needs to understand — without the PhD

6.1.1

Why Data Engineers Need to Understand This

Large Language Models (LLMs) — the AI systems behind ChatGPT, Claude, Gemini, and GitHub Copilot — are no longer external tools that sit beside the data stack. They are being embedded inside it. LLMs generate SQL from natural language descriptions. They summarise pipeline run logs and flag anomalies. They power chatbots that query your Gold layer semantic model in plain English. GitHub Copilot autocompletes your dbt models. Azure Fabric's AI features generate data transformations from business descriptions. Microsoft Copilot in Power BI writes DAX measures when you describe what you want.

A data engineer who treats LLMs as black boxes — who uses them without understanding their failure modes — will eventually deploy one in a context where it fails quietly and expensively. The most dangerous LLM failure mode in data engineering is not the model refusing to answer. It is the model confidently producing a wrong answer that looks correct: a SQL query that references a non-existent table, a column mapping that silently drops records, or a data quality summary that inverts the finding.

You do not need to understand transformer architecture or backpropagation to use LLMs safely in production. You need to understand tokens, context windows, hallucination, and temperature. These four concepts explain the vast majority of production LLM failures in data engineering contexts.
6.1.2

Tokens, Context Windows, and Embeddings

Tokens: The Unit of Everything

LLMs do not process text character by character or word by word. They process tokens — subword units that fall somewhere between characters and words. A common rule of thumb: one token is roughly three-quarters of a word in English, or approximately four characters. The word "running" is one token. "Uncharacteristically" is likely two or three. A SQL keyword like "SELECT" is one token. A code block of 200 lines might be 600 tokens.

Tokens matter for two reasons. First, API costs: LLM APIs charge per million tokens in (input) and per million tokens out (output). Sending a 10,000-row CSV as context to an LLM will cost significantly more than sending a 500-row summary. For data engineers building LLM pipelines, token counting is cost engineering. Second, context limits: every LLM has a maximum number of tokens it can process in a single request — the context window. Anything beyond that limit is silently truncated or results in an error.

Context Windows: The Model's Working Memory

The context window is the total amount of text (measured in tokens) that an LLM can "see" simultaneously — the conversation history, the system instructions, the retrieved documents, and the current question, all together. Think of it as the model's working memory. Context window size has increased dramatically since 2022: early GPT models had 4,096-token limits. By 2025, context windows of 128,000 to 1 million tokens are common across major models. Meta's Llama 4 Scout introduced context windows of up to 10 million tokens — enough to process an entire large codebase in a single request.

For data engineers, context window size determines: how much schema documentation you can include when asking an LLM to generate SQL (too little context = wrong table and column names), how long a pipeline log you can pass for error diagnosis, and whether an entire dbt project's models can be included in a single prompt for cross-model analysis.

Embeddings: Meaning as Numbers

An embedding is a numerical representation of text — a vector of hundreds or thousands of floating-point numbers that encodes the semantic meaning of the text. The critical property: text with similar meaning produces similar embeddings (close together in vector space); text with different meaning produces dissimilar embeddings (far apart). "Customer churn" and "subscriber cancellation" will have similar embeddings even though they share no words.

Embeddings are the foundation of Retrieval-Augmented Generation (RAG, section 6.3) — the technique used to give LLMs access to your organisation's data without fine-tuning. The embedding models used by Azure OpenAI (text-embedding-3-small and text-embedding-3-large) produce vectors that are stored in a vector database and queried at runtime to find the most semantically relevant context for a given question.

6.1.3

How Text Generation Works — and Why Hallucination Is Inevitable

An LLM generates text by predicting the next token, given all the tokens that came before. It does this one token at a time: produce a token, add it to the context, predict the next token from the expanded context, repeat. The prediction is a probability distribution over the entire vocabulary of possible next tokens — the model assigns a probability to every possible word or subword, and the output token is sampled from that distribution.

The critical implication: an LLM is not retrieving facts from a database. It is not running a search engine. It is predicting what text is most likely to follow the given input — based on patterns learned from an enormous corpus of human-generated text. When you ask it a question, it generates the most plausible-sounding continuation of the conversation. "Plausible" and "correct" are not the same thing.

Why Hallucination Is Structural, Not a Bug

Hallucination — when an LLM produces confident, fluent, factually wrong output — is not a defect that will be fixed in the next model version. It is an emergent property of the generation mechanism. A model that predicts the most plausible next token will occasionally produce text that sounds completely correct but refers to a non-existent fact, a wrong table name, or an incorrect calculation — because those wrong answers were the highest-probability continuation of the prompt, given the model's training.

For consumer use cases (writing assistance, summarisation), occasional hallucination is an acceptable quality issue. For data engineering use cases — LLM-generated SQL executed against a production database, LLM-summarised data quality reports distributed to executives, LLM-generated ETL logic — hallucination is a pipeline failure mode. A hallucinated table name silently drops records. A hallucinated column mapping produces wrong numbers. These failures do not raise exceptions; they produce incorrect output that may not be discovered for days or weeks.

6.1.4

Temperature and Determinism

Temperature is a parameter that controls how the model samples from the probability distribution over next tokens. At temperature 0, the model always selects the highest-probability token — the output is deterministic and reproducible. At temperature 1.0, the model samples according to the actual probability distribution — more varied, more creative, less predictable. At temperatures above 1.0, even low-probability tokens get amplified — outputs become increasingly random and often incoherent.

For data engineering tasks: set temperature to 0 (or as close as the API allows). When asking an LLM to generate SQL, classify a data quality issue, or extract structured information from text, you want the most probable correct answer every time — not creative variation. A SQL query that generates different results on each call is not a reliable pipeline component. Temperature 0 makes LLM outputs deterministic, repeatable, and testable — essential properties for any production system.

Temperature above 0 is appropriate for: creative writing assistance, brainstorming data product names, generating varied example data for testing. Never for SQL generation, code writing, data extraction, or any task where the output feeds directly into a pipeline.

6.1.5

The Prompt Structure — System, User, and Assistant Turns

Modern LLM APIs structure interaction as a conversation with three distinct role types. Understanding the purpose of each role is fundamental to writing effective prompts for data engineering use cases.

System prompt: The instructions that define the model's behaviour, role, and constraints for the entire conversation. Set once at the start. The system prompt is where you tell the model what it is ("You are a SQL assistant for our Azure Databricks environment"), what it knows ("The following tables exist in the Gold schema: fact_sales, dim_customer, dim_product, dim_date"), and what it must not do ("Never generate DROP or DELETE statements. Never reference tables not listed above"). For production data pipelines, the system prompt is your primary tool for constraining the model's behaviour and reducing hallucination surface area.

User turn: The current request. "Generate a SQL query that shows revenue by product category for the last 90 days, excluding returns." This is what changes with each request; the system prompt stays constant.

Assistant turn: The model's response. In a multi-turn conversation, prior assistant responses are included in the context for subsequent requests — allowing follow-up questions and iterative refinement.

Go DeeperPrompt engineering patterns for SQL generation and data tasks

Four prompt patterns that materially improve LLM output quality for data engineering tasks:

1. Schema grounding in the system prompt. The most important technique for eliminating SQL hallucination is listing the exact available tables and their key columns in the system prompt: "Available tables: fact_sales (columns: sale_sk, date_sk, customer_sk, product_sk, revenue_net, quantity, is_return). dim_customer (columns: customer_sk, customer_id, name, region, segment)." A model that is given the schema explicitly produces far fewer hallucinated table or column names than one given only the question. Include data types for columns where the type determines behaviour (date columns, numeric types).

2. Few-shot examples. Provide 2–3 examples of the exact input-output format you want. "Example: User asks 'show me top 10 customers by revenue last month'. Correct SQL: [example]. User asks 'what is the average order value by region'. Correct SQL: [example]." Few-shot examples teach the model the dialect (Spark SQL vs T-SQL vs BigQuery SQL), the table naming conventions, and the level of complexity you expect. This is more reliable than describing the format in words.

3. Chain-of-thought for complex queries. For multi-step analytical questions, add "Think step by step: first identify the tables and joins required, then the filters, then the aggregation, then write the query" to the user turn. Chain-of-thought prompting reduces errors on complex multi-join or window function queries by forcing the model to decompose the problem before writing the SQL.

4. Output format specification. For programmatic use (parsing the model's response in a pipeline), specify the exact output format: "Return only the SQL statement. Do not include markdown code fences, explanations, or comments. Return nothing other than valid Spark SQL." Without this, models frequently wrap output in markdown code blocks or include explanatory text that breaks downstream parsing.

6.1.6

Fine-Tuning, Prompting, and RAG — The Decision Framework

Three approaches exist for making an LLM useful in a specific domain. They are often confused or conflated. Understanding when each is appropriate prevents significant wasted engineering effort.

Prompting (Few-Shot and Zero-Shot)

The cheapest and fastest approach: write a good system prompt, provide context, use few-shot examples. Requires no training data, no infrastructure beyond API access, and can be iterated in hours. For most data engineering use cases — SQL generation, log analysis, data quality description, documentation generation — good prompting is sufficient. Always try this first. The time spent building a fine-tuning dataset is almost always better spent on prompt engineering.

RAG (Retrieval-Augmented Generation)

The middle path: retrieve relevant information from your data at query time and include it in the prompt as context. An analyst asks "what was the North region's Q3 performance?" — a RAG system retrieves the relevant data from your warehouse and includes it in the prompt before asking the LLM to summarise or answer. The LLM never needs to know your data in advance; it sees it at query time. RAG is the correct approach when the LLM needs access to your organisation's proprietary data without the data being baked into the model. Covered in full in section 6.3.

Fine-Tuning

Training the model on a domain-specific dataset to permanently change its behaviour. Appropriate when: your domain has language that is genuinely unusual enough that no existing model understands it (specialist medical terminology, proprietary financial instrument naming), you need consistent stylistic output that prompting cannot achieve, or you are running at such scale that smaller, cheaper fine-tuned models produce significant cost savings. Almost never the right first choice for enterprise data engineering questions. Fine-tuning requires: hundreds to thousands of high-quality labelled examples, infrastructure for the training job, and model hosting — all before you know whether it outperforms a well-prompted base model. The majority of teams that start fine-tuning projects find that the time investment was not justified compared to improved prompting.

🏢Field Note
A Fintech Company  ·  The Hallucinated Table Name

A fintech data engineering team integrated an LLM into an internal data catalogue tool that allowed analysts to describe a report in natural language and receive a SQL query in return. The system prompt included general instructions but did not list the available tables — the team reasoned that analysts could see the table list in the UI and would catch any wrong table names.

In the third week of use, an analyst ran a generated query that included a reference to fact_transaction_summary — a table that sounded plausible but did not exist in the production schema. The query executed against a development database where the table happened to exist (with incomplete test data), returned 47 rows, and the analyst included the figures in a client report. The error was discovered three days later when the figures were challenged in a client call.

The fix was adding the complete table list to the system prompt, along with the instruction "Only reference tables listed below. If the required data is not available in these tables, say so explicitly rather than generating a query." Zero hallucinated table references occurred in the following three months of production use.

The Lesson

In data engineering contexts, schema grounding in the system prompt is not optional — it is the primary control for SQL hallucination. Never deploy an LLM SQL generator without explicitly listing the available tables. The UI showing a table list does not constrain the model; the system prompt does.

6.1.7

Azure OpenAI — Why Enterprise Data Teams Use It

The same GPT-4 and GPT-4o models available through OpenAI's direct API are also available through Azure OpenAI Service — hosted on Microsoft's Azure infrastructure rather than OpenAI's own servers. For enterprise data teams, this distinction is material, and Azure OpenAI is the default choice in most regulated and enterprise environments for four specific reasons.

Private endpoints: Azure OpenAI can be deployed within your Azure Virtual Network (VNet) with a private endpoint, meaning no traffic leaves your private network through the public internet. For data teams handling PII (Personally Identifiable Information), financial data, or any data subject to compliance requirements, this is frequently a non-negotiable requirement.

Data does not train the model: When you use Azure OpenAI Service, Microsoft explicitly commits that your prompts and completions are not used to train or improve the model. This is the opposite of the default behavior for OpenAI's consumer API (where opt-in to training is the default for some tiers). For enterprise data — customer records, financial figures, proprietary business metrics — this commitment is required by many data governance policies.

Azure RBAC and Microsoft Entra ID: Access to Azure OpenAI resources is controlled through the same Azure Role-Based Access Control (RBAC) system used for every other Azure resource. Existing Azure AD (Microsoft Entra ID) identities, groups, and conditional access policies apply. No separate user management system is required.

Compliance and certifications: Azure OpenAI Service is covered by Azure's compliance framework — ISO 27001, SOC 2, GDPR compliance, UK NCSC guidance. For teams in regulated industries (financial services, healthcare, insurance), Azure's existing compliance posture extends to Azure OpenAI.

Go DeeperToken counting and cost estimation for data engineering LLM pipelines

LLM API costs compound quickly in data pipeline contexts where the model is called at scale. Estimating costs before building is essential; discovering them in production is expensive.

Token estimation rules of thumb: 1 token ≈ 4 characters ≈ 0.75 words in English. A 500-word document ≈ 650 tokens. A full database schema with 20 tables and their column definitions ≈ 1,500–2,500 tokens. A 100-line Python script ≈ 800–1,000 tokens. For exact counts before production: use the tiktoken Python library (OpenAI's tokenizer) or provider-specific tokenizers.

Cost structure for Azure OpenAI GPT-4o (approximate, verify current pricing): Input tokens cost less than output tokens. A typical data engineering request — 2,000 tokens of schema context + 50-token question = 2,050 input tokens, ~200 output tokens (SQL response). At roughly $2.50 per million input tokens and $10.00 per million output: this request costs approximately $0.007. If your pipeline makes this call 10,000 times per day (one per analyst query), daily cost ≈ $70. At 100,000 calls per day: $700/day → $255,000/year. Build your cost model before scaling.

Cost optimization techniques: Cache common requests (the same question against the same schema produces the same result at temperature 0 — cache by prompt hash). Use smaller/cheaper models for simple tasks (GPT-4o mini for classification; GPT-4o for complex SQL generation). Include only the tables relevant to the question in the context rather than the full schema (reduces input tokens by 60–80% for targeted queries). Batch offline requests where latency is not critical.

The context window cost trap: Sending a 100,000-token context (e.g., a large document or log file) multiplied by 1,000 API calls = 100 million input tokens. At $2.50 per million = $250. For a single day's pipeline run. Always ask: "Does this context need to be this large? Can RAG retrieve only the relevant 1,000 tokens instead of passing 100,000?"

🏢Field Note
A Financial Services Firm  ·  The Private Endpoint Requirement

A financial services firm's data team built a prototype internal analyst assistant using the OpenAI direct API. The prototype worked well technically and received enthusiastic feedback from the analyst team. The IT security team reviewed the deployment six weeks later and raised an immediate blocker: the prototype was sending prompts — which included customer names, account numbers, and transaction summaries — to OpenAI's US-based servers over the public internet. This violated the firm's data handling policy for customer financial data and the firm's interpretation of its FCA regulatory obligations on data storage.

The rebuild on Azure OpenAI Service with a private endpoint took four weeks. The Azure OpenAI endpoint sat entirely within the firm's Azure VNet. No traffic left the private network. Microsoft's commitment that prompt data was not used to train models satisfied the compliance team. The existing Azure RBAC roles (which the analysts already had for other Azure resources) controlled access without any additional user management. The rebuilt assistant was identical in capability to the prototype — different only in where it was hosted.

The Lesson

For any data that falls under regulatory, contractual, or organisational data handling requirements — which is most enterprise data — validate the compliance posture of the LLM provider before building the prototype, not after. For Azure-based data teams, Azure OpenAI is the compliance-first default. Build everything else as a deliberate exception.

Go DeeperHallucination in data contexts — why the stakes are higher than consumer AI

Consumer-facing LLM hallucination is a nuisance. A chatbot that makes up a fact about a historical event can be corrected by the user. Data engineering LLM hallucination is a pipeline failure mode — and often a silent one.

Why data hallucination is more dangerous:

No human in the loop at scale. A consumer user reads the LLM's output. A data pipeline executes it. If an LLM generates a SQL query with a hallucinated table name and no human reviews it before execution, the query either fails (best case) or executes against a similarly-named but different table (worst case). The worst case produces wrong data that looks correct.

Wrong data propagates silently. A hallucinated fact in a chatbot response affects one user's understanding. A hallucinated column mapping in an ETL pipeline propagates through every downstream table, every report, and every dashboard that reads from that table — until someone notices the numbers are wrong. This is why the silent 15% record drop (section 5.4 field note) is so dangerous: it looks like data, behaves like data, and only reveals itself when someone with domain knowledge questions a figure.

Specific data engineering hallucination failure modes: SQL table or column names that don't exist; business logic that sounds correct but inverts a comparison operator (> vs <); date range logic that uses the wrong reference date; aggregation functions that double-count due to missing GROUP BY; join conditions that produce a cartesian product; filter conditions that silently exclude the wrong records.

Mitigation at the pipeline level: Parse and validate LLM-generated SQL using a SQL parser before execution (sqlglot is a Python library that parses and validates SQL across dialects). Run EXPLAIN on generated queries before execution to catch cartesian products and missing indexes. Test against a development environment with known record counts before promoting to production. Add a human review step for any LLM-generated transformation before it runs in the Gold layer.

✓ Section Summary

LLMs generate text by predicting the next token from a probability distribution — they are not retrieving facts from a database. This mechanism makes hallucination structurally inevitable: the model produces plausible-sounding text, not verified correct text. Tokens are the unit of LLM input/output (~4 characters each); context windows define how much the model can see simultaneously (128k to 1M+ tokens across current models); embeddings encode semantic meaning as numerical vectors used in RAG. Temperature at 0 gives deterministic outputs — required for data engineering tasks. Prompt structure: system prompt (persistent instructions — where hallucination is controlled via schema grounding), user turn (the request), assistant turn (prior responses). Fine-tuning is almost never the right first choice — try prompting, then RAG. Azure OpenAI is the enterprise standard: same models as OpenAI direct, plus private endpoints (data stays in your VNet), no training on your data, Azure RBAC, and Microsoft compliance certification. In data engineering contexts, hallucination is not a nuisance but a pipeline failure mode — SQL generators must include explicit schema grounding, output validation, and human review before production execution.

LLMs are powerful but isolated — they cannot access your files, databases, or APIs without explicit integration. Understanding MCP is quickly becoming a core data engineering skill.

Question 5 of 5

A data team needs an LLM to answer questions about their company's proprietary product catalogue data (10,000 products, updated daily). The team is considering fine-tuning vs RAG. Which is the more appropriate approach and why?

A
Fine-tuning — training the model on the product catalogue will give it deep knowledge of the data
B
Fine-tuning — it is cheaper and faster than building a RAG pipeline
C
RAG — it retrieves current product data at query time, handling daily updates without retraining; fine-tuning would require daily retraining, costs significantly more, and produces a model that cannot update when the catalogue changes
D
Both approaches are equivalent — choose based on team preference

C is correct. This is a textbook RAG use case. The data is: (1) proprietary (not in the model's training data), (2) frequently updated (daily changes), and (3) large enough that it cannot all fit in a single prompt. RAG retrieves the relevant product records at query time, includes them in the prompt as context, and returns a grounded answer. When the catalogue updates, the RAG index is refreshed — no model retraining required. Fine-tuning would bake the product catalogue into model weights at training time. This means: daily retraining to keep current (expensive and operationally complex), no way to update the model's knowledge without another training run, and the fine-tuned model is still likely to hallucinate specific product details since fine-tuning changes model behaviour, not its factual knowledge in the way most teams expect. RAG is cheaper, faster to build, and correctly handles dynamic data. Fine-tuning is justified when the task requires a different output style or the domain vocabulary is genuinely unusual — not when the requirement is access to current proprietary data.
