NLP Task Types in Practice

6.7.1

A Working Taxonomy

Section 6.6 covered how to customize an LLM's behavior. This section covers what kind of problem you're actually solving when you reach for NLP — the umbrella term for any technique that processes or generates human language. Being able to name the task type correctly is not academic precision for its own sake: it directly determines which evaluation metric is meaningful, which failure modes to watch for, and — tying back to 6.6 — whether prompting alone is realistic or whether the task needs heavier investment.

Every NLP task this section covers falls into one of two families: understanding tasks, which assign a label or extract information from existing text (classification, NER, sentiment), and generation tasks, which produce new text (summarization, conversational responses). That distinction matters because generation tasks carry a structural risk — hallucination — that understanding tasks simply don't have, covered in 6.7.4.
6.7.2

Understanding Tasks: Classification, NER, and Sentiment

Text classification assigns a predefined category to a piece of text — the support-ticket routing example from section 6.6 is a classification task. Named Entity Recognition (NER) identifies and labels specific spans of text as belonging to a category — person names, company names, dates, monetary amounts — rather than classifying the whole document at once; it's the technique behind automatically extracting "invoice number," "vendor name," and "due date" from a scanned invoice. Sentiment analysis is a specialized form of classification that assigns an emotional polarity (positive/negative/neutral, or a finer-grained scale) to text, commonly used on customer reviews and support transcripts.

All three are understanding tasks: the model's job is to assign labels to text that already exists, not to produce new text. This matters because it means their output space is constrained and enumerable — there is a fixed, known list of possible categories or entity types — which is exactly what makes them straightforward to evaluate using the same precision, recall, and F1 framework from section 8.3.

Figure 6.13Named Entity Recognition tagging four distinct entity types within a single sentence. Each tag is independently evaluable against ground truth, which is exactly what makes precision/recall-style metrics a natural fit for understanding tasks.
+
Go DeeperWhy raw accuracy is often the wrong metric for classification tasks like these

Section 8.3 introduced precision, recall, F1, and AUC-ROC in the context of classical ML classifiers, and the same logic applies directly to NLP classification tasks — with the same trap waiting for anyone who reaches for plain accuracy instead. Consider a sentiment model screening support transcripts for genuinely urgent complaints, where only 4% of transcripts are actually urgent. A model that simply predicts "not urgent" for everything achieves 96% accuracy while catching zero real complaints — a textbook case of class imbalance making accuracy a misleading headline number.

The fix is the same one covered in 8.3: report precision (of the transcripts flagged urgent, how many really were?) and recall (of the transcripts that really were urgent, how many did the model catch?) separately, and choose which one to prioritize based on the actual cost of each error type. For urgent-complaint screening, missing a real complaint (a false negative) is usually far costlier than over-flagging a normal one (a false positive) for a human to quickly dismiss — which argues for optimizing recall, possibly at some cost to precision, exactly the threshold-tuning logic covered in 8.3's classification threshold discussion.

The takeaway that generalizes across every NLP classification task in this section: always check the class balance of your evaluation data before trusting a single accuracy number, and report the metric that actually reflects which error type is more costly for your specific business problem.

🏦Field Note
An Insurance Company  ·  The 98% Accurate Model That Missed Almost Every Fraud Case

A claims-processing team built a classifier to flag potentially fraudulent claims for human review, training it on a year of historical claims where roughly 2% had been confirmed fraudulent. The first reported result — 98% accuracy — was presented to leadership as a strong success, until a more careful look at the confusion matrix (the same tool from section 8.3) revealed the model was flagging almost nothing as fraud at all, simply because predicting "not fraud" for every single claim already achieves 98% accuracy given the underlying 2% base rate.

Recomputing the same model's recall specifically on the fraud class showed it was catching only 11% of actual fraud cases — a result the headline accuracy number had completely obscured. The team re-ran evaluation reporting precision and recall on the fraud class specifically, retrained with class-weighting to penalize missed fraud cases more heavily during training, and only then started making real progress on the metric that actually mattered to the business.

The Lesson

On any meaningfully imbalanced classification task — fraud, churn, urgent-complaint detection — a high accuracy number is close to meaningless on its own. Always demand the per-class precision and recall, exactly as section 8.3 establishes, before accepting any classification result as a genuine success.

6.7.3

Generation Tasks: Summarization

Summarization is the first generation task covered in this section: producing a shorter text that captures the key content of a longer source. It splits into two genuinely different techniques. Extractive summarization selects and stitches together actual sentences or phrases lifted directly from the source document. Abstractive summarization generates entirely new sentences that paraphrase and condense the source — the way a human asked to summarize an article would naturally write, rather than copy-pasting selected lines from it.

+
Go DeeperExtractive vs. abstractive: the accuracy/risk tradeoff

Extractive summarization's core guarantee is that every word in its output actually appeared in the source document — it cannot introduce a fact that wasn't there, because it never writes new sentences, only selects existing ones. Its weakness is fluency and coherence: stitched-together sentences from different parts of a document can read disjointedly, and the technique cannot paraphrase or condense in the genuinely flexible way a human summarizer can.

Abstractive summarization, almost always implemented today via an LLM, produces far more fluent, naturally condensed output — closer to what a human would actually write. Its weakness is exactly the flip side of extractive's strength: because it generates new text rather than selecting existing text, it can state something that sounds plausible but doesn't actually appear in, or even contradicts, the source document. This is hallucination, covered fully in section 6.7.4, and it is structurally possible in abstractive summarization in a way it simply is not in extractive summarization.

The practical decision: extractive summarization is the safer default whenever factual fidelity to the source is non-negotiable (legal document summaries, medical record summaries) even at some cost to fluency. Abstractive summarization is the better fit when fluency and genuine condensation matter more than a guarantee of zero added content, and when downstream review can catch the occasional hallucinated detail.

6.7.4

Conversational AI and the Hallucination Problem

Conversational AI — chatbots, virtual assistants, support agents — is the broadest and most generation-heavy NLP task type covered here, combining intent recognition (a classification sub-task: what is the user actually trying to do?) with open-ended response generation. It's also where hallucination causes the most direct business damage, because a generated response that sounds confident and plausible but is factually wrong is, from the user's perspective, indistinguishable from a correct one until something goes visibly wrong.

Figure 6.14Hallucination risk drops sharply when a generative response is grounded in retrieved source text (RAG, section 6.3) rather than generated purely from the model's training data — but grounding reduces the risk, it does not eliminate it, since a model can still misread or over-generalize from retrieved text.
+
Go DeeperWhy hallucination is structural, not an occasional bug

Section 6.1 covered how an LLM generates text: predicting the most statistically probable next token given everything before it, learned from patterns across its training data. That mechanism has no built-in concept of "I don't actually know this" — it produces the most plausible-sounding continuation regardless of whether the underlying claim is true, because plausibility, not truth, is what the model was trained to optimize.

This is why hallucination cannot be fully "fixed" by better prompting or a larger model alone: it is a direct consequence of how generation works, not a defect introduced by insufficient training. What can genuinely reduce it: grounding the generation in retrieved source text via RAG (so the model has something concrete to draw from rather than relying purely on its training-data statistics), explicitly instructing the model to say "I don't know" when retrieved context doesn't contain the answer, and — for the highest-stakes use cases — a human review step before a generated response reaches a customer or a regulator.

The practical implication for choosing between conversational AI architectures: an LLM answering purely "from memory" with no retrieval step is taking on unmitigated hallucination risk on every single response. The moment factual accuracy genuinely matters — policy details, account information, anything with legal or financial consequences — that risk is exactly why section 6.3's RAG architecture, not a bare LLM, is the appropriate default.

💬Field Note
A Telecom Customer Support Team  ·  The Chatbot That Invented a Refund Policy

A support chatbot, built as a bare LLM with a system prompt describing the company's general tone and a brief written summary of refund policy, confidently told a customer they were entitled to a full refund outside the actual policy window — a specific dollar amount and a specific timeframe that simply did not exist in any actual company document. The customer, reasonably, held the company to the chatbot's stated answer, and the dispute escalated to a manual review.

A root-cause review found the underlying issue precisely matched the mechanism in this section's deep dive: the bare LLM had no actual policy document to draw from at answer time, only a brief paraphrased summary in its system prompt, and filled remaining detail with statistically plausible-sounding specifics that happened to be fabricated. The team rebuilt the system as a RAG pipeline retrieving the actual current refund policy document at every query, with an explicit instruction to state "I'd need to check on that" whenever the retrieved policy text didn't directly answer the specific question asked.

The Lesson

A confident-sounding answer is not evidence of a correct one. Any conversational AI use case with real financial, legal, or safety consequences needs a grounding source (RAG) and an explicit "don't know" escape hatch — not just a well-written system prompt describing the policy in general terms.

6.7.5

Matching Task to Metric, and Back to the Build/Buy Decision

Pulling the section together: the right evaluation approach and the right failure modes to watch for both follow directly from which task type you're actually solving — which is exactly why naming the task correctly, rather than reaching for "AI" as an undifferentiated catch-all, is the first real step in scoping any NLP project.

That last row is the explicit bridge back to section 6.6: a stable, well-defined classification task with plenty of labeled history is a reasonable fine-tuning candidate, while any conversational task with real factual stakes should default to RAG, not a bare LLM, precisely because of the structural hallucination risk this section has covered in detail.

✓ Section Summary — Part 6 Complete

NLP tasks split into understanding tasks (classification, NER, sentiment) with a fixed, enumerable output space evaluable via precision/recall/F1, and generation tasks (summarization, conversational AI) that produce new text and therefore carry hallucination risk that understanding tasks structurally cannot. Extractive summarization trades fluency for a guarantee against added content; abstractive trades that guarantee for more natural output. Hallucination is a direct consequence of how LLMs generate text — optimizing plausibility, not truth — and is meaningfully reduced, though never eliminated, by RAG grounding. Naming the correct task type up front determines the right metric, the right failure modes to watch for, and which of section 6.6's four customization approaches actually fits. Two Families

Check Class Balance First

High accuracy on an imbalanced task can mean almost nothing. Always check per-class precision and recall before trusting a headline number.

Confident ≠ Correct
