6.4

AI Agents — Architecture, Frameworks, and Production Reality

How agents use tools and memory to act across multiple steps, the ReAct reasoning loop, framework options from LangChain to bare metal, guardrails that keep them safe, and the observability you need before trusting them in production

6.4.1

What Makes Something an Agent?

A single call to an LLM is not an agent. It is a function: you send text in, you get text out. The call is stateless, single-step, and side-effect-free. This is enormously useful, but it has a hard ceiling: the model can reason about what should be done but cannot actually do anything.

An agent is a system in which the LLM is embedded in a loop — observe the environment, reason about what to do next, act by calling a tool, observe the result, reason again, repeat until the task is complete or a stopping condition is reached. The key word is loop. An agent operates across multiple reasoning and action steps, with the output of each step feeding into the next.

The minimal definition of an agent: an LLM that can call tools and use the results to inform further reasoning. Everything else — memory systems, multi-agent coordination, planning horizons — is built on that foundation.

This changes the engineering problem substantially. With a single LLM call, you reason about a prompt and an output. With an agent, you reason about a trajectory: a sequence of actions and observations that may branch, backtrack, or run for dozens of steps. The scope for failure is correspondingly larger. Agents can take irreversible actions, consume significant compute by looping, hallucinate tool calls that don't exist, or get stuck in unproductive reasoning cycles. The power and the risk scale together.

💭 Hmm moment: A useful mental model is the difference between asking a colleague a question versus delegating a task. A question expects a single answer. A delegation expects the person to figure out the steps, handle surprises, and come back with a result — possibly after going back and forth with other people or systems. Agents are the delegation model.

6.4.2

The ReAct Loop — Reasoning + Acting

The dominant pattern for single-agent reasoning is ReAct (Reason + Act), introduced in a 2022 paper from Princeton and Google Brain. The insight is that interleaving reasoning traces with tool calls produces substantially better results than either chain-of-thought (reasoning alone) or plain tool use (acting without visible reasoning).

The ReAct Cycle

In each cycle the agent produces a Thought — a private reasoning trace about what the current state is and what to do next — followed by an Action — a specific tool call with parameters — then receives an Observation — the tool's return value — which feeds the next Thought. This continues until the agent decides to produce a final Answer.

Figure 6.7. The ReAct loop. Each cycle appends a Thought, Action, and Observation to the context window. The loop exits when the model decides to emit a final Answer rather than another tool call.

The reasoning traces are visible to developers (via logging) and improve reliability by giving the model a scratchpad to plan before committing to an action. They also make debugging tractable: if an agent makes the wrong tool call, the Thought trace usually reveals exactly why the model was confused.

6.4.3

Function Calling — How Tools Work at the API Level

Function calling (also called tool use) is the mechanism by which an LLM signals its intent to invoke an external function. The host application defines a set of tools — each with a name, description, and JSON schema for its parameters — and passes them to the model at call time. If the model decides a tool is needed, instead of generating prose it generates a structured tool-call object: {"name": "search_documents", "arguments": {"query": "Q3 revenue"}}. The application intercepts this, executes the real function, and returns the result back to the model for the next step.

This creates a clean division of responsibility. The LLM decides which tool to call and what arguments to pass — pure reasoning. The host application handles execution and authorization. The model never directly touches a database, API, or file system; it only produces structured intent, and your code carries that intent out.

▶ Deep Dive — Tool Schema Design for Reliable Agents

Poor tool schemas are one of the most common causes of agentic failure. Models choose tools based on their descriptions and call them based on parameter schemas. Every gap in the schema is an opportunity for the model to hallucinate.

Write descriptions as if briefing a junior colleague. The description for a run_sql_query tool should not just say "runs SQL". It should say: "Execute a read-only SELECT query against the data warehouse. Returns up to 1000 rows. Do NOT use for INSERT, UPDATE, or DELETE operations. The database dialect is Spark SQL." The specificity prevents misuse.

Use enums aggressively. If a parameter can only take a finite set of values, enumerate them in the schema. A format parameter that accepts ["csv", "json", "markdown"] will never hallucinate "excel". An open string field will.

Document error modes. Tell the model what errors the tool can return and what they mean. If a search returns empty results it means the query found nothing — not that the data doesn't exist. This prevents the model from concluding "no data" when the real issue is a bad query.

Limit tool sets per agent. An agent with 40 tools has high cognitive overhead — the model must scan through 40 descriptions before choosing. Prefer specialized agents with 5–10 tightly relevant tools over a generalist agent with a massive tool belt. Route tasks to the right specialized agent at the orchestration layer.

6.4.4

Memory Architectures

An agent's ability to maintain context over time depends on its memory architecture. There are four categories, and production systems typically combine several.

In-Context Memory

The simplest form: everything the agent has seen in the current session is accumulated in the context window. Observations, prior tool calls, user messages — all in-context. This is zero-infrastructure but has a hard ceiling: context windows, even large ones (128k–1M tokens), are finite, and longer contexts increase latency and cost. For tasks that run beyond ~100 turns, in-context memory alone is not enough.

External Memory (Vector / Key-Value)

The agent writes summaries or facts to an external store and retrieves them by semantic search or key lookup. This is exactly the RAG retrieval pattern applied to the agent's own history. It scales to arbitrarily long task histories but requires infrastructure (a vector database or key-value store) and introduces retrieval latency and potential retrieval failure. Embeddings from prior sessions are stored and retrieved as needed.

Episodic Memory

Structured records of past episodes — completed tasks, decisions made, outcomes achieved. Different from raw vector memory in that episodes are semantically organized and often summarized. An agent that handled a similar request last week can retrieve that episode, see how it resolved, and apply the same pattern. Useful for customer-facing agents that interact with the same users repeatedly.

Procedural Memory (Learned Rules)

Encoded patterns that update the agent's behavior across sessions — either as fine-tuned weights, system prompt updates, or structured rule stores. The agent learns that certain tool call patterns fail in certain contexts, and its default behavior adjusts. This is the hardest to implement robustly and risks introducing drift or instability if the update mechanism is poorly designed.

6.4.5

Multi-Agent Systems

Single agents hit a ceiling: context windows fill, task complexity exceeds what one reasoning chain can manage reliably, and generalist agents with too many tools become unreliable. Multi-agent architectures address this by decomposing a complex task into subtasks handled by specialized agents coordinated by an orchestrator.

Orchestrator-Worker Pattern

An orchestrator agent receives the high-level goal, decomposes it into subtasks, routes each subtask to a specialized worker agent, collects the results, and synthesizes the final output. Workers are narrow specialists: a SQL agent, a document retrieval agent, a code execution agent, a summarization agent. Each has a focused tool set and a focused system prompt. The orchestrator does not need to know how to write SQL — it needs to know when to call the SQL agent.

Peer / Debate Architecture

Multiple agents of similar capability engage with the same problem from different angles and critique each other's outputs. A drafting agent proposes an answer; a review agent critiques it; the drafting agent revises. This is particularly effective for tasks where correctness matters more than speed — code generation, contract review, data quality rule definition — because the critique step catches errors the first agent missed.

📋 Field Note — Customer Service Agent, Retail Banking

A large retail bank deployed an orchestrator-worker agent system to handle first-tier customer inquiries. The orchestrator classified intent (balance inquiry, transaction dispute, product question, fraud report) and routed to a specialist: a read-only banking data agent for account queries, a RAG agent backed by product documentation for product questions, and a human escalation agent for anything involving money movement or fraud.

The read-only constraint on the banking data agent was the key safety mechanism. The agent could see account information but had no tools capable of initiating transactions, approvals, or data changes. All consequential actions required human review. This architectural boundary — not just a prompt instruction — ensured the system could not take harmful actions even if the LLM was tricked or confused.

After six months, first-contact resolution on eligible inquiries improved from 62% to 81%. The remaining 19% were escalated to human agents with a pre-populated context summary, which reduced average handle time on escalated calls by 34%.

6.4.6

Guardrails and Safety Architecture

Agents that can take actions can take the wrong actions. A guardrail system is not optional in production — it is a first-class component of the architecture. The categories to address are:

Input Guardrails

Screen the user's request before the agent processes it. Categories include prompt injection detection (is the user trying to override the system prompt?), PII (personally identifiable information) redaction before the text hits the LLM, jailbreak classification, and topic scope enforcement (the agent only handles X; anything outside X is declined before processing).

Output Guardrails

Screen the agent's final response before it reaches the user. Content moderation classifiers, PII checks in the output, factual consistency checks (does the output contradict retrieved documents?), and citation verification for RAG-backed answers.

Tool-Level Authorization

The most important guardrail is structural: only give agents tools they are authorized to use. An agent that handles customer questions about product pricing should not have a tool that modifies pricing data. Authorization at the tool-definition layer is more reliable than instructions in the system prompt, because instructions can be overridden by adversarial input; a missing tool cannot.

Human-in-the-Loop Gates

For high-consequence actions — sending emails to customers, modifying production data, initiating financial transactions — the agent should not act autonomously. Instead it presents its proposed action to a human for approval and waits for explicit confirmation. The gate can be asynchronous (the agent pauses and resumes after approval) or synchronous (a human reviews and clicks confirm before the tool call executes).

▶ Deep Dive — Observability for Agents: Traces, Spans, and LLM-Specific Metrics

Standard application observability tracks latency, error rate, and throughput. Agent observability needs all of that plus a new dimension: trace-level visibility into multi-step reasoning sequences.

Traces and spans. Every agent run is a trace. Every tool call within the run is a span. You need to see the full sequence: Thought → Action → Observation for every cycle, with timestamps, token counts, latency per tool, and whether the run completed or timed out. OpenTelemetry (OTel) is the standard transport; frameworks like LangSmith, Langfuse, Phoenix (Arize), and Azure AI Foundry all export OTel-compatible traces.

LLM-specific metrics to track:

  • Token consumption per run — agents with feedback loops can consume orders of magnitude more tokens than a single call. Budget tracking is essential.
  • Tool call success rate — what % of tool invocations return valid results vs errors? A falling success rate often indicates schema drift (the tool changed, the schema didn't).
  • Loop count distribution — how many reasoning cycles does a typical run take? A fat tail with many high-loop runs signals tasks the agent cannot reliably resolve and should be escalated.
  • Hallucinated tool calls — calls to tool names that don't exist in the tool set. A non-zero rate means the model is confusing tool names; fix with better schema descriptions or narrower tool sets.
  • Human escalation rate — what % of tasks are being routed to human-in-the-loop review? Rising rates indicate expanding scope beyond the agent's reliable envelope.

Production tooling (mid-2026): LangSmith and Langfuse are the dominant open-source/SaaS options for trace storage and visualization. Arize Phoenix provides evaluation-focused analytics. Azure AI Foundry includes native tracing for agents built on Azure OpenAI. All of these can ingest OTel spans from custom agents built outside any framework.

📋 Field Note — Data Quality Monitoring Agent, E-Commerce Platform

A large e-commerce company replaced a rule-based data quality monitoring system with an agent-based system capable of investigating anomalies rather than merely flagging them. When a metric breached a threshold, the agent had access to tools for querying the data warehouse, inspecting pipeline run logs, checking upstream source system APIs, and searching a Confluence-backed knowledge base of known data issues.

The agent's task on each alert was: determine whether the anomaly is a true data quality issue or an expected business event (e.g. a promotion causing a volume spike). For genuine issues, identify the probable root cause from pipeline logs and upstream source data. Draft a Slack summary with root-cause hypothesis and proposed remediation steps for the on-call engineer.

The engineering team deliberately kept the agent out of the remediation loop — it could investigate and recommend but not execute pipeline reruns, schema changes, or incident tickets. This was a design choice, not a capability limit. The investigating agent reduced mean time to root cause from 47 minutes to 8 minutes on the anomaly classes within its scope, while the on-call engineer retained decision authority on all remediations.

✎ Section 6.4 — Check Your Understanding
Five questions. Answers and explanations are revealed after you submit.
1. What is the defining characteristic that makes a system an "agent" rather than a simple LLM call?
An agent is defined by the loop: the LLM observes state, reasons, calls a tool, observes the result, and iterates. A larger model or a vector database are optional enhancements; the loop is the defining feature.
2. In the ReAct pattern, what is the role of the "Thought" step, and why does it improve reliability?
The Thought is a scratchpad the model uses to reason through the situation before deciding on an action. Interleaving reasoning with tool calls outperforms acting without visible reasoning. It also makes debugging tractable — the Thought trace reveals why the model chose an action.
3. In function calling architecture, who is responsible for executing the tool, and what does the model produce?
The model never directly touches external systems. It produces a structured tool-call object (name + arguments); the host application intercepts this, runs the real function, and feeds the result back to the model. Authorization and execution live in application code, not in the model.
4. A team is building a customer support agent that will interact with the same users repeatedly across many sessions. Which memory architecture should they prioritize adding beyond in-context memory?
For repeat-user agents, episodic memory — structured records of past sessions — allows the agent to personalize based on prior interactions. In-context memory cannot persist across session boundaries. Fine-tuning bakes averages across all users into weights, not per-user history.
5. A team is building an agent that will access production financial data. Which of the following is the most robust safety mechanism?
Structural constraints — the absence of harmful tools — are more reliable than instructions. System prompt instructions can be overridden by adversarial input or ignored under certain conditions. A tool that doesn't exist cannot be called. Logging catches violations after harm; structural limits prevent them.
