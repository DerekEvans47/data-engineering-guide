The orchestration layer — moving data reliably from hundreds of sources into your data lake

1.4.1

What ADF Is — and Critically, What It Is Not

Azure Data Factory is an orchestration and data movement service. It connects to source systems, schedules when data moves, handles connection failures with retry logic, logs what happened, and alerts when something goes wrong. It is the Layer 1 (Ingestion) tool in the four-layer architecture from Section 1.1.

What ADF is not: a transformation engine. This boundary is the single most important thing to understand about ADF, and it is where the most common architectural mistakes originate. ADF has a feature called Mapping Data Flows that provides a visual interface for transformations. Practitioners discover this feature, use it for logic that belongs in Databricks, and then wonder why their pipeline is slow, expensive, and hard to test. The guidance is direct: use ADF to move data, use Databricks to transform it. The one overlap zone — simple, low-volume transformations where the overhead of a Databricks cluster is not justified — is a narrow exception, not a general pattern.

The fact that ADF doesn't transform data is considered a limitation by beginners and a deliberate design by architects. Clean separation of concerns means you can change your transformation logic without touching your ingestion logic, and vice versa. Systems that mix the two are harder to debug, harder to scale, and harder to hand off to a team member.
🏢Field Note
A European Logistics Company  ·  Migrating from SSIS

A European logistics company ran their entire data warehouse ETL on SSIS — SQL Server Integration Services, Microsoft's on-premise predecessor to ADF. Their SSIS packages had accumulated over a decade and mixed data movement, transformation, business logic, and even email notifications in the same package files. Maintaining them required specialists who knew SSIS's proprietary visual interface, and version control was nearly impossible — SSIS packages are XML blobs that do not diff cleanly.

Their ADF migration took eighteen months, not because ADF was difficult, but because the SSIS packages had never been properly documented. For every package migrated, the team discovered undocumented business rules embedded in transformations, hardcoded connection strings, and dependencies on SQL Server Agent jobs that no one remembered creating. The migration forced a documentation exercise that the organization had been deferring for years.

The outcome: ADF pipelines backed by Git, parameterized templates that replaced dozens of nearly-identical SSIS packages, and transformation logic properly separated into Databricks notebooks that could be unit-tested. The migration cost was significant. The maintainability improvement was transformational.

The Lesson

SSIS migrations are often documentation exercises disguised as technical projects. The cost is real, but the alternative — maintaining undocumented legacy pipelines indefinitely — compounds with each passing year.

1.4.2

The ADF Object Model

ADF has a specific set of objects that every practitioner must understand. They have a clear hierarchy and a clear purpose at each level. The analogy of a logistics company maps cleanly to each layer:

A Linked Service is the address book — how ADF connects to a specific system. Each Linked Service stores credentials and connection configuration for one endpoint: an ADLS storage account, an on-premise SQL Server, a Salesforce org, an SAP system, an HTTP API. The Linked Service does not specify what data to read — only how to connect. You create one Linked Service per connection, then reuse it across many Datasets.

A Dataset is the cargo manifest — it specifies what data within a connected system. A Dataset references a Linked Service and adds specifics: which table, which file path, which format (Parquet, CSV, JSON, Delta), which schema. Datasets are used as inputs and outputs for Activities.

An Activity is a single task: copy this data, call this REST API, run this SQL stored procedure, execute this Databricks notebook, wait for this condition, send this email notification. Activities are the individual steps of work inside a pipeline.

A Pipeline is the route — an ordered sequence of Activities with control flow logic (if-then conditions, loops, parallel execution). A pipeline might copy files from an API, then trigger a Databricks notebook to transform them, then send a success notification. Pipelines are the primary unit of work in ADF.

A Trigger fires a pipeline on a schedule. There are three trigger types: scheduled (run at 2am daily), tumbling window (run every hour for the previous hour's data window), and storage event (run when a file arrives in ADLS). The right trigger type depends on your ingestion pattern.

An Integration Runtime (IR) is the compute infrastructure that ADF uses to physically perform the data movement. This is where most practitioners get confused, so the next section covers it in full.

Figure 1.10 The ADF object model. A Trigger fires a Pipeline on schedule or event. The Pipeline runs Activities in sequence. Activities reference Datasets (what data) backed by Linked Services (how to connect). The Integration Runtime is the compute that physically moves the data. Each object has exactly one responsibility.
1.4.3

Integration Runtimes: The Three Types

The Integration Runtime is the component most practitioners underestimate until it becomes the blocking issue on a project. It is the compute infrastructure that ADF uses to physically perform data movement and transformation. There are three types, each designed for a different connectivity scenario.

Azure Integration Runtime (Azure IR) is the default. It handles cloud-to-cloud connectivity: ADF moving data from one Azure service to another, from a public REST API to ADLS, from Azure SQL to ADLS. Azure IR is fully managed by Microsoft — you specify how many Data Integration Units (DIUs) to use and ADF handles the rest. No infrastructure to maintain.

Self-Hosted Integration Runtime (SHIR) is the most important and least understood. It handles connectivity between ADF and systems that are not on the public internet: on-premise SAP ERP systems, SQL Servers behind firewalls, manufacturing execution systems in plant networks, on-premise Oracle databases. You install the SHIR software on a Windows server inside your network. That server becomes the bridge between your internal systems and Azure. ADF sends instructions to the SHIR; the SHIR executes them locally and sends results to ADLS. For any organization with on-premise ERP or operational systems — which is most manufacturing and logistics companies — the SHIR is not optional.

If someone asks why you can't just use Azure IR to connect to an on-premise SAP system, the answer is network topology. Azure IR runs in Microsoft's cloud. Your SAP system is behind your corporate firewall. There is no inbound route from Azure into your network. The Self-Hosted IR reverses the connection: it sits inside your network, calls out to Azure (outbound), and pulls instructions. Outbound connections through firewalls are generally permitted. Inbound from unknown cloud endpoints are not.

SSIS Integration Runtime exists specifically for organizations with existing SSIS package investments. It runs SSIS packages natively in Azure, providing a lift-and-shift path for legacy pipelines. Organizations use this during a migration period while rebuilding SSIS logic as proper ADF pipelines and Databricks notebooks. It is not a long-term target architecture — it is a migration bridge.

Figure 1.11 The three Integration Runtime types. Azure IR handles cloud connectivity with no infrastructure overhead. Self-Hosted IR bridges on-premise systems to Azure — required for any ERP, SQL Server, or proprietary system that sits behind a corporate firewall. SSIS IR is a migration tool, not a target architecture.

Go DeeperThe Self-Hosted IR in practice — network setup and high availability

Installing a Self-Hosted IR requires a Windows Server 2012 R2 or later machine with outbound internet access on ports 443 and 1433. The installation process involves downloading the SHIR agent from ADF, registering it with an authentication key from your ADF instance, and configuring the machine to run the service. The SHIR calls out to ADF every few seconds to check for new pipeline run instructions — this is the outbound connection that eliminates the need for inbound firewall rules.

High availability: for production use, you should install the SHIR on at least two machines. ADF automatically load-balances between available SHIR nodes and routes around unavailable nodes. A single SHIR node creates a single point of failure for all on-premise connectivity — if that server needs a Windows update reboot, all on-premise pipelines pause.

Common issues: the most frequent SHIR problem in manufacturing environments is network connectivity to on-premise SAP systems. The SHIR machine needs access to the SAP application server port (typically 3300 + system number) and the SAP message server port. Coordinating this with corporate network security teams is usually the longest task in an ERP integration project.

Credentials: the SHIR machine needs a service account with read access to the source systems it connects to. These credentials are stored in the SHIR's encrypted local store, not in ADF. They never leave your network.

1.4.4

Incremental Load Patterns

The most consequential design decision in any ADF pipeline is the load pattern: how do you identify and move only the new or changed records since the last run? Three patterns cover the vast majority of real scenarios.

Full Load copies the entire source dataset on every run. Simplest to implement, most expensive at scale. Appropriate for small reference tables that change infrequently (product catalogs, lookup tables, customer segments under 100,000 rows). Becomes problematic when the source table exceeds a few million rows or when the source system cannot sustain the query load of a full extract multiple times per day.

Watermark-Based Incremental Load uses a stored value — the watermark — to track how far the pipeline has processed. On each run: read the current watermark from a tracking table, query the source for records modified after that value, copy those records to Bronze, update the watermark to the current high value. This pattern requires a reliable "last modified" timestamp column in the source table. If the source system updates records without changing the timestamp (or if there is no timestamp), the watermark pattern cannot detect those changes.

Change Data Capture (CDC) reads the source database's transaction log to capture every insert, update, and delete as it happens. CDC is the most precise pattern: it captures deletes (which a watermark cannot detect), captures multiple changes to the same row as separate events, and adds minimal load to the source system. The trade-off: CDC requires the source database to have change tracking or CDC features enabled, and the configuration varies significantly between database vendors (SQL Server CDC, Oracle LogMiner, Debezium for open-source databases).

The ADF Cost Model — Where Bills Surprise People

ADF charges across four dimensions, and the interaction between them creates costs that new practitioners consistently underestimate:

Pipeline runs: a fixed cost per pipeline execution. Running a pipeline 100 times per day creates 100 pipeline run charges per day. This is usually the smallest cost component but adds up for high-frequency pipelines.

Activity runs: a cost per activity executed within a pipeline. A pipeline with 20 activities that runs 48 times per day generates 960 activity runs per day. For organizations running hundreds of pipelines, this is often the largest cost component.

DIU hours: Data Integration Units are the measure of compute power for Copy Activities. More DIUs mean faster data transfer but higher cost. ADF auto-selects DIU count but you can cap it.

Self-Hosted IR: you pay for the server running your SHIR (Azure VM or on-premise hardware). This is a fixed cost regardless of whether pipelines are running.

The most effective ADF cost optimization: parameterization. Instead of separate pipelines for each source table (common in organizations that let pipelines proliferate), build one generic pipeline that takes source name, schema, watermark column, and destination path as parameters. Run it 50 times per day with different parameter values. The activity count stays constant; the flexibility is unlimited.

Go DeeperADF and Git — the publish button trap

ADF has a Git integration mode where pipeline definitions are stored as JSON files in a Git repository (GitHub or Azure DevOps). This is the correct approach for any team with more than one developer. But it introduces a behavior that trips up nearly every team the first time they encounter it.

In Git mode, ADF has two states: "in development" (changes saved to your feature branch in Git) and "published" (changes deployed to the ADF live service). A pipeline saved to Git is not running in production. A pipeline must be published to run. The Publish button in the ADF UI is what deploys your changes from the collaboration branch (usually main) to the live service.

The trap: a developer merges changes into main. They expect the pipeline to start running with the new logic. It does not — because they forgot to click Publish. The live ADF service is still running the previously published version. This causes real incidents when teams assume deployment happened automatically.

The proper pattern: automate publishing via Azure DevOps or GitHub Actions pipelines that trigger on merges to main, run ADF's ARM template deployment, and publish to the live service. This eliminates the manual publish step and makes ADF deployment behave like normal CI/CD. See Section 5.4 for the full CI/CD implementation.

🏢Field Note
A UK Financial Services Firm  ·  The Activity Run Bill

A financial services firm built their data ingestion layer on ADF over the course of a year. They had approximately 340 source tables across four systems, and the team had created individual pipelines for each table — 340 pipelines, each with between 8 and 15 activities, running hourly. The architecture worked fine technically. Then the monthly Azure invoice arrived.

340 pipelines × 12 activities average × 24 runs per day × 30 days = approximately 2.9 million activity runs per month. At ADF's activity run pricing, this was a significant and unexpected cost. The platform team had built the pipelines independently without considering the aggregate cost model.

The remediation took six weeks: a data engineer designed a generic parameterized pipeline template that handled all 340 source tables. The refactored architecture ran one pipeline 340 times per hour with different parameters, rather than 340 pipelines once per hour each. Activity runs dropped from 2.9 million to approximately 350,000 per month. The savings paid for the refactoring effort within the second month.

The Lesson

Design for parameterization from day one. One generic pipeline running many times costs dramatically less than many specific pipelines running once each. This principle is the single most impactful ADF cost optimization in practice.

+
Go DeeperADF vs Streaming — when ADF is the wrong tool

ADF is a batch orchestration tool. It moves data in discrete runs — once per hour, once per day, once per week. It is not designed for real-time data movement. This distinction matters more than it might initially seem, because the desire for "near real-time" data is common in business stakeholders, and the correct architectural response is not to make ADF run more frequently.

The streaming alternative in Azure is Event Hubs (Microsoft's Kafka-compatible managed streaming service) feeding into Databricks Structured Streaming, which writes to Delta Lake continuously. This architecture achieves latency of seconds to minutes. ADF is not in this stack at all.

The practical question: does your business actually need streaming data? "Near real-time" often means "I want my dashboard to update faster." If the underlying business decision happens daily or weekly, a 5-minute lag from ADF running frequently is often sufficient and dramatically simpler than a full streaming architecture. True streaming is warranted when the business action must happen within seconds of the event — fraud detection, real-time inventory alerts, live operational monitoring. For financial reporting, inventory analysis, and most BI dashboards, ADF running at 15-minute intervals is more than adequate.

Introducing streaming prematurely is one of the most expensive over-engineering mistakes in data engineering. Evaluate the actual business decision latency requirement before reaching for Event Hubs.

✓ Section Summary

ADF is an orchestration and data movement tool — not a transformation engine. Its object model has six layers: Trigger (when) → Pipeline (what order) → Activity (one task) → Dataset (what data) + Linked Service (how to connect) → Integration Runtime (the compute engine). Three IR types cover different connectivity scenarios: Azure IR for cloud-to-cloud, Self-Hosted IR for on-premise systems (required for ERP), and SSIS IR for legacy migration only. Three load patterns handle most ingestion scenarios: full load for small tables, watermark incremental for large tables with timestamps, and CDC for high-volume or delete-sensitive sources. The cost model charges per pipeline run, activity run, and DIU hours — parameterization is the key optimization. ADF is batch only; true streaming requires Event Hubs + Databricks Structured Streaming.

Core Rule

ADF moves. Databricks transforms. Every time you are tempted to use ADF Data Flows for complex logic, ask: should this be a Databricks notebook instead? The answer is almost always yes.

Key Term

Self-Hosted IR — required for any on-premise source. Installed on your server. Calls out to Azure; no inbound firewall rules needed. Run at least two for high availability.

Interview Insight

"What triggers a pipeline in ADF?" — three types: Schedule (run at 2am daily), Tumbling Window (run for each time window sequentially), Storage Event (run when a file arrives). Knowing the distinction shows operational depth.

Cost Warning

Activity runs compound. 20 pipelines × 15 activities × 24 runs/day = 7,200 activity runs/day. Design with parameterization from the start.

The Publish Trap

In ADF Git mode, saved ≠ deployed. Changes saved to Git do not run until you click Publish (or automate publishing via CI/CD). This confuses nearly every team at least once.
