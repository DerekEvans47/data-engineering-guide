1.1

The Modern Data Stack

How everything connects, why it exists, and what your role is within it

✓

Expected Learning Outcomes

When you have completed this section, you should be able to:

  1. Explain why the modern data stack emerged and what specific failure it solved
  2. Identify and describe the four fundamental layers of any modern data platform
  3. Map Azure's primary tools to each layer in the stack
  4. Articulate the principle of storage/compute separation and why it matters
  5. Connect the role of a data practitioner to architectural decision-making in this stack
1.1.1

The Problem That Created the Stack

Every powerful idea in data engineering exists because someone had a painful problem and got creative about solving it. The modern data stack is no exception. To understand what it is — and more importantly, why it works the way it does — you need to understand what came before it and why that broke.

For much of the 2000s and early 2010s, a typical enterprise data setup looked like this: a SQL Server database sat at the center of everything. Tools like SSIS — SQL Server Integration Services, Microsoft's legacy pipeline tool — pulled data from source systems, transformed it on the way in, and loaded it into the same server that later powered the reports your executives relied on. The entire operation was tightly coupled: one system did the storing, the computing, the transforming, and the serving, all at once. This pattern is called ETL (Extract, Transform, Load).

For modest data volumes, this worked. Then something changed. Systems of record — SAP, Salesforce, custom APIs, IoT sensors, web logs — started generating data at volumes nobody anticipated. Running a complex multi-million-row transformation job on the same server simultaneously serving a CFO's quarterly dashboard became a recipe for everything slowing to a crawl, or crashing at exactly the wrong moment.

+
Go DeeperETL vs ELT — Does the order actually matter?

ETL (Extract → Transform → Load) was designed for a world where compute and storage were the same machine. You had to know the shape of your data before loading it. The transformation happened in a dedicated staging area, and only clean, structured data landed in the destination.

ELT (Extract → Load → Transform) flips this. Raw data lands first — exactly as it arrived from the source, no transformations applied. Compute (Databricks) then reads that raw data and transforms it in place. This is the modern default for one key reason: cloud compute is cheap and elastic, so you no longer need to commit to a schema before you understand your data.

When ELT wins: messy or unpredictable source data, evolving schemas, reprocessing historical data with new logic, or any situation where "we might want this data later" is true. When ETL still makes sense: strict compliance requirements where raw PII data must never touch your platform, or high-volume real-time streaming where transformation must happen before landing.

In the Azure stack, the default is ELT: ADF loads raw to ADLS Bronze, Databricks transforms toward Silver and Gold.

The tightly coupled model assumed data volume was predictable and manageable. That assumption stopped being true around 2010, and it has gotten less true every year since.
🏢 Field Note
eBay · Mid-2000s

By the mid-2000s, eBay was processing hundreds of millions of transactions across its auction marketplace — and all of it ran through a central Oracle database cluster. The engineering team faced a problem that had no clean solution within the existing architecture: running analytical queries on that data required touching the same infrastructure that handled live bidding. Heavy reporting jobs would slow the marketplace. Overnight batch processes — meant to run while users slept — started extending into business hours as data volumes grew. The data team's ability to answer business questions was directly capped by their ability to avoid disrupting auctions.

eBay's engineering team eventually built a distributed data platform they called Singularity — an early precursor to what the industry would formalize as the modern data stack. It wasn't that Oracle was the wrong database. It was that the assumption underneath Oracle — that storage, compute, and serving could share one machine — had a ceiling. And eBay had hit it.

The Lesson

Tightly coupled architectures don't fail suddenly. They degrade slowly — batch jobs creep longer, query windows shrink, and eventually the data team starts making architectural concessions to avoid taking down the business.

The insight that changed everything was deceptively simple: storage and compute don't have to live in the same place. Cloud providers proved you could store virtually unlimited data in cheap object storage — think of it as an infinitely large, infinitely expandable filing cabinet that costs pennies per gigabyte — and spin up powerful compute clusters only when you actually needed to process something. When you were done, you shut them down and stopped paying. You weren't renting the filing cabinet and the forklift on one lease anymore. Each scales on its own terms.

+
Go DeeperBlock vs File vs Object Storage — three different philosophies

Storage is not one thing. There are three fundamentally different architectures, and the choice between them determines what you can build on top:

TypeHow it worksBest forAzure example
BlockRaw disk presented to an OS. You format it yourself.VM operating system disks, databases needing raw disk performanceAzure Managed Disks
FileFamiliar folder hierarchy over a network (NFS/SMB). Everyone sees the same directory.Shared drives, lift-and-shift of on-prem file serversAzure Files
ObjectFlat namespace. Each file is an independent object with a unique URL and metadata. No real directories.Data lakes, backups, static web assets, anything at scaleAzure Blob Storage / ADLS

Why object storage won for data lakes: it scales infinitely, costs pennies per gigabyte, and separates storage cost from compute cost entirely. The trade-off: no native directory operations (listing a folder on pure Blob Storage requires iterating every object). ADLS Gen2 adds a hierarchical namespace on top of Blob Storage to fix this — making directory operations atomic and Databricks-compatible without losing the infinite scale of object storage.

This decoupling of storage from compute is the philosophical foundation of the modern data stack. Worth noting: the same fundamental insight independently drove the rise of Databricks, Snowflake, and BigQuery in roughly the same five-year window. When a problem is painful enough, the industry converges on the solution almost simultaneously.

1.1.2

The Four Layers of Any Data Platform

Regardless of which specific tools a company uses — whether it's Azure, AWS, Google Cloud, or Snowflake — every modern data platform organizes itself into the same four layers. This isn't convention for the sake of convention. It's a direct consequence of the decoupling principle: when you separate storage from compute, you naturally end up with distinct concerns that deserve distinct homes. Understanding these four layers means you can look at any unfamiliar architecture diagram and immediately orient yourself — every data platform you'll encounter in your career maps onto them, even when it doesn't call them by these names.

Layer 1  —  Sources

Sources are where data originates: ERP (Enterprise Resource Planning) systems like SAP, CRMs like Salesforce, REST APIs, flat files dropped by vendors, IoT sensors, manufacturing line systems, web event logs. The critical thing about sources: you don't own them — you read from them. Your relationship with sources is one of careful extraction, not management. Everything downstream inherits the quality and latency of how cleanly you extract from Layer 1.

Layer 2  —  Ingestion

Ingestion is the movement of data from sources into your platform. Think of it as a postal service: it doesn't rewrite your letters, it moves them from Point A to Point B reliably, on a schedule, at scale. Azure Data Factory (ADF) is Microsoft's ingestion orchestrator. It connects to hundreds of source types, schedules pipeline runs, handles retry logic when something fails, and logs everything. What ADF does not do is complex transformation — and that boundary is intentional. The fact that ADF doesn't transform is seen as a limitation by beginners and a feature by architects.

Layer 3  —  Storage & Transform

This is the heart of the modern stack. Azure Data Lake Storage (ADLS) provides the storage: an infinitely large, cloud-based file system where data lives at every stage of its lifecycle. Databricks, running on Apache Spark, provides the compute that transforms data inside that lake. The two work together as a pair: ADLS holds data at rest, Databricks moves it forward.

This is also where the Medallion Architecture pattern plays out — and it is worth being precise about this, because it spans both layers. The three tiers (Bronze → Silver → Gold) all physically reside in ADLS. But transitioning between them requires Databricks compute. ADF drops raw data into Bronze — no compute, just a landing zone. Databricks then reads Bronze, cleans and validates it, and writes the result to Silver. Databricks again reads Silver, applies business logic and aggregations, and writes Gold. The tiers are data states. The transitions between them are compute events. Both layers are always in play. Figure 1.3 maps this flow in detail.

Layer 4  —  Consumption

Consumption is where business value is finally realized. Analysts query here. Executives view dashboards. Data scientists pull features for models. Power BI is the primary consumption tool in this stack, sitting atop the Gold layer. This is the only layer most of your stakeholders will ever see — which means it carries disproportionate weight in their perception of whether "data" is working. Everything upstream exists to make this layer fast, accurate, and trustworthy enough to act on.

The Medallion tiers — Bronze, Silver, Gold — live inside the Storage layer and represent increasing levels of data quality. Section 1.3 covers them in full detail.
Figure 1.3 The Medallion Architecture as a data refinery. Bronze is a pure landing zone — ADF drops data there with no transformation. Each subsequent tier requires Databricks compute to produce. All three states live in ADLS, but only Bronze is "free" — Silver and Gold carry a compute cost every time they're rebuilt.
1.1.3

Why Azure? Why This Particular Stack?

The natural question at this point: why not Snowflake? Why not AWS Redshift or Google BigQuery? The architectural answer is that the principles are identical across all of them — storage/compute separation, layered refinement, centralized consumption. The tools differ; the layers don't.

Snowflake's clever move, for instance, was collapsing Layers 2 and 3 into a single managed service — compelling until you need to do something it wasn't designed for. That's a useful pattern to understand: vendors often differentiate by collapsing or specializing layers, but the underlying layer model is always there, whether visible or not.

In most enterprise environments, the stack was already decided before you arrived. What matters is that concepts transfer regardless of the vendor. An engineer who understands Medallion Architecture in Azure can orient themselves in a Snowflake or Databricks-on-AWS design in an afternoon, because the thinking underneath is the same.

That said, the Azure + Databricks combination is one of the most prevalent in mid-to-large enterprise environments — particularly organizations already running on the Microsoft ecosystem. Power BI's deep integration with Azure Active Directory, Microsoft Teams, and Office 365 makes it a natural fit wherever Microsoft infrastructure already exists. If you're in a manufacturing, logistics, or ERP-heavy environment — which the role you're targeting signals — you are very likely in an Azure-first world.

🏢 Field Note
Moderna · 2020

When COVID-19 emerged, Moderna was a relatively small biotech company that had never brought a drug to market. One of the factors that enabled them to design their mRNA vaccine sequence in 48 hours and move to clinical trials faster than any vaccine in recorded history was an investment they had made years earlier: a cloud-based data infrastructure where genomic data, clinical trial results, manufacturing quality metrics, and regulatory documentation all lived in one unified platform — with compute that could scale to any workload on demand.

Traditional pharmaceutical data infrastructure, by contrast, was built on on-premise systems where biology teams, clinical teams, and manufacturing teams each had their own siloed data stores. Moving data between them required formal extraction jobs, format conversions, and reconciliation processes that could take days or weeks. Moderna's leadership has been publicly explicit in calling their digital infrastructure a strategic advantage. Their chief digital officer described it as the difference between being able to ask a question and get an answer versus submitting a request and waiting for a report.

The architecture was built on AWS rather than Azure — the specific cloud provider matters less than the principle. Storage and compute were decoupled. Data was accessible across teams. Workloads could scale to meet the moment.

The Lesson

The value of a properly decoupled data platform is often invisible until the moment when speed matters and legacy infrastructure is what stands between you and being able to move. You don't build this infrastructure for the normal days.

⚡ The Tool vs. The Principle

Throughout this guide, we separate the tool from the principle it embodies. Tools evolve, get replaced, and fall out of favor. Principles don't. When you understand why ADF exists — to decouple data movement from transformation — you can evaluate any competing ingestion tool against the same criteria. That judgment is what separates a practitioner from someone who only knows a product.

Go DeeperWhen Snowflake wins — an honest comparison

The Azure + Databricks stack is powerful but carries real complexity: cluster management, Delta Lake tuning, Unity Catalog governance, ADF pipeline design. Snowflake is a legitimate alternative — and in some situations, the better choice.

Snowflake wins when: the team is primarily SQL analysts rather than Python engineers, the workload is ad-hoc queries and BI dashboards rather than complex pipeline engineering, simplicity and managed infrastructure are valued over flexibility, and there is no existing Microsoft ecosystem lock-in (no Azure AD, no Office 365, no Power BI investment).

Azure + Databricks wins when: workloads include large-scale Python and ML work, Delta Lake's open format and time travel features matter, the organization is already invested in Azure services and wants tight integration, or the team needs full control over compute configuration and cost.

The pattern in the market: analytics-first companies (e-commerce, SaaS, media) often choose Snowflake. Engineering-heavy enterprises (manufacturing, finance, healthcare, logistics) more often land on Azure + Databricks. Neither is wrong. The job description you're targeting signals an Azure + Databricks environment — which is why that's the focus of this guide.

1.1.5

Your Role in This Architecture

Understanding architecture isn't just technical credibility — it's the practical mechanism by which you move from "the business has a problem" to "here's exactly what we build and where." Without it, you're reacting. With it, you're directing.

Consider a real scenario: a finance team reports that their inventory report is three days out of date. A junior analyst escalates it as "the report is broken." Someone with architectural fluency translates that immediately: three days of latency is almost certainly an ingestion problem. The data isn't reaching the platform on time, which means the issue lives in Layer 1 — not in the Power BI dashboard. That diagnosis narrows the investigation from "anything could be wrong" to "let's examine ADF pipeline scheduling and whether we're doing full table loads when incremental loads would be faster." That's a conversation that takes hours, not weeks.

The practitioner's core job is translation: from business symptom → to architectural layer → to specific tool → to actionable solution. The four-layer model is the translation key.

Every section that follows in this guide builds on this mental model. When you learn Databricks in Part 2, you'll know it lives in Layer 3 and its job is to transform data already sitting in ADLS. When you learn Power BI in Part 3, you'll know it reads from the Gold layer. When something breaks — and in data engineering, something always eventually breaks — you'll know which layer to examine first, and why.

✓ Section Summary

The modern data stack emerged from the failure of tightly coupled architectures to scale with data volume. Its core innovation — separating storage from compute — allows each layer to scale independently. Azure implements this through ADF (ingestion), ADLS (storage), Databricks (compute/transform), and Power BI (consumption). The four-layer mental model is how you diagnose problems, evaluate solutions, and direct teams — regardless of which tools are on the table.

What was the core architectural failure of legacy monolithic data platforms that led to the modern data stack?

A

The defining innovation of the modern data stack is decoupling storage from compute, so each can scale independently. When a transformation job was making a CFO's dashboard slow, it was because they shared the same machine. That problem disappears when compute spins up on demand and shuts down when done.

Question 2 of 5

Which of the following is NOT a responsibility of Azure Data Factory (ADF)?

A
Scheduling pipeline runs on a defined cadence
B
Connecting to hundreds of source system types
C
Applying complex business logic and aggregation transformations to data
D
Handling retry logic when a source connection fails

C is correct. ADF's job is to move data, not transform it. Complex business logic and aggregations belong in Databricks (Layer 3). This is a deliberate architectural boundary — and one of the most common mistakes practitioners make when first learning ADF. The phrase worth remembering: "ADF moves. Databricks transforms." Beginners see ADF's Data Flow feature and try to do everything there. Experienced practitioners do light-touch transformations in ADF at most, and push anything meaningful to Databricks.

Question 3 of 5

Your team needs to calculate monthly revenue by region, joining sales transactions with a customer dimension table. Where in the Medallion Architecture does this aggregation belong?

A
Bronze — process it immediately on arrival to avoid doing it twice later
B
Silver — alongside the cleaning and validation work
C
Gold — this is business-aggregated, query-optimized data ready for Power BI
D
Directly in Power BI using DAX measures, keeping the lake simple

C is correct. Gold is purpose-built for business-ready aggregates — joined, modeled, and optimized for the specific questions the business asks. Silver is for cleaning and standardizing individual source datasets. Bronze is raw and untouched. While Power BI can perform joins using DAX, doing them in Gold means the computation runs once at pipeline refresh time rather than on every dashboard load by every user — which is significantly more efficient at scale.

Question 4 of 5

A finance team's Power BI dashboard shows correct numbers, but the data is always exactly 24 hours stale. Your Databricks jobs show successful runs overnight. Using the four-layer model, where do you look first?

A
Layer 4 — the Power BI refresh schedule may be set to run weekly instead of daily
B
Layer 1 — the ADF ingestion pipeline is likely running once daily, so data always arrives 24 hours late
C
Layer 2 — the data may be sitting in the Archive tier of ADLS with high retrieval latency
D
Layer 3 — the Databricks transformation logic may have an off-by-one error in the date filter

B is correct. If Databricks (Layer 3) ran successfully overnight, the transformation layer is healthy. The 24-hour lag is upstream — in ingestion. ADF pipeline scheduling determines when source data arrives in Bronze. A pipeline set to run once per day at midnight means the data is always yesterday's by the time it reaches Power BI. The four-layer mental model narrows your investigation from "anything could be wrong" to "check ADF scheduling." This is the diagnostic value of understanding architecture before looking at individual components.

B is correct — and this question is designed to make a point: knowing when not to use a stack is part of being a genuine practitioner. ETL vs ELT — ETL transforms data before loading. Modern stacks often flip to ELT: load raw first, transform after. ELT is generally faster to implement and easier to debug. Knowing which pattern your organization uses tells you a lot about platform maturity.
