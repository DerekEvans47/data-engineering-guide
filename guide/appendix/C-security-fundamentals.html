Security Fundamentals for Data Engineers

The authentication, authorisation, encryption, and compliance concepts every senior data engineer must understand — from Managed Identity and Key Vault to Unity Catalog column masking, Private Endpoints, and GDPR data residency obligations

C.1

Why Security Is a Data Engineering Concern

Security is not the sole domain of a security team. Data engineers build and operate the systems that hold the most sensitive data in an organisation — customer PII (personally identifiable information), financial transactions, health records, intellectual property. The decisions made at the data platform level — who can access what, how credentials are managed, where data is stored, how it flows across network boundaries — determine the organisation's data security posture.

Senior data engineering roles require fluency with the security mechanisms of the Azure data stack specifically. Hiring panels test this through scenario questions: "how would you prevent an analyst from reading raw payment card data?", "what is wrong with storing a database password in a pipeline config file?", "how would you implement row-level security in a shared Power BI dataset?". This appendix gives you the vocabulary and patterns to answer these confidently.

The principle of least privilege is the most important security concept in data engineering: every user, service, and pipeline should have the minimum level of access required to perform its function — no more. This single principle, applied consistently, prevents most data breaches that occur through credential compromise or insider threat.
C.2

Identity and Access — RBAC vs ACL

RBAC (Role-Based Access Control) assigns permissions to roles, and users are assigned to roles. No direct user-to-permission mapping. This is the dominant model in Azure and most enterprise platforms. A data engineer has the "Data Engineer" role which grants read/write to specified containers; a data analyst has the "Data Analyst" role which grants read-only access to Silver and Gold layers. When an employee changes jobs, you change their role assignment — permissions follow automatically.

ACL (Access Control List) assigns permissions directly to individual identities on individual objects. More granular — you can set different permissions per user per file in ADLS Gen2 — but exponentially harder to manage at scale. In practice: use RBAC for coarse-grained access (who can access which containers, databases, workspaces), use ACLs for fine-grained exceptions (this specific analyst needs access to this specific folder).

Figure C.1. Azure identity hierarchy. Managed Identities are the preferred authentication mechanism for Azure services — they authenticate via Entra ID token exchange with no stored credentials. Service Principals require secret management and are appropriate when Managed Identity is unavailable (e.g. external CI/CD runners).
C.3

Managed Identity — No Credentials in Code

A Managed Identity is an identity automatically managed by Azure for an Azure resource. When you enable a Managed Identity on an Azure Data Factory (ADF) pipeline, a Databricks workspace, or an Azure Function, that resource gets an identity in Entra ID. Other Azure services can grant that identity RBAC roles. The ADF pipeline can then authenticate to ADLS Gen2, Azure SQL, or Key Vault without any password, connection string, or client secret stored anywhere in code or config.

This is the correct pattern for all service-to-service authentication within Azure. The alternative — storing connection strings in pipeline configs, environment variables, or (worst of all) checked into source code — is one of the most common causes of credential exposure in data engineering projects.

System-Assigned vs User-Assigned

System-assigned: tied to a single resource; deleted when the resource is deleted. Use when one service needs to authenticate to another and you don't need to share the identity. User-assigned: an independent identity resource that can be attached to multiple services. Use when several services (e.g. multiple ADF pipelines, multiple Functions) need the same set of permissions — manage one identity rather than many.

C.4

Azure Key Vault — Secret Management

Key Vault is the correct location for all secrets, connection strings, API keys, and certificates used by Azure data pipelines. The pattern is: pipeline authenticates to Key Vault via Managed Identity, retrieves the secret at runtime, uses it. The secret never appears in code, config files, logs, or environment variables. It is never stored locally. It can be rotated in Key Vault without any code change — the pipeline always fetches the current version.

ADF + Key Vault Pattern

In Azure Data Factory, linked services (connections to external systems) support Key Vault references directly in the connection string fields. The ADF Managed Identity needs the "Key Vault Secrets User" role on the vault. When the pipeline runs, ADF fetches the secret from Key Vault before initiating the connection. No human ever sees the credential in the pipeline configuration.

▶ Deep Dive — Private Endpoints and Network Isolation

By default, Azure services are accessible over the public internet (though protected by authentication). For organisations handling regulated data, this is insufficient — the data must never traverse the public internet, even if the traffic is encrypted and authenticated.

Private Endpoints solve this by giving an Azure resource (a Storage Account, SQL Server, Key Vault, etc.) a private IP address inside your Virtual Network (VNet). Traffic from your Databricks cluster or ADF integration runtime to that resource routes entirely over the Azure backbone and your private network — never over the public internet. The resource's public endpoint can be disabled entirely.

VNet Integration for services like ADF and Databricks ensures that outbound traffic from these services also exits through your VNet rather than the public internet. Combined with Private Endpoints on the target services, this creates a fully private data plane.

Service Endpoints are a lighter alternative that restricts a storage account or SQL server to accept traffic only from specified subnets. Easier to configure than Private Endpoints but does not provide a private IP — traffic still goes to the service's public IP (just from an allow-listed network).

In practice, enterprise Azure data platforms use Private Endpoints for storage accounts, Key Vault, Azure SQL, and Databricks workspace. The configuration is done by the platform/networking team, but data engineers need to understand the pattern because it affects connectivity: a pipeline that works in development (public access enabled) may fail in production (private endpoints, public access disabled) if the integration runtime is not on the approved VNet.

C.5

Unity Catalog — Fine-Grained Data Access in Databricks

Unity Catalog is Databricks' unified governance layer, providing centralised access control, data lineage, and auditing across all Databricks workspaces. For data engineers on the Azure Databricks stack, Unity Catalog is the mechanism for implementing column-level security, row-level security, and dynamic data masking — the controls that prevent analysts from reading raw PII while still allowing them to work with the data.

Figure C.2. Unity Catalog hierarchy and security controls. Permissions are granted at any level — catalog, schema, table, or column. Dynamic data masking applies masking functions at query time based on the caller's role, without modifying the underlying data.

Row-Level Security (RLS)

Row-level security restricts which rows a user can see based on their identity. In Unity Catalog, RLS is implemented using row filters — SQL functions applied to a table that evaluate the current user's group membership or attributes and filter rows accordingly. A regional sales analyst sees only the rows for their region; the filter is applied transparently at query time. They cannot circumvent it by writing different SQL.

-- Unity Catalog row filter example
CREATE FUNCTION region_filter(region_col STRING)
  RETURN IS_ACCOUNT_GROUP_MEMBER('emea_analysts') = (region_col = 'EMEA')
      OR IS_ACCOUNT_GROUP_MEMBER('apac_analysts') = (region_col = 'APAC')
      OR IS_ACCOUNT_GROUP_MEMBER('data_admins');  -- admins see all

ALTER TABLE gold.sales_facts SET ROW FILTER region_filter ON (region);
C.6

Encryption — At Rest and In Transit

Encryption at rest: all Azure storage services (Blob, ADLS Gen2, Azure SQL, Cosmos DB) encrypt data at rest by default using Microsoft-managed keys. For organisations requiring customer control, Azure Storage encryption supports Customer-Managed Keys (CMK) via Key Vault. The data engineer's responsibility here is mainly: ensure CMK is configured where required by policy, ensure key rotation is automated, and ensure the Key Vault holding the CMK is backed up and access-controlled.

Encryption in transit: all Azure service endpoints enforce HTTPS/TLS. Data engineers should ensure that: connection strings use encrypted protocols (not http:// for storage, not non-SSL database connections), Azure Storage accounts have "Secure transfer required" enabled (rejects HTTP), and any custom data movement code uses TLS explicitly rather than relying on defaults.

C.7

GDPR and Data Residency Obligations

GDPR (General Data Protection Regulation) imposes specific obligations on organisations processing personal data of EU residents. For data engineers, the most operationally relevant requirements are:

Data Residency

Personal data of EU residents must not be transferred outside the EU/EEA without adequate protection. In Azure terms: deploy storage and compute resources in EU Azure regions (West Europe, North Europe, Germany West Central, etc.). Ensure replication policies don't copy data to non-EU regions. Azure paired regions within the EU are acceptable; cross-region replication to the US is not, without additional contractual protections.

Right to Erasure (Right to be Forgotten)

Individuals can request deletion of their personal data. For data engineers, this means pipelines must be designed for erasure: identify all tables and Delta partitions containing data for a given individual, execute targeted deletes, and handle the consequences for downstream models that used that data. Delta Lake's DELETE command handles this at the table level; the challenge is identifying all locations where a person's data has propagated across a multi-layer platform.

Data Minimisation and Retention

Only collect and retain what is necessary for a stated purpose. In practice: implement retention policies (auto-delete raw PII after 90 days if no longer needed), avoid copying PII into analytics layers unless necessary (use pseudonymised keys instead), and document the purpose and retention period for each personal data element in your data catalog.

📋 Field Note — GDPR Erasure at a Retail Analytics Platform

A UK-based retail company received a surge of "right to be forgotten" requests following a high-profile news story about data retention practices. Their existing platform had no erasure capability — customer data existed in twelve Delta tables across bronze, silver, and gold layers, several dbt mart tables, a Databricks Feature Store, and historical snapshots in a secondary analytics database.

The data engineering team spent six weeks building an erasure pipeline: a parameterised Databricks job that accepted a customer_id, looked up all table locations from Unity Catalog lineage, executed targeted DELETEs, ran VACUUM to remove the physical files, and updated a compliance audit log. The critical design decision was not to cascade deletes automatically through all downstream tables — instead, the pipeline identified affected records and required explicit approval before deleting from Gold layer tables that fed live dashboards. This human gate prevented accidental destruction of data that the subject had not actually requested be deleted.

The lesson: build erasure capability into the platform from day one, not as a retrofit. The right time to design for the right to erasure is when you design your data model, not when a request arrives.

Least Privilege First

Every user, service, and pipeline gets the minimum access needed to do its job. Scoped RBAC — container-level, not subscription-level — is the single most effective security practice you can apply to a data platform.

Managed Identity Always

For any service-to-service auth within Azure — ADF to ADLS, Databricks to Key Vault, Functions to SQL — use Managed Identity. No secrets to rotate, no credentials in config, no exposure risk. It costs nothing extra.

Design for Erasure Early

GDPR right-to-erasure is a data engineering problem. If you don't build erasure capability at design time, you will spend weeks retrofitting it later. Know which tables contain PII, how it propagates, and how to delete it cleanly.

Interview Pattern

"How do you prevent analysts from reading raw payment card numbers?" Answer: Unity Catalog dynamic data masking on the column, role-based — analysts get masked values, DE admins get raw. No code change needed when access patterns change.
