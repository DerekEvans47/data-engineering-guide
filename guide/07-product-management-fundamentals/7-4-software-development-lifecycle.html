The Software Development Lifecycle

7.4.1

The Process That Contains Every Other Process

Everything covered so far in this Part — writing a good story, tracking it in Jira, validating its design with the right users — happens inside a larger container: the Software Development Lifecycle (SDLC), the end-to-end process by which a piece of work moves from an idea to something running reliably in production and eventually retired. Section 5.1 covered Agile as a delivery philosophy; this section covers the broader structural question Agile is one possible answer to — how does work actually flow, end to end, from "someone has an idea" to "it's live, stable, and someone is on call for it"?

The SDLC has the same handful of phases regardless of which methodology wraps around them: Requirements (Section 7.1's territory), Design (Section 7.3's territory), Development, Testing, Deployment, and Maintenance. What differs between methodologies is whether these phases happen once, in strict sequence, for an entire project — or whether they happen repeatedly, in tight loops, for small slices of a project, with feedback flowing continuously back into earlier phases rather than only at the very end.

Figure 7.7The same six phases, structured two ways. Waterfall runs them once, in sequence, for an entire project — feedback from testing only reaches requirements after the full build is done, when it's most expensive to act on. Agile and DevOps-integrated models run small, repeated loops, feeding monitoring and production learnings directly back into the next loop's requirements — the same continuous-improvement principle behind Section 5.1's sprint retrospectives and Section 5.4's CI/CD pipelines, applied at the level of the whole lifecycle rather than just the code deployment step.

Where Data Engineering Actually Sits

Data engineering work is frequently treated as an afterthought inside a broader SDLC designed primarily with application software in mind — "the app is done, now hook up the data" — which causes a predictable set of problems: pipelines built reactively under time pressure, data models that fight the application's existing schema rather than being designed deliberately, and testing that focuses entirely on the application layer while treating data correctness as someone else's problem to verify after the fact. A senior data practitioner's job inside an SDLC conversation is frequently to insist that data requirements, data design, and data testing get scheduled as first-class phases alongside (not after) the application work — exactly the kind of structural insistence Section 5.1's Agile principles and this Part's PM artifacts are meant to give a practitioner the vocabulary to make.

7.4.2

Shift-Left Testing

Shift-left testing means moving testing activities earlier in the lifecycle — "left" on a diagram that flows left to right through the phases — rather than treating testing as a gate that happens only after development is "finished." For a data pipeline specifically, this means writing dbt tests alongside the transformation logic itself (not after), validating schema expectations against source systems before a single line of transformation code is written, and catching a data quality assumption that's wrong during design review rather than discovering it three weeks later when a stakeholder notices a number looks impossible.

The economic argument for shifting left is the same one that justifies Section 7.1's acceptance criteria and Definition of Ready: a defect caught during design costs a conversation; the same defect caught in production costs an incident, a root-cause investigation, a fix, a redeploy, and — if the bad data already reached a dashboard a stakeholder acted on — a much harder conversation about trust.

+
Go DeeperShift-left testing for a real data pipeline, step by step

Shift-left is a principle, but it only matters once it's turned into specific habits at specific points in a pipeline's build.

At requirements time: write the negative-case acceptance criteria (Section 7.1) before development starts — "what should happen if this source field is null?" is a testing question disguised as a requirements question, and answering it now is far cheaper than discovering the gap in production.

At design time: validate the source schema against assumptions before writing a single transformation. A quick exploratory query against the actual source data — checking for nulls, duplicate keys, unexpected value ranges — frequently surfaces a wrong assumption ("this field is always populated") that would otherwise have been baked silently into the transformation logic and discovered only when it broke something downstream.

During development: write dbt tests (not_null, unique, relationships, custom business-rule tests) in the same pull request as the model itself, not as a follow-up ticket. A model and its tests reviewed together in one PR makes it structurally harder to ship untested logic, because the absence of tests is visible to the reviewer in the same diff.

Before deployment: run the full test suite against a realistic-volume Test or UAT environment, not just a small local sample — a transformation that handles ten rows correctly can still fail at production volume due to a join that fans out unexpectedly or a window function that behaves differently against a much larger partition.

The common thread across all four points: each one moves a question that would otherwise surface as a production incident into a stage where it's a normal, low-stakes part of the existing workflow — a code review comment instead of a 3am page.

🏢Field Note
A Healthcare Network  ·  The Shift-Left Habit That Paid for Itself in One Sprint

A healthcare network's data team adopted a rule, after a string of production data quality incidents: no model gets merged without dbt tests in the same pull request, and no source table gets built against without a quick exploratory profiling query run first, documented in the story itself. The team expected this to slow delivery down — more upfront work per story, on paper.

During the very first sprint under the new rule, the exploratory profiling step on a new insurance claims source caught that a "claim status" field, assumed to have five clean values matching the source system's documentation, actually contained eleven distinct values in practice — including several legacy codes the documentation never mentioned, still actively in use for a subset of older claim types. Building the transformation logic against the documented five values, as originally planned, would have silently misclassified roughly 8% of claims in production.

The fix took twenty extra minutes during design — updating the transformation's case logic to handle all eleven values correctly. Had it shipped against the original five-value assumption, the data team estimated, based on a near-identical incident from the previous quarter, that the production discovery and correction cycle would have taken roughly two weeks, involved a stakeholder-facing data quality apology, and required a backfill of several months of misclassified historical claims.

The Lesson

Shift-left testing's upfront cost is real but small and predictable — minutes per story. The cost it prevents is unpredictable in timing but reliably large when it lands. The trade only looks unfavourable if you only count the visible upfront cost and ignore the (much larger, just less visible until it happens) cost being avoided.

7.4.3

Environment Promotion — Dev, Test, UAT, Prod

A clean environment promotion path is the structural backbone that makes shift-left testing actually enforceable rather than aspirational. Each environment exists to catch a specific category of problem before it reaches the next, more consequential, stage.

Figure 7.8Each environment catches a different class of problem. Dev and Test catch technical defects — bugs, broken tests, regressions — through automation and the original engineer. UAT is the only stage where someone outside the build team checks the output against their actual real-world expectations, the same check Section 7.3 argued automated tests structurally cannot perform. Skipping straight from Test to Prod removes that one human check entirely — everything that passed automated testing but still doesn't match what the business actually meant only gets discovered live.
+
Go DeeperStructuring environment promotion for ADF, Databricks, and dbt specifically

The Dev / Test / UAT / Prod concept is generic, but each tool in the Azure stack implements it slightly differently, and a senior practitioner needs to know the concrete mechanics, not just the abstract principle.

Azure Data Factory: ADF supports separate factories (or separate folders within one factory, using ARM template parameterization) per environment, with linked services pointing at environment-specific connection strings. The standard pattern is Git-integrated development in a Dev factory, with ARM template export and deployment via Azure DevOps pipelines promoting the same pipeline definitions into Test and Prod factories — the pipeline logic itself doesn't change between environments, only the parameterized connection details it points at.

Databricks: separate workspaces per environment is the cleanest pattern for genuine isolation, though smaller teams sometimes use a single workspace with environment-specific catalogs or schemas (Unity Catalog makes this considerably cleaner than older Hive metastore-based separation). Notebooks or Databricks Asset Bundles get deployed via CI/CD (Section 5.4) rather than manually copy-pasted between workspaces, for the same auditability reasons covered there.

dbt: environment separation typically happens through dbt's built-in target configuration — the same models run against dev, test/staging, and prod targets, each pointing at a different database or schema, with environment-specific variables controlling things like sample-size limits in dev for faster iteration. dbt Cloud or a CI/CD pipeline runner triggers the test-environment run automatically on every pull request, which is what makes the Test stage in Figure 7.8 actually automatic rather than a manual step someone has to remember.

The cross-cutting principle: in every case, the goal is that promoting to the next environment is a configuration and deployment change, never a rewrite of the underlying logic. If promoting from Test to UAT requires touching the actual transformation code rather than just the environment it points at, the environments aren't truly separated yet — and that gap is exactly where "it worked in Test" surprises happen in Prod.

🏢Field Note
An Energy Utility  ·  "We're Confident, Skip UAT"

An energy utility's data team rebuilt a customer billing reconciliation pipeline under deadline pressure ahead of a regulatory reporting date. The new logic passed every automated dbt test in the Test environment cleanly. With the deadline two days away and the team confident in the rebuild, the project lead made the call to skip the planned UAT cycle and promote directly to production — "the tests are green, we're confident, we don't have time."

The rebuilt logic was technically correct against every test the team had written — and every test the team had written reflected the team's own understanding of the business rules, not the finance team's. A change in how partial payments were allocated across multiple outstanding invoices, technically correct by the data team's interpretation of the specification, did not match how the finance team had actually been doing it manually for years — a nuance nobody on the data team had reason to know, because it had never been written down anywhere, including in the original requirements.

The discrepancy was discovered four days after go-live, when finance noticed reconciliation totals didn't match their own manual figures — after the regulatory submission had already gone out using the new pipeline's numbers. The correction required an amended regulatory filing, a multi-day reconciliation effort to identify every affected account, and a substantially more difficult conversation with the regulator than a standard submission would have required.

The Lesson

"All tests pass" only proves the code matches the team's own understanding of the requirements — it cannot prove that understanding itself is correct. UAT exists precisely to catch the gap between "technically correct, by our interpretation" and "actually matches how the business has really been doing this." Skipping it under deadline pressure trades a known, bounded delay now for an unknown, frequently much larger cost later — and the cost lands disproportionately on a regulatory or financial deadline being the worst possible moment to discover the gap.

+
Go DeeperThe SDLC is a process, not a tool — tying 7.2's tooling back to this section

It is entirely possible to own every tool covered in Section 7.2 — a perfectly configured Jira workflow, a tidy Confluence space, polished Figma mockups — and still run a broken SDLC, because the tools track and document the process; they don't enforce it. A team can have a Jira board with a "UAT" column that every ticket dutifully passes through in name, while the actual sign-off step is a rubber stamp nobody takes seriously, because the organizational discipline behind the column, not the column itself, is what makes UAT meaningful.

What actually enforces an SDLC stage, in practice: a Definition of Done (Section 7.1) that explicitly requires a named stakeholder's sign-off before a ticket can move out of UAT — not just a board column the ticket passes through. A CI/CD pipeline (Section 5.4) configured so that a production deployment is technically blocked without a passing Test-environment run, removing the option to skip it under pressure rather than relying on willpower. An on-call rotation and incident review process that explicitly asks, for any production data incident, "which SDLC stage should have caught this, and why didn't it?" — turning each incident into a structural improvement to the process itself rather than just a one-off fix.

The throughline across this entire Part: a story (7.1) becomes trackable, visible work (7.2), gets validated against the right humans before being built (7.3), and moves through a process with enforced checkpoints rather than honour-system ones (7.4). Each section closes a gap the previous one leaves open. Skipping any single one reopens a version of the same underlying failure — building the wrong thing, confidently, and finding out only after it's already live.

✓ Section Summary — Part 7 Complete

The SDLC is the container every other process in this Part operates inside — Requirements, Design, Development, Testing, Deployment, and Maintenance, run once in strict sequence under Waterfall, or in small repeated loops under Agile and DevOps-integrated models, with continuous feedback flowing back into the next loop. Data engineering work is frequently treated as an afterthought inside SDLCs designed primarily around application software, and a senior practitioner's job is often to insist data requirements, design, and testing get scheduled as first-class phases. Shift-left testing moves defect discovery earlier — into requirements, design, and code review — where it's cheap, rather than into production, where it's expensive and visible. A clean Dev → Test → UAT → Prod promotion path matters because each stage catches a different class of problem, and UAT specifically is the only stage where a real stakeholder verifies output against their actual expectations rather than against the team's own interpretation of a specification — skipping it under deadline pressure trades a small known delay for an unknown, frequently much larger cost. Tools (Section 7.2) document and track a process; only explicit enforcement mechanisms — Definition of Done sign-offs, blocking CI/CD gates, incident reviews that ask which stage should have caught this — actually make the process real rather than aspirational. With this section, Part 7 closes the loop this guide opened in Section 7.1: turning a vague stakeholder wish into something built, validated, and shipped through a process that catches its own mistakes before they become expensive.

Question 1 of 5

What is the fundamental structural difference between Waterfall and Agile/DevOps-integrated SDLC models, given that both contain the same core phases (Requirements, Design, Development, Testing, Deployment, Maintenance)?

A
Waterfall includes a Testing phase and Agile does not
B
Waterfall runs the phases once, in strict sequence, for the entire project, with feedback only available after the full build; Agile and DevOps-integrated models run small, repeated loops through the same phases, feeding feedback continuously back into the next loop much earlier
C
Agile is only used for software, while Waterfall is the only model appropriate for data engineering work
D
DevOps-integrated models skip the Requirements phase entirely in favour of immediate development

B is correct. Both models contain identical phases — the difference is shape and feedback timing, not content. Waterfall's single-pass, full-project structure means a wrong assumption made during early requirements surfaces only once the entire build is complete and tested, by which point it's expensive to correct. Agile and DevOps-integrated models repeat the same phases in small loops, so a wrong assumption surfaces within days or weeks rather than months. A is factually wrong — Waterfall includes Testing as a phase, it just runs it once at the end. C is an oversimplification — both models are used in data engineering depending on context (see the comparison table). D is also incorrect — DevOps-integrated models still gather requirements, just in smaller, more frequent increments rather than skipping the phase.

Question 2 of 5

A data engineer discovers, during an exploratory profiling query run before writing any transformation code, that a source field assumed to have five clean values actually contains eleven, including undocumented legacy codes. What does this scenario best illustrate, and what would have happened if the same discovery had instead occurred after production deployment?

A
This illustrates Definition of Ready specifically, and the cost would have been identical whether caught before or after deployment
B
This illustrates shift-left testing — catching the data quality issue during design, at the cost of a short profiling query, instead of in production, where the same gap would likely have caused silent data misclassification, a stakeholder-facing incident, and a costly backfill of historical data
C
This illustrates UAT, since the engineer is verifying the output matches stakeholder expectations
D
This illustrates a documentation failure only, with no broader implication for the testing process

B is correct. Profiling source data before writing transformation logic is a textbook shift-left habit — moving defect discovery as early as possible in the lifecycle. The cost asymmetry is the entire point: catching this pre-development costs minutes; catching it in production (as the related field note describes) costs a multi-week incident cycle and a historical data backfill — the costs are explicitly not identical (contradicting A). This is not UAT (C) — UAT involves a stakeholder verifying output against their expectations, whereas this is a technical data-profiling step done by the engineer alone, earlier in the process. Calling it purely a documentation failure (D) misses the actionable lesson — the fix wasn't better documentation, it was building the habit of verifying assumptions against real data before coding against them.

Question 3 of 5

A pipeline rebuild passes every automated dbt test in the Test environment. Under deadline pressure, the team skips UAT and promotes directly to production. A discrepancy with how the business actually performs a calculation manually is discovered four days after go-live. What does this reveal about what automated tests can and cannot verify?

A
The automated tests were poorly written and should be rewritten to be more comprehensive before this kind of issue can be prevented
B
Automated tests verify that code matches the team's own understanding of the requirements — they cannot verify that understanding itself is correct. UAT exists specifically to catch the gap between "technically correct by our interpretation" and "actually matches how the business has been doing this," a check no automated test, however comprehensive, can perform on its own
C
This proves automated testing is generally not worth the investment for data pipelines
D
The issue was caused by insufficient Dev-environment testing, not a missing UAT stage

B is correct. No amount of additional automated test coverage (A) fixes this specific gap, because the tests were written against the team's own (incorrect, in this case) interpretation of the business rule — more tests of the same flawed understanding just confirms the flawed understanding more thoroughly, it doesn't surface the flaw. This does not argue against automated testing generally (C) — Dev/Test-stage automation correctly caught everything it was designed to catch (genuine code defects); it simply was never designed to catch a requirements-interpretation gap, which is a fundamentally different problem requiring a different kind of check. D misattributes the failure — more Dev-stage testing wouldn't have surfaced a gap that only a business stakeholder, comparing the output to their own manual process, could catch; that's specifically UAT's job, which was skipped.

Question 4 of 5

A team has a Jira board with a clearly labeled "UAT" column, and every ticket passes through it before being marked Done. However, no stakeholder is actually required to review or approve anything while a ticket sits in that column — it's treated as a formality. What is the most accurate description of this team's actual process, and what would fix it?

A
The team has a fully functioning UAT process, since the column exists and every ticket passes through it
B
The team has the appearance of a UAT stage without its substance — the tool tracks that a column was passed through, but doesn't enforce the actual human verification step that gives UAT its value. The fix is a Definition of Done requiring an explicit, named stakeholder sign-off before a ticket can leave that column, not just the column's existence
C
The team should remove the UAT column entirely, since it isn't being used meaningfully and its presence is misleading
D
The team needs a more sophisticated tool than Jira, since Jira itself is the root cause of the weak UAT process

B is correct. This is precisely the "tools document the process, they don't enforce it" point — a board column that every ticket passes through by default, without a required human action attached, is theater rather than verification. The fix is organizational (a Definition of Done requiring named sign-off), not technical. Removing the column (C) would make the gap more honest but doesn't fix the underlying missing verification step — it just stops pretending one exists. A different tool (D) misdiagnoses the problem; the issue is process discipline, not Jira's capability — Jira can absolutely support a sign-off requirement if the team chooses to enforce one.

Question 5 of 5

Which organization is the better fit for a Waterfall SDLC model rather than Agile or DevOps-integrated, and why?

A
A startup building an MVP dashboard with rapidly evolving stakeholder requirements week to week
B
A team building a fixed-scope, heavily regulated reporting system against requirements that are genuinely stable, well-documented, and unlikely to change mid-project — the kind of context where Waterfall's slower feedback loop is an acceptable tradeoff for its more rigorous upfront sign-off process
C
A data team iterating weekly on a Power BI dashboard based on ongoing user feedback
D
No real-world data engineering context ever justifies Waterfall; it should be considered obsolete in all cases

B is correct. Waterfall's core tradeoff — slower feedback in exchange for thorough, locked-down upfront specification and sign-off — actually pays off when requirements are genuinely stable and the cost of getting something wrong is high enough to justify extensive upfront validation, which describes many regulated reporting contexts well. A and C both describe situations with rapidly evolving requirements, exactly where Waterfall's slow feedback loop becomes a liability rather than an advantage — Agile fits both far better. D overstates the case — Waterfall is the wrong default for most modern data product work, but "wrong default" is different from "never appropriate"; dismissing it entirely ignores the genuine contexts (like B) where its tradeoffs are the right ones.

Same Phases, Different Shape

Waterfall runs Requirements → Maintenance once, for the whole project. Agile and DevOps-integrated models run the same phases in small, repeated loops — the structure changes, the phases don't.

Shift-Left Math

A defect caught at design time costs a conversation. The same defect caught in production costs an incident, a fix, a redeploy, and a harder conversation about trust. The earlier stage is always cheaper.

Tests Verify Code, Not Understanding

"All tests pass" proves the code matches the team's interpretation of requirements — not that the interpretation is correct. Only UAT, with a real stakeholder, catches that gap.

Interview Insight

"Tell me about a time a process step you skipped caused a problem" is a near-universal senior interview question. A strong answer names the specific SDLC stage skipped and the structural fix applied afterward — not just "we were more careful next time."
