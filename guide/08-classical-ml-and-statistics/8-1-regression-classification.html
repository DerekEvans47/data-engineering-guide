8.1

Regression & Classification

The two fundamental problems all supervised machine learning solves, the four algorithms practitioners reach for first, and why the shape of the thing you're predicting matters more than the business question itself

  2. Explain how linear regression fits a line to data and why its coefficients are directly interpretable

8.1.1

Supervised Learning: The Target Variable Decides Everything

Supervised learning is the branch of machine learning (ML) where a model learns a mapping from input variables — called features — to a known output variable, called the label or target, using historical examples where the correct answer is already known. You show the model thousands of past loan applications along with whether each one defaulted, and it learns the pattern well enough to estimate the risk on a new application it has never seen. This is different from unsupervised learning (covered in section 8.2), where there is no known label at all — the goal is to discover structure in the data rather than predict a known outcome.

Within supervised learning, there is exactly one question that determines which family of algorithms applies, and it has nothing to do with how important or complex the business problem feels: is the target variable a continuous number, or is it a category? Predicting next month's warehouse throughput in units is a number on a continuous scale — that's regression. Predicting whether a customer will churn this quarter is one of a fixed set of categories (yes or no) — that's classification. A surprising amount of confusion in this field — including the name "logistic regression," which you'll meet in section 8.1.3 — comes from not asking this question first.

The business question never tells you whether you need regression or classification. The shape of the target variable does. "How much revenue will we make next month?" is regression. "Will this specific deal close this month?" is classification. Both questions can come from the exact same sales pipeline data — what differs is what you're asking the model to output.

Classification itself splits further: binary classification has exactly two possible categories (churn / no churn, fraud / not fraud), while multiclass classification has three or more (which of five support queues should this ticket route to). The distinction matters because some algorithms handle multiclass problems natively and others need an extension — a detail covered in the deep dive after section 8.1.3.

Where does this sit relative to the rest of the guide? The Gold-layer tables you've spent Parts 1 through 7 learning to build — clean, well-modeled, business-logic-applied — are exactly what becomes training data for these models. A senior data engineer is rarely the person choosing between logistic regression and a random forest, but is very often the person whose pipeline determines whether the features reaching the data science team are correct, complete, and available at the right point in time. That last detail — timing — turns out to be one of the most common and most expensive mistakes in production ML, and it gets a full treatment in section 8.1.6.

8.1.2

Linear Regression

Linear regression is the oldest and most interpretable algorithm in this section, and very often the right first model to try on a regression problem before reaching for anything more complex. It fits a straight line — or, with multiple input features, a flat hyperplane — through the training data, of the form:

y = b₀ + b₁x₁ + b₂x₂ + … + bₙxₙ
Predicted value = intercept + (weight × feature) for every feature in the model

The model "learns" by finding the set of weights (b₁, b₂, …) that minimizes the gap between its predictions and the actual values across all training rows — typically measured as the mean squared error, the average of the squared differences between predicted and actual. Why squared rather than just the difference? Squaring makes every error positive (so overshooting and undershooting don't cancel out) and penalizes large errors disproportionately more than small ones, which is usually the behaviour you want — a model that is wildly wrong on one row is worse than one that is slightly wrong on many.

The reason linear regression remains so widely used despite being decades old is interpretability: each weight tells you, directly, the effect of that feature on the prediction, holding everything else constant. If the weight on "years of customer tenure" is +42, you can say to a stakeholder, with no hedging, "each additional year of tenure is associated with $42 more in predicted annual spend." Decision trees and random forests, covered later in this section, cannot make a claim that clean.

Linear regression's core assumption — and its main limitation — is that the relationship between each feature and the target really is a straight line. Demand that doubles every time price drops by 10% is not a linear relationship; advertising spend with strongly diminishing returns past a certain point is not linear either. Forcing a straight line onto a curved relationship produces a model that is systematically wrong in predictable ways — a problem you can usually spot by plotting the residuals (the errors) and seeing a curved pattern rather than random scatter.

Go DeeperHow does a model actually "learn" a line? Gradient descent in plain terms

For small datasets, the exact best-fit line for linear regression can be solved directly with linear algebra (a method called ordinary least squares). But most real ML training — including logistic regression, neural networks, and gradient-boosted trees — uses an iterative method called gradient descent instead, because a direct solution either doesn't exist or is too computationally expensive at scale.

The analogy: imagine you're standing on a hillside in thick fog, trying to reach the lowest point in the valley. You can't see the whole landscape, but you can feel which direction the ground slopes beneath your feet. So you take a step downhill, feel the slope again, take another step, and repeat — gradually working your way to the bottom. The "hillside" is the model's cost function (how wrong its current predictions are, plotted against every possible combination of weights), and "downhill" is the direction that reduces that error the most.

The learning rate controls how big each step is. Too large, and you overshoot the bottom of the valley and bounce around without ever settling (the model fails to converge, or its error oscillates wildly). Too small, and you crawl toward the bottom so slowly that training takes far longer than necessary, or you get stuck in a shallow dip that isn't the true minimum. Tuning the learning rate is one of the most common practical levers in ML training, and is the first thing to check when a model's training loss isn't decreasing as expected.

This same hill-descending process is what's happening, conceptually, every time you see a training job logging a decreasing "loss" value epoch after epoch — whether it's a simple linear regression or a billion-parameter LLM (Large Language Model) being trained. The math gets more elaborate at scale, but the core idea — measure the slope, take a step downhill, repeat — does not change.

8.1.3

Logistic Regression: A Classifier Wearing a Regression's Name

The single most common point of confusion for anyone new to this material: logistic regression is not used for regression problems. It is one of the most widely used classification algorithms in production systems today, and the name is a historical artifact rather than a description of what it does.

Here's how it actually works. Logistic regression starts exactly like linear regression — it computes a weighted sum of the input features. But instead of returning that number directly as the prediction, it passes the result through the sigmoid function, an S-shaped curve that squashes any number, no matter how large or small, into a value strictly between 0 and 1. That output is interpreted as a probability — "this transaction has an 87% estimated probability of being fraudulent" — and a threshold (commonly, but not always, 0.5) converts that probability into a final class label: above the threshold, predict "fraud"; below it, predict "not fraud."

This probability output is precisely why logistic regression is so heavily used in practice, beyond just its accuracy. A binary yes/no prediction discards information — a 51% churn probability and a 98% churn probability are both "predict churn," but they call for very different business responses. A retention team can triage its limited outreach budget toward the 98%-probability customers first, something a model that only outputs "yes" or "no" cannot support.

Like linear regression, logistic regression's weights remain reasonably interpretable: each one tells you the direction and approximate strength of a feature's effect on the log-odds of the positive class. It's a slightly less intuitive quantity to explain to a non-technical stakeholder than a linear regression coefficient, but it is still far more transparent than the algorithms covered next in this section — which matters enormously in regulated industries where a model's decision sometimes has to be explained to a regulator or a rejected customer by law.

Go DeeperBeyond yes/no: handling more than two classes

Logistic regression, as described above, naturally handles binary classification. Real classification problems are frequently multiclass — routing a support ticket to one of five queues, or classifying a product image into one of dozens of categories. Two standard strategies extend binary classifiers to handle this:

One-vs-Rest (OvR): train one binary classifier per class, each asking "is this Class A, or is it anything else?" For five ticket queues, that's five separate logistic regression models. At prediction time, run all five and pick whichever model is most confident. Simple to implement and reason about, but the separate models never directly compare against each other, which can produce inconsistent calibration across classes.

Softmax (multinomial) regression: rather than training separate models, extend the sigmoid function to output a full probability distribution across all classes simultaneously, guaranteed to sum to exactly 1. This is the standard approach inside neural networks' final layer for multiclass problems, and is generally preferred over One-vs-Rest when training a single unified model is feasible, because the classes are learned jointly rather than independently.

Decision trees and random forests, by contrast, handle multiclass classification natively with no extension required — a leaf node can simply store a probability distribution over any number of classes directly. This is one of several reasons tree-based methods are often the practical default for multiclass tabular problems.

8.1.4

Decision Trees

A decision tree makes predictions through a sequence of simple yes/no questions about the input features, arranged in a branching tree structure — conceptually identical to a flowchart. "Is account tenure greater than 18 months? If no, is monthly usage below 20 hours? If yes, predict churn." Each question is a split, and the tree keeps splitting the data into progressively more homogeneous (purer) groups until it reaches a leaf node, which holds the final prediction — a class label for classification, or an average value for regression.

The algorithm chooses each split by testing many candidate questions (across every feature and many possible threshold values) and picking whichever one creates the purest resulting groups. For classification, "purity" is most often measured using Gini impurity or entropy — both quantify how mixed the classes are within a group, with a value of zero meaning the group is now 100% one class. For regression, the equivalent measure is the reduction in variance within each resulting group.

The single biggest advantage of a decision tree is that it is the most interpretable model covered in this section after linear and logistic regression — arguably more so, because it can be drawn as an actual diagram and walked through, question by question, with a business stakeholder who has never heard the word "algorithm" in their life. The biggest disadvantage is that a tree grown without any constraints will happily keep splitting until every single leaf contains exactly one training row — at which point it has achieved 100% accuracy on the data it was trained on by essentially memorizing it, and has learned nothing generalizable about new data it hasn't seen. This failure mode, overfitting, is one of the most important concepts in this entire section and gets dedicated treatment in 8.1.6.

In practice, decision trees are constrained during training to prevent this: a maximum depth limits how many questions deep the tree can go, a minimum samples per leaf stops splits that would create tiny, statistically meaningless groups, and post-training pruning removes branches that don't meaningfully improve validation performance. Even with these constraints, a single tree's accuracy on unseen data is rarely state-of-the-art — which is exactly the gap random forests close.

🏢Field Note
A Manufacturing Firm  ·  The Tree the Engineers Trusted Immediately

A manufacturing company's data science team built a predictive maintenance model to flag machines likely to fail within the next 7 days, using sensor data — vibration, temperature, run-hours since last service — engineered into Gold-layer features by the data engineering team. The first model the team tried was a single decision tree, deliberately, before reaching for anything more sophisticated.

The result wasn't the most accurate model they would eventually build, but it was the most immediately trusted. The maintenance engineers could look at the printed tree diagram and say, within seconds, "yes — vibration above that threshold combined with run-hours past that point is exactly the pattern I've seen before a bearing fails." That validation from domain experts, who could read the model's logic directly rather than being told to trust a black box, was what got the project funded for a second phase.

The second phase replaced the single tree with a random forest once accuracy needed to improve beyond what the engineers' intuition alone could validate — but the team kept the original tree as a standalone "explainability layer" they could show to plant managers, alongside the more accurate forest doing the actual flagging in production.

The Lesson

A simpler, more interpretable model that domain experts can validate against their own intuition is often the right first step — even when you already know you'll eventually need something more powerful. Trust, once lost on an unexplainable model, is expensive to rebuild.

8.1.5

Random Forests: Many Trees, One Vote

A random forest is an ensemble method — rather than training one model, it trains many decision trees (often hundreds) and combines their outputs into a single prediction. The mechanism that makes this work, rather than just multiplying the same overfitting problem, is called bagging (bootstrap aggregating), and it introduces two distinct sources of randomness into how each tree is built:

1. Each tree trains on a random sample of the rows — drawn with replacement from the full training set (a "bootstrap sample"), so each tree sees a slightly different, overlapping subset of the data. 2. Each tree's splits only consider a random subset of the features at each decision point, rather than all of them. Both forms of randomness mean every tree in the forest ends up slightly different from every other tree — and, critically, each tree tends to overfit to different, essentially random quirks of its own training subset.

At prediction time, every tree in the forest votes (classification) or contributes its predicted value to an average (regression), and the forest's final output is the combined result. This is the practical payoff of bagging: the random, idiosyncratic mistakes that any individual tree makes tend to cancel out across hundreds of trees, while the genuine underlying signal — the pattern that's actually true across the data, which every tree picks up on regardless of its specific random subset — gets reinforced. The result, in the large majority of tabular ML problems, is meaningfully higher accuracy on unseen data than any single tree achieves, with substantially lower risk of overfitting.

Figure 8.2Bagging in a random forest. Each tree is trained on its own bootstrap sample of rows and considers only a random subset of features at each split, producing many slightly different, individually imperfect trees. Combining their votes (classification) or averaging their outputs (regression) typically beats any single tree's accuracy on unseen data.

The tradeoff for that accuracy gain is the same interpretability random forests lose relative to a single decision tree. There is no longer one diagram you can draw and walk a stakeholder through — there are hundreds of trees, each voting independently, and no single one of them represents "the" model's logic. Random forests do provide feature importance scores (a ranking of which features most influenced predictions across the whole forest), which recovers some explainability, but nowhere near the row-level, plain-English explanation a single decision tree or a logistic regression coefficient can offer.

Go DeeperBagging vs boosting: random forest vs gradient-boosted trees

Random forests train every tree independently and in parallel — none of the trees know anything about how the others performed. This strategy, bagging, is excellent at reducing variance (the tendency to overfit to noise in the training data) but does nothing to correct a systematic bias shared by every tree.

Boosting takes the opposite strategy: trees are trained sequentially, and each new tree is trained specifically to correct the mistakes the previous trees collectively made, by focusing on the training rows the ensemble so far has gotten most wrong. Gradient-boosted trees — implemented in popular libraries like XGBoost, LightGBM, and CatBoost — apply this idea using the same gradient descent logic from the deep dive in section 8.1.2, but to minimize the ensemble's error tree by tree rather than weight by weight.

In practice, gradient-boosted trees frequently outperform random forests on structured, tabular data — they are the most consistently winning model family in tabular ML competitions on platforms like Kaggle, and are heavily used in production systems for exactly that reason. The cost is that boosted models are more sensitive to hyperparameter tuning (learning rate, tree depth, number of boosting rounds) and more prone to overfitting if those settings aren't carefully validated, compared to a random forest, which tends to perform reasonably well even with fairly default settings.

Rule of thumb for an interview answer: "Start with a random forest as a strong, fairly low-maintenance baseline. Move to gradient boosting (XGBoost/LightGBM) when you need to squeeze out additional accuracy and have the time to tune it properly." Both are still classical ML — section 8.5 covers when the added complexity of deep learning is actually justified instead of either.

8.1.6

Overfitting, Underfitting, and the Train/Test Split

Overfitting occurs when a model learns the training data too well — including its noise and random quirks — to the point that it performs excellently on the data it was trained on but poorly on new data it hasn't seen. An unconstrained decision tree memorizing every training row, described in section 8.1.4, is the textbook example. Underfitting is the opposite failure: a model too simple to capture the real pattern in the data, performing poorly on both training and new data alike — for example, forcing a straight line onto a relationship that is clearly curved.

The practical danger of overfitting is that it's invisible if you only ever measure a model's performance on the same data it was trained on. A model can report 99% accuracy on its training set while being barely better than a coin flip on real, unseen data — and the only way to catch this before it costs the business something is to evaluate the model on data it has never encountered during training.

This is the entire purpose of the train/test split: before training begins, a portion of the available data — commonly 20–30% — is set aside and never shown to the model during training. Once training is complete, the model's accuracy is measured only on this held-out test set, giving an honest estimate of how it will perform on genuinely new data. A common refinement splits the data three ways — train, validation, and test — using the validation set to tune the model's settings (like a decision tree's maximum depth) and reserving the test set purely as a final, untouched check, so that not even the tuning process can leak information from the data meant to simulate "the real world." An even more robust approach, k-fold cross-validation, repeats this split several times over different partitions of the data and averages the results, reducing the chance that one lucky or unlucky split gives a misleading performance estimate.

If a model's training accuracy and its test accuracy are nearly identical, that's a healthy sign. A large gap — 98% training accuracy, 61% test accuracy — is the single clearest signal of overfitting an interviewer can ask you to diagnose. The fix is rarely "get a fancier algorithm"; it's usually simplify the model (less depth, more regularization), get more training data, or fix a leaking feature.

That last possibility — a leaking feature — is where data engineering and data science responsibilities directly intersect, and it's worth being explicit about it here rather than treating it as a purely modeling concern. Target leakage happens when a feature in the training data is, accidentally, only available after the outcome you're trying to predict has already occurred — for example, including "total repair cost" as a feature when predicting "will this machine fail," when repair cost is only ever recorded once a failure has already happened. The model appears to perform brilliantly in testing, because that feature is an almost-perfect proxy for the label, and then collapses in production, because at actual prediction time — before the failure has happened — that feature simply isn't available yet. Catching this requires understanding exactly when, in the pipeline, each feature's value becomes known relative to the label's timestamp, which is precisely the kind of detail a Gold-layer feature pipeline's design needs to get right.

🏢Field Note
A Logistics Company  ·  96% Accuracy That Wasn't Real

A logistics company's data science team built a random forest to predict which shipments would arrive late, intended to trigger proactive customer notifications. Initial testing showed 96% accuracy — high enough that the team scheduled a production rollout within two weeks, well ahead of their original timeline.

During a pre-launch review, a senior data engineer asked a simple question about one of the features: "actual_delivery_duration_hours" — when, exactly, is that value populated in the source system? The answer was uncomfortable: it was written by the warehouse system only once a shipment had actually been delivered — meaning, for every single training row, the model had effectively been given a feature that directly encoded almost the entire answer to the question it was supposedly predicting in advance.

The random/test split, drawn the normal way (a random sample of historical rows), hadn't caught this, because in every historical row — train and test alike — that feature was already populated by the time the data was pulled. The leak only became visible once someone checked the feature's actual availability timeline against the label's, rather than just trusting the held-out accuracy number. The fix was a strict temporal split instead of a random one: train on shipments before a cutoff date, test only on shipments after it, using only features that would genuinely have been known at the moment of prediction. Real accuracy on the corrected setup: 71% — still useful, but a very different number than 96%.

The Lesson

A held-out test set protects against overfitting to noise. It does not, by itself, protect against target leakage — a feature that is technically present in every historical row but wouldn't actually exist yet at real prediction time. Auditing exactly when each feature becomes known, relative to the label, is a data engineering responsibility that no train/test split can substitute for.

✓ Section Summary

Supervised learning splits into two families based purely on the shape of the target variable: regression for continuous numbers, classification for categories. Linear regression fits an interpretable straight line and minimizes squared error, typically via gradient descent. Logistic regression — despite its name — is a classification algorithm, passing a linear combination of features through the sigmoid function to output a probability. Decision trees split data into progressively purer groups via simple yes/no questions, offering excellent interpretability but a high risk of overfitting if unconstrained. Random forests bag together many trees trained on random row and feature subsets, trading away single-tree interpretability for substantially better accuracy and lower overfitting risk. Overfitting — a model that has memorized training data rather than learned a generalizable pattern — is caught using a held-out test set or cross-validation; target leakage, where a feature inadvertently encodes information only available after the outcome occurs, is a distinct failure that a clean train/test split alone will not catch and is squarely a data engineering concern.

Question 2 of 5

Why is logistic regression classified as a classification algorithm rather than a regression algorithm, despite the name?

A
It is actually a regression algorithm; "logistic regression for classification" is a common misconception
B
It uses decision trees internally instead of a weighted sum of features, which makes it a classifier
C
It computes a weighted sum of features like linear regression, but passes that sum through the sigmoid function to produce a probability between 0 and 1, which is then converted into a discrete class label via a threshold
D
It only works when the target variable has exactly two possible numeric values, such as 0 and 1, which makes it technically a form of regression on a restricted numeric range

C is correct. Logistic regression shares its linear core with linear regression — a weighted sum of input features — but the sigmoid function applied afterward squashes that sum into a 0–1 probability, and a threshold then converts the probability into a final class prediction. This is a classification pipeline end to end: the model's output is used to assign a category, not to estimate a continuous quantity. The "regression" in the name is a historical holdover from the statistical literature describing how the model is fitted, not a description of the type of prediction it produces.

Question 3 of 5

A decision tree, trained with no depth limit, achieves 99.8% accuracy on its training data but only 64% accuracy on a held-out test set. What is happening, and how would a random forest address it?

A
The tree is underfitting; a random forest would address this by training an even deeper single tree
B
The tree has overfit — it has grown deep enough to essentially memorize the training data's noise and specific quirks rather than learning a generalizable pattern. A random forest addresses this by bagging many trees, each trained on a random subset of rows and features, so each tree's idiosyncratic overfitting tends to cancel out when their predictions are combined
C
The test set is corrupted, since a properly trained model should perform identically on both training and test data
D
This is expected behaviour for decision trees and does not indicate a problem with the model

B is correct. The large gap between training accuracy (99.8%) and test accuracy (64%) is the textbook signature of overfitting — the model has learned the training data too specifically, including noise that doesn't generalize. An unconstrained decision tree is especially prone to this because it can keep splitting until every leaf is a single training row. A random forest doesn't eliminate any individual tree's tendency to overfit, but because each tree overfits to a different random subset of rows and features, those overfitting errors are largely uncorrelated across trees — averaging or voting across hundreds of such trees cancels out much of that noise while preserving the genuine shared signal.

Question 4 of 5

A bank must, by regulation, provide applicants with a specific, human-readable reason whenever a loan application is denied. Which model is the more appropriate choice for this use case, and why?

A
A random forest, because it generally achieves the highest accuracy of the algorithms covered in this section
B
Logistic regression (or a single, depth-constrained decision tree), because its weights or splits translate directly into a specific, explainable reason for the decision — a regulatory requirement that a forest's hundreds of voting trees cannot satisfy as cleanly
C
Either model works equally well, since regulatory explainability requirements apply only to deep learning models, not classical ML
D
A random forest, because feature importance scores fully satisfy any regulatory explainability requirement

B is correct. This is the interpretability-versus-performance tradeoff in its most consequential real-world form. A regulatory requirement to explain a specific denial to a specific applicant needs a row-level explanation — "your application was denied primarily because of X" — which logistic regression's coefficients (or a shallow decision tree's specific path of splits) can produce directly. A random forest's feature importance scores describe which features mattered across the whole model on average; they do not cleanly explain any single individual prediction, which is what the regulation actually requires. Choosing the highest-accuracy model without checking whether its explanations satisfy the actual regulatory or business requirement is a common and costly mistake.

Question 5 of 5

A model to predict equipment failure reports 97% test accuracy using a properly randomized train/test split, but performs far worse once deployed in production. An investigation finds one feature, "total_downtime_hours," is only populated by the source system after a failure has already occurred. What is the correct diagnosis and fix?

A
The model is overfitting; the fix is to limit the random forest's tree depth further
B
The test set was too small; the fix is to collect more historical rows and re-run the same random split
C
This is target leakage — the feature encodes information that wouldn't actually be available at real prediction time. A random train/test split does not catch this because the feature is present in every historical row regardless of split. The fix is to remove the leaking feature and verify, for every remaining feature, exactly when it becomes available relative to the label's timestamp — ideally using a temporal (not random) train/test split going forward
D
The model needs to be retrained as a gradient-boosted tree instead of a random forest, since boosting is more robust to this type of data issue

C is correct. A high test accuracy that collapses in production, combined with a feature that is only populated after the outcome has already happened, is the defining pattern of target leakage. Crucially, a standard random train/test split does not catch this kind of leak, because the problematic feature is populated identically in both the training rows and the randomly held-out test rows — the leak is invisible to any metric computed on historical data alone. No choice of algorithm (A, D) fixes a leaking feature, and more data (B) would simply make the same flawed signal more confidently learned. The only real fix is identifying and removing features that wouldn't genuinely be known at prediction time, which requires understanding the data's actual availability timeline — a data engineering responsibility, not a modeling one.

The Only Question That Matters First

Continuous number → regression. Category → classification. Decide this before anything else — it determines which entire family of algorithms is even on the table.

Interpretability vs Accuracy

Linear/logistic regression and single decision trees: explainable, lower ceiling on accuracy. Random forests and gradient boosting: higher accuracy, much harder to explain a single prediction. Pick based on the actual requirement, not just the leaderboard score.

Overfitting's Tell

Training accuracy far above test accuracy is the signature. The fix is rarely a fancier algorithm — usually it's less model complexity, more data, or a leaking feature to track down.

"How would you choose between logistic regression and a random forest for this problem?" tests whether you reach for accuracy by default or actually weigh interpretability, regulatory needs, and training cost.
