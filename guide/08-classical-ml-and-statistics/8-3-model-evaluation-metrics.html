8.3

Model Evaluation Metrics

Why "accuracy" is the metric most likely to mislead you, the confusion matrix that every other classification metric is built from, and how to choose the right number for a problem where false positives and false negatives don't cost the same

  1. Explain why accuracy is a misleading metric on imbalanced classification problems, and construct a concrete example showing why
  2. Read a confusion matrix and define true positive, false positive, true negative, and false negative in terms of a specific business scenario
  3. Calculate precision and recall from a confusion matrix, and explain the real-world tradeoff between them
  4. Explain what the F1 score balances, and when reporting it is more useful than reporting precision and recall separately
  5. Explain what an ROC curve plots and what the AUC summarizes, including why AUC alone can be misleading on highly imbalanced data

8.3.1

Why Accuracy Lies on Imbalanced Data

Accuracy — the percentage of predictions a model got right — is the first metric almost everyone reaches for, and on many problems it's perfectly reasonable. It becomes actively misleading the moment the classes you're predicting aren't roughly balanced, which describes most of the highest-stakes classification problems in practice: fraud, churn, disease screening, and equipment failure all share the property that the thing you actually care about predicting is rare.

Take a concrete number. Suppose 2% of transactions in a dataset are fraudulent and 98% are legitimate. A model that has learned absolutely nothing — one that simply predicts "not fraud" for every single transaction, regardless of any input feature — achieves 98% accuracy. That number sounds excellent on a slide. It is also completely useless: the model catches zero fraud, every single time, by construction. Any genuinely useful fraud model that flags real fraudulent transactions will, in the process, inevitably also flag some legitimate transactions by mistake — and that tradeoff will very often produce an accuracy number that's mathematically lower than the "always predict not fraud" baseline, despite being the only one of the two models providing any real value.

On an imbalanced dataset, the question "what accuracy did the model achieve?" is the wrong question to lead with. The right first question is: "what's the class balance, and what does a model that predicts nothing useful score on accuracy alone?" If that baseline number is already high, accuracy cannot be trusted as the primary metric — full stop.

This is precisely the gap section 8.1.3 flagged in passing when introducing logistic regression for fraud detection. The rest of this section builds the toolkit that replaces accuracy with metrics that remain meaningful even when one class vastly outnumbers the other.

8.3.2

The Confusion Matrix

Every classification metric in this section is built from the same underlying structure: the confusion matrix, a simple table comparing what the model predicted against what actually happened, broken into four distinct outcomes. For a binary problem (say, predicting fraud), each prediction lands in exactly one of four cells:

Figure 8.5The four confusion matrix outcomes for a binary fraud classifier. Precision asks, of everything flagged as fraud, how much actually was fraud (TP over TP+FP — the top row). Recall, covered next, asks of everything that actually was fraud, how much got caught (TP over TP+FN — the left column).

The reason this table matters more than a single accuracy number is that it separates two fundamentally different kinds of mistake — a false positive and a false negative — that very rarely cost the same amount in the real world. A false positive on a fraud model blocks a legitimate customer's transaction, causing friction and possibly a support call. A false negative lets actual fraud through entirely, with a direct financial loss. Those two outcomes are not equally bad, and no single accuracy number tells you which kind of mistake a model is actually making more of.

8.3.3

Precision and Recall

Precision answers: of everything the model flagged as positive, how much actually was positive? Precision = TP / (TP + FP). A fraud model with high precision rarely cries wolf — when it flags a transaction, it's very likely to genuinely be fraud.

Recall answers a different question: of everything that actually was positive, how much did the model catch? Recall = TP / (TP + FN). A fraud model with high recall misses very little real fraud — though it may also flag a fair number of legitimate transactions along the way.

These two numbers trade against each other, and the classification threshold — the cutoff probability, mentioned in section 8.1.3, above which a prediction counts as "positive" — is the dial that controls where on that tradeoff a model sits. Lower the threshold, and the model flags more things as positive: recall goes up (catching more real positives) but precision typically falls (also catching more false alarms along the way). Raise the threshold, and the reverse happens. Neither direction is universally correct — which one a given business should prefer depends entirely on which type of mistake is more expensive for that specific problem.

Recall-prioritized problems are ones where missing a true positive is the costly mistake: cancer screening (missing an actual case is far worse than an unnecessary follow-up test), fraud detection at most banks (a missed large fraud case can cost more than the friction of a few blocked legitimate transactions), and critical equipment failure prediction (an unplanned shutdown is far more expensive than an unnecessary inspection). Precision-prioritized problems are ones where a false alarm carries the real cost: spam filtering (a legitimate, important email wrongly sent to spam is often worse than the occasional spam message reaching the inbox), and content moderation systems where wrongly flagging a legitimate user's content erodes trust faster than the harm of a small amount of unflagged borderline content.

Go DeeperWhy a 99% accurate fraud model can be worthless

Let's make section 8.3.1's point fully concrete with numbers. Take 100,000 transactions, of which 1,000 (1%) are genuinely fraudulent. A model that achieves "99% accuracy" sounds excellent — but 99% accuracy is also exactly what you'd get by predicting "not fraud" for every single transaction, since 99,000 of the 100,000 transactions are legitimate. That baseline model has a recall of 0% — it catches not a single fraud case — while reporting an accuracy figure that looks nearly perfect on a dashboard.

Now suppose a real model, trained with actual signal, catches 600 of the 1,000 fraud cases (recall = 60%) while also incorrectly flagging 2,000 of the 99,000 legitimate transactions (precision = 600 / (600+2000) = 23%). Its overall accuracy is actually lower than the do-nothing baseline — roughly 97.4% versus 99% — purely because it makes 2,000 false-positive mistakes the baseline never makes by never flagging anything at all. By accuracy alone, the genuinely useful model looks worse than the useless one.

This is exactly why fraud, churn, and similar imbalanced classification problems are reported using precision, recall, and F1 in production dashboards, and accuracy is rarely, if ever, the headline number a data team presents to stakeholders for this kind of problem. If you ever see a single accuracy figure presented as the primary success metric for a rare-event classification problem, the first follow-up question worth asking is what the class balance is — and what a do-nothing baseline would score.

🏢Field Note
A Health Insurance Provider  ·  The Metric That Matched the Mistake's Cost

A health insurance provider built a model to flag claims likely to be fraudulent before payout, and the initial version was tuned and evaluated using accuracy, which came in at a reassuring 96%. In production, the model was quietly letting through a meaningful share of large fraudulent claims while also flagging a number of entirely legitimate claims from elderly policyholders for manual review — adding weeks of delay to claims that were, in the end, completely valid.

A review reframed the problem around the actual cost of each type of mistake. A missed fraudulent claim cost the company, on average, far more than the cost of one additional manual reviewer-hour spent clearing a flagged legitimate claim. That made this explicitly a recall-prioritized problem: missing real fraud was the expensive mistake, not the operational cost of extra review.

The team lowered the classification threshold to deliberately trade some precision for higher recall, accepting a meaningfully higher false-positive rate in exchange for catching a much larger share of true fraud. Reported accuracy actually dropped slightly as a result — exactly as the earlier deep dive predicts — but estimated annual fraud losses fell by a double-digit percentage, and the review team's added workload from extra false positives was modest in comparison. The new headline metric in monthly reporting became recall at a fixed, deliberately chosen precision floor, not accuracy.

The Lesson

The right metric is the one that mirrors the real cost of each type of mistake to the business — not the one that produces the most reassuring-looking number. Choosing precision versus recall as the optimization target is a business decision about which error is more expensive, made explicit, not a default left to whichever metric a model happened to report well on first.

8.3.4

F1 Score: Balancing Precision and Recall

Reporting precision and recall as two separate numbers is the most informative option, but sometimes a single summary number is genuinely useful — for comparing many candidate models at a glance, or for a leaderboard where two numbers are awkward to rank by. The F1 score is the standard answer: the harmonic mean of precision and recall.

F1 = 2 × (Precision × Recall) / (Precision + Recall)

The harmonic mean, rather than a simple average, matters here: it penalizes a large imbalance between precision and recall much more heavily than a regular average would. A model with 90% precision and 10% recall has a simple average of 50%, which sounds moderate — but its F1 score is only about 18%, correctly reflecting that a model catching just 10% of real positives is performing badly overall, regardless of how trustworthy its few flagged cases are. F1 effectively refuses to give a high score to a model that's lopsided in either direction.

The honest limitation of F1 is exactly what makes it convenient: it weights precision and recall equally, which is rarely true of the actual business cost structure, as the health insurance field note above illustrates plainly — that problem was not a 50/50 tradeoff. An F-beta score generalizes F1 by introducing a weighting parameter that lets recall count more or less heavily than precision, but in practice, many teams find it clearer to simply report precision and recall directly alongside the specific threshold chosen, rather than collapsing the decision into a single weighted number that obscures exactly what tradeoff was made.

8.3.5

ROC Curves and AUC

Precision, recall, and F1 are all calculated at one specific classification threshold. The ROC curve (Receiver Operating Characteristic — a name with origins in WWII-era radar signal detection, now used purely as a fixed term of art) takes a different view: it plots how a model's true positive rate (equivalent to recall) and false positive rate trade off against each other across every possible threshold simultaneously, producing a single curve rather than one number.

Figure 8.6The ROC curve plots true positive rate against false positive rate across every classification threshold at once. The diagonal represents a model with no predictive power. AUC (Area Under the Curve) condenses the entire curve into one number summarizing performance across all thresholds simultaneously — useful for comparing models without committing to a single operating threshold up front.

The single number derived from this curve, AUC (Area Under the Curve), summarizes the model's discriminative power across every threshold in one value: 1.0 is a perfect classifier, 0.5 is exactly equivalent to flipping a coin, and anything meaningfully below 0.5 almost always indicates a labeling bug (such as the positive and negative classes being accidentally swapped) rather than a model that has somehow learned to be reliably wrong. AUC's appeal is that it lets you compare two candidate models without having to first commit to a specific operating threshold — useful early in model development, before a business has decided exactly where on the precision/recall tradeoff it wants to operate.

AUC's most important caveat: on a heavily imbalanced dataset, the false positive rate's denominator (all actual negatives) is so large that AUC can look deceptively high even when precision — which divides by the much smaller number of predicted positives — is poor. For severely imbalanced problems like fraud detection, many practitioners prefer the closely related Precision-Recall AUC, which plots precision against recall directly instead of false positive rate, and is generally considered the more honest summary metric when positives are rare.

+
Go DeeperTuning the classification threshold: moving beyond the default 0.5

Most ML libraries default a binary classifier's threshold to 0.5 — predict positive if the model's estimated probability exceeds 50%. This default is a coincidence of convenience, not a business decision, and treating it as fixed is one of the more common ways teams ship a poorly calibrated model without realizing it.

The practical process: after training, compute precision and recall at a range of candidate thresholds — say, every value from 0.1 to 0.9 — and plot how they trade off against each other (a precision-recall curve, the threshold-sweep companion to the ROC curve above). From that curve, pick the threshold that achieves the business's actual target — for example, "the highest recall achievable while keeping precision above 80%," if false positives have a known, bounded operational cost the business is willing to tolerate up to that point.

A concrete illustration: the health insurance field note above is exactly this process in action. The team didn't retrain the model at all — they kept the same trained classifier and moved its threshold lower, deliberately trading precision for recall because the cost analysis showed that was the right direction to move for their specific problem. Threshold tuning is often a faster, cheaper lever to pull than retraining an entirely new model, and it's worth checking before assuming a model needs to be rebuilt from scratch to hit a different precision/recall target.

This is also why reporting a model's AUC alone, without a chosen operating threshold, is incomplete for a production decision — AUC describes the curve, but a live system has to commit to one specific point on it.

8.3.6

Regression Metrics: RMSE, MAE, and R²

Everything covered so far in this section applies to classification. Regression problems — predicting a continuous number, from section 8.1 — need a different set of metrics, since there's no confusion matrix when the prediction is a number rather than a category.

RMSE (Root Mean Squared Error) takes the same squared-error logic linear regression uses internally to fit its line (section 8.1.2) and applies it as an evaluation metric: square every prediction error, average them, then take the square root to bring the result back into the original units of the target variable. Because the errors are squared before averaging, RMSE penalizes large individual errors disproportionately more than small ones — a model that's wildly wrong on a handful of rows will be punished by RMSE far more than a model that's mildly wrong everywhere, even if both models have a similar average error in absolute terms.

MAE (Mean Absolute Error) instead averages the plain absolute value of each error, with no squaring. This makes MAE more robust to outliers than RMSE — a single wildly-off prediction moves MAE only modestly, while the same prediction can dominate RMSE entirely. The right choice between the two depends on whether large individual errors are disproportionately costly for the specific business problem (favoring RMSE, which punishes them more) or whether a more typical, representative sense of "how far off is the model usually" is what matters (favoring MAE).

R² (R-squared, the coefficient of determination) answers a different question entirely: what proportion of the variation in the target variable does the model explain, on a scale from 0 (the model explains nothing — it does no better than just always predicting the average) to 1 (the model explains all the variation perfectly). R² is popular specifically because it's scale-free and immediately interpretable as a percentage, unlike RMSE and MAE, which are expressed in the same units as the target variable and therefore aren't directly comparable across different problems — an RMSE of "200" means something completely different when predicting house prices in dollars versus predicting delivery times in minutes.

+
Go DeeperRMSE vs MAE: why squaring the error changes what "good" means

Consider a demand forecasting model evaluated on five days, with absolute errors (in units) of 2, 3, 2, 4, and 40 — four days of solid, ordinary performance and one day with a major miss, perhaps due to an unflagged promotional spike the model didn't see in its features.

MAE = (2+3+2+4+40) / 5 = 10.2 units. The one bad day pulls the average up noticeably, but the other four days still meaningfully shape the result.

RMSE = the square root of the mean of (4 + 9 + 4 + 16 + 1600) / 5 = the square root of 326.6 ≈ 18.1 units. The single 40-unit miss, once squared to 1,600, almost completely dominates the calculation — RMSE here is nearly double MAE, driven almost entirely by one outlier day.

Neither number is "wrong" — they're answering different questions. If a 40-unit miss on one day causes a genuinely disproportionate business problem (a stockout that damages a major customer relationship, say), RMSE's heavier penalty on large errors is the more honest reflection of what actually matters. If that one bad day was a known, explainable, one-off anomaly and what really matters is "how good is this model on a typical day," MAE gives a less outlier-distorted picture. Reporting both, rather than picking one in isolation, is common practice precisely because the gap between them is itself informative — a large RMSE/MAE gap signals that a small number of large errors, rather than consistent everyday error, are driving overall performance.

🏢Field Note
An E-Commerce Platform  ·  The AUC That Hid a Precision Problem

An e-commerce platform's trust and safety team built a model to flag fraudulent seller listings, and reported a strong AUC-ROC of 0.91 to leadership as evidence the model was production-ready. On a dataset where fraudulent listings made up under 0.5% of all listings, the team moved forward with launch based primarily on that one number.

Within the first week, the manual review queue — every listing the model flagged as fraudulent required human review before removal — was overwhelmed. The model's precision at the threshold actually deployed in production turned out to be just 12%: for every genuinely fraudulent listing it correctly caught, it also flagged roughly seven legitimate sellers' listings for review. The AUC of 0.91 was real and not miscalculated — the model did have strong overall discriminative power across thresholds — but on a dataset this imbalanced, the false positive rate's denominator (the enormous pool of legitimate listings) was so large that even a fairly small false-positive rate translated into an overwhelming raw number of false positives once converted to actual flagged listings.

The fix was twofold: switching the primary reported metric to Precision-Recall AUC, which is far less forgiving on this kind of imbalance and immediately revealed the precision problem that the standard AUC-ROC had obscured, and then re-tuning the threshold specifically to hit a precision target the review team could sustainably handle, accepting a lower recall in exchange.

The Lesson

AUC-ROC can be a genuinely strong, accurate number while still hiding a serious precision problem on highly imbalanced data — because its false positive rate is computed against a large denominator of negatives that swamps a real precision issue. Precision-Recall AUC, or simply checking precision and recall directly at the actual deployed threshold, is the more honest check before shipping on a rare-event problem.

✓ Section Summary

Accuracy is unreliable on imbalanced classification problems, since a model that predicts nothing useful can still score deceptively well. The confusion matrix's four outcomes — true positive, false positive, true negative, false negative — underpin every classification metric that follows: precision (of what was flagged positive, how much was right) and recall (of what was actually positive, how much was caught) trade off against each other via the classification threshold, and which one to prioritize depends entirely on which type of error costs more in a given business context. F1 balances both into one number, assuming they matter equally — which is often not true in practice. AUC-ROC summarizes performance across every threshold at once but can be misleadingly high on severely imbalanced data, where Precision-Recall AUC is the more honest alternative. Regression problems use a different toolkit entirely: RMSE penalizes large errors disproportionately, MAE is more robust to outliers, and R² gives a scale-free sense of how much variance the model explains.

A
It achieves 99% accuracy despite having zero recall and providing no value at all, demonstrating that accuracy is unreliable on imbalanced data because the majority class dominates the score regardless of whether the model does anything useful for the minority class

A is correct. Accuracy simply counts the percentage of all predictions that were correct. Since 99% of transactions are legitimate and the model predicts "not fraud" for all of them, it is correct on all 99,000 of the 100,000 transactions in this example — a 99% accuracy score — while catching exactly zero of the actual fraud cases. Question 3 of 5

A model has 90% precision and 10% recall. Why is its F1 score (approximately 18%) much closer to the lower of the two numbers rather than their simple average (50%)?

A
This indicates a calculation error, since F1 should always equal the simple average of precision and recall
B
F1 is the harmonic mean, not the simple average, of precision and recall, and the harmonic mean penalizes a large imbalance between the two numbers far more heavily — correctly reflecting that a model catching only 10% of real positives is performing poorly overall, regardless of how trustworthy its few correct flags are
C
F1 always equals whichever of precision or recall is lower, ignoring the higher value entirely
D
F1 only applies to regression problems, so this calculation is not meaningful for a classification model

B is correct. F1 = 2 × (Precision × Recall) / (Precision + Recall) is the harmonic mean, which mathematically penalizes imbalance between the two inputs much more severely than a simple arithmetic average would. A model with 90% precision but only 10% recall is, in practical terms, missing 90% of the real positives — a serious shortfall that F1's low score of roughly 18% correctly reflects, in contrast to a simple average of 50% that would make the model look far more balanced and useful than it actually is.

Question 4 of 5

A fraud detection model on a severely imbalanced dataset (0.5% fraud) reports a strong AUC-ROC of 0.91, but its precision at the deployed threshold turns out to be only 12%. How can both of these be true at once?

A
They cannot both be true; an AUC of 0.91 mathematically guarantees precision above 80% at any threshold
B
AUC-ROC is computed using the false positive rate, whose denominator is the very large number of actual negatives. On a severely imbalanced dataset, even a fairly low false positive rate can still produce a large raw number of false positives relative to the small number of true positives, which is exactly what drags precision down — a problem the false-positive-rate-based ROC curve does not directly expose
C
The AUC must have been calculated incorrectly, since a high AUC and low precision are mutually exclusive by definition
D
Precision and AUC measure unrelated, independent aspects of completely different models and cannot be meaningfully compared

B is correct. This is exactly the e-commerce field note's scenario. AUC-ROC is built from the false positive rate (FP divided by all actual negatives), and on a dataset where negatives vastly outnumber positives, that denominator is enormous — so even a quite low false positive rate can still mean a raw false-positive count that's many times larger than the true-positive count, which is precisely what drives precision down. AUC-ROC genuinely measured strong overall discriminative ability across thresholds; it simply isn't the metric that exposes a precision problem on severely imbalanced data. Precision-Recall AUC is the standard fix for this blind spot.

Question 5 of 5

A demand forecasting model has near-identical RMSE and MAE on one dataset, but a much larger RMSE than MAE on a second dataset. What does the second dataset's gap most likely indicate?

A
The second model is performing better overall, since RMSE is always a more favorable metric than MAE
B
The second dataset likely contains a small number of unusually large prediction errors. Because RMSE squares each error before averaging, large individual misses are disproportionately amplified, while MAE — which uses absolute, unsquared error — stays comparatively unaffected; a wide RMSE/MAE gap is a signal that a few outlier errors, not consistent everyday error, are driving overall performance
C
RMSE and MAE should always be equal for any well-performing model, so this gap indicates a bug in the evaluation code
D
The second dataset must have more rows than the first, since RMSE increases with dataset size while MAE does not

B is correct. RMSE squares each individual error before averaging, which means a small number of unusually large errors get amplified disproportionately in the final number, while MAE — using unsquared absolute error — treats every error proportionally to its actual size. A large gap between the two metrics on the same dataset is itself a useful diagnostic: it tells you that overall performance is being driven by a handful of large misses rather than broadly consistent, everyday error, which is exactly the pattern shown in this section's RMSE vs MAE deep dive.

Check the Baseline First

Before trusting an accuracy number on a classification problem, ask what "always predict the majority class" would score. If that's already high, accuracy cannot be the primary metric.

Precision vs Recall, In One Line

Precision: of what I flagged, how much was right? Recall: of what was real, how much did I catch? The threshold is the dial that trades one for the other.

AUC Can Hide a Precision Problem

On severely imbalanced data, a strong AUC-ROC can coexist with poor real-world precision. Check precision and recall at the actual deployed threshold before shipping — don't stop at AUC alone.

"Walk me through how you'd choose an evaluation metric for this problem" is testing whether you start from the cost of false positives versus false negatives, not whether you can recite formulas. Naming the business tradeoff first is the senior-level answer.
