One word, three completely different concepts — and why confusing them causes real problems

2.2.1

Three Things Called "Partition"

No word in data engineering causes more confusion than "partition." It is used freely in conversations, documentation, and job descriptions — and it means three completely different things depending on context. A senior data engineer discussing partitions in a storage context means something entirely different from one discussing partitions in a Spark context, and both mean something different from a DBA discussing SQL table partitions. All three concepts are important. Conflating them leads to misdiagnosed performance problems and mis-designed architectures.

Storage partitioning is how files are organised in a data lake folder hierarchy. SQL table partitioning is a database-level feature that splits a table into physical segments for query performance. Spark compute partitioning is how Spark distributes data across executor cores for parallel processing. The word is the same. The mechanism, the purpose, and the optimisation levers are entirely different in each case.
2.2.2

Storage Partitioning: Organising Your Data Lake

When you write data to a data lake (ADLS, S3, GCS), you can organise files in a folder hierarchy that reflects a key column's values. A table partitioned by date might be stored as:

/sales/
  year=2024/
    month=01/
      day=01/  ├ part-00001.parquet
               └ part-00002.parquet
      day=02/  └ part-00001.parquet
    month=02/
      day=01/  └ part-00001.parquet

The benefit is partition pruning: when a query includes a filter on a partition column, the engine reads only the relevant folders. A query with WHERE year = 2024 AND month = 01 reads only two files instead of the entire dataset. For large tables, this can reduce data scanned from terabytes to gigabytes — directly translating to faster queries and lower Databricks compute cost.

In Delta Lake, you declare the partition column at table creation:

Spark SQL (Databricks)  ·  Create a partitioned Delta table
CREATE TABLE silver.sales_transactions (
    transaction_id   BIGINT,
    customer_id      BIGINT,
    amount           DECIMAL(12,2),
    status           STRING,
    transaction_date DATE
)
USING DELTA
PARTITIONED BY (transaction_date)
LOCATION 'abfss://silver@yourlake.dfs.core.windows.net/sales/';

-- When writing with PySpark, partition columns go in partitionBy()
df.write \
  .format("delta") \
  .mode("append") \
  .partitionBy("transaction_date") \
  .save("abfss://silver@yourlake.dfs.core.windows.net/sales/")

Partition pruning only fires when the filter is on the exact partition column. A query filtering on MONTH(transaction_date) = 1 does not trigger pruning because the engine cannot determine which partition folders satisfy a derived function without reading them all. This is a very common performance mistake: engineers assume a date function filter will prune partitions, and it does not.

The Small Files Problem

The most expensive partition mistake is partitioning by a column with high cardinality or by too many columns simultaneously. If you partition a sales table by customer_id and you have two million customers, you create two million partition folders — each potentially containing tiny files. Reading this table requires opening two million directory listings before a single data file is touched. Query planning time alone can exceed query execution time.

🏢Field Note
A SaaS Analytics Platform  ·  Over-Partitioning

A SaaS company building an analytics platform for their customers partitioned their event table by customer_id, reasoning that customers would always query their own data and partition pruning would make those queries fast. With 850,000 customer accounts, this created 850,000 partition directories. Each directory averaged three files of around 200KB.

Queries that filtered by a single customer_id were fast. Queries that aggregated across all customers — needed for platform-level reporting — were catastrophically slow. The query planner had to resolve 850,000 directory paths before reading any data. A full-table scan that should have taken 40 seconds took 18 minutes. The fix was to remove the customer_id partition and replace it with a date partition (which was the actual query pattern for most reports) plus Liquid Clustering on customer_id (covered in Section 2.2.5). Platform-wide queries dropped from 18 minutes to 55 seconds.

The Lesson

Partition columns should have low-to-moderate cardinality and should match your most common filter pattern. Date (year/month) is the canonical correct choice for most transactional data. Customer IDs, product IDs, and user IDs almost never make good partition columns.

Figure 2.3 Partition pruning in action. An unpartitioned table requires scanning all 12 files to find records from a specific date. A table partitioned by date lets the engine skip directly to the relevant folder and read only the one matching file. The performance difference scales with table size — on petabyte-scale tables, partition pruning can reduce data scanned by 99%.
2.2.3

SQL Table Partitioning: The Database-Level Feature

SQL table partitioning (available in SQL Server, Azure SQL, PostgreSQL, and others) is a relational database feature where a single logical table is physically split into multiple storage segments called partitions. Unlike storage partitioning in a data lake, SQL table partitioning is invisible to the query writer — queries use the table name, the database engine decides which partitions to read.

The typical use case is a large transaction table partitioned by month. Each month is a separate physical segment. Rolling off old data is a metadata operation (switch the old partition out) rather than a row-by-row DELETE. Maintenance operations like index rebuilds can target one partition at a time. For very large SQL Server tables, this is a genuine performance and manageability win.

In Databricks and modern lakehouses, SQL table partitioning as a distinct concept is less relevant because the storage partitioning and Delta Lake together achieve the same query optimisation goal. The distinction matters most when you are working with SQL Server, Azure SQL Managed Instance, or Azure Synapse SQL pools.

2.2.4

Spark Compute Partitioning: Parallelism Under the Hood

When Spark reads a dataset, it divides it into chunks called partitions and assigns each chunk to an executor core for processing. This is Spark compute partitioning — it has nothing to do with the physical file organisation in the lake, even though the term is the same. A Spark job on a dataset with 200 compute partitions runs 200 tasks in parallel (up to the number of available cores). The split happens in memory; no files are reorganised.

The default number of Spark shuffle partitions (created during operations like GROUP BY, JOIN, and DISTINCT that require redistributing data across executors) is controlled by spark.sql.shuffle.partitions. The default value is 200. For small datasets, 200 partitions means 200 tiny tasks — the overhead of launching each task exceeds the work it does. For very large datasets, 200 partitions may be too few, leaving cores idle while large tasks process slowly.

PySpark  ·  Tuning compute partitions
# Check current partition count on a DataFrame
print(df.rdd.getNumPartitions())   # e.g. 200

# Increase partitions before a heavy shuffle (large dataset)
df_repartitioned = df.repartition(800)

# Decrease partitions after filtering (avoids many tiny tasks)
df_small = df.filter(df.region == "UK").coalesce(10)
# coalesce() reduces without a full shuffle (cheaper than repartition)
# repartition() does a full shuffle — use when you also want to redistribute data

# Set shuffle partitions for the whole session
spark.conf.set("spark.sql.shuffle.partitions", "400")

# Or let Adaptive Query Execution (AQE) decide automatically (Spark 3.0+)
spark.conf.set("spark.sql.adaptive.enabled", "true")
# AQE dynamically coalesces shuffle partitions at runtime
# This is now the default in Databricks Runtime 7.3+

Go DeeperData skew — when one partition ruins the whole job

A perfectly set partition count does not guarantee good performance if data is unevenly distributed. If one partition receives 80% of the data — a common result when partitioning by a column with a highly skewed distribution, like country when 90% of records are from one country — one executor task runs for minutes while all others finish in seconds. The job's total time is determined by the slowest task.

This is called data skew, and it is one of the most common causes of unexpectedly slow Spark jobs. Symptoms: the Spark UI shows tasks completing quickly except for one or two that drag on far longer than the rest.

Common fixes: salting (adding a random suffix to skewed keys before joining, then removing it after), using Adaptive Query Execution's skew join optimisation (enabled by default in Databricks 7.3+), or pre-aggregating the skewed data before the problematic join.

Broadcast joins are a separate but related optimisation: if one side of a join is small enough to fit in memory, Spark can broadcast it to every executor rather than shuffling the large table. Controlled with spark.sql.autoBroadcastJoinThreshold (default 10MB). For lookups against small dimension tables, broadcast joins can eliminate shuffle entirely.

+
Go Deeperrepartition() vs coalesce() — which reorganises data and which does not

repartition(n) performs a full shuffle: every record is redistributed across the new n partitions. Use it when you want to increase partition count, or when you want to redistribute data by a specific column (e.g. repartition(200, "customer_id") ensures all records for a customer land on the same partition before a join).

coalesce(n) reduces partition count without a full shuffle. It merges partitions on the same executor, avoiding the network transfer cost. Use it when you want to reduce partition count after filtering a large dataset down to a small result. The trade-off: coalesced partitions may be uneven in size because it avoids the equalising effect of a full shuffle.

The rule: use coalesce() to reduce, repartition() to increase or to redistribute by a key. Calling repartition() to reduce is wasteful but not incorrect.

+
Go DeeperChoosing the right partition column — a decision framework

Choosing a partition column is a one-time decision with long-term consequences. The right framework has three questions:

1. What column appears in WHERE clauses on 80%+ of queries? Partition pruning only benefits queries that filter on the partition column. If your analysts mostly filter by date, partition by date. If they filter by region, partition by region. A partition column that does not appear in common query filters provides no benefit and adds file overhead.

2. What is the cardinality of that column? Low-to-moderate cardinality is ideal. Partition by year/month gives 12–120 partitions over a decade — manageable. Partition by day gives ~3,650 — still fine. Partition by hour gives ~87,600 — getting problematic. Partition by individual customer ID (millions of values) — always wrong.

3. How will data be written? Partition columns that match your write pattern avoid rewriting across many partitions. If you write daily batches, partitioning by date means each write lands in one folder. If your writes span many dates (backfills, late-arriving data corrections), each write touches many partition folders, causing fragmentation and small files over time.

The default recommendation for most transactional tables: partition by year and month at the top two levels, not by date alone. Querying a full month reads one folder. Querying a year reads twelve. Querying a specific day reads one. This gives efficient pruning at multiple granularities without folder explosion.

2.2.5

Liquid Clustering: The Modern Databricks Alternative

Traditional storage partitioning has a design flaw: you must commit to one partition column at table creation. If your primary query pattern changes — or if users query by multiple different columns — the partition scheme becomes a liability. Re-partitioning an existing large table means rewriting every file: an expensive, disruptive operation.

Databricks introduced Liquid Clustering in Databricks Runtime 13.3 as a modern replacement. Instead of physically organising files into partition folders, Liquid Clustering uses Z-ordering and file-level statistics to cluster data by one or more columns while keeping the table in a flat, non-hierarchical layout. The clustered columns can be changed at any time without rewriting the data. The optimiser uses file statistics to skip irrelevant files at query time — achieving similar pruning benefits without the partition folder hierarchy.

SQL (Databricks)  ·  Liquid Clustering vs traditional partitioning
-- TRADITIONAL PARTITIONING (partition column fixed at creation)
CREATE TABLE silver.orders
USING DELTA
PARTITIONED BY (order_date)          -- committed forever
LOCATION 'abfss://...';

-- Cannot easily change the partition column later.
-- Querying by customer_id instead of order_date = full scan.

-- ─────────────────────────────────────────────────────────────

-- LIQUID CLUSTERING (Databricks 13.3+)
CREATE TABLE silver.orders_clustered
USING DELTA
CLUSTER BY (order_date, customer_id)  -- multiple columns supported
LOCATION 'abfss://...';

-- Change clustering columns anytime (no data rewrite required):
ALTER TABLE silver.orders_clustered
CLUSTER BY (customer_id, product_id);

-- Trigger clustering optimisation (runs incrementally):
OPTIMIZE silver.orders_clustered;
🏢Field Note
A Retail Analytics Team  ·  Re-partitioning Pain

A retail company's data team partitioned their 5TB transactions table by order_date when it was first built. This was the right call — 90% of queries at the time filtered by date range. Two years later, the business changed: the data science team needed to query by product_category, the finance team by store_id, and marketing by customer_segment. None of these benefited from date partitioning. Full-table scans ran daily against a 5TB table.

The team evaluated re-partitioning by a composite key. The rewrite would take 14 hours of Databricks compute and require a maintenance window. They migrated to Liquid Clustering instead. The initial OPTIMIZE run took 6 hours. Subsequent incremental OPTIMIZE runs (triggered after each daily load) took under 15 minutes. Queries filtering by product_category that previously scanned the entire 5TB table now scanned an average of 200GB, achieved by file-level statistics skipping 96% of files.

The Lesson

Liquid Clustering is the right default for new Databricks tables where query patterns may evolve, multiple filter columns are common, or you want to avoid committing to a partition column permanently. Traditional partitioning by date remains valid for append-only tables where the filter pattern is extremely stable.

✓ Section Summary

"Partition" means three different things. Storage partitioning organises lake files in folder hierarchies by a column value, enabling partition pruning that skips irrelevant files at query time — choose low-cardinality columns, avoid high-cardinality columns that cause the small files problem. SQL table partitioning is a relational database feature for manageability and query optimisation on very large RDBMS tables. Spark compute partitioning divides data across executor cores for parallel processing — tune with spark.sql.shuffle.partitions or let Adaptive Query Execution manage it automatically. In Databricks, Liquid Clustering is the modern alternative to storage partitioning: clustering on multiple columns without a folder hierarchy, changeable without rewriting data, and well-suited to tables with multiple query patterns.

A Databricks query filters with WHERE YEAR(transaction_date) = 2024. The table is partitioned by transaction_date. Does partition pruning fire?

A
Yes — the engine is smart enough to know YEAR(transaction_date) = 2024 maps to the year=2024 partition folder
B
No — partition pruning requires a direct filter on the exact partition column (e.g. WHERE transaction_date BETWEEN ...), not a derived function applied to it
C
It depends on whether the table uses Delta Lake or plain Parquet
D
Yes, but only if the partition is on a date type rather than a timestamp

B is correct. Partition pruning requires the engine to resolve which partition folders satisfy a filter without reading the data. A derived function like YEAR(transaction_date) cannot be resolved to specific folder paths without evaluating every row. The engine falls back to a full scan. To use partition pruning with date ranges, use WHERE transaction_date >= '2024-01-01' AND transaction_date < '2025-01-01' — the engine can resolve this to the year=2024 folders directly.

Question 3 of 5

A Spark job runs 200 shuffle tasks. In the Spark UI you notice 198 tasks complete in 30 seconds, but 2 tasks take 8 minutes each. What is the most likely cause?

A
200 is too low a partition count for this dataset — increase spark.sql.shuffle.partitions to 2000
B
The two slow tasks are running on underpowered executor nodes
C
Data skew — two partition keys contain a disproportionate share of the data, causing two tasks to process far more records than the others
D
The Spark shuffle spilled to disk on those two tasks due to insufficient executor memory

C is correct. The pattern — almost all tasks fast, two tasks dramatically slow — is the textbook signature of data skew. A small number of partition keys have an outsized share of the data; the tasks assigned to those keys process significantly more work than all others. Solutions include salting, Spark's AQE skew join optimisation, or pre-aggregating the skewed key before the join. Option D (memory spill) would typically manifest as many slow tasks, not just two.

Question 4 of 5

What is the key advantage of coalesce() over repartition() when reducing the number of Spark partitions?

A
coalesce() produces more evenly sized partitions than repartition()
B
coalesce() can increase partition count as well as decrease it
C
coalesce() avoids a full shuffle by merging partitions on the same executor, making it significantly cheaper than repartition() for reducing partition count
D
coalesce() is the only method that works on Delta Lake tables

C is correct. repartition() performs a full shuffle — every record crosses the network to be redistributed. coalesce() merges partitions that are already on the same executor node, avoiding network traffic. The trade-off is that coalesced partitions may be uneven in size. Use coalesce() when reducing after filtering, and repartition() when you also need to redistribute data by a key or increase partition count.

Question 5 of 5

You are creating a new Delta Lake table in Databricks. The business currently queries primarily by order_date, but the data science team expects to query by customer_segment and product_category within six months. Which approach gives the most future flexibility?

A
Partition by order_date and add ZORDER BY customer_segment, product_category in OPTIMIZE runs
B
Partition by a composite of all three columns to cover all query patterns
C
Create separate copies of the table partitioned differently for each team
D
Use Liquid Clustering on CLUSTER BY (order_date, customer_segment, product_category) — clustering columns can be changed at any time without rewriting data as query patterns evolve

D is correct. Liquid Clustering supports multiple cluster columns and, crucially, allows changing those columns with an ALTER TABLE ... CLUSTER BY statement without rewriting the data. Traditional partitioning commits you to one layout at creation. If query patterns evolve — which they almost always do — changing a traditional partition scheme requires a full table rewrite. Option A (ZORDER) on a traditionally-partitioned table is workable but still commits you to the partition column. Liquid Clustering is the right default for tables with evolving or multiple query patterns.

Three Different Things

Storage partition: folder hierarchy in ADLS. SQL partition: physical table segment in an RDBMS. Spark partition: data chunk on an executor core. Same word — completely different concepts.

Pruning Rule

Pruning only fires on a direct filter on the partition column — not on a function applied to it. WHERE date = '2024-01-01' prunes. WHERE YEAR(date) = 2024 does not.

Never Partition By

High-cardinality columns: user IDs, product IDs, order IDs. Millions of folders means millions of directory lookups before a single file is read. Use Liquid Clustering instead.

Spark Tuning Default

Enable Adaptive Query Execution (spark.sql.adaptive.enabled=true) for most workloads. AQE dynamically coalesces shuffle partitions and handles skew automatically. It is on by default in Databricks Runtime 7.3+.

Liquid Clustering

Databricks 13.3+. Replaces traditional partitioning. Multiple cluster columns. Changeable without data rewrite. Best for tables with evolving query patterns. Run OPTIMIZE after each load.

Interview Insight

"Why was your Spark job slow?" — first check the Spark UI for task duration distribution. All tasks similar = partition count problem. A few tasks much longer = data skew. Night-and-day different root causes.
