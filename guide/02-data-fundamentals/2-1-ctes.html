2.1

Common Table Expressions

Named, temporary, readable — how CTEs transform the way you structure complex queries

✓

Expected Learning Outcomes

When you have completed this section, you should be able to:

  1. Define a CTE and explain the problem it was designed to solve
  2. Write basic and chained CTEs in both T-SQL and Spark SQL syntax
  3. Build a recursive CTE to traverse an organisational hierarchy
  4. Choose between CTEs, subqueries, temp tables, and views based on the scenario
  5. Explain why CTEs are not always materialised and what that means for performance
2.1.1

The Problem: Nested Subquery Hell

Before CTEs existed, SQL developers solved complex multi-step query problems by nesting subqueries inside subqueries. The logic was correct. The code was unreadable. A query that ranked customers by spend, filtered to the top decile, joined to a region table, and then aggregated by product category might be five levels deep — each subquery only interpretable by reading from the inside out, with no names for intermediate results, and no easy way to debug which level was producing unexpected output.

A Common Table Expression (CTE) is a named temporary result set defined at the beginning of a query with the WITH keyword. It exists only for the duration of that single query. You give a name to each intermediate step, define it clearly before using it, and then reference it by name in the main query or in subsequent CTEs. The output is identical to an equivalent nested subquery. The difference is entirely in readability, maintainability, and the ability to debug one step at a time.

A CTE is not a performance optimisation. It is a readability and maintainability tool. The database engine may or may not execute a CTE differently from an equivalent subquery — that depends on the query optimiser. What CTEs reliably do is make complex SQL legible, testable in parts, and maintainable by someone who did not write it.
Figure 2.1 The same four-step query written as nested subqueries (left) versus CTEs (right). Both return identical results. The CTE version names each intermediate step, reads top-to-bottom, and lets you run any individual CTE in isolation to debug it. The subquery version requires reading inside-out with no named anchors.
2.1.2

Basic Syntax and Chaining CTEs

A CTE opens with the WITH keyword, followed by a name, the word AS, and the query in parentheses. The main query follows immediately after. Multiple CTEs are separated by commas — the WITH keyword appears only once, at the very beginning.

T-SQL  ·  Single CTE
WITH customer_totals AS (
    SELECT
        customer_id,
        COUNT(*)       AS order_count,
        SUM(amount)    AS total_spend
    FROM orders
    WHERE status = 'completed'
    GROUP BY customer_id
)
SELECT
    c.name,
    ct.order_count,
    ct.total_spend
FROM customers AS c
JOIN customer_totals AS ct ON c.id = ct.customer_id
WHERE ct.total_spend > 1000
ORDER BY ct.total_spend DESC;

Chaining multiple CTEs is where the real power emerges. Each CTE can reference any CTE defined above it. This lets you break a complex transformation into a sequence of named, readable steps — each one building on the previous.

T-SQL  ·  Chained CTEs (three steps)
WITH
-- Step 1: aggregate orders per customer
order_summary AS (
    SELECT customer_id,
           SUM(amount)  AS total_spend,
           COUNT(*)     AS order_count
    FROM   orders
    GROUP  BY customer_id
),
-- Step 2: rank customers by spend
ranked AS (
    SELECT customer_id,
           total_spend,
           order_count,
           RANK() OVER (ORDER BY total_spend DESC) AS spend_rank
    FROM   order_summary
),
-- Step 3: keep only top 10%
top_decile AS (
    SELECT *
    FROM   ranked
    WHERE  spend_rank <= (SELECT COUNT(*) * 0.1 FROM ranked)
)
-- Final query: join to customer names
SELECT c.name, c.region, t.total_spend, t.spend_rank
FROM   customers AS c
JOIN   top_decile AS t ON c.id = t.customer_id
ORDER  BY t.spend_rank;

Spark SQL uses identical CTE syntax. If you are writing Databricks notebooks, the WITH clause works exactly as in T-SQL — no translation needed. The one practical difference: in Databricks SQL cells, each cell runs as a single statement, so a CTE defined in cell 3 is not available in cell 4. If you need a result to persist across cells, that is a case for a temp view or a Delta table, not a CTE.

Spark SQL (Databricks)  ·  Identical CTE syntax
-- Spark SQL: CTEs work exactly as in T-SQL
WITH daily_revenue AS (
    SELECT
        TO_DATE(order_timestamp) AS order_date,
        SUM(amount)              AS revenue
    FROM   bronze.orders
    WHERE  status = 'completed'
    GROUP  BY 1
),
rolling_avg AS (
    SELECT
        order_date,
        revenue,
        AVG(revenue) OVER (
            ORDER BY order_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ) AS revenue_7d_avg
    FROM   daily_revenue
)
SELECT  *
FROM    rolling_avg
ORDER   BY order_date;
+
Go DeeperT-SQL vs Spark SQL — where CTE syntax actually diverges

For standard CTE definitions, T-SQL and Spark SQL are identical. The WITH name AS (...) syntax and multi-CTE chaining with commas work the same in both. The divergence appears in a few specific areas.

Table Variables (T-SQL only): T-SQL has DECLARE @tbl TABLE (...) syntax that Spark SQL does not support at all. Table variables behave like single-session temp tables but have no equivalent in Spark. If you are porting T-SQL that uses table variables, the Spark equivalent is typically a CTE or a cached DataFrame.

Recursive CTEs: Both T-SQL and Spark SQL 3.0+ support recursive CTEs with WITH RECURSIVE. T-SQL uses WITH cte_name AS (the RECURSIVE keyword is implied). Spark SQL requires the explicit WITH RECURSIVE keyword. One of the more common porting errors when migrating SSIS/T-SQL pipelines to Databricks.

CTE inside DML (T-SQL): T-SQL allows CTEs inside UPDATE, DELETE, and MERGE statements. Spark SQL does not support this — in Spark, you perform the same logic with a MERGE INTO using a subquery or a CTE in the SELECT arm only.

2.1.3

Recursive CTEs: Traversing Hierarchies

A recursive CTE calls itself. It consists of two parts joined by UNION ALL: an anchor member that defines the starting rows (base case), and a recursive member that references the CTE itself to traverse the hierarchy level by level. The recursion terminates automatically when the recursive member returns zero rows.

The canonical use case is an organisational hierarchy: each employee has a manager, who has a manager, who has a manager. A recursive CTE can traverse this structure to any depth — finding all direct and indirect reports of a given manager, calculating reporting chain length, or building a full org tree.

T-SQL  ·  Recursive CTE — Organisational Hierarchy
-- Find all direct and indirect reports under Manager ID 42
WITH org_tree AS (

    -- ANCHOR MEMBER: start with the manager themselves
    SELECT
        employee_id,
        name,
        manager_id,
        job_title,
        0    AS depth,           -- depth = 0 at root
        CAST(name AS VARCHAR(500)) AS reporting_path
    FROM   employees
    WHERE  employee_id = 42      -- start node

    UNION ALL

    -- RECURSIVE MEMBER: join back to the CTE to go one level deeper
    SELECT
        e.employee_id,
        e.name,
        e.manager_id,
        e.job_title,
        ot.depth + 1,
        CAST(ot.reporting_path + ' > ' + e.name AS VARCHAR(500))
    FROM   employees   AS e
    JOIN   org_tree    AS ot  ON e.manager_id = ot.employee_id
    -- Termination: when no more employees report to anyone in the current set,
    -- the JOIN returns zero rows and recursion stops automatically.

)
SELECT
    REPLICATE('  ', depth) + name  AS indented_name,
    job_title,
    depth                           AS org_level,
    reporting_path
FROM   org_tree
ORDER  BY reporting_path;
Figure 2.2 The structure of a recursive CTE. The anchor member selects the root nodes (no self-reference). The recursive member joins back to the CTE itself to go one level deeper on each iteration. Execution terminates when the recursive member returns zero rows — when no more children exist. The result is a flat table containing every node in the hierarchy with its depth and any accumulated columns.

The result is a flat table with every employee under Manager 42, at every level of depth, with their full reporting path. A structure that would require application code or multiple round trips to the database can be expressed as a single SQL statement.

🏢Field Note
A Professional Services Firm  ·  Budget Roll-Up

A professional services firm needed to roll up project budgets through their management hierarchy for a quarterly review. Each project was assigned to a project manager, who reported to a practice lead, who reported to a regional director, who reported to a global head. The organisation had seven levels in some chains and three in others. The original implementation loaded the entire employees and projects tables into a Python script that traversed parent relationships in a loop, re-querying the database at each level. For 4,000 employees it took several minutes and required the Python environment to be running.

A single recursive CTE replaced the entire Python script. The query executed in under two seconds. The anchor member selected projects with no parent project (top-level initiatives). The recursive member joined each project to its children by matching parent_project_id. A SUM(budget) OVER in the final select rolled up values through each path. The Power BI report that read from this query went from a 4-minute refresh to a 6-second one.

The Lesson

When you find yourself writing Python loops that make repeated database calls to traverse a hierarchy, a recursive CTE is almost always the right replacement. The database engine does the traversal far more efficiently than application code.

🏢Field Note
An FMCG Manufacturer  ·  Bill of Materials Explosion

A fast-moving consumer goods manufacturer needed to calculate the total raw material cost for each finished product. Their bill of materials was recursive: a finished product contained sub-assemblies, each sub-assembly contained components, and some components were themselves assembled from raw materials. The depth varied by product and was not stored anywhere — it had to be discovered by traversal.

The data engineering team used a recursive CTE to "explode" the bill of materials. The anchor member selected all finished products. The recursive member joined each assembly to its child components, accumulating a quantity multiplier at each level. The final query multiplied each raw material's unit cost by the accumulated quantity multiplier and summed per finished product. A calculation that previously ran nightly as a stored procedure taking 45 minutes was replaced by a CTE-based query running in 90 seconds as part of a Databricks pipeline.

The Lesson

Recursive CTEs are not just for org charts. Any parent-child relationship with unknown depth — product hierarchies, geographic hierarchies, account roll-ups, network graphs — is a candidate.

2.1.5

Performance: What the Optimiser Actually Does

The most persistent CTE misconception among intermediate SQL writers is that a CTE is always materialised — computed once and cached. It is not. In most database engines, including SQL Server and Spark, a CTE is treated by the optimiser as a macro substitution. The engine replaces each CTE reference with its definition inline and optimises the whole query together. If a CTE is referenced three times, the engine may execute it three times.

+
Go DeeperWhen to use a temp table instead of a CTE

The "CTE vs temp table" decision comes down to one question: will this intermediate result be referenced more than once, and is it large enough that recomputing it is expensive?

If a CTE is referenced three times in a single query and the query optimiser does not choose to materialise it, the underlying computation runs three times. For a CTE that reads one million rows and performs an expensive window function, three executions means three million rows processed. A temp table forces materialisation: the result is computed once, written to temporary storage, and all three references read from that stored copy.

Use a CTE when: the intermediate result is used once or twice, the data volume is small-to-medium, or readability of a single query is the priority.

Use a temp table when: the intermediate result is large, used more than twice in downstream logic, or when you need to add an index to the intermediate result to speed up subsequent joins.

In Databricks / PySpark, the equivalent of a temp table is df.cache() or writing to a Delta table and reading from it. The same principle applies: if you reference the same expensive computation repeatedly in a pipeline, materialise it.

+
Go DeeperCTEs and window functions — the most common pairing

The most frequent CTE pattern in production data engineering is using a CTE to first compute a window function, then filtering on its result in the outer query. This is necessary because you cannot reference a window function alias in a WHERE clause — the WHERE clause is evaluated before SELECT in SQL's logical execution order.

The pattern: define the window function inside a CTE, then SELECT * FROM that_cte WHERE window_col = value. The CTE makes this idiomatic and readable. The alternative — a subquery — works identically but buries the window logic inside an anonymous nested block.

A practical example: ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) gives the most recent order per customer. To keep only the most recent, you cannot WHERE row_num = 1 in the same SELECT. You wrap it in a CTE and filter in the outer query. This pattern appears in virtually every SCD Type 2 implementation, deduplication pipeline, and "latest record per entity" query in the industry.

✓ Section Summary

A CTE (WITH name AS (...)) is a named temporary result set scoped to a single query. It solves the readability problem of nested subqueries by letting you name and sequence each logical step. Multiple CTEs chain with commas after the first. Recursive CTEs traverse parent-child hierarchies of unknown depth using an anchor member (base case) and a recursive member (self-reference via UNION ALL). CTEs are not always materialised — for large intermediate results used multiple times, a temp table forces materialisation and avoids repeated computation. In Spark SQL, syntax is identical to T-SQL with one exception: recursive CTEs require the explicit WITH RECURSIVE keyword.

Question 1 of 5

What does the WITH keyword signal at the start of a SQL statement?

A
The query will use a stored procedure
B
One or more CTEs are being defined — named temporary result sets that exist only for the duration of this query

B is correct. The WITH keyword introduces a CTE block. What follows is one or more named result sets (separated by commas) that exist only for the query that follows. Nothing is stored permanently. The query is complete when the final SELECT (or INSERT/UPDATE/MERGE in T-SQL) after the CTE block finishes executing.

Question 2 of 5

You write a CTE that computes an expensive window function over 10 million rows. The CTE is referenced four times in the main query. What should you consider?

A
Nothing — CTEs are always materialised, so the computation runs once regardless of reference count
B
Renaming the CTE so the optimiser recognises it as a single object
C
Replacing the CTE with a temp table, which forces materialisation and ensures the 10M-row computation runs once regardless of reference count
D
Splitting it into four separate CTEs, one per reference

C is correct. CTEs are not guaranteed to materialise. The query optimiser treats a CTE as a macro and may inline it at every reference point, running the underlying computation up to four times. A temp table writes the result once and all four references read from the stored copy. For large intermediate results referenced multiple times, a temp table is the appropriate tool. In Databricks, df.cache() or writing to a Delta table serves the same purpose.

Question 3 of 5

In a recursive CTE, what is the role of the anchor member?

A
It defines the stopping condition that terminates the recursion
B
It defines the base case — the starting set of rows from which recursion begins (e.g. the root node of a hierarchy)
C
It references the CTE itself to traverse one level deeper in the hierarchy
D
It defines the maximum recursion depth to prevent infinite loops

B is correct. A recursive CTE has two parts joined by UNION ALL. The anchor member is the first part: it selects the starting rows with no self-reference (e.g. WHERE employee_id = 42 for the root of an org tree). The recursive member is the second part: it joins back to the CTE itself to go one level deeper. The recursion terminates automatically when the recursive member returns zero rows — typically when no more children exist at the current level.

Question 4 of 5

Why can't you filter on a window function result in the same SELECT clause where you define it — e.g. WHERE row_num = 1 in the same query that defines ROW_NUMBER() AS row_num?

A
Window functions cannot be aliased in T-SQL
B
The WHERE clause only accepts aggregate functions, not window functions
C
SQL's logical execution order evaluates WHERE before SELECT, so the alias defined in SELECT does not yet exist when WHERE runs
D
Window functions cannot be used in queries that also contain a WHERE clause

C is correct. SQL has a logical order of execution that differs from the written order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. The alias defined in SELECT (row_num) does not exist yet when WHERE is evaluated. The solution is to wrap the window function in a CTE or subquery and filter in the outer query. This is one of the most common reasons to use a CTE in production SQL.

Question 5 of 5

A

C is correct. Spark SQL 3.0+ supports recursive CTEs, but the RECURSIVE keyword is required, whereas T-SQL implies it. The syntax is otherwise identical. Definition

A CTE is a named temporary result set scoped to a single query. It exists only while that query runs. Nothing is stored permanently.

Common Misconception

CTEs are not always materialised. The optimiser may execute a CTE multiple times if referenced multiple times. For large intermediate sets, use a temp table instead.

Syntax Rule

WITH appears once at the top. Multiple CTEs are separated by commas. Only the final SELECT/INSERT/MERGE comes after the last CTE.

Recursive Structure

Anchor member (base case) UNION ALL recursive member (self-join). Terminates when recursive member returns zero rows. Use for org charts, BOM explosions, network graphs.

Spark SQL Note

Spark SQL requires WITH RECURSIVE for recursive CTEs. T-SQL implies it. This is the #1 CTE porting error when migrating to Databricks.

Interview Insight

"When would you use a view instead of a CTE?" — A view is persistent and shared across sessions; a CTE is ephemeral and query-scoped. Views encode shared business logic. CTEs encode query-local logic.
