Data Terminology

The shared vocabulary of data engineering — 36 terms every practitioner must own

✓

Expected Learning Outcomes

When you have completed this section, you should be able to:

  1. Define and distinguish terms commonly confused with each other: SLA vs SLO, schema drift vs schema evolution, soft delete vs tombstone vs archive
  2. Explain pipeline operational concepts: backfill, watermark, upsert, dead letter queue, late-arriving data
  3. Describe governance and quality concepts: data contract, data lineage, data catalog, data observability
  4. Apply modeling terminology correctly: grain, surrogate vs natural vs business key, cardinality

2.3.1

Why Vocabulary Is a Professional Edge

Technical vocabulary is not jargon for its own sake. Shared terms compress complex concepts into single words that everyone in a conversation understands identically. When a team member says "the pipeline needs to handle late-arriving data," a practitioner with shared vocabulary immediately knows this refers to records whose event timestamps belong to a closed processing window, not simply records that arrived after a scheduled cut-off time. Without that shared understanding, the same conversation takes ten minutes of clarification.

This section covers 36 terms that appear regularly in data engineering conversations, documentation, and interviews. Terms covered in depth in earlier sections are cross-referenced rather than repeated. New terms are given full entries with definitions, practical context, and common points of confusion. Use this section as a reference: read it through once for broad familiarity, then return to individual entries when a term comes up in context.

Several terms in this section are commonly conflated by practitioners with years of experience: SLA vs SLO, natural key vs surrogate key, schema drift vs schema evolution, soft delete vs tombstone. Knowing not just the definitions but the distinctions between related terms is what signals senior-level fluency in an interview or architecture review.
2.3.2

Pipeline Operations & Patterns

Core patterns — all covered in detail in earlier sections

ETL vs ELT

Extract-Transform-Load vs Extract-Load-Transform. ETL transforms before loading (traditional DW). ELT loads raw first and transforms after (modern data lake / Medallion).  → Covered in Section 1.1.modern.data.stack

Batch vs Streaming

Batch processes data in discrete scheduled runs. Streaming processes data continuously as events arrive. ADF is batch. Event Hubs + Databricks Structured Streaming is the Azure streaming stack.  → Covered in Section 1.5.data.mesh

Full Load vs Incremental Load

Full load copies the entire source on every run. Incremental load copies only new or changed records since the last run, tracked via a watermark.  → Covered in Section 1.4.adf

Watermark

A stored value (timestamp or sequence number) tracking how far an incremental pipeline has processed. Read before each run, updated after.  → Covered in Section 1.4.adf

Change Data Capture (CDC)

Reading the source database's transaction log to capture every insert, update, and delete as it happens. More precise than watermark; required when deletions must be detected.  → Covered in Section 1.4.adf

Idempotency

A pipeline that produces the same output no matter how many times it runs on the same input. Critical for Bronze ingestion; achieved via partition-overwrite mode.  → Covered in Section 1.2.medallion.architecture

New terms

Backfill

Re-running a pipeline over a historical time range it missed or that needs correction.

Backfills are triggered by: pipeline downtime during a period, a logic bug that has been fixed and history needs reprocessing, or a new dataset that requires historical data loaded before go-live.

The key requirement: backfills must be idempotent. Re-running a backfill for January must produce exactly the same result as running it the first time. Partition-overwrite mode achieves this for Bronze layers. For Silver and Gold, MERGE operations with deduplication logic handle it. A non-idempotent backfill doubles historical data — a very common and painful bug.

Hmm. A frequently missed edge case: backfilling a table that downstream consumers are actively querying. The backfill may temporarily produce inconsistent query results as partial history is loaded. The safe pattern is to backfill into a shadow table, validate, then swap.

Upsert / MERGE

Insert if the record does not exist; update if it does. The standard pattern for synchronising source data to Silver and Gold.

In Delta Lake: MERGE INTO target USING source ON target.id = source.id WHEN MATCHED THEN UPDATE SET ... WHEN NOT MATCHED THEN INSERT .... This is how CDC records (inserts, updates, deletes from a source) are applied to Silver tables.

Upserts are more expensive than appends because each incoming record requires a key lookup against the target. For high-volume tables, this cost is justified. For append-only tables (like raw event logs), an upsert adds unnecessary overhead. Know which pattern your table requires before designing the pipeline.

Soft deletes in a source system are handled as updates in the upsert: a record with is_deleted = true is applied as an update to the target row, not a deletion of it.

Hot Path vs Cold Path

Two parallel processing paths in a Lambda architecture: one for real-time (hot) and one for batch (cold).

In a Lambda architecture, the hot path processes streaming data with low latency (seconds to minutes) for real-time dashboards and alerts. The cold path processes the same data in large batches (hours) with higher accuracy and completeness. Results from both paths are merged in a serving layer.

The Lambda architecture was the dominant pattern before unified streaming/batch systems like Spark Structured Streaming and Delta Lake matured. Today, most organisations use a Kappa architecture (streaming-only, with batch replaced by replaying the stream) or a Medallion Architecture where the same Delta tables serve both near-real-time and batch queries. Lambda is worth knowing because legacy systems still use it — and because "hot path" and "cold path" remain common vocabulary even outside strict Lambda implementations.

A Financial Services Firm  ·  When "Backfill" Means Four Different Things

During a data platform migration project, a firm's project manager called a meeting to discuss "the backfill plan." In the room were a data engineer, a business analyst, a DBA, and a product owner. Each of them understood "backfill" differently: the data engineer meant reprocessing historical pipeline runs; the business analyst meant loading historical data for a new report; the DBA meant rebuilding fragmented index pages (an entirely different SQL Server term); and the product owner meant filling in gaps in data coverage from source systems that hadn't been connected yet.

The meeting produced a project plan that was four incompatible plans merged into one document. The confusion cost two weeks of rework when the implementation team started executing on different assumptions. The resolution: the project adopted a shared vocabulary document — every significant technical term was defined with an example before it appeared in project documentation.

The Lesson

Vocabulary failures are not vocabulary problems — they are coordination problems. Shared definitions are engineering infrastructure. Define terms early and explicitly, especially on projects where data engineers, analysts, DBAs, and business stakeholders work together.

2.3.3

Data Quality & Lifecycle

How data degrades, fails, and gets preserved

Schema Drift

A source system changing its schema without notice, causing downstream pipelines to fail or silently produce wrong data.

Schema drift is distinct from intentional schema evolution (adding a column as part of a planned release). Drift is unplanned: a source developer renames cust_id to customer_id, changes a column type from VARCHAR to INTEGER, or removes a field that has been deprecated internally but is still used by downstream pipelines.

In a Medallion architecture, Bronze absorbs schema drift gracefully (schema-on-read). The failure surface is Silver: if the Silver transformation references a column by its old name, it breaks silently or raises an error. Detection tools run schema comparison checks on each pipeline run and alert when column names, types, or counts change from the last successful run.

Schema evolution is the managed process of deliberately changing a schema — adding a new column, deprecating an old one — with advance notice and backward-compatible changes. Delta Lake supports schema evolution via mergeSchema and overwriteSchema options. Schema drift is the unmanaged version of the same event.

Late-Arriving Data

Records whose event timestamp belongs to a past processing window that has already been closed.

A mobile order placed at 11:58pm while the user is offline syncs to the backend at 12:04am. It arrives in January's pipeline but belongs in December. An IoT sensor with a slow connection sends yesterday's readings today. These records are not wrong — they are genuinely late.

Handling strategies depend on how much lateness is acceptable. For daily reporting, accepting records up to 48 hours late and MERGEing them into the correct partition may be sufficient. For financial close, a hard cutoff is required and late records may need to be held for the next reporting period. For streaming, Apache Spark's watermarking and windowing features define how long to wait for late events before closing a time window.

The worst handling strategy is silently dropping late records. The second-worst is unknowingly including them in wrong time periods without a flag. Always add an is_late_arrival boolean and an arrival_timestamp column to Silver for auditability.

Dead Letter Queue

A holding area for records that a pipeline could not process, preserved for investigation rather than lost or blocking.

Without a DLQ, a pipeline has two bad choices when a record fails: stop the entire pipeline (blocking all subsequent records) or silently drop the bad record (data loss). A DLQ provides a third option: capture the failed record with its error message and timestamp, continue processing the rest of the batch, and let engineers investigate the failures separately.

A mature DLQ includes: the original record bytes, the error message and stack trace, the pipeline run ID, the timestamp, and a status field (PENDING, RESOLVED, DISCARDED). Engineers review DLQ contents daily, fix the root cause (a new enum value the source started sending, a null in a required field, a schema change), and replay the records through the corrected pipeline.

In Azure, DLQs are commonly implemented as a dedicated ADLS container or as a secondary Delta table. ADF's fault tolerance settings can route failed records to a log file automatically.

Soft Delete vs Hard Delete vs Tombstone vs Archive

Four ways to handle record removal — all of which mean "not deleted" in different technical senses.

Hard delete: the record is physically removed from the table. A DELETE FROM orders WHERE id = 42 with no record of the deletion. For analytics, hard deletes in source systems are the hardest to handle — CDC can detect them from the transaction log, but downstream tables need explicit removal logic.

Soft delete: the record remains in the table but is flagged as deleted with a column like is_deleted = true and a deleted_at timestamp. The application filters these out in normal queries. For analytics pipelines, soft deletes are changes (updates), not removals — they are captured cleanly by CDC and watermark incremental loads.

Tombstone: a positive record asserting that deletion occurred. Common in event-sourced systems and Apache Kafka (a null-value message for a key tombstones that key's history in a compacted topic). Used when you need to communicate deletion across distributed systems.

Archive: the record is moved to a separate storage location (an archive table, a Cold-tier ADLS container) rather than deleted. Primarily a storage cost and compliance concept. Archived records are not deleted — they are moved. Under GDPR right-to-erasure, archiving is not sufficient; the data must be genuinely deleted from all copies.

Snapshot

A complete copy of data state at a specific point in time, preserved for comparison or audit.

A daily snapshot of a customer table captures every customer record as of midnight, regardless of what changes tomorrow. Snapshots enable point-in-time queries ("what did our customer list look like on January 1st?"), historical comparisons ("how did churn rates differ between Q1 and Q3?"), and regulatory compliance ("what did we report on this filing date?").

Delta Lake's time travel feature provides snapshot access to any previous version of a table without storing explicit snapshot copies: SELECT * FROM table TIMESTAMP AS OF '2024-01-01'. For very long retention requirements or cross-system snapshots, explicit snapshot tables written as INSERT INTO snapshots SELECT *, current_date() AS snapshot_date FROM source remain necessary.

Snapshots grow linearly with retention period. A table with 10 million rows taking daily snapshots for 5 years stores 18 billion rows. Columnar formats and partition pruning on snapshot_date keep this manageable but it remains the primary cost of snapshot-based historical tracking.

Fill and Kill

A financial trading term: execute as much of an order as is immediately available, and cancel the remainder.

In algorithmic trading systems, a Fill and Kill (FAK) order instructs the exchange: "buy 10,000 shares right now if they're available at this price; fill whatever you can immediately and cancel the unfilled portion." This contrasts with Fill or Kill (FOK, must fill entirely or not at all) and Good 'Til Cancelled (GTC, remain open until filled or explicitly cancelled).

For data engineers working in financial services, trading, or capital markets, this term appears in order management system (OMS) data. A fill_kill status in a trade table is not a data quality flag — it is a valid order execution type. Misinterpreting it as an error or incomplete record is a real and documented cause of incorrect financial reporting.

Why it's in a data engineering guide: domain vocabulary leaks into data. The names of database columns, status codes, and event types in financial source systems often carry trading-specific meaning. Getting these wrong produces incorrect analytics and risk reports.

Go DeeperSchema drift vs schema evolution — detecting the difference automatically

Schema evolution is planned: a team adds a customer_segment column to the CRM, notifies the data team, and the Silver pipeline is updated to include the new column before the next run. Schema drift is unplanned: a developer renames a column in a hotfix, no notification is sent, and the Silver pipeline fails silently at 2am.

Automated schema comparison solves this. On each Bronze write, compare the schema of the incoming data to the schema of the last successful run. If column names, types, or counts changed: alert the team and optionally fail the pipeline rather than write potentially broken data to Silver.

In Databricks, Delta Live Tables schema enforcement raises errors on schema mismatches automatically. Great Expectations' schema validation suite can be added as a pipeline step. dbt tests include not_null, accepted_values, and custom schema checks. The goal: detect drift before it propagates from Bronze to Gold and appears in a business dashboard as unexplained missing data.

2.3.4

Performance Concepts

Already covered in depth — cross-references

Data Skew

Uneven data distribution where some Spark partitions receive far more records than others, causing slow tasks. Detected via Spark UI task duration variance. See Section 2.2 — Partitions.  → Covered in Section 2.2.partitions

Shuffle

Data redistribution across Spark executors during GROUP BY, JOIN, and DISTINCT operations. Expensive because it involves network transfer. Minimised with broadcast joins and AQE.  → Covered in Section 2.2.partitions

Broadcast Join

Spark optimisation: send a small dimension table to every executor to avoid shuffling the large fact table. Controlled via spark.sql.autoBroadcastJoinThreshold.  → Covered in Section 2.2.partitions

New terms

Pushdown Predicate

A WHERE clause filter applied at the storage layer before data is read into memory, reducing data transfer.

Without predicate pushdown, the compute engine reads the entire dataset and then applies the filter. With pushdown, the storage layer applies the filter and returns only matching records. Parquet achieves this via row group statistics: if a filter is WHERE amount > 1000 and a row group's maximum amount value is 500, the entire row group is skipped without being read.

Partition pruning (Section 2.2) is the most extreme form of predicate pushdown — entire files are skipped, not just row groups. Column statistics pushdown is more granular — individual page-level skipping within a file. Delta Lake maintains statistics on the first 32 columns of every file, enabling very effective predicate pushdown without partition folders.

ADF pushdown: when ADF copies data from a SQL database, the source query filter is pushed to the database rather than reading all rows and filtering in ADF. This dramatically reduces the network transfer between the source database and ADLS.

Cardinality

The number of distinct values in a column — a fundamental factor in partition design, join performance, and index effectiveness.

High cardinality: many distinct values relative to total rows. Customer ID, order ID, email address, UUID. High-cardinality columns make good surrogate keys and join keys. They make poor partition columns.

Low cardinality: few distinct values. Status (ACTIVE/INACTIVE), country code (200 values), boolean flags (2 values). Low-cardinality columns make good partition columns. They make poor index keys because the selectivity is low — a filter on is_active = true might still return 90% of the table.

In query optimisation: the planner estimates how many rows a filter will return using cardinality statistics. A WHERE customer_id = 12345 filter on a high-cardinality column is highly selective (returns one row). A WHERE country = 'US' filter on a low-cardinality column may be non-selective (returns 60% of rows). The planner uses these estimates to choose the right join strategy and index access path.

2.3.5

Modelling, Governance & Security

Data modelling

Grain

The level of detail that a single row in a fact table represents — the most important decision in dimensional modelling.

Declaring the grain means answering: "What is one row?" For a sales fact table, one row might represent one order line item (fine grain) or one order (coarser grain). For a web analytics fact table, one row might represent one page view (fine) or one session (coarser). The grain must be declared before choosing which facts and dimensions to include.

Mixing grains in a single fact table is one of the most common dimensional modelling errors. If a fact table has both order-level facts (total_order_amount) and order-line-level facts (line_amount), querying both simultaneously produces incorrect results. Analysts SUM(total_order_amount) and get a number multiplied by the number of line items per order.

Hmm. Many practitioners can define "grain" abstractly but cannot apply it. The practical test: given a new fact column someone wants to add, ask "what does one row mean?" If the answer changes depending on which column you look at, the grain is mixed and the table needs redesign.

Surrogate Key vs Natural Key vs Business Key

Three types of identifier used in dimensional models, each with different stability, meaning, and join performance characteristics.

Surrogate key: a system-generated integer (or UUID) with no business meaning. Generated by the data platform, not the source. Stable when source systems change. Enables SCD Type 2 (multiple versions of the same entity with different surrogate keys). Used as the primary join key in dimensional models.

Natural key: the identifier from the source system. Customer email, order number, product SKU. Has business meaning but may change (customer updates their email), may not be unique across systems (order #1001 exists in both the CRM and the ERP), and is often a long string (slow joins).

Business key: a stable, unique identifier agreed upon at the business level, used to match the same entity across multiple source systems. May or may not match the natural key. Often the first column you reach for when resolving the "same customer appears twice" problem. A customer might have different IDs in the CRM, ERP, and eCommerce platform — the business key is the agreed single identifier.

Normalised vs Denormalised

Two opposing schema design philosophies with different write efficiency, query efficiency, and storage cost tradeoffs.

Normalised schemas eliminate redundancy by splitting data into many related tables. A customer appears once in the customers table; their orders appear in the orders table referencing the customer by ID. Write efficiency is high (one place to update). Query efficiency is lower (JOINs required). Dominant in OLTP databases.

Denormalised schemas embed related data into fewer, wider tables. A customer's name and region are repeated on every order row. Write efficiency is lower (updating a customer's name requires updating many rows). Query efficiency is higher (no JOINs). Dominant in analytics and OLAP workloads where queries read many rows and JOINs are expensive.

Gold tables in a Medallion architecture are typically denormalised for query performance. Silver tables may be partially normalised to avoid data duplication across multiple Gold layers. Bronze is whatever the source sent.

Materialised View

A pre-computed query result stored as a physical table and refreshed on a schedule or trigger.

A regular view re-executes its query every time it is queried. A materialised view executes once, stores the result as a physical table, and subsequent queries read the stored result. This trades storage cost and staleness (the view is only as fresh as its last refresh) for dramatically faster query performance on expensive aggregations.

In Databricks: CREATE MATERIALIZED VIEW gold.monthly_revenue AS SELECT .... In Power BI: calculated tables and aggregation tables implement materialised views. In Azure Synapse: materialised views on dedicated SQL pools provide the same benefit for large aggregations.

The key question before materialising: how often is the view queried vs how often does the underlying data change? A materialised view that's queried 1,000 times per day but refreshed once per hour is very efficient. A materialised view queried once per day but refreshed every 5 minutes has the costs of materialisation with minimal benefit.

Semantic Layer

A business-friendly abstraction that translates physical data models into business concepts and metric definitions.

Without a semantic layer, "revenue" might be defined differently in every Power BI report, every Looker dashboard, and every ad-hoc SQL query. The semantic layer centralises metric logic: "revenue" is defined once as SUM(order_amount - discount_applied) WHERE status = 'completed' and every consumer references that definition. Changing the definition in one place propagates everywhere.

Tools: dbt Semantic Layer (formerly dbt Metrics), Microsoft Analysis Services (SSAS / Power BI datasets), Looker's LookML, AtScale, Cube.dev. The common feature: a modelling layer that sits between physical tables and BI tools, where business logic is written once and exposed to all consumers through a stable interface.

Practically: if your Power BI reports and your SQL reports consistently show different revenue numbers, the root cause is almost always a missing or inconsistently implemented semantic layer — multiple teams defining the same metric differently.

Lakehouse

An architecture that combines data lake storage flexibility with data warehouse reliability and query performance.

A data lake stores data in open file formats on cheap object storage. Flexible and cheap, but brittle: no ACID transactions, no schema enforcement, poor query performance for analytical workloads. A data warehouse provides ACID transactions, strong schema enforcement, and excellent query performance but is expensive, inflexible, and proprietary.

A Lakehouse achieves both by adding a transaction layer (Delta Lake's transaction log, Apache Iceberg's metadata, Apache Hudi's timeline) on top of object storage files. ACID transactions are enforced via optimistic concurrency control in the metadata layer. Schema enforcement, query optimisation, and time travel are all handled by the transaction layer without changing the underlying Parquet files.

Databricks coined the term. Microsoft Fabric, Snowflake, and Apache Iceberg-based platforms implement variants. The key insight: the architectural gap between data lakes and data warehouses is not in the storage format but in the metadata management layer above it.

Governance & quality
🏢Field Note
Spotify  ·  Data Contracts at Scale

Spotify, which runs one of the largest data platforms in the industry with thousands of data producers and consumers, documented their data contract journey publicly around 2023. Their core problem: when producers changed schemas without notice, downstream consumers broke silently. With hundreds of teams producing data and thousands of pipelines consuming it, the informal "just let the data team know if you change something" coordination mechanism had completely broken down.

Spotify implemented data contracts as machine-readable YAML specifications stored in Git alongside the code that produces each dataset. Each contract declared the schema, the freshness SLA, the quality guarantees, and the breaking-change policy. Automated CI/CD checks validated that any proposed schema change was backward-compatible before the change could be merged. Consumers referenced the contract version in their pipeline configurations, so breaking changes triggered explicit migration workflows rather than silent failures.

The measurable outcome they reported: data-related incidents caused by unexpected schema changes dropped significantly, and the time to diagnose and resolve remaining incidents decreased because the contract immediately surfaced which consumers were affected.

The Lesson

Data contracts shift the cost of schema changes from discovery (finding out something broke after it breaks) to negotiation (agreeing on a change before it ships). The upfront cost is higher. The total cost across the organisation is lower.

Data Lineage

The documented trail of where data came from, every transformation it underwent, and where it went.

Table-level lineage: "This Gold table was produced from Silver table A and Silver table B, which were produced from Bronze tables C, D, and E, which were ingested from SAP and Salesforce." Column-level lineage: "The 'revenue' column in the Gold table is SUM(Silver.order_amount) - SUM(Silver.discount_applied) where order_amount came from Bronze.total_price from SAP field NET_VALUE."

Lineage matters for two reasons. Impact analysis: if a source column changes, lineage tells you every downstream table and report that may be affected without manually tracing the pipelines. Regulatory audit: a regulator asks "where did this reported figure come from?" Lineage provides the answer without manual investigation.

Tools: Microsoft Purview (Azure-native, scans ADF and Databricks automatically), Databricks Unity Catalog (column-level lineage for Databricks workloads), dbt (documents lineage as part of its DAG). The most valuable lineage is automated — lineage documented manually in a spreadsheet is always out of date.

Data Catalog

A searchable inventory of data assets: what exists, what it means, who owns it, and who can access it.

A data catalog answers the question "where is the data for X?" without needing to ask a person. It documents every table, view, and data product: column descriptions, sample values, data owner, freshness, access controls, quality metrics, and lineage. Without a catalog, data engineers spend 20-30% of their time answering discovery questions from analysts.

In Azure: Microsoft Purview is the native data catalog, automatically scanning Azure SQL, ADLS, ADF, and Databricks to populate metadata. Databricks Unity Catalog serves as both the governance layer and catalog for Databricks workloads. Open-source options include Apache Atlas and DataHub (LinkedIn open-source).

A catalog without governance is just documentation. The most effective catalogs are also the enforcement layer: access controls defined in the catalog are enforced at query time, not just documented.

Data Contract

A formal, versioned agreement between a data producer and its consumers, defining schema, quality guarantees, and breaking-change policies.

A data contract typically includes: the schema (column names, types, nullability), quality SLAs (freshness, null rates, value ranges), versioning semantics (what constitutes a breaking change, how much notice is given), ownership (who to contact when something breaks), and the consumer registry (who is using this dataset and therefore must be notified of changes).

Contracts are typically stored as YAML or JSON in Git alongside pipeline code. CI/CD checks validate that proposed changes are contract-compliant before merging. Consumers reference the contract version they depend on, enabling producers to detect when a change would break a consumer before shipping it.

The cultural shift data contracts require is significant: producers must accept that they have accountability to their consumers, not just to their own product. This is the same shift as Data Mesh's "Data as a Product" principle (Section 1.5) — data contracts are the technical mechanism that makes that principle operational.

Data Observability

Automatically monitoring pipeline health across five dimensions — freshness, volume, schema, distribution, and lineage.

Freshness: did the data arrive on time? Volume: is the row count within the expected range? Schema: did any column types or names change unexpectedly? Distribution: are value ranges, null rates, and categorical distributions consistent with historical norms? Lineage: when something breaks, which upstream change caused it?

Observability differs from monitoring in granularity. Monitoring checks if the pipeline finished. Observability checks if the data it produced is correct. A pipeline can complete successfully and still produce wrong data if a source changed format, a business logic rule produced incorrect results, or a join produced unexpected duplicates.

Rule of thumb: if a business user has ever been the first person to notice a data quality problem that should have been detected by the data team, you have a data observability gap.

Go DeeperImplementing GDPR right-to-erasure in a Medallion architecture

GDPR Article 17 gives EU residents the right to request deletion of their personal data. For a data platform with a Medallion architecture, this is technically one of the most demanding obligations because data has been copied, transformed, and replicated across many layers.

The deletion surface for one customer typically includes: raw records in Bronze (partitioned files — the customer's data is embedded in Parquet files with other customers), cleaned records in Silver, aggregated records in Gold (which may contain the customer's contribution to aggregate metrics), Delta Lake time travel history (previous versions of tables are stored in the Delta log and remain queryable until VACUUM'd), backups and disaster recovery copies in secondary regions, and any materialized views or snapshots derived from these tables.

The implementation pattern most commonly used: pseudonymisation at Bronze. Instead of storing PII directly in Bronze, customer identifiers are stored as pseudonymous tokens mapped to real identities in a separate, tightly controlled "identity vault." To fulfill a right-to-erasure request, delete the customer's record from the identity vault. All Bronze data containing their pseudonymous token is now inaccessible and effectively anonymised — without requiring rewriting every Parquet file that contains their data.

VACUUM timing matters: Delta Lake's time travel retains historical table versions. The default retention is 7 days. For GDPR compliance, you must either reduce retention to zero for PII-containing tables (at the cost of losing time travel) or design PII out of Delta tables and keep it only in the identity vault. Regulators consider data in accessible time travel history to be retained data subject to erasure requests.

Service levels

SLA vs SLO vs SLI

Three related but distinct concepts for measuring and committing to service quality — often confused by practitioners.

SLI (Service Level Indicator): a specific, measurable metric. "Percentage of pipeline runs completing within 4 hours." "Null rate on the customer_id column." "Row count variance from yesterday's run." An SLI is the raw measurement.

SLO (Service Level Objective): a target value for an SLI. "95% of pipeline runs must complete within 4 hours." "Null rate on customer_id must be below 0.01%." "Row count must not vary by more than 5% from yesterday's run." An SLO is the engineering team's internal target. The data engineering team sets and monitors SLOs.

SLA (Service Level Agreement): a commitment made to business consumers, typically less strict than the SLO to provide a buffer. "The Gold tables will be ready for reporting by 8am." If the SLO is 95% of runs completing by 7am, the SLA of 8am provides a 1-hour buffer. Breaking an SLA has business consequences. Breaking an SLO is an internal signal to improve reliability before it becomes an SLA breach.

Go DeeperData contracts in practice — what one actually looks like

A data contract is a YAML or JSON file stored in Git. A minimal example for a sales transaction dataset might contain four sections.

Schema: a list of fields with name, type, description, nullable flag, and accepted values for enums. Specifies what producers promise to deliver and what consumers can depend on.

Quality: freshness SLA ("updated within 2 hours of source"), null rate thresholds per column ("customer_id null rate < 0.001"), row count expectations ("daily row count between 50,000 and 500,000"), and value range constraints ("amount > 0").

Versioning: the current version number, a changelog of past versions, what constitutes a breaking change (removing a column, changing a type) vs a non-breaking change (adding a nullable column), and the required notice period for breaking changes (e.g. "30 days notice, with a 90-day deprecation window").

Ownership and consumers: the team that owns the dataset, the Slack channel for incidents, and the list of known consumers. When a producer wants to make a breaking change, CI/CD checks this list and automatically opens GitHub issues or sends Slack notifications to every registered consumer team.

Security & privacy

Row-Level Security (RLS)

Restricting which rows of a table a user can see, based on their identity or group membership.

RLS allows different users to query the same table and see different results. A regional sales manager running SELECT * FROM gold.sales sees only their region's records. A global head running the same query sees all regions. The filtering is applied transparently by the database engine before the query results are returned — the user is unaware of the restriction.

In Power BI: RLS is defined in the data model with DAX filter expressions. In Databricks: Unity Catalog row filters apply policies to Delta tables. In Azure SQL: the built-in RLS predicate function mechanism. The implementation must be tested carefully — an incorrectly defined row filter can either expose data that should be hidden or hide data that should be visible.

Column-Level Security & Data Masking

Restricting access to specific columns, or hiding sensitive values from unauthorised users while preserving the row.

Column-level security grants or denies access to entire columns. A contractor querying a customer table may see demographic and preference columns but not the email address or phone number. Unity Catalog and Microsoft Purview both support column-level access policies.

Data masking allows the column to be queried but returns a transformed value instead of the real one. An analyst might query SELECT email FROM customers and see j***@***.com instead of the actual email address. Analysts can still work with the data shape and structure without seeing real PII. The same query by a privileged user returns the unmasked value.

Data encryption is distinct from masking: the data is encrypted at rest and decryption requires a key. Encryption protects against storage-layer breaches. Masking protects against query-layer exposure. Both may be required for PII depending on regulatory requirements.

PII (Personally Identifiable Information)

Any data that can identify an individual person — subject to significant regulatory obligations.

PII includes obvious identifiers (name, email, phone, national ID number, passport number, IP address in some jurisdictions) and indirect identifiers that can identify someone when combined (age + zip code + gender can uniquely identify many individuals). The definition varies by regulation: GDPR's definition is broader than HIPAA's.

Data engineering obligations for PII: identify and tag PII columns in the data catalog. Mask or pseudonymise PII in non-production environments (dev and test should never contain real customer data). Apply column-level access controls and data masking in production. Implement right-to-erasure (RTBF) workflows so that when a user requests deletion, their PII is removed from all tables including historical snapshots. Maintain a record of where PII flows through the system via data lineage.

GDPR Article 17 (Right to Erasure) is the most technically demanding PII obligation for data engineers. Deleting a record from a Bronze partition that was snapshotted 3 years ago, replicated to a disaster-recovery region, and referenced in 15 downstream tables requires a documented deletion workflow across every copy — including Delta Lake's time travel history, which must be VACUUM'd.

Question 1 of 5

Your ADF pipeline processes last night's orders but three records from 11:45pm are missing — they arrived in the source system at 12:20am. What is this called and how should you handle it?

A
Schema drift — the source changed its ingestion timing; add a schema check and alert
B
A dead letter queue failure — the records failed processing and need to be investigated
C
Late-arriving data — records whose event timestamp belongs to a closed processing window; handle by either extending the window, using MERGE to insert them into the prior partition, or accepting them in the current run with an is_late_arrival flag
D
A watermark error — the watermark was not updated correctly and skipped these records

C is correct. The records exist in the source system but arrived after the pipeline's processing window closed. This is the definition of late-arriving data. The records are not failures (so DLQ is wrong), the schema has not changed (schema drift is wrong), and the watermark worked correctly — it simply did not capture records that arrived after the run. The correct response depends on business requirements: how much lateness is acceptable and whether the prior period's data can be amended.

Question 2 of 5

Your Gold table's "Revenue" figure differs between the Finance Power BI report and the Operations SQL report. Both query from the same Gold table. What is the most likely structural cause?

A
The Gold table is not being refreshed frequently enough
B
The Gold table has mixed grain — some rows represent orders and some represent order line items
C
Each report applies different filters or aggregations to define "Revenue" — what is missing is a semantic layer or data contract that encodes the single agreed definition
D
Data lineage is not configured, so the reports cannot access the same rows

C is correct. When two reports query the same physical table and get different answers, the root cause is almost always different metric definitions applied at the report layer. Finance may exclude returns; Operations may include intercompany. The structural fix is a semantic layer that centralises the "Revenue" definition as a single, agreed calculation. Both reports then reference the same definition and cannot diverge. A data contract for the Gold table would also formalise what "Revenue" means so both teams are explicitly aligned.

Question 3 of 5

A data engineering team tells a business stakeholder "we have an SLO of 95% of pipeline runs completing by 7am." The stakeholder responds "but you committed to having data ready by 8am." Are these statements contradictory?

A
Yes — if the SLO is 7am, the SLA must also be 7am or earlier
B
No — the SLO (7am internal target) is more strict than the SLA (8am business commitment) to provide a buffer. Missing the SLO occasionally does not breach the SLA
C
Yes — SLOs and SLAs must be identical or the governance model is incorrect
D
It depends on how frequently the SLO is missed

B is correct. The SLO (7am) is intentionally stricter than the SLA (8am). This buffer means the team can miss their internal target occasionally without immediately breaching the business commitment. An SLO equal to the SLA leaves no room for recovery — any miss is immediately a breach. Well-designed SLAs include buffer above the SLO to account for the fact that SLOs are sometimes missed even in well-run teams.

Question 4 of 5

A source CRM system uses a "soft delete" pattern — deleted customers have is_deleted = true but remain in the table. Your incremental pipeline uses a watermark on the last_modified column. Will deleted customers be captured?

A
Yes — a soft delete updates the row (setting is_deleted = true and updating last_modified), so the watermark pattern captures it as an update and the change is replicated to Silver via upsert
B
No — the watermark pattern cannot detect any type of deletion, soft or hard
C
No — soft deletes require CDC to detect because the watermark only sees inserts
D
It depends on whether the source database updates the last_modified timestamp on soft deletes

A is correct (with the caveat in D being the critical nuance). A soft delete that updates last_modified will be captured by a watermark pipeline because the watermark filters on last_modified > last_run_watermark — the soft delete appears as a changed row and is replicated to Silver as an upsert. The Silver table then shows is_deleted = true. If the source does NOT update last_modified on soft delete, option D would be correct and CDC would be required. Option B is wrong because soft deletes are updates, not deletions. Hard deletes, not soft deletes, require CDC.

Question 5 of 5

What is the grain of a fact table that contains one row per customer per day showing their total orders, total spend, and most recent order date?

A
One row per order
B
One row per customer
C
One row per customer per day — the grain is the unique combination of customer and date
D
The grain cannot be determined without knowing the primary key

C is correct. The grain is defined by the granularity of a single row: what unique combination of dimensions does one row represent? "One row per customer per day" means the grain is customer × date. This determines what facts can exist in the table: total_orders and total_spend are valid (they aggregate to customer × day). An individual order amount is not valid at this grain because one customer may have many orders on the same day — including that column would require a finer grain (one row per order). When in doubt about grain, find the natural primary key of the fact table: (customer_id, date) in this case.

Key Distinctions

SLA vs SLO: business commitment vs engineering target. Schema drift vs evolution: unplanned vs managed. Soft delete vs tombstone vs archive: row flagged vs event recorded vs data moved.

Backfill Rule

Every backfill must be idempotent. Running it twice must produce the same result as running it once. Use partition-overwrite mode for Bronze, MERGE for Silver and Gold.

Grain Test

For any column you want to add to a fact table: ask "does a single row at this grain always have exactly one value for this column?" If the answer is sometimes "it depends," the grain is wrong.

SLI / SLO / SLA

SLI: what you measure. SLO: what you target. SLA: what you promise. The SLA should always be more generous than the SLO to allow a buffer for recovery.

GDPR & PII

Archiving PII does not satisfy right-to-erasure. The data must be deleted from all copies: production, backup, Delta time travel history (VACUUM), and replicated regions.
