Git & GitHub for Data Teams

Version control for every artefact in the data stack — the mental model, core commands, branching strategy, merge conflicts, pull requests, and Git integration across Databricks, dbt, and ADF

  1. Explain what version control is and describe Git's four areas: working directory, staging area, local repository, and remote
  2. Execute the core daily Git workflow: branch, add, commit, push, pull request, merge
  3. Distinguish between git fetch, git pull, git merge, and git rebase — and explain when to use each
  4. Read a merge conflict marker and resolve a conflict correctly

5.2.1

Why Version Control Exists

Every data engineering team has a version of this story. A SQL file named customer_revenue_final.sql gets modified. Someone renames it customer_revenue_final_v2.sql. A week later there is a customer_revenue_FINAL_USE_THIS_ONE.sql and nobody is certain which one is actually used in production. A pipeline fails on a Monday morning and the engineer who knows which file is live is on holiday.

Version control solves this by making the history of every change the source of truth. Not the filename, not a folder called "archive," not a Slack message saying "I updated the model last Thursday." The repository. Every file in a Git-managed project has a complete, auditable history: who changed what, when, and with what description. Rolling back to any previous state takes one command. Working on a new feature without disrupting what's in production takes one command. Deploying a reviewed and approved change takes one command.

Every artefact a data engineering team produces — SQL models, Python scripts, dbt YAML configs, ADF pipeline JSON, Databricks notebooks, YAML workflow definitions — is a text file. Every text file belongs in version control. There is no exception to this rule for data engineering work.

Three things version control gives you that nothing else does: History — every version of every file, forever, with context. Collaboration — multiple engineers can work simultaneously without overwriting each other's changes. Rollback — any file, any state, any point in time, recoverable in seconds.

5.2.2

The Mental Model: Git's Four Areas

The most important thing to understand about Git before learning any commands is the mental model. Git has four distinct areas where your code can exist. Understanding which area each command moves code between is the key that makes all the commands coherent. Without this model, Git commands feel like arbitrary incantations. With it, they feel inevitable.

Figure 5.3Git's four areas and the commands that move code between them. Working directory is your local disk. git add stages specific changes (this is the "intent to commit" step — you choose which changes to include). git commit creates a permanent snapshot in the local repository with a message describing what changed and why. git push sends local commits to the remote (GitHub) where teammates can see and review. git fetch/pull retrieves changes from the remote — fetch downloads without merging, pull fetches and merges in one step.

Working directory: Files on your disk as they currently exist. Edits you haven't yet staged. The state you see in your code editor.

Staging area (index): A holding area for changes you intend to commit. Running git add dim_customer.sql moves that file's changes to staging. This intermediate step exists so you can commit related changes together, leaving unrelated changes unstaged. It is the "intent to commit" step — deliberate, selective, and yours to control.

Local repository: The permanent history on your machine. A commit is a snapshot of the staged changes at a point in time, with a message, an author, a timestamp, and a unique hash (e.g., c3f9a1b). Commits are private to your machine until pushed. Running git log shows your local commit history.

Remote (origin): The shared repository on GitHub, Azure DevOps, GitLab, or similar. This is what your team sees. Push sends your local commits here. Pull (or fetch + merge) retrieves remote changes to your local repository.

5.2.3

Core Commands for Data Work

The following commands cover approximately 95% of a data engineer's day-to-day Git usage. Each is illustrated with a real data engineering context rather than a generic software example.

Getting Started

git clone <url> — copies a remote repository to your local machine. First command run when joining a project. Creates a local copy of the repo with the full history. Example: git clone https://github.com/org/data-warehouse.git

git status — shows what has changed since the last commit: modified files (not yet staged), staged files (ready to commit), and untracked files (new files Git doesn't know about yet). Run this constantly. It is your orientation tool.

The Daily Workflow

git checkout -b feature/add-dim-product — creates a new branch and switches to it. Never work directly on main. Branch names should be descriptive: feature/ for new work, fix/ for bug fixes, chore/ for maintenance. This keeps main deployable at all times.

git add <file> or git add . — stages changes. Prefer staging specific files (git add models/dim_product.sql) over staging everything at once (git add .) to ensure your commit contains only related changes. A commit that touches a dbt model, a Python utility function, and a config file is harder to review and harder to revert than three separate focused commits.

git commit -m "add dim_product with SCD Type 2 columns" — creates a snapshot. Write commit messages in the imperative present tense as a description of the change: "add", "fix", "update", "remove". A good commit message completes the sentence: "If applied, this commit will: [your message]." Bad: "fixed stuff". Good: "fix null handling in fact_sales revenue calculation".

git push origin feature/add-dim-product — pushes your branch to the remote. After this, open a Pull Request on GitHub or Azure DevOps.

Staying Up to Date

git pull — fetches changes from the remote and merges them into your current branch. Use before starting new work and before pushing to avoid conflicts. On a branch that's been open several days: pull from main regularly to avoid diverging far from the base. git pull origin main pulls the latest main into your current branch.

git fetch — downloads remote changes without merging them. Safer than pull when you want to inspect what changed before incorporating it. After fetch: git log origin/main..HEAD shows what's in main that you don't have. Then git merge origin/main when you're ready.

Undoing Work

git stash — temporarily shelves uncommitted changes so you can switch context. Useful when an urgent prod fix interrupts ongoing work. git stash pop restores the stashed changes when you return.

git revert <commit-hash> — creates a new commit that undoes the changes of a specific commit. Safe for shared branches because it adds to history rather than rewriting it. Use this to undo a broken pipeline deployment in production.

git reset — moves the current branch pointer. Three modes with different consequences: --soft (uncommit but keep changes staged), --mixed (uncommit and unstage but keep changes), --hard (uncommit, unstage, and discard changes permanently). Only use --hard when you are certain you want to destroy the changes. Never use reset on commits that have been pushed to a shared branch.

Useful Inspection Commands

git log --oneline — compact history view. One commit per line with hash and message. Add --graph to see branch topology: git log --oneline --graph.

git diff — shows exact line-by-line changes between working directory and staging. git diff --staged shows differences between staging and last commit.

git blame <file> — shows who wrote each line and when. Invaluable for understanding why a specific SQL transformation or pipeline config was written a particular way. "Who wrote this and in what context?" — git blame answers both.

git cherry-pick <commit-hash> — applies a specific commit from another branch to the current branch without merging the whole branch. Use case: a fix was made on a feature branch but needs to be applied to main immediately without merging the entire feature.

5.2.4

Fetch, Pull, Merge, and Rebase — The Confused Quartet

These four operations are related but distinct, and the confusion between them causes more lost work and more Git anxiety than almost anything else. A clear mental model of each resolves the confusion permanently.

Fetch

Downloads changes from the remote to your local repository without incorporating them into any branch. Your working directory and local branches are unchanged. After fetch, you can inspect remote changes with git log origin/main before deciding to merge. Think of fetch as "show me what's out there without touching my work." Use it when you want to look before you leap.

Pull

Fetch + merge in one step. Downloads remote changes and immediately merges them into your current branch. The most convenient option for routine updates on branches with no local conflicts. Risk: if there are conflicts between remote and local changes, pull drops you into conflict resolution immediately with no chance to inspect first. On a branch with significant local work, fetch then inspect then merge is safer than pull.

Merge

Combines two branch histories into one. Creates a "merge commit" that has two parents, preserving the fact that two parallel lines of work were combined at a specific point. git merge feature/dim-product merges the feature branch into your current branch. Merge is safe for shared branches — it doesn't rewrite history, just adds to it.

Figure 5.4Merge vs rebase. Merge (top): feature commits D, E, F retain their original hashes; a merge commit with two parents is created, preserving the full branching history. Rebase (bottom): commits D, E, F are rewritten as D', E', F' with new hashes — replayed on top of main as if the feature branch had been started from the latest main commit. Rebase creates a clean linear history but rewrites commits, making it unsafe on any branch that other engineers have already pulled.

Rebase

Rebase replays your commits on top of another branch, rewriting their history in the process. The result is a clean linear log — it looks as if your feature was developed starting from the current tip of main, with no merge commit. The golden rule: never rebase a branch that other engineers have pulled. When you rebase, you rewrite commit hashes. Anyone who pulled the original commits now has a history that conflicts with yours in a way that is deeply confusing to resolve. Safe uses of rebase: tidying up your own local feature branch before opening a PR, squashing several small "fix typo" commits into one clean commit. Unsafe: rebasing main, rebasing a branch that the team has been pulling from.

5.2.5

Merge Conflicts

A merge conflict occurs when two branches have modified the same lines of the same file in incompatible ways. Git cannot automatically decide which version is correct — it marks the conflict and waits for a human decision. Conflicts are not errors or failures. They are a normal part of collaborative development and take minutes to resolve once you know how to read them.

Reading a Conflict Marker

When a conflict occurs, Git inserts markers directly into the file:

<<<<<<< HEAD — the start of your current branch's version
revenue = SUM(amount) WHERE currency = 'GBP'
======= — the divider between the two versions
revenue = SUM(amount_gbp) WHERE is_settled = TRUE
>>>>>>> feature/settlement-logic — the end of the incoming branch's version

Your job: decide which version is correct (your version, their version, or a combined new version), remove all three marker lines, and save the file. Then git add <file> and git commit to complete the merge.

+
Go DeeperMerge conflicts in VS Code and preventing them upstream

VS Code conflict resolution: VS Code renders merge conflicts visually with four buttons above each conflict block: "Accept Current Change" (HEAD version), "Accept Incoming Change" (other branch), "Accept Both Changes" (concatenate both), and "Compare Changes" (diff view). For most data engineering conflicts — two engineers who both modified the same dbt model or pipeline config — "Compare Changes" first, then accept the correct version or write a combined solution manually.

Preventing conflicts before they happen:

The most effective prevention is short-lived branches. A feature branch open for three days has a much lower chance of conflicting than one open for two weeks. Pull from main into your feature branch regularly: git pull origin main while on your feature branch keeps you close to the current state of main and resolves small conflicts incrementally rather than accumulating a large one.

For dbt specifically: conflicts in schema.yml files are common when two engineers add tests or documentation to the same model simultaneously. Structure your schema.yml files by model — one file per directory or one file per model group — so that engineers working on different models are editing different files, eliminating the conflict surface entirely.

Conflict resolution for data files: If the conflict is in a pipeline config or dbt model and both changes are semantically correct (both engineers added different business logic), the resolution often requires a conversation, not just a text merge. The conflict marker shows you what changed — understanding why it changed requires talking to the engineer who made the other change. For data work, a five-minute conversation before resolving a complex conflict is always faster than resolving the wrong version and discovering the data issue two weeks later.

🏢Field Note
A Data Consultancy  ·  The final_v2_FINAL.sql Problem

A data consultancy's client project had a shared network drive (not a Git repository) as its "version control" for SQL transformation scripts. The folder for the client's revenue model contained: revenue_model.sql, revenue_model_v2.sql, revenue_model_v2_FIXED.sql, revenue_model_FINAL.sql, revenue_model_FINAL_CLIENT_APPROVED.sql, and revenue_model_FINAL_CLIENT_APPROVED_deployed.sql. Six files representing one model. Nobody was certain which was actually running in production.

A discrepancy in the client's month-end revenue figures triggered an audit. The audit revealed that revenue_model_FINAL_CLIENT_APPROVED.sql (what everyone thought was live) differed from revenue_model_FINAL_CLIENT_APPROVED_deployed.sql by a single line — a currency conversion multiplier that had been added during a late-stage client approval session and applied only to the deployed copy, without updating the "approved" copy. The consultancy spent four days and significant goodwill explaining this to the client.

The project migrated to Git in the week that followed. One file per model, one source of truth. Every change tracked, every version recoverable, every deployment traceable to a commit with an author, a timestamp, and a message.

The Lesson

File-naming as version control fails the moment two people work on the same file or two changes are made in close succession. Git replaces the filename with the commit history as the source of truth. There is one file, one name, infinite history.

5.2.6

Pull Requests and Branching Strategy

A Pull Request (PR) — called a Merge Request in GitLab — is a formal proposal to merge a branch into main. It is the gate through which work passes from an engineer's branch to the shared codebase. For data engineering, PRs serve three purposes: code review (catching SQL logic errors, pipeline bugs, and schema issues before they reach production), documentation (the PR description captures why a change was made, not just what changed), and deployment trigger (CI/CD pipelines can run dbt tests or deploy Databricks notebooks when a PR is approved and merged).

What to Include in a Data Engineering PR Description

A PR description should answer four questions: What does this change do? Why was it needed (linked to a Jira/Linear ticket)? How was it tested? Are there any risks or caveats? For data changes specifically: include the row counts or sample output before and after the change, note any downstream impacts (other models that depend on this one), and flag any changes to grain, schema, or key columns that consuming reports depend on.

GitHub Flow — The Right Strategy for Most Data Teams

Three branching strategies are commonly discussed: GitFlow (main + develop + feature + release + hotfix branches — complex, heavyweight, better suited to versioned software releases), GitHub Flow (main + feature branches only — simple, fast, the right choice for most data teams), and Trunk-Based Development (everyone commits directly to main with feature flags — fastest iteration, requires high test coverage and CI confidence). GitHub Flow is the recommended default for data engineering teams:

One rule: main is always deployable. Feature branches are where work in progress lives. When a feature is complete and reviewed, it merges to main, and main is the source of truth for production. No develop branch, no release branch — just feature branches and main. This simplicity reduces cognitive overhead and removes the common confusion about which branch to branch from.

Squash, Merge, and Rebase — PR Merge Strategies

GitHub offers three ways to merge a PR: Merge commit (preserves all commits from the feature branch plus a merge commit — full history, messy log for branches with many small commits); Squash and merge (combines all feature branch commits into one commit on main — clean log, loses individual commit history); Rebase and merge (replays feature commits onto main individually — linear history, no merge commit, loses the branch topology). For data teams: squash and merge is usually the best choice. It keeps main's log clean (one commit per feature, not seventeen "fix typo" commits) while the PR itself preserves the full development history for anyone who needs it.

5.2.7

.gitignore for Data Projects

The .gitignore file lists patterns for files and directories that Git should not track. For data engineering projects, the gitignore is both a security control (preventing credentials from being committed) and a repository hygiene tool (preventing large binary files and generated artefacts from bloating the repo).

What to Exclude

Security-critical: .env, *.env, secrets.yaml, credentials.json, *.pem, *.key — environment files and credential files must never enter the repository. Even in a private repo, credentials in Git history are a security debt that is expensive to clean up and carries real breach risk if the repo is ever made public or accessed by the wrong person.

Large data files: *.csv, *.parquet, *.xlsx, *.feather — data files belong in ADLS or object storage, not in Git. Even "small" CSVs used for testing accumulate quickly and make cloning slow.

Generated artefacts: __pycache__/, *.pyc, target/ (dbt compilation output), .databricks/, dist/, *.egg-info/ — these are generated from source files that are tracked, so tracking the generated versions creates redundancy and conflicts.

What to Always Commit

Everything that defines how the pipeline, model, or report is built: all .sql files (dbt models), all .py files (scripts, utilities), all .yml/.yaml files (dbt project config, schema tests, GitHub Actions workflows, Databricks bundle configs), all .json files that represent pipeline definitions (ADF pipeline JSON, ARM templates), requirements.txt and pyproject.toml (Python dependencies), Makefile and Dockerfile if present. If another engineer cannot reproduce your environment and pipeline from the repository alone, something is missing from version control.

🏢Field Note
A SaaS Startup  ·  Credentials in GitHub

A data engineer at a SaaS startup was connecting a new Databricks notebook to an Azure SQL Database. Rather than using environment variables or Azure Key Vault, they hardcoded the connection string — including username and password — directly in the notebook and committed it. The repository was private, but the company used a GitHub organization account that was accidentally set to public visibility during a settings change three weeks later.

Within 48 hours of the accidental public exposure, an automated credential-scanning bot had found the connection string. The Azure SQL Database received login attempts from six different geographic regions. The database was not breached — the SQL firewall limited access to known Azure IP ranges — but the credential had to be rotated immediately, triggering an emergency change window and a four-hour outage of the downstream reporting system that depended on that connection.

The fix took 15 minutes. The retrospective, the credential rotation, the stakeholder communication, and the code review process changes took three days. The engineer involved had not done anything unusual — many people commit credentials "just temporarily" — but temporary commits have a way of becoming permanent history.

The Lesson

Never commit credentials to any repository — private or public. Add .env to .gitignore before the first commit. Use Azure Key Vault, environment variables, or Databricks secrets. A credential in Git history is not easily removed — even after deletion, it remains in the history and must be explicitly purged with git filter-branch or BFG Repo-Cleaner.

5.2.8

Git in the Azure Data Stack

Git integration works differently in each tool of the Azure stack. Understanding the specific integration pattern for each tool prevents the most common mistakes.

dbt + Git

dbt is Git-native by design. Every dbt model is a .sql file. Every test is a YAML entry. Every documentation string is a YAML field. The entire dbt project is a folder structure of text files, ready to be committed. The recommended workflow: one branch per feature or model change, PR to main triggers a CI run (dbt build --select state:modified+ on the changed models only — the "slim CI" pattern), merge to main triggers deployment to production. This gives you code review, automated testing, and deployment traceability with no additional tooling.

Databricks + Git

Databricks has first-class Git integration built into the workspace. Under Workspace → Repos, you can connect a Databricks workspace directly to a GitHub or Azure DevOps repository. Changes made in Databricks notebooks are synced to the Git repository. Databricks Asset Bundles (DABs) extend this to full CI/CD: define jobs, clusters, and workflows in YAML (bundle.yml), commit to Git, and deploy via databricks bundle deploy from a GitHub Actions or Azure DevOps pipeline. This replaces the older pattern of manually creating jobs in the Databricks UI — DABs make the entire Databricks environment reproducible from code.

ADF + Git — The Publish Button Trap

ADF's Git integration is the most commonly misunderstood in the stack. When you connect ADF to a Git repository (Settings → Git configuration), ADF stores pipeline definitions as JSON files in the connected repository. Changes to pipelines are automatically saved as commits on a feature branch. This sounds straightforward — but there is a critical distinction between "saving to Git" and "deploying to production" in ADF Git mode.

In ADF Git mode, the pipeline definitions in your Git repository are not what runs in production. What runs in production is the "published" version — a separate copy maintained in the adf_publish branch, updated only when you click the blue "Publish" button in the ADF UI. This means: a pipeline committed to Git and merged to main is not yet live. You must also click Publish. Engineers who are used to "merge = deploy" patterns (as in dbt or Databricks) regularly make the mistake of merging a pipeline change and assuming it is live — only to find production is still running the old version.

The correct ADF CI/CD pattern bypasses the Publish button entirely: export ADF ARM (Azure Resource Manager — Microsoft's infrastructure-as-code format) templates from Git, use Azure DevOps Pipelines or GitHub Actions to deploy the ARM template to each environment (dev → test → prod) on PR merge. This makes ADF deployments as reproducible and controllable as any other infrastructure deployment. The Publish button should only be used in development — never as the deployment mechanism for test or production.

+
Go Deeperdbt slim CI with GitHub Actions — the full pattern

Slim CI is the dbt pattern for running only the models and tests affected by a PR, rather than rebuilding the entire project on every change. At scale (hundreds of dbt models), full project runs take 30–90 minutes — too slow for a PR gate. Slim CI brings that to under five minutes for most PRs.

Prerequisites: dbt artifacts from the production run (specifically manifest.json) must be stored somewhere the CI runner can access. Common pattern: upload to Azure Blob Storage after each production run, download in CI before comparing state.

GitHub Actions workflow (`.github/workflows/dbt-ci.yml`):

name: dbt Slim CI
on: [pull_request]
jobs:
  dbt-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: pip install dbt-databricks
      - run: az storage blob download --account-name $STORAGE --container dbt-artifacts --name manifest.json --file target/manifest.json
      - run: dbt build --select state:modified+ --defer --state target/

What this does: state:modified+ selects only models that changed in this PR, plus their downstream dependents. --defer uses the production run's compiled SQL for unmodified upstream models rather than re-running them. The result: a PR that changes dim_product.sql will run dim_product and any models that depend on it — not the entire warehouse. Fast feedback, targeted testing, automated gate before merge.

Go DeeperThe ADF Git integration publish trap — full explanation and correct CI/CD pattern

Why the trap exists: ADF Git integration stores pipeline definitions in a repository branch you specify (usually main or a collaboration branch). ADF reads pipeline definitions from Git for development — you see the pipeline, you can edit it, it saves to Git. But ADF's execution runtime reads from a different source: the published ARM template, stored in the adf_publish branch. Clicking "Publish" compiles the Git-based pipeline definitions into ARM templates and overwrites the adf_publish branch. Production ADF reads only from adf_publish, never directly from your collaboration branch.

The consequence: A pipeline change committed to main and merged via PR is visible in the ADF UI — but it is NOT running in production. The old published version is still live. Engineers who merge-and-assume-deployed create a silent divergence between what Git says should run and what production is actually running. This is particularly dangerous for data pipelines where the wrong version quietly produces incorrect results rather than failing visibly.

The correct pattern — ARM template CI/CD:

Step 1: In ADF Git mode, develop on feature branches, collaborate on main. Never click Publish in production ADF instances.

Step 2: In Azure DevOps Pipelines or GitHub Actions, create a release pipeline triggered on merge to main. The pipeline: checks out the ADF repository, generates ARM templates from the pipeline JSON (ADF provides a Node.js utility for this, or use the adf-ci-option from the ADF team), and deploys to dev, test, then prod using az deployment group create with the ARM template.

Step 3: Use ADF-specific parameters (linked service connection strings, etc.) per environment — the ARM template parameterises these automatically when ADF generates it.

This pattern makes ADF deployments reproducible, reviewable (all changes go through PR), and environment-consistent. The Publish button is used only in development ADF instances for initial testing before a PR is raised.

✓ Section Summary

Version control exists to give teams history, collaboration, and rollback. Git has four areas: working directory (your edits), staging area (changes you intend to commit), local repository (committed history on your machine), remote (GitHub — what the team sees). Core workflow: branch → add → commit → push → PR → merge. git fetch downloads remote changes without merging (inspect first); git pull fetches and merges in one step (convenient but less controlled). Merge creates a merge commit and preserves history — safe for shared branches. Rebase rewrites commits for a clean log — never rebase shared branches. Merge conflicts are normal: read the markers, decide which version is correct, remove the markers, add and commit. GitHub Flow (main + feature branches only) is the right branching strategy for most data teams. .gitignore must exclude credentials, data files, and generated artefacts — never commit .env or any file containing a connection string or password. In the Azure stack: dbt is Git-native by design; Databricks uses Repos + DABs; ADF's publish button is a trap — use ARM template CI/CD instead.

Question 3 of 5

Why is it dangerous to use git rebase on a branch that other engineers have already pulled?

A
Rebase cannot be run on branches with more than one contributor
B
Rebase deletes the branch entirely once complete
C
Rebase rewrites commit hashes — anyone who has already pulled the original commits now has a diverged history that conflicts with the rebased version in ways that are confusing to resolve and can result in duplicate or lost commits
D
Rebase on shared branches is slow because it must re-run all CI checks on each replayed commit

C is correct. When you rebase, Git takes your commits, temporarily removes them, moves the branch pointer to the new base, then replays the commits — creating new commits with new hashes (D becomes D', E becomes E'). Engineers who pulled the original D and E now have commits in their local history that no longer exist on the remote. When they next try to push or pull, Git sees completely different histories for the same branch and produces deeply confusing conflicts. The golden rule: rebase is safe only on commits you have not yet pushed, or on a personal branch that nobody else is pulling. Once commits are shared, use merge instead.

The pipeline change was not committed correctly — ADF saves to Git only when the Save All button is explicitly clicked

B is correct. This is the ADF publish button trap. ADF Git integration creates a separation between development (collaboration branch in Git) and execution (published ARM templates in the adf_publish branch). Merging to the collaboration branch updates the development state visible in the ADF UI — but it does not update what production ADF runs. Production ADF reads only from the adf_publish branch, which is updated only when a human clicks Publish or when an automated ARM template deployment pipeline runs. This is the most common confusion for engineers moving from dbt or Databricks (where Git + run = deploy) to ADF (where Git ≠ deploy until published). The production CI/CD pattern bypasses the Publish button entirely, using ARM template deployments triggered by pipeline events instead.

Question 5 of 5

Which of the following should be added to a data engineering project's .gitignore?

A
dbt model files (*.sql) — they contain business logic that should be version controlled separately
B
schema.yml — this is auto-generated and should not be committed
C
.env files, *.parquet, *.csv, __pycache__/, and target/ (dbt compilation output) — credentials must never be committed; data files belong in object storage; generated artefacts are rebuilt from tracked source files
D
requirements.txt — Python dependencies should be managed externally, not tracked in Git

C is correct. .env files contain credentials — committing them, even once, is a security risk that persists in Git history even after deletion. Data files (*.csv, *.parquet) belong in ADLS or other object storage, not in Git repositories. __pycache__/ and *.pyc are Python compilation artefacts generated from tracked .py source files — tracking the generated versions creates unnecessary conflicts. target/ is dbt's compilation output directory, rebuilt on every dbt run — tracking it bloats the repository. dbt model files (A) are exactly what should be committed — they are the source of truth for transformations. schema.yml (B) is hand-authored configuration and test definitions, not auto-generated — it must be committed. requirements.txt (D) should absolutely be committed — without it, nobody can reproduce your Python environment.

The Four Areas

Working directory → (git add) → Staging → (git commit) → Local repo → (git push) → Remote. Every Git command moves changes between these areas. The mental model is more important than memorizing flags.

Never Rebase Shared

Rebase rewrites commit hashes. Anyone who pulled the original commits now has a diverged history. Golden rule: rebase only commits you haven't pushed, or branches only you are working on.

ADF Publish Trap

In ADF Git mode, merging to main does NOT deploy. Production ADF reads from the adf_publish branch only, updated by the Publish button or ARM template CI/CD. Merge ≠ deploy in ADF.

Interview Insight

"Walk me through your Git workflow" is standard in senior interviews. Describe GitHub Flow: branch from main, commit with clear messages, open PR with description, squash merge. Mention dbt slim CI and the ADF publish distinction as evidence of production experience.
