DataOps & CI/CD

Applying engineering rigour to data pipelines — automated testing, deployment, and observability for dbt, Databricks, and ADF; data quality tooling; and building the monitoring that tells you your pipeline is broken before your stakeholders do

5.4.1

What DataOps Actually Means

DataOps is a set of practices that apply the principles of DevOps — automation, collaboration, fast feedback, and continuous improvement — to data pipelines and analytics workflows. The term is often misused by vendors to describe any modern data tooling. The concept itself is valuable and specific: treat data pipelines as production software, not as ad hoc scripts.

For a data engineering team, this means three things in practice. First, automated CI/CD: pipeline changes are tested automatically when a PR is opened, and deployed automatically when that PR is merged — nobody manually runs a notebook to deploy a change. Second, data quality monitoring: automated checks run against the data itself (not just the code), verifying that the records loaded match expected patterns of volume, completeness, and values. Third, observability: the team knows the pipeline is healthy without manually inspecting it — dashboards, alerts, and logs provide a continuous picture of pipeline state.

A data pipeline without monitoring is not a production system — it is a script that may or may not be working, with no mechanism to find out until a stakeholder notices a wrong number. DataOps turns "may or may not be working" into "working, and here is the evidence."

This section builds directly on section 5.2 (Git and GitHub). CI/CD for data requires version control. If your team's data pipelines are not in Git, DataOps is not possible yet — address that first, then return here.

5.4.2

CI/CD for Data Pipelines

CI (Continuous Integration) means that every change to the codebase is automatically tested before it can be merged. CD (Continuous Deployment or Delivery) means that every approved change is automatically deployed to the target environment without manual steps. Applied to data pipelines, CI/CD gives you: confidence that a change will not break production before it reaches production, deployment traceability (every change in production is linked to a PR with a reviewer and a description), and fast rollback capability (the previous version is always reachable via Git).

Figure 5.7The DataOps loop. Left to right: a developer makes a change on a feature branch and opens a PR, triggering the CI pipeline (dbt slim CI, unit tests, linting, YAML validation). Passing CI allows peer review. Approved merge triggers CD (automated deployment through dev → test → production). Below: continuous observability monitoring runs independently of deployments — volume, freshness, schema, and distribution checks generate alerts that feed back to the developer when production data deviates from expected patterns. The loop closes when an alert triggers investigation, a fix is made, and the cycle repeats.

dbt CI/CD Pattern

dbt CI runs on PR open: dbt build --select state:modified+ --defer --state target/ (slim CI, covered in section 5.2). This runs only the changed models and their downstream dependents, using production artifacts for unmodified upstream nodes. Test failures block the PR. dbt CD runs on merge to main: dbt build (full production run) or dbt build --select state:modified+ --defer depending on environment strategy. The dbt Cloud Orchestration layer or a scheduled GitHub Actions workflow handles the production cadence.

Databricks CI/CD with DABs

Databricks Asset Bundles (DABs) are the current standard for Databricks CI/CD. A bundle.yml file in the repository defines jobs, cluster configurations, notebook paths, and environment-specific parameters. Deployment: databricks bundle deploy --target dev promotes all resources to the dev workspace; databricks bundle deploy --target prod promotes to production. GitHub Actions or Azure Pipelines trigger the deployment command on PR merge. Every job definition, cluster spec, and scheduled run is in version control — the entire Databricks workspace configuration is reproducible from code.

For notebook testing: the pytest framework with databricks-connect allows unit tests to run against a Databricks cluster as part of CI. This catches Python logic errors before deployment. Alternatively, extract pure Python logic from notebooks into importable modules and test those modules without a cluster connection — faster CI and better code organization.

ADF CI/CD Pattern

ADF CI/CD uses ARM (Azure Resource Manager) template deployment. The ADF Git integration (covered in section 5.2) stores pipeline definitions as JSON files in the collaboration branch. A CI/CD pipeline (Azure Pipelines or GitHub Actions) exports the ARM template from the ADF JSON files and deploys it to each environment using az deployment group create. Environment-specific linked service connections (dev database vs. production database) are parameterised in the ARM template. The Publish button in the ADF UI is used only in development — never as the production deployment mechanism.

5.4.3

Data Observability

Observability is the property of a system that allows you to infer its internal state from its external outputs — without looking at the code or manually inspecting tables. For a data pipeline, observability means: you can know whether the pipeline ran correctly, whether the data it produced is healthy, and whether anything has changed unexpectedly — from a dashboard or alert, not from a morning of manual checking.

The practical question is: what should you monitor? Four categories cover the vast majority of production data health issues.

dbt compile + schema contracts · Alert: column_count != expected OR dtype changed Foundation 2. Freshness Was the table updated when it should have been? Stale data is invisible failure. dbt: source freshness block in schema.yml · Alert: MAX(updated_at) > threshold minutes ago Reliability 3. Volume Did the expected number of records load? Drop of 15% is often invisible without a count check. Alert: row_count < (7-day avg × 0.85) OR row_count = 0 · Compare today vs same-day last week Completeness 4. Distribution Are values within expected ranges? Null rates acceptable? No unexpected new values? dbt: accepted_values, not_null tests · Alert: null_rate > 5% OR new categorical value appears Accuracy Catch most issues Catch subtle issues
Figure 5.8Four-layer data health monitoring framework, ordered from most fundamental (schema, bottom) to most nuanced (distribution, top). Schema and freshness checks catch failures that make data unusable — wrong columns or stale tables. Volume checks catch silent data loss — the missing 15% of records that no error log captures. Distribution checks catch semantic drift — values that are technically valid but statistically anomalous. Most teams should implement all four layers, starting from the bottom. dbt provides native support for schema contracts, source freshness, and distribution tests; volume checks require custom monitoring logic or an observability platform.

Alert Design and Avoiding Alert Fatigue

Alert fatigue is the condition where a monitoring system generates so many alerts that engineers begin ignoring them — including the important ones. It is the single most common failure mode of data observability implementations. A team that receives 50 alerts per day acknowledges none of them meaningfully. A team that receives 3 alerts per day treats each one as significant.

Alert on deviation from baseline, not on absolute thresholds. An alert that fires when a table has fewer than 10,000 rows is brittle — it fires every time business activity is seasonally low and adds no signal. An alert that fires when today's row count is more than 15% below the rolling 7-day average for the same day of week is informative — it signals something unusual happened, regardless of absolute scale.

Every alert must be actionable. If the person receiving the alert cannot take a specific action in response, the alert should not exist. "Revenue table freshness is 4 hours overdue" is actionable (investigate the ADF pipeline run). "Revenue table has 247,892 rows" is not actionable — it conveys no signal about whether that is correct or not. Alerts should say something is wrong, not just that something exists.

Start with fewer alerts than you think you need. The correct starting point for observability is not comprehensive coverage — it is the three or four checks that would have caught the three or four production incidents your team has actually experienced in the past year. Build from known failure modes, not hypothetical ones. Add alerts as new failure modes occur.

🏢Field Note
A Retail Bank  ·  The Silent 15% Drop

A retail bank's data engineering team ran a daily ingestion pipeline from their core banking system into the Gold layer. The pipeline had been running in production for 14 months without incident. One quarter, the risk team noticed that their loan portfolio exposure figures were consistently lower than the credit team's independent calculation — by approximately 15%.

Investigation revealed that three months earlier, a schema change in the core banking system had added a new loan_status code ("SECURITISED") to the loan table. The ingestion pipeline had a filter excluding loan records with status values not in a hardcoded list — a list that had been written at implementation and never updated. The new status code was silently excluded from every load. Three months of loan records, totalling approximately 15% of the portfolio, had been dropped without any error, any warning, or any alert.

The pipeline had run correctly — it had done exactly what it was coded to do. The problem was that what it was coded to do no longer matched what it should do. No monitoring had been in place to detect the volume anomaly. Adding a single dbt test — accepted_values for the loan_status column — would have flagged the new status value on its first appearance.

The Lesson

Pipelines that run without errors are not necessarily pipelines that produce correct data. Volume monitoring and distribution checks catch silent failures that error logs never surface. A pipeline that drops 15% of records silently is harder to detect than one that crashes visibly — and more dangerous, because the data looks plausible.

5.4.4

Data Quality Tools

Data quality tooling operates at different layers of the stack and solves different problems. Choosing correctly requires understanding what problem each tool is actually designed for — and not conflating testing (does the data meet defined rules?) with observability (is the data behaving consistently over time?). The planning doc for this guide explicitly called out this comparison, and it is worth covering in full because the market is fragmented and the tool names are often used loosely.

The Decision Framework

Start with dbt tests. If you use dbt, you have no excuse not to have not_null, unique, and relationships tests on every primary key and foreign key in your Gold layer — they are four lines of YAML each and run as part of every CI pipeline. Add accepted_values for any categorical column you care about. This catches a large proportion of production data issues with minimal investment.

Add Great Expectations when you need rule-based testing outside the dbt layer — in Bronze ingestion (before transformation), in Python pipelines that don't use dbt, or for complex validation logic that dbt's generic tests cannot express. Great Expectations works well in Databricks notebooks as a validation step after ingestion.

Consider Soda when your business stakeholders want visibility into data quality without reading dbt test output. Soda's dashboard and notification integrations (Teams, Slack, email) are business-friendlier than dbt's CLI output. Use it as the presentation layer for data quality checks that non-engineers need to monitor.

Monte Carlo and Datafold are appropriate at scale and maturity levels where the cost is justified. Monte Carlo makes sense for large data platforms (50+ tables in production, 10+ engineers) where manual rule-writing cannot cover the surface area and automated ML-based anomaly detection is needed. Datafold makes sense for teams with frequent model changes who need to see the downstream data impact of a PR before it merges, not after.

Go DeeperGreat Expectations vs dbt tests — when each is the right choice

Both tools perform rule-based data validation, and teams sometimes wonder whether they need both or whether one replaces the other. The answer: they are complementary, not competitive, and the choice of which to reach for first depends on where in the pipeline the validation needs to happen.

dbt tests are the right choice when: The data you want to test has passed through dbt — it is a Gold or Silver table built by a dbt model. Tests are defined as YAML in the same repository as the model, run as part of dbt build and therefore automatically included in your CI pipeline. Zero new tooling, zero new infrastructure, minimal learning curve. The test types are limited (generic: unique, not_null, accepted_values, relationships; singular: custom SQL assertions) but those four cover 80–90% of what most teams need. dbt tests are always the starting point for teams already using dbt.

Great Expectations is the right choice when: You need to validate data at the Bronze layer — before dbt has run, directly on the raw ingested data. For example: after an ADF pipeline loads a CSV to ADLS, before any transformation, run a Great Expectations suite to verify that the expected columns exist, the row count is within range, and key fields are not null. GE is also the right choice for non-dbt transformation stacks (Python scripts, Spark notebooks not managed by dbt) where there is no dbt project to add tests to. GE's Expectations can be defined in Python code and run anywhere Python runs — in a Databricks notebook, in a GitHub Actions job, in an Azure Function.

The combined pattern: Great Expectations at the Bronze ingestion layer (raw data validation before transformation), dbt tests at the Gold layer (business logic validation after transformation). This gives you a two-layer quality gate: bad source data is caught at ingestion, bad transformation logic is caught at the model layer. An issue caught by GE at Bronze points to the source system or ingestion pipeline. An issue caught by dbt at Gold points to transformation logic. The two layers give you root-cause locality — you know where to look.

Go DeeperDIY data observability without Monte Carlo — using dbt + Azure Monitor

For teams that cannot justify Monte Carlo's enterprise pricing, a functional observability stack can be built from tools you already have. The result won't have ML-based anomaly detection, but it will catch the 90% of production issues that rule-based monitoring covers.

Volume monitoring with dbt: Create an audit_table_stats model in your dbt project that runs after your main build and records row counts, null rates, and max(updated_at) for each production table into a monitoring Gold table. Query this table in Power BI or a simple Python script to track trends. A chart of daily row counts per table, visible to the team, is enough to see drops visually before they become incidents.

Freshness monitoring: dbt's source freshness feature (defined in schema.yml with freshness: warn_after: {count: 6, period: hour}) generates a JSON report on each dbt run. Parse this JSON in a post-run step and send an alert to Teams or Slack via webhook if any source fails freshness. This runs as part of your existing CI/CD pipeline — no additional tool required.

Azure Monitor for pipeline health: ADF, Databricks, and dbt (via dbt Cloud) emit diagnostic logs to Azure Monitor. Create alert rules in Azure Monitor that trigger on: ADF pipeline run failure (sends Teams message), Databricks job failure (sends Teams message), dbt build failure (sends Teams message via GitHub Actions). This gives you infrastructure-level observability at no additional cost beyond Azure Monitor's standard billing.

The escalation path: Start with this DIY stack. When your team grows beyond 5 data engineers, your Gold layer exceeds 50 tables, and you are spending more time triaging noisy alerts than building product — that is when the Monte Carlo or Soda investment conversation becomes justified. Until then, dbt tests + source freshness + Azure Monitor covers the essentials.

🏢Field Note
A Fintech Company  ·  dbt CI Cuts Production Incidents by 70%

A fintech data team was deploying dbt model changes directly to production without any CI gate. A developer would make a change locally, test it manually against their dev environment, and push to main. Production ran the next scheduled dbt job and picked up the change. This worked until it didn't: a change that passed local testing failed in production because it relied on a materialized table that existed in the developer's dev schema but not in production (it had been manually created during a debugging session and never cleaned up).

The team implemented a slim CI pipeline: on every PR to main, GitHub Actions ran dbt build --select state:modified+ --defer --state target/ against a dedicated CI database environment, using the previous production run's manifest as the state reference. The pipeline blocked merge if any test or build failed. The CI environment was isolated from production and automatically cleaned up after each run.

In the four months following implementation: three PRs were blocked by CI failures that would have caused production incidents. Two were logical errors (a join condition that produced duplicates — caught by a unique test). One was a missing column reference (a model referenced a field that didn't exist in the production source). All three would have generated incorrect data in production reports. The team's tracked production data incidents dropped by approximately 70% in the first quarter after implementation.

The Lesson

The CI investment is two days to set up — a GitHub Actions YAML file, a CI database environment, and dbt state artifacts from the last production run stored in Azure Blob Storage. The return is measured in production incidents that don't happen. The three failures caught in this team's first quarter of CI would each have required a weekend fix and an apologetic email to stakeholders.

Go DeeperDatabricks CI/CD with DABs — bundle.yml structure and deployment pattern

Databricks Asset Bundles (DABs) are the successor to the older dbx library and the even older "copy notebooks to a folder" deployment pattern. They define the complete Databricks workspace configuration — jobs, clusters, notebooks, libraries, and secrets — in a YAML bundle, versioned in Git, deployed via CLI.

Basic bundle.yml structure:

bundle:
  name: data-warehouse
targets:
  dev:
    workspace:
      host: https://adb-dev.azuredatabricks.net
  prod:
    workspace:
      host: https://adb-prod.azuredatabricks.net
resources:
  jobs:
    daily_silver_load:
      name: daily-silver-load
      schedule:
        quartz_cron_expression: "0 0 4 * * ?"
        timezone_id: Europe/London
      tasks:
        - task_key: silver_transform
          notebook_task:
            notebook_path: notebooks/silver_transform

Deployment commands:
databricks bundle validate — checks YAML syntax and resource references
databricks bundle deploy --target dev — deploys to dev workspace
databricks bundle run daily_silver_load --target dev — runs a job manually
databricks bundle deploy --target prod — deploys to production

GitHub Actions integration: Add a databricks/setup-cli action step, authenticate via service principal (client ID and secret stored as GitHub secrets), then call databricks bundle deploy --target prod on push to main. Every job definition, cluster config, and schedule is now in version control — reproducible, reviewable, and deployable in minutes after a workspace reset.

✓ Section Summary

DataOps applies DevOps principles to data pipelines: automated CI/CD, data quality monitoring, and observability. Requires version control (section 5.2) as a prerequisite. CI/CD pattern: dbt (slim CI on PR via state:modified+, full build on merge), Databricks (DABs — bundle.yml defines jobs/clusters, databricks bundle deploy on merge), ADF (ARM template export + az deployment group create — never the Publish button for production). Four observability categories: schema (columns present, types correct), freshness (table updated on schedule), volume (row counts within expected range), distribution (null rates, value ranges). Alert design: alert on deviation from baseline, not absolute thresholds; every alert must be actionable; start with fewer alerts than you think you need. Data quality tools: dbt tests first (free, YAML, catches most known-bad patterns); Great Expectations for non-dbt stacks; Soda for business-friendly alerting; Monte Carlo for enterprise ML-based anomaly detection; Datafold for PR-level data diff. CI/CD platforms: GitHub Actions (easiest for most data teams), Azure DevOps Pipelines (native Azure, best for Microsoft enterprises), GitLab CI (self-hosted option), Jenkins (legacy, high overhead).

Question 2 of 5

What is the correct approach for deploying an Azure Data Factory pipeline change to production — and what is wrong with clicking the Publish button after merging to the collaboration branch?

A
Clicking Publish is the correct approach; it packages and deploys the pipeline to production automatically
B
Merging to the collaboration branch deploys to production automatically in ADF Git mode
C
The correct approach is ARM template CI/CD: export the ARM template from ADF JSON files and deploy using az deployment group create via Azure Pipelines or GitHub Actions. Publish is problematic because it is a manual step that can be skipped, bypasses environment promotion (dev → test → prod), and is performed by whoever is logged in rather than tied to an approved PR
D
ADF changes cannot be automated — they must always be deployed manually via the Publish button

C is correct. The ADF Publish button has three problems in a production context: (1) it is manual — it can be forgotten, meaning the collaboration branch and production diverge silently; (2) it bypasses environment promotion — a developer can Publish directly to the production ADF instance without going through test; (3) it is not tied to the PR approval process — anyone with ADF Author permissions can Publish, regardless of whether their change has been reviewed. ARM template CI/CD solves all three: deployment is automated (no manual step to forget), goes through defined environments in sequence, and is triggered only on approved PR merge. Merging to the collaboration branch (B) does NOT deploy — this is the core misunderstanding covered in section 5.2.

Question 3 of 5

A data team is receiving 60 monitoring alerts per day. Engineers have started ignoring most of them. What is the fundamental design problem, and how should it be addressed?

A
The monitoring tool is misconfigured; switch to a different observability platform
B
The team needs more engineers to handle the alert volume
C
Alert fatigue: too many alerts, most non-actionable. Fix by reducing to only alerts that are actionable (a specific response is possible) and that signal genuine deviation from baseline — not absolute thresholds that fire on normal variance
D
Alerts should be sent to a dedicated channel that engineers check once per week rather than in real-time

C is correct. Alert fatigue is not a staffing problem — it is a design problem. When alerts fire for non-issues (seasonal volume variation, known acceptable null rates, informational events), engineers correctly learn that most alerts don't require action. But when a real issue fires, it is lost in the noise. The fix: audit every alert. For each one, ask: "If this alert fires at 2am, what specific action does the on-call engineer take?" If the answer is "nothing" or "it depends," the alert should not exist. Reduce to the minimum set of alerts that are genuinely actionable and signal genuine deviation. A monitoring system generating three alerts per day is more effective than one generating sixty — because every alert from the three-alert system is treated as significant.

Question 4 of 5

A data engineering team at a startup uses dbt, Databricks, and GitHub. They have no data quality monitoring in place. What is the correct order to implement monitoring, starting from the highest value for lowest effort?

A
Implement Monte Carlo first for ML-based anomaly detection, then add dbt tests
B
Implement Great Expectations across all pipeline layers simultaneously for comprehensive coverage
C
Add dbt tests (not_null, unique, accepted_values, relationships) to Gold tables first — free, YAML, runs in existing CI. Then add source freshness monitoring. Then add volume checks via a dbt audit model. Add observability platforms only when scale justifies the cost.
D
Build a custom Python observability framework specific to the team's stack before adopting any third-party tool

C is correct. The correct sequence follows the principle: highest value at lowest cost first, then expand as maturity grows. dbt tests on Gold tables are four lines of YAML per test, run for free in the CI pipeline that likely already exists, and catch the most common data quality failures (duplicate PKs, missing FKs, unexpected NULL values, new categorical values). Source freshness monitoring is another few lines of YAML in the dbt schema.yml. Both are zero incremental tooling cost for a team already running dbt. Volume monitoring via an audit model adds a small dbt model. Only when these layers are in place and scale demands more — more tables, more engineers, more sources — does the Monte Carlo or Soda investment conversation become justified. Starting with Monte Carlo (A) for a startup is the wrong tool for the maturity level and is unlikely to be used correctly without a foundation of basic dbt tests beneath it.

A
Azure DevOps Pipelines — source control is already in Azure DevOps Repos (native integration, no cross-platform auth needed), ADF has a built-in deployment task, and all Azure resources use service connections that are natively managed within Azure DevOps
B
GitHub Actions — it has a larger marketplace and is used by more data engineering teams globally
C
Either platform is equally suitable — the choice makes no practical difference
D
Neither — Jenkins is the enterprise standard for complex multi-tool CI/CD pipelines

A is correct. When source control is already in Azure DevOps Repos, Azure DevOps Pipelines is the path of least resistance: the pipeline can reference repository files, branch policies, and service connections without any cross-platform authentication configuration.
