3.2.1

SQL's Role in the Modern Data Stack

SQL is not going away. Despite the rise of Python and Spark, SQL remains the dominant language for data transformation — and for good reason. SQL executes set-based operations at the database or compute engine level. A GROUP BY in SQL pushes the aggregation work into Databricks' Photon engine or SQL Server's query processor, which applies vectorised computation across millions of rows simultaneously. The equivalent Python loop iterates row by row on a single thread. For data transformation on large datasets, SQL is almost always faster than the Python alternative.

In a Medallion architecture, SQL is the language of Silver and Gold layer transformations: in Databricks notebooks (spark.sql()), in dbt models (every model is a .sql file), and in Power BI calculated tables. Advanced SQL competency — specifically window functions, MERGE, and cross-platform awareness — is the single most transferable technical skill a data engineer can have.

The key insight about SQL: it has a logical execution order that differs from the written order. Understanding this order resolves most SQL confusion about what you can reference where. FROM and JOIN execute first. WHERE filters rows. GROUP BY aggregates. HAVING filters groups. SELECT defines the output columns. ORDER BY sorts. This is why you cannot reference a SELECT alias in a WHERE clause — WHERE runs before SELECT assigns the alias.
Figure 3.3SQL's logical execution order. Written top-down but executed bottom-up from the data source. The most important practical implication: SELECT aliases are defined in step 5, which means WHERE (step 2), GROUP BY (step 3), and HAVING (step 4) cannot reference them. This is why filtering on a window function alias requires wrapping it in a CTE (Section 2.1) — the alias only exists after SELECT runs.
3.2.2

Window Functions: Computing Across Rows Without Losing Them

A window function performs a calculation across a set of rows related to the current row, without collapsing those rows into a single output row. This is the fundamental distinction from GROUP BY: a GROUP BY produces one output row per group, discarding the individual rows. A window function preserves every input row and appends the computed value alongside each row.

All window functions share the same syntax structure: FUNCTION_NAME() OVER (PARTITION BY ... ORDER BY ... ROWS BETWEEN ...). The OVER clause is what makes it a window function. Without OVER, the same function (e.g. SUM) is an aggregate.

T-SQL / Spark SQL  ·  Window function anatomy and common functions
-- The window function skeleton
FUNCTION_NAME() OVER (
    PARTITION BY partition_col    -- defines the group (the "window")
    ORDER BY    order_col         -- defines row order within the group
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- the frame
)

-- ── Ranking functions ─────────────────────────────────────
SELECT
    customer_id,
    order_date,
    amount,

    -- Assigns a unique integer per row, no ties
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date) AS row_num,

    -- Ties get same rank, next rank skips (1, 1, 3)
    RANK()       OVER (PARTITION BY customer_id ORDER BY amount DESC) AS spend_rank,

    -- Ties get same rank, no skip (1, 1, 2)
    DENSE_RANK() OVER (PARTITION BY customer_id ORDER BY amount DESC) AS spend_dense,

    -- Divide into 4 buckets
    NTILE(4)     OVER (ORDER BY amount DESC) AS spend_quartile

FROM orders;

-- ── Offset functions ──────────────────────────────────────
SELECT
    order_date,
    amount,

    -- Previous row's value (NULL for first row)
    LAG(amount, 1, 0)  OVER (ORDER BY order_date) AS prev_amount,

    -- Next row's value (NULL for last row)
    LEAD(amount, 1, 0) OVER (ORDER BY order_date) AS next_amount,

    -- First value in the window
    FIRST_VALUE(amount) OVER (ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS first_amount

FROM orders;

-- ── Running aggregates ────────────────────────────────────
SELECT
    order_date,
    amount,

    SUM(amount)   OVER (ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total,

    AVG(amount)   OVER (ORDER BY order_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)         AS rolling_7_avg

FROM orders;
Figure 3.4Window function anatomy. PARTITION BY divides rows into groups (like GROUP BY but without collapsing them). ORDER BY defines row sequence within each partition. The FRAME clause (ROWS BETWEEN) defines which rows contribute to aggregate calculations. For ranking functions (ROW_NUMBER, RANK), no frame is needed. For running aggregates (SUM OVER, AVG OVER), the frame controls the rolling window size.

The most common practical application of ROW_NUMBER is deduplication and "latest record per entity" queries — two operations that appear constantly in Silver layer transformations:

T-SQL / Spark SQL  ·  Deduplication with ROW_NUMBER
-- Pattern: keep only the most recent record per customer
-- (cannot use WHERE row_num = 1 in same SELECT — must use CTE or subquery)
WITH ranked_customers AS (
    SELECT
        *,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id
            ORDER BY updated_at DESC    -- most recent first
        ) AS row_num
    FROM bronze.crm_customers
)
SELECT * FROM ranked_customers WHERE row_num = 1;

-- Pattern: detect changes between current and previous row (SCD change detection)
WITH ordered AS (
    SELECT
        customer_id,
        region,
        updated_at,
        LAG(region) OVER (PARTITION BY customer_id ORDER BY updated_at) AS prev_region
    FROM bronze.crm_customers
)
SELECT * FROM ordered WHERE region != prev_region OR prev_region IS NULL;
3.2.3

MERGE: The Upsert Statement

A MERGE statement (cross-referenced from Section 2.3: Upsert) combines INSERT and UPDATE logic in a single atomic operation. It reads a source dataset and applies changes to a target: updating rows that exist in both (WHEN MATCHED), inserting rows that exist only in source (WHEN NOT MATCHED), and optionally deleting rows that exist only in target (WHEN NOT MATCHED BY SOURCE).

T-SQL  ·  MERGE for SCD Type 1 (overwrite)
-- Apply CRM updates to Silver customer dimension (SCD Type 1 — overwrite)
MERGE INTO silver.dim_customer AS target
USING (
    -- source: latest record per customer from Bronze
    SELECT customer_id, name, email, region, updated_at
    FROM bronze.crm_customers
    WHERE ingestion_date = CAST(GETDATE() AS DATE)
) AS source
ON target.customer_id = source.customer_id

WHEN MATCHED AND (
    target.email  != source.email  OR
    target.region != source.region
) THEN UPDATE SET
    target.name       = source.name,
    target.email      = source.email,
    target.region     = source.region,
    target.updated_at = source.updated_at

WHEN NOT MATCHED BY TARGET THEN INSERT (
    customer_id, name, email, region, updated_at
) VALUES (
    source.customer_id, source.name, source.email,
    source.region, source.updated_at
);
Spark SQL (Delta Lake)  ·  MERGE INTO
-- Delta Lake MERGE (identical concept, slightly different syntax)
MERGE INTO silver.dim_customer AS target
USING (
    SELECT customer_id, name, email, region, updated_at
    FROM bronze.crm_customers
    WHERE ingestion_date = current_date()
) AS source
ON target.customer_id = source.customer_id

WHEN MATCHED THEN UPDATE SET *   -- update all matching columns

WHEN NOT MATCHED THEN INSERT *;  -- insert all columns from source

-- WHEN NOT MATCHED BY SOURCE (T-SQL only — not in Spark SQL):
-- THEN DELETE  -- removes target rows with no source match
3.2.4

Performance: Anti-Patterns and EXPLAIN Plans

SQL performance problems are almost always caused by a small set of recurring mistakes. Knowing these patterns by name lets you identify them in unfamiliar code immediately.

Functions on indexed/partition columns in WHERE. A filter like WHERE YEAR(order_date) = 2024 prevents the engine from using the index or partition pruning on order_date. The function must be evaluated for every row before filtering can happen. Replace with range filters: WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'.

SELECT *. Selects all columns, including those not needed. For wide tables in columnar formats (Parquet), SELECT * defeats column pruning — every column is read from disk. In analytical queries, specify only the columns you need.

Implicit conversions. When a filter compares columns of different types (e.g. a string literal compared to an integer column), the engine must convert every value before comparing. This defeats indexes. Match types explicitly.

Correlated subqueries. A subquery that references the outer query executes once per outer row. For a 10 million-row outer table, a correlated subquery runs 10 million times. Replace with a JOIN or a window function.

OR in JOIN conditions. A JOIN ON a.id = b.id OR a.id = b.alt_id cannot use an index efficiently. Rewrite as a UNION of two simple joins.

Go DeeperReading EXPLAIN plans in T-SQL and Spark SQL

An EXPLAIN plan shows how the database engine will execute a query before running it. It is the first tool to reach for when a query is unexpectedly slow.

In SQL Server / Azure SQL: run SET SHOWPLAN_XML ON before your query, or use SQL Server Management Studio (SSMS) to display the graphical execution plan. Key things to look for: any operator with a thick arrow (many rows flowing), an Index Scan instead of Index Seek (scanning the whole index vs looking up specific values), and the cost percentage next to each operator (the highest-cost operator is usually where to optimise).

In Spark SQL / Databricks: run EXPLAIN EXTENDED your_query to see the logical plan and physical plan as text. More usefully, run the query and then click the "View" link next to the SQL command in the Databricks UI to see the Spark DAG visualisation. Look for: large shuffle stages (wide arrows between stages in the DAG), skewed tasks (one task much longer than others in the Spark UI task view), and Sort steps immediately before joins (can indicate missing broadcast hint).

The most valuable EXPLAIN pattern for data engineers: a full table scan where you expected partition pruning. If EXPLAIN shows every partition being read despite a date filter in your WHERE clause, the filter is on a derived expression and predicate pushdown is not firing. This is Section 2.2's partition pruning content applied in practice.

🏢Field Note
A Retail Analytics Team  ·  The Window Function Refactor

A retail company's data team had a weekly sales report query that took 47 minutes to run. The query calculated each store's rank by weekly revenue, compared this week's revenue to last week's, and flagged stores whose revenue had dropped more than 10% week-over-week. The original implementation used multiple GROUP BY subqueries: one to calculate weekly revenue per store, joined to itself on the same store ID with a date offset for last week's figures, joined again to a separately calculated rank table.

A senior engineer rewrote the query using window functions: SUM(amount) OVER (PARTITION BY store_id ORDER BY week) for weekly totals, LAG(weekly_revenue, 1) over the same partition for prior week's figure, and RANK() OVER (ORDER BY weekly_revenue DESC) for the ranking. The four-subquery join became a single CTE with three window function columns. Runtime dropped from 47 minutes to 3 minutes. No schema changes, no indexing, no hardware upgrades — just correct use of window functions where multiple self-joins had been.

The Lesson

Window functions are not just a syntactic convenience. They enable the query engine to make a single pass through the data for what previously required multiple passes and joins. When you find yourself joining a table to itself for date offsets or ranks, reach for LAG/LEAD/RANK instead.

🏢Field Note
A Financial Services Team  ·  The Hidden Full Table Scan

A financial services team had an overnight pipeline that processed transaction data for regulatory reporting. The main query filtered on WHERE CONVERT(VARCHAR, transaction_date, 112) = '20240115' — converting the date column to a string before comparing, a pattern common in legacy T-SQL that developers had been copying without questioning. The transactions table had 800 million rows partitioned by transaction_date. The query consistently took 4 hours.

An EXPLAIN plan review showed an Index Scan across all partitions rather than an Index Seek on the specific date. The CONVERT function on the indexed column prevented the engine from using partition elimination. Changing the filter to WHERE transaction_date = '2024-01-15' (a direct date comparison) reduced the query to 6 minutes by reading only the 20 million rows in the January 15th partition.

The Lesson

Functions applied to columns in WHERE clauses defeat indexes and partition pruning. This is the single most common SQL performance mistake in legacy pipelines migrated from on-premise. EXPLAIN before you optimise. The fix is usually a one-line change.

Go DeeperPIVOT and UNPIVOT — rotating rows into columns and back

PIVOT rotates rows into columns: useful when source data has one row per metric per entity but the report needs one column per metric per entity. UNPIVOT does the reverse. Both appear commonly when integrating data from systems that store data in "long format" (many rows, few columns) but analysts need "wide format" (few rows, many columns).

In T-SQL, PIVOT is a native operator: SELECT * FROM (SELECT customer_id, month, revenue FROM orders) AS src PIVOT (SUM(revenue) FOR month IN ([Jan],[Feb],[Mar])) AS pvt. The limitation: column list in FOR must be static at query write time — dynamic PIVOT requires dynamic SQL string construction.

In Spark SQL, there is no native PIVOT keyword (though PySpark DataFrames have a .pivot() method). The SQL equivalent uses CASE WHEN month = 'Jan' THEN revenue END repeated for each pivot column — verbose but explicit. For data engineering transformations, the PySpark .pivot() method with an explicit values list is cleaner and allows runtime-determined column names.

+
Go DeeperROWS BETWEEN vs RANGE BETWEEN — the subtle frame difference

The frame clause (ROWS BETWEEN or RANGE BETWEEN) controls which rows contribute to a window aggregate. The difference between ROWS and RANGE matters when there are ties in the ORDER BY column.

ROWS BETWEEN defines the frame by physical row position — exactly N rows before and after the current row, regardless of whether those rows have the same ORDER BY value. Predictable, deterministic, usually what you want for running totals.

RANGE BETWEEN defines the frame by logical value range — all rows with ORDER BY values within the specified range of the current row's value. With duplicate ORDER BY values, RANGE includes ALL rows with the same value in the frame, even if that means including more rows than expected. The default for most window functions when you omit the frame clause is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — which for a running total with ties will sum to the same value as the last tied row for all tied rows.

Practical guidance: when writing running totals or running averages, always specify ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW explicitly. This gives deterministic, tie-insensitive results. Never rely on the default RANGE behaviour unless you specifically want tie-inclusive frames.

Question 1 of 5

You write SELECT customer_id, SUM(amount) AS total, ROW_NUMBER() OVER (ORDER BY total DESC) AS rn FROM orders GROUP BY customer_id. What happens?

A
It works — total is computed by GROUP BY and then used in the window function
B
It fails — the window function alias total is referenced in ORDER BY before SELECT defines it, violating SQL logical execution order
C
It works in Spark SQL but fails in T-SQL
D
It fails only if the table has NULL values

B is correct. SQL logical execution order: GROUP BY runs before SELECT. The alias total is defined in SELECT (step 5) and the window function's ORDER BY runs as part of SELECT (step 5) — but the issue is that ROW_NUMBER() OVER (ORDER BY total) references the alias total which doesn't exist at the point the window function is being computed. The engine raises a column not found error. The fix: wrap in a CTE. WITH agg AS (SELECT customer_id, SUM(amount) AS total FROM orders GROUP BY customer_id) SELECT *, ROW_NUMBER() OVER (ORDER BY total DESC) AS rn FROM agg.

Question 2 of 5

What is the difference between RANK() and DENSE_RANK() when two rows have the same value?

A
RANK() and DENSE_RANK() are identical — different names for the same function
B
RANK() assigns the same number to ties and skips the next rank (1,1,3). DENSE_RANK() assigns the same number to ties and does not skip (1,1,2)
C
DENSE_RANK() assigns the same number to ties and skips the next rank. RANK() does not skip
D
RANK() works row by row. DENSE_RANK() collapses tied rows into one

B is correct. With values [100, 100, 80]: RANK() produces [1, 1, 3] (two rows tied at first, next rank is 3). DENSE_RANK() produces [1, 1, 2] (two rows tied at first, next rank is 2 with no gap). RANK() reflects true ordinal position (if two people tie for gold, the next person is bronze — no silver). DENSE_RANK() reflects tier position (if two people tie for first, the next person is second — no gap). Choose based on which business definition of "rank" applies.

Question 3 of 5

A MERGE statement's WHEN NOT MATCHED BY SOURCE THEN DELETE clause deletes rows from the target that have no match in the source. What is a critical risk of this clause?

A
It only works in Spark SQL, not T-SQL
B
If the source query returns an incomplete dataset (e.g. only today's changed records, not all records), rows not in today's source are incorrectly deleted from target even though they are valid historical records
C
It is limited to deleting 1,000 rows per execution
D
It deletes target rows before the WHEN MATCHED UPDATE runs, causing referential integrity errors

B is correct. WHEN NOT MATCHED BY SOURCE deletes target rows with no source match. This is correct when source is a full extract of all current records. It is dangerously wrong when source is an incremental extract (only changed records). In that case, unchanged records are simply absent from today's source — and would all be deleted. This is a very common production bug. Note: this clause is T-SQL only; Delta Lake's MERGE INTO does not support it.

Question 4 of 5

What does LAG(amount, 1, 0) OVER (PARTITION BY customer_id ORDER BY order_date) return for the first order of a customer?

A
NULL, because there is no previous row
B
0, because the third argument to LAG is the default value returned when there is no previous row
C
The value from the last row of the previous customer's partition
D
An error, because LAG requires at least two rows in the partition

B is correct. LAG(column, offset, default) takes three arguments: the column, the number of rows to look back, and the default value to return when there is no previous row (the offset goes before the first row of the partition). Without the third argument, LAG returns NULL for the first row. With 0 as the third argument, it returns 0 instead. This is useful for running difference calculations where NULL would propagate incorrectly.

Question 5 of 5

You are migrating a T-SQL query to Spark SQL in Databricks. The query includes WITH history AS (...) using a recursive CTE. What change is required?

A
No change — Spark SQL supports the same recursive CTE syntax as T-SQL
B
Change to WITH RECURSIVE history AS (...) — Spark SQL requires the explicit RECURSIVE keyword
C
Recursive CTEs are not supported in Spark SQL — rewrite as a Python loop
D
Change UNION ALL to UNION in the recursive member

B is correct. Spark SQL 3.0+ supports recursive CTEs, but requires the explicit RECURSIVE keyword: WITH RECURSIVE cte_name AS (...). T-SQL implies recursion without the keyword. This is covered in Section 2.1 (CTEs) and is the most common CTE porting error when migrating T-SQL pipelines to Databricks.

✓ Section Summary

SQL remains the dominant language for data transformation. Window functions (ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM OVER) compute across row sets without collapsing them — the key distinction from GROUP BY. SQL's logical execution order (FROM → WHERE → GROUP → HAVING → SELECT → ORDER) explains why SELECT aliases are not available in WHERE. MERGE combines INSERT and UPDATE atomically; WHEN NOT MATCHED BY SOURCE is safe for full extracts but dangerous for incremental ones. The five major anti-patterns: functions on indexed columns, SELECT *, implicit conversions, correlated subqueries, OR in JOIN conditions. Key T-SQL vs Spark SQL differences: LIMIT vs TOP, WITH RECURSIVE, date function syntax.

Execution Order

SQL written order ≠ SQL execution order. FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. This is why you cannot reference a SELECT alias in WHERE.

Function on Column

WHERE YEAR(date_col) = 2024 defeats partition pruning and indexes. Always use range filters: WHERE date_col >= '2024-01-01'.

Window vs GROUP BY

GROUP BY collapses rows. Window functions preserve them and append the computed value. When you need both the row detail AND a rank/running total, window functions are always the right tool.

MERGE Risk

WHEN NOT MATCHED BY SOURCE deletes unmatched target rows. Safe for full extracts; catastrophic for incremental ones.
