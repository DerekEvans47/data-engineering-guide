3.5.1

PySpark: The Python Interface to Spark

PySpark is the Python API for Apache Spark. Everything in Section 3.4 — the Driver, Executors, DAG, shuffle — is the engine. PySpark is how you program that engine in Python. Most data engineers on Databricks write PySpark rather than Scala (Spark’s native language) because Python’s ecosystem, hiring pool, and ML library compatibility outweigh Scala’s modest speed advantage for the vast majority of production workloads.

In Databricks notebooks, spark is pre-initialised — no setup needed. In standalone scripts or unit tests, you create it explicitly:

PySpark  ·  SparkSession
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("SilverTransform") \
    .config("spark.sql.shuffle.partitions", "200") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

# In Databricks notebooks, spark is already available
print(spark.version)

DataFrames and Schema

A Spark DataFrame is a distributed, immutable dataset organised into named columns. Operations return new DataFrames rather than modifying the original. Reading with schema inference is convenient but fragile in production — it reads data twice and can infer wrong types. Explicit schemas are the production best practice.

PySpark  ·  Reading data with explicit schema
from pyspark.sql.types import (StructType, StructField, LongType,
                                StringType, DecimalType, DateType)
from pyspark.sql import functions as F

order_schema = StructType([
    StructField("order_id",    LongType(),        nullable=False),
    StructField("customer_id", LongType(),        nullable=False),
    StructField("order_date",  DateType(),        nullable=False),
    StructField("amount",      DecimalType(12,2), nullable=True),
    StructField("status",      StringType(),      nullable=True),
    StructField("region",      StringType(),      nullable=True),
])

# Read from ADLS with explicit schema
df = spark.read.schema(order_schema).format("parquet") \
         .load("abfss://bronze@lake.dfs.core.windows.net/orders/")

# Read from Delta via Unity Catalog (preferred in production)
df = spark.table("prod.silver.orders")
3.5.3

Transformations vs Actions: Lazy Evaluation in Practice

Every PySpark method is either a transformation (lazy — adds a node to the DAG) or an action (triggers the DAG to execute). This is the most important distinction in PySpark and the source of most debugging confusion for newcomers.

PySpark  ·  Transformations and Actions
# ── Transformations: nothing executes ────────────────────
df2 = df.filter(F.col("status") == "completed")           # lazy
df3 = df2.withColumn("year", F.year(F.col("order_date"))) # lazy
df4 = df3.groupBy("region","year").agg(                   # lazy
    F.sum("amount").alias("total_revenue"),
    F.count("*").alias("order_count")
)
# At this point: zero bytes have been read. A DAG exists.

# ── Actions: trigger full DAG execution ──────────────────
df4.count()                    # executes, returns integer
df4.show(5)                    # executes, prints rows
df4.collect()                  # executes, returns list to Driver (careful!)

# Write is the most common production action
df4.write.format("delta") \
   .mode("overwrite") \
   .save("abfss://gold@lake.dfs.core.windows.net/revenue/")
Figure 3.9PySpark lazy evaluation. Every transformation adds a node to the DAG without executing. The action (.write, .count, .show) triggers Spark to optimise and execute the full accumulated plan. Time Spark pipelines at the action — not at individual transformation calls, which appear instantaneous because they do nothing yet.

Common DataFrame Operations

PySpark  ·  Core operations reference
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# ── Selection and column work ─────────────────────────────
df.select("order_id", "customer_id", "amount")
df.withColumn("year",  F.year("order_date"))
df.withColumn("month", F.month("order_date"))
df.drop("internal_flag")
df.withColumnRenamed("cust_id", "customer_id")

# ── Filtering ──────────────────────────────────────────────
df.filter(F.col("status") == "completed")
df.filter(F.col("amount") > 100)
df.filter(F.col("region").isin(["EMEA", "AMER"]))
df.filter(F.col("email").isNotNull())

# ── Joins ──────────────────────────────────────────────────
df_orders.join(df_customers, on="customer_id", how="inner")
df_orders.join(df_customers, on="customer_id", how="left")

# Broadcast join — small lookup table to every executor (no shuffle)
from pyspark.sql.functions import broadcast
df_orders.join(broadcast(df_country_codes), on="country_code")

# ── Aggregation ────────────────────────────────────────────
df.groupBy("region").agg(
    F.sum("amount").alias("total_revenue"),
    F.count("*").alias("order_count"),
    F.countDistinct("customer_id").alias("unique_customers"),
    F.max("order_date").alias("latest_order"),
)

# ── Window functions ───────────────────────────────────────
w = Window.partitionBy("customer_id").orderBy("order_date")
df.withColumn("row_num",     F.row_number().over(w)) \
  .withColumn("prev_amount", F.lag("amount", 1).over(w)) \
  .withColumn("running_total", F.sum("amount").over(
      w.rowsBetween(Window.unboundedPreceding, Window.currentRow)))

# ── Writing Delta ──────────────────────────────────────────
df.write.format("delta").mode("append") \
    .partitionBy("order_date") \
    .save("abfss://silver@lake.dfs.core.windows.net/orders/")

df.write.saveAsTable("prod.silver.orders")   # Unity Catalog
3.5.5

Delta MERGE from PySpark

PySpark  ·  Upsert with Delta MERGE
from delta.tables import DeltaTable

target = DeltaTable.forPath(
    spark, "abfss://silver@lake.dfs.core.windows.net/customers/")

source = spark.table("prod.bronze.customers") \
              .filter(F.col("ingestion_date") == F.current_date())

target.alias("t") \
    .merge(source.alias("s"), "t.customer_id = s.customer_id") \
    .whenMatchedUpdate(set={
        "t.name":       "s.name",
        "t.email":      "s.email",
        "t.region":     "s.region",
        "t.updated_at": "s.updated_at",
    }) \
    .whenNotMatchedInsertAll() \
    .execute()
Figure 3.10PySpark join types. inner, left, full, and left_anti follow SQL semantics. The broadcast hint eliminates shuffle for small tables — the most impactful single-line performance optimisation in PySpark. Spark auto-broadcasts tables under 10MB; use the explicit hint for small dimension tables above that threshold.
🏢Field Note
A Manufacturing Team  ·  The Broadcast Join That Saved 4 Hours

A data engineering team joined a 2.4-billion-row transaction fact table to a 40,000-row product dimension in their nightly Gold pipeline. The pipeline used a standard inner join with no hint. The resulting sort-merge join — where Spark sorted and shuffled the fact table across all Executors — took 4 hours 22 minutes. The Spark UI showed this one join stage consuming 78% of total runtime.

The fix was one line: wrapping the product dimension in broadcast(). Spark sent the 40,000-row table (around 12MB serialised) to every Executor, eliminating the 2.4-billion-row shuffle. Runtime dropped to 28 minutes. No schema changes, no infrastructure changes, no additional cost.

The Lesson

Always check the Spark UI for sort-merge joins where one side is orders of magnitude smaller. The fix is almost always a broadcast hint. It is the highest return-on-investment single-line change in PySpark.

🏢Field Note
A Retail Team  ·  Schema Inference Breaking Overnight Pipelines

A retailer's Bronze-to-Silver pipeline used schema inference for all Parquet reads. The pipeline ran stably for six months. In November, a source system change added a column called discount_pct that was populated with integers in most rows but contained a string "N/A" for legacy product lines. Schema inference sampled the first rows, saw integers, inferred the column as LongType, and then failed with a type cast error when it encountered the string value mid-file.

The pipeline failed silently at 3am, was discovered at 8am by the Finance team when their daily revenue report was empty, and took two hours to diagnose because the error message pointed to a serialisation exception rather than the source column. The fix was an explicit schema definition where discount_pct was declared as StringType with downstream cleaning logic in Silver.

The Lesson

Schema inference is a development convenience, not a production pattern. Every Bronze read in a production pipeline should have an explicit schema. When source schemas change, you get a clear error at the schema definition rather than a cryptic serialisation failure mid-execution.

Go DeeperCaching and persisting DataFrames

If you reference the same DataFrame multiple times in a pipeline (once for a count check, once for a write), Spark re-executes the full lineage from the source for each reference by default. df.cache() materialises the DataFrame in Executor memory after the first action, so subsequent actions use the cached version.

df.persist(StorageLevel.MEMORY_AND_DISK) allows overflow to disk if the DataFrame is too large for Executor memory. df.unpersist() releases the cache explicitly. On Databricks, cached DataFrames persist across notebook cells until unpersisted or until the cluster restarts — accumulated caches silently consume Executor memory over a session.

Use caching when: the same expensive DataFrame (complex join, large read) is referenced in two or more downstream actions. Do not cache DataFrames that are used only once — caching adds overhead with no benefit.

+
Go Deeperrepartition() vs coalesce() for DataFrames

Covered in detail in Section 2.2 for storage partitioning — the same principles apply to PySpark DataFrames in memory.

df.repartition(n) performs a full shuffle: all data is redistributed across n new partitions. Use to increase partition count or to redistribute data by a specific column (df.repartition(200, "customer_id") ensures all rows for a customer land on the same partition before a groupBy or join).

df.coalesce(n) reduces partition count without a full shuffle by merging partitions on the same Executor. Use after filtering a large DataFrame down to a small result. Coalesced partitions may be uneven in size.

The typical production pattern: after a large filter that reduces rows by 90%, coalesce before writing to avoid writing many tiny Parquet files. df.filter(...).coalesce(20).write.format("delta")...

+
Go Deeperschema evolution in Delta writes

Delta Lake enforces schema-on-write by default: a write fails if the incoming DataFrame has columns not present in the existing table. Two options to allow evolution:

.option("mergeSchema", "true") on the write: adds new columns to the Delta table schema, writes NULL for those columns in existing rows. Safe for additive changes. Use on Bronze tables to absorb source schema changes.

.option("overwriteSchema", "true") with .mode("overwrite"): replaces the table schema entirely. Dangerous — destroys existing column definitions. Only appropriate when intentionally redesigning a table.

Production guidance: Bronze allows mergeSchema. Silver schema changes should be reviewed (they represent changes to the shared trusted data contract). Gold enforces strict schema because Power BI reports depend on column names and types remaining stable.

Question 1 of 5

You write df.filter(F.col("status") == "completed"). What happens when this line executes?

A
Spark scans the DataFrame and removes non-matching rows immediately
B
Spark records this transformation in the DAG — no data is read or processed yet
C
Spark validates that the "status" column exists and contains string values
D
Spark reads a sample of rows to estimate filter selectivity

B is correct. filter() is a transformation — it is lazy. Spark adds a filter node to the DAG and returns immediately. No data is read, no computation occurs. The transformation executes only when an action (write, count, show) is called. This is why transformation calls appear instantaneous in notebook timing.

Question 2 of 5

A 250MB customer dimension needs to be joined to a 2-billion-row fact table. What is the most effective PySpark optimisation?

A
Increase spark.sql.shuffle.partitions to 2000
B
Use df_fact.join(broadcast(df_customers), on="customer_sk") to send the dimension to every Executor and eliminate the fact table shuffle
C
Write the dimension to Delta first, then join using spark.table()
D
Repartition the fact table by customer_sk before joining

B is correct. The broadcast hint sends the dimension (250MB) to every Executor, eliminating the need to shuffle the 2-billion-row fact table. The automatic broadcast threshold is 10MB — the hint explicitly tells Spark to broadcast above that threshold. This is the single highest-impact optimisation for large-to-small-table joins. Increasing shuffle partitions (option A) changes partition count but does not eliminate the shuffle itself.

Question 3 of 5

What does .option("mergeSchema", "true") do on a Delta write?

A
Merges duplicate rows by a specified key before writing
B
Adds new columns from the source DataFrame to the Delta table schema; existing rows receive NULL for the new columns
C
Overwrites the entire Delta table schema with the new DataFrame schema
D
Prevents writing if the schema has changed since the last run

B is correct. mergeSchema enables additive schema evolution: new columns are added to the Delta table, existing rows get NULL for those columns. It does not merge rows (that is MERGE INTO). It does not overwrite the schema (that is overwriteSchema). Without it, a write with new columns fails with a schema mismatch error. Use mergeSchema on Bronze to absorb source schema changes; use strict enforcement on Silver and Gold.

Question 4 of 5

You cache a DataFrame and use it in two write operations. The notebook finishes running. When is the cache released?

A
When the notebook cell containing .cache() finishes
B
After 60 minutes of inactivity
C
When explicitly called with df.unpersist(), or when the cluster restarts
D
When the next notebook opens on the same cluster

C is correct. Spark caches persist in Executor memory until explicitly released with df.unpersist() or until the cluster restarts. They do not release when a cell or notebook completes. Over a long session, accumulated caches consume Executor memory and eventually cause out-of-memory errors or slow processing. Always call df.unpersist() when a cached DataFrame is no longer needed.

Question 5 of 5

A PySpark pipeline reads Bronze Parquet without specifying a schema. A source system adds a new column containing mostly integers but "N/A" strings for legacy records. What happens?

A
Spark reads the column as StringType and the pipeline continues without error
B
Spark infers LongType from early rows, fails with a type cast error when it encounters "N/A", producing a confusing runtime exception
C
Spark skips rows containing "N/A" and processes the rest
D
Spark automatically widens the type to StringType when it encounters the mixed values

B is correct. Schema inference samples early rows, sees integers, and infers LongType. When it later encounters "N/A" it attempts to cast it to Long, fails, and throws a serialisation exception — often with an error message that does not clearly identify the column. This is why explicit schemas are the production standard. With an explicit schema declaring the column as StringType, the pipeline accepts all values and the cleaning logic in Silver handles the "N/A" to NULL conversion.

✓ Section Summary

PySpark is Python's interface to Apache Spark. Transformations (filter, select, join, groupBy, withColumn) are lazy — they build a DAG. Actions (write, count, show) trigger DAG execution. Always define explicit schemas on Bronze reads rather than relying on inference. broadcast() on a small dimension table eliminates the shuffle of a large fact table — the single most impactful single-line PySpark optimisation. Delta MERGE handles upserts atomically. Cache DataFrames used multiple times; always call unpersist() when done. mergeSchema allows additive column additions to existing Delta tables.

Lazy Rule

Transformations build the DAG. Actions run it. If a Spark line appears instantaneous, it is a transformation. Time your pipeline at the write step.

Schema Inference

Never use schema inference in production. It reads data twice and infers wrong types on mixed-value columns. Always define an explicit StructType.

Broadcast First

Before adding cluster nodes to fix a slow join, check the Spark UI. A sort-merge join where one side is small is a broadcast candidate. One line fix, 10x speed improvement.

unpersist()

df.cache() persists until df.unpersist() or cluster restart. Accumulated caches silently consume Executor memory. Release caches when done.
