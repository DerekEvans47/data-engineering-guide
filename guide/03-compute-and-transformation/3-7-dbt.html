dbt — data build tool

SQL transformation with software engineering practices: tests, documentation, version control, and CI/CD for your data models

3.7.1

What dbt Is — and Why It Exists

dbt (data build tool) is a transformation framework that applies software engineering practices to SQL-based data transformation. It does one thing: it takes .sql SELECT statements and runs them against your data warehouse or lakehouse, materialising the results as tables or views. What makes dbt different from a folder of SQL files is everything around the SELECT: version control (every model is a file in Git), automated dependency management (ref() builds the DAG), built-in testing (schema.yml), generated documentation (dbt docs), and CI/CD integration (dbt on pull requests).

dbt does not move data. It does not orchestrate pipelines end-to-end. It does not ingest from sources. It transforms data that is already in your lakehouse or data warehouse. In a Medallion architecture, dbt typically operates on Silver data to produce Gold, or on staged Bronze data to produce cleaned Silver. ADF brings data in; Databricks (or dbt on Databricks) transforms it.

The most powerful thing dbt does that a SQL file cannot: the ref() function. When you write ref("dim_customer") instead of the table name, dbt knows this model depends on dim_customer. dbt builds a directed acyclic graph (DAG) of all dependencies, executes models in the correct order, and knows which models are affected when you change another. This is the core of dbt's value proposition.
🏢Field Note
A Financial Services Team  ·  Replacing 200 Stored Procedures

A financial services firm's Silver-to-Gold transformation layer was implemented as 200 SQL Server stored procedures, organised by a verbal convention ("run spr_staging_customers before spr_dim_customer") documented in a Word document. The procedures had been written over five years by six different engineers. No procedure had tests. None generated documentation automatically. Changing one procedure required manually tracing all downstream consumers to understand the blast radius.

The team migrated to dbt over six months. Each procedure became a dbt model: a single .sql file containing a SELECT statement. Dependencies were declared via ref() rather than assumed by convention. dbt generated the execution DAG automatically. Schema tests (not_null, unique, accepted_values) were added to critical columns. Running dbt docs generate produced an interactive documentation site with the full DAG visualisation and column-level descriptions. When engineers proposed changes, dbt ls --select state:modified+ showed exactly which downstream models would be affected.

The Lesson

dbt transforms a collection of undocumented SQL scripts into a governed, tested, documented, DAG-aware transformation layer. The migration cost is significant but the long-term maintainability benefit compounds with every new model added.

Project Structure and Core Concepts

dbt project  ·  Directory structure
my_dbt_project/
├── dbt_project.yml          # Project configuration (name, version, model paths)
├── packages.yml             # Third-party package dependencies
├── profiles.yml             # Connection config (usually in ~/.dbt/ not in repo)
│
├── models/                  # .sql transformation files
│   ├── staging/             # 1:1 with source tables, light cleaning
│   │   ├── stg_crm_customers.sql
│   │   └── schema.yml       # source definitions + tests
│   ├── intermediate/        # Business logic, joins
│   │   └── int_customer_orders.sql
│   └── marts/               # Final Gold-equivalent tables
│       ├── dim_customer.sql
│       └── fact_orders.sql
│
├── seeds/                   # Static CSV files loaded as reference tables
│   └── country_codes.csv
├── tests/                   # Singular (custom SQL) tests
│   └── assert_no_negative_revenue.sql
├── macros/                  # Reusable Jinja functions
│   └── generate_surrogate_key.sql
└── snapshots/               # SCD Type 2 definitions
    └── snap_customers.sql
SQL  ·  dbt model — a SELECT statement that becomes a table
-- models/marts/dim_customer.sql
-- dbt materialises this SELECT as a table (based on dbt_project.yml config)

WITH source AS (
    -- source() references a declared external source (Bronze / external table)
    SELECT * FROM {{ source('bronze', 'crm_customers') }}
),

cleaned AS (
    SELECT
        customer_id,
        TRIM(LOWER(email))          AS email,
        INITCAP(name)               AS name,
        COALESCE(region, 'UNKNOWN') AS region,
        CAST(created_date AS DATE)  AS created_date
    FROM source
    WHERE customer_id IS NOT NULL
),

final AS (
    -- ref() references another dbt model — builds the DAG automatically
    SELECT
        {{ dbt_utils.generate_surrogate_key(['customer_id']) }} AS customer_sk,
        c.*,
        o.order_count,
        o.total_spend
    FROM cleaned AS c
    LEFT JOIN {{ ref('int_customer_orders') }} AS o
        ON c.customer_id = o.customer_id
)

SELECT * FROM final

The two key functions in every dbt model:

{{ source('source_name', 'table_name') }} references a table defined in a sources.yml declaration. dbt tracks freshness, generates documentation, and creates lineage nodes for source references. Use source() for anything external to dbt (Bronze Delta tables, external databases).

{{ ref('model_name') }} references another dbt model. This is how dbt builds the DAG: every ref() creates a dependency edge. dbt resolves the correct target schema and database automatically based on the environment (dev vs prod), and ensures all upstream models run before the current one.

schema.yml: Documentation and Tests in One File

dbt schema.yml  ·  Tests and documentation together
# models/marts/schema.yml
version: 2

models:
  - name: dim_customer
    description: "One row per customer. Slowly changing (Type 1 overwrite)."
    columns:
      - name: customer_sk
        description: "Surrogate key. Primary join key for all fact tables."
        tests:
          - unique
          - not_null

      - name: email
        description: "Customer email. Normalised to lowercase."
        tests:
          - not_null
          - unique

      - name: region
        description: "Business region."
        tests:
          - accepted_values:
              values: ["EMEA", "AMER", "APAC", "UNKNOWN"]

      - name: total_spend
        description: "Lifetime spend in GBP. NULL if customer has no orders."
        tests:
          - not_null:
              where: "order_count > 0"  # conditional test

sources:
  - name: bronze
    description: "Raw Bronze layer in Unity Catalog."
    tables:
      - name: crm_customers
        freshness:
          warn_after: {count: 2, period: hour}
          error_after: {count: 6, period: hour}
3.7.3

Materializations: How dbt Persists Results

A materialisation is how dbt writes the result of a model to the database. Four materialisation types cover most use cases, and choosing correctly has significant performance and cost implications.

Figure 3.13dbt project DAG. source() references external tables (Bronze Delta tables, external databases). Staging models (typically views) perform light cleaning 1:1 with source tables. Intermediate models (typically tables) join and apply business logic. Mart models (Gold-equivalent) are the final business-facing outputs consumed by Power BI and other tools. ref() between models creates the dependency edges — dbt executes the DAG in topological order.
dbt  ·  Materialization types and when to use each
-- dbt_project.yml — set defaults per folder
models:
  my_project:
    staging:
      +materialized: view           # No storage cost; always fresh
    intermediate:
      +materialized: table          # Pre-computed; faster for downstream
    marts:
      +materialized: incremental    # Only process new/changed rows

-- ── view ─────────────────────────────────────────────────
-- No storage. SQL executed fresh on every query.
-- Use for: staging models, simple transformations, anything
-- where the underlying table is already small and fast.
{{ config(materialized='view') }}
SELECT * FROM {{ source('bronze', 'crm_customers') }}
WHERE is_deleted = false

-- ── table ─────────────────────────────────────────────────
-- Full refresh every dbt run. Fast to query; expensive at scale.
-- Use for: intermediate models, dimensions, small-to-medium facts.
{{ config(materialized='table') }}
SELECT c.*, o.order_count
FROM {{ ref('stg_customers') }} c
LEFT JOIN {{ ref('int_customer_orders') }} o ON c.customer_id = o.customer_id

-- ── incremental ───────────────────────────────────────────
-- Only processes new/changed rows on subsequent runs.
-- First run: full load. Subsequent: incremental only.
-- Use for: large fact tables, event tables, anything expensive to full-refresh.
{{ config(
    materialized='incremental',
    unique_key='order_id',
    on_schema_change='append_new_columns'
) }}
SELECT * FROM {{ source('bronze', 'orders') }}
{% if is_incremental() %}
  WHERE order_date > (SELECT MAX(order_date) FROM {{ this }})
{% endif %}

-- ── ephemeral ─────────────────────────────────────────────
-- Compiled as a CTE in the referencing model. No storage.
-- Use for: intermediate CTEs you want to name without creating a table.
{{ config(materialized='ephemeral') }}
SELECT * FROM {{ source('bronze', 'products') }}
WHERE product_status = 'active'
Figure 3.14dbt materialization comparison. view runs SQL fresh on every query — no storage cost but slow for large underlying tables. table fully rebuilds on every dbt run — fast to query, expensive for very large models. incremental processes only new rows after the first run — the correct default for large fact tables. ephemeral compiles to a CTE in the referencing model — no storage, cannot be queried directly.
3.7.4

Testing and Documentation

dbt has two test types. Generic tests are defined in schema.yml and apply pre-built rules to columns: unique, not_null, accepted_values, relationships (referential integrity check). They run with dbt test and fail the pipeline if any assertion is violated.

Singular tests are custom SQL files in the tests/ directory. A singular test must return zero rows to pass. Any returned row is a test failure. Use them for business logic that cannot be expressed as a generic test:

dbt  ·  Singular test example
-- tests/assert_no_negative_revenue.sql
-- Returns rows where revenue is negative — test fails if any rows returned
SELECT
    order_id,
    amount
FROM {{ ref('fact_orders') }}
WHERE amount < 0

-- tests/assert_order_count_matches_source.sql
-- Checks that dbt model row count matches source (catches silent data loss)
SELECT 1
WHERE (SELECT COUNT(*) FROM {{ ref('fact_orders') }})
    != (SELECT COUNT(*) FROM {{ source('bronze', 'orders') }}
        WHERE status = 'completed')

Documentation is written in schema.yml description: fields. Running dbt docs generate creates a static documentation site with the full DAG visualisation, column descriptions, test coverage, and source freshness status. dbt docs serve hosts it locally. In production, the generated site is typically deployed to a static hosting service or internal wiki.

3.7.5

dbt in Production: CI/CD and 2026 Trajectory

GitHub Actions  ·  dbt slim CI pattern
# .github/workflows/dbt_ci.yml
name: dbt CI
on: [pull_request]

jobs:
  dbt-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dbt
        run: pip install dbt-databricks

      - name: dbt deps
        run: dbt deps

      # Slim CI: only run models changed in this PR + all downstream
      # "state:modified+" means: changed models and everything that depends on them
      - name: dbt build (changed models only)
        run: |
          dbt build \
            --select state:modified+ \
            --defer \
            --state prod_artifacts/    # artifacts from last successful prod run
        env:
          DBT_TARGET: ci
          DBT_DATABRICKS_TOKEN: ${{ secrets.DATABRICKS_TOKEN }}

The slim CI pattern is the most important dbt operational pattern for teams at scale. Instead of running the full dbt project on every pull request (which can take 30+ minutes for large projects), --select state:modified+ selects only the models that changed in the current PR plus all downstream models that depend on them. Combined with --defer, which reads upstream results from the last production run rather than rebuilding them, CI runs that would take 40 minutes take 3.

dbt Core v2.0 was announced in alpha at Snowflake Summit in June 2026. It is a Rust-based rewrite (the "Fusion engine") that ships as two distributions: dbt-core (open source, Apache 2.0) and dbt (Fusion distribution, proprietary). The Fusion engine delivers significantly faster compilation and execution. Section 3.7 teaches the stable 1.x syntax, which v2.0 is designed to be backward-compatible with. Monitor the dbt Labs release notes for v2.0 GA timeline before upgrading production deployments.
+
Go DeeperIncremental models in depth — append vs merge strategy

Incremental models are the most complex dbt feature and the most commonly misconfigured. Two strategies exist: append and merge (the dbt default on Databricks).

Append strategy: new rows from the SELECT are appended to the existing table. Simple and fast. Correct only when source rows never change after initial load (immutable event logs, for example). If rows can be updated, append creates duplicates.

Merge strategy (default on Databricks): dbt uses a unique_key column (or list of columns) to MERGE incoming rows into the existing table. Rows matching the unique key are updated; new rows are inserted. This is correct for tables where rows can be updated. Requires a unique_key configuration.

on_schema_change controls what happens when the SELECT adds new columns not in the existing table. Options: fail (default — prevents silent schema drift), append_new_columns (adds them), ignore (ignores the new columns). Choosing fail in development catches accidental schema changes; append_new_columns is appropriate for stable production Bronze tables absorbing source additions.

+
Go DeeperJinja templating — the minimum viable knowledge

dbt models use Jinja templating to enable dynamic SQL. Most dbt users need only a small subset of Jinja: the is_incremental() macro, ref() and source(), config(), and basic if/else.

{% if is_incremental() %} ... {% endif %}: the most important block. SQL inside runs only on incremental runs (not the first full build). Used to add a WHERE clause filtering to new rows only.

{{ config(...) }}: sets model-level configuration inline. Overrides project defaults. Common: materialized, unique_key, partition_by, cluster_by, on_schema_change.

Macros extend dbt with reusable functions. Define in macros/my_macro.sql and call with {{ my_macro(arg1, arg2) }}. The dbt_utils and dbt_date packages provide widely used macros (generate_surrogate_key, date spine, pivot). Install packages via packages.yml and run dbt deps.

+
Go DeeperSCD Type 2 with dbt snapshots

dbt snapshots implement Slowly Changing Dimension Type 2 (SCD Type 2 — full history with version rows) using a declarative YAML-like configuration. Define a snapshot in the snapshots/ directory:

{{ config(target_schema='snapshots', unique_key='customer_id', strategy='timestamp', updated_at='updated_at') }} followed by a SELECT from the source. On the first run, dbt inserts all rows with dbt_valid_from = current_timestamp and dbt_valid_to = NULL. On subsequent runs, dbt checks for changed rows using the unique_key and updated_at. Changed rows get the prior version's dbt_valid_to stamped with the current timestamp, and a new row is inserted as the current version.

The result is a Type 2 dimension table where every historical version of a customer record exists with valid_from and valid_to timestamps. The current version has dbt_valid_to IS NULL. Query current customers: WHERE dbt_valid_to IS NULL. Query as-of a specific date: WHERE dbt_valid_from <= '2024-01-01' AND (dbt_valid_to > '2024-01-01' OR dbt_valid_to IS NULL).

Snapshots run separately from dbt models: dbt snapshot. They should be scheduled before model runs that depend on the snapshot table, and they should be excluded from slim CI runs because they capture point-in-time state from the source, not transformed data.

🏢Field Note
A Data Engineering Team  ·  dbt CI Catching a Business Logic Error

A data engineering team maintained a dbt project with 140 models. A PR was opened adding a new column to the fact_orders mart model to capture a promotional discount flag. The developer introduced a logic error: the WHERE clause on the incremental filter was changed from order_date > (SELECT MAX(order_date) FROM {{ this }}) to order_date >= CURRENT_DATE() — a change that would cause the incremental model to only process orders from today rather than all unprocessed orders.

The GitHub Actions CI workflow ran dbt build --select state:modified+ on a CI target that pointed to a Databricks dev schema containing a small representative dataset. The dbt test suite included a singular test asserting that the model row count after a run matched the expected cumulative order count. The test failed: the incremental model produced 847 rows (today's orders only) rather than the expected ~45,000 cumulative rows.

The reviewer saw the test failure in the PR before merge and the error was corrected. Without CI testing, the bug would have silently truncated the fact_orders model on the next production run, removing all historical orders from the Gold layer.

The Lesson

dbt tests exist to catch logic errors before they reach production. Running dbt build on every PR with a representative dataset and a meaningful test suite is the difference between catching a silent data loss bug in CI and discovering it when the Finance team calls.

Question 1 of 5

What does ref("dim_customer") do inside a dbt model, beyond substituting the correct table name?

A
It only substitutes the correct schema-qualified table name in the compiled SQL
B
It creates a dependency edge in the dbt DAG, ensuring dim_customer runs before the current model, and enabling impact analysis and lineage tracking
C
It creates a foreign key constraint between the current model and dim_customer
D
It triggers a data freshness check on dim_customer before running the current model

B is correct. ref() does two things: (1) substitutes the correct environment-aware table name (dev schema in development, prod schema in production), and (2) creates a dependency edge in the DAG. dbt uses these edges to determine execution order (topological sort), to identify which models are affected by a change (state:modified+), and to render the lineage graph in dbt docs. Without ref(), two models that use the same table have no declared relationship and dbt cannot manage their dependency.

Question 2 of 5

A fact_orders table has 500 million rows and grows by 2 million rows daily. Which dbt materialization is most appropriate?

A
view — always reflects the latest data with no storage overhead
B
table — ensures the model is always fully rebuilt with the latest logic
C
incremental — processes only new rows on each run, avoiding the cost of rebuilding 500M rows daily
D
ephemeral — compiles the logic as a CTE to avoid storage costs

C is correct. An incremental model processes only new or changed rows on each run after the initial full build. For a 500M-row table growing by 2M daily, rebuilding the entire table every day (table materialization) would process 500M rows for 2M new rows — 250× more work than necessary. The incremental model, with a date-based filter in is_incremental(), processes only the 2M new rows. ephemeral cannot be queried directly and is inappropriate for a final mart model.

Question 3 of 5

What happens when a dbt singular test returns zero rows?

A
The test fails — zero rows indicates no data was tested
B
The test passes — zero rows means no violations were found
C
dbt raises a warning but does not fail the pipeline
D
The test is skipped and marked as inconclusive

B is correct. Singular tests are written as SQL that SELECTs violation rows. The test passes when the query returns zero rows (no violations found) and fails when it returns any rows. This is the inverse of what many developers expect initially. The pattern: write a SELECT that finds rows that SHOULD NOT EXIST — negative revenue orders, NULL customer IDs in a required field, orders with dates in the future. If any rows are returned, a data quality violation exists.

Question 4 of 5

What does --select state:modified+ do in dbt slim CI?

A
Runs all models that have been modified in the last 24 hours
B
Runs only models changed in the current PR, plus all downstream models that depend on them (transitively)
C
Runs only the models marked as modified in dbt_project.yml
D
Runs the modified models in parallel to reduce CI runtime

B is correct. state:modified selects models whose .sql files changed compared to the last production state artifacts. The + suffix means "and all downstream descendants." So if you changed stg_customers, dbt also runs int_customer_orders, dim_customer, and fact_orders (everything that depends on stg_customers, transitively). Combined with --defer (which reads upstream results from production rather than rebuilding them), this limits CI runs to only the affected subgraph of the DAG, dramatically reducing CI time for large projects.

✓ Section Summary

dbt applies software engineering practices to SQL transformation: version control (every model is a .sql file in Git), dependency management (ref() builds the DAG), testing (schema.yml generic tests + singular SQL tests), and documentation (dbt docs). Four materializations: view (no storage, always fresh), table (full rebuild), incremental (new rows only — correct for large facts), ephemeral (CTE, no storage). Slim CI runs only changed models + downstream via --select state:modified+ --defer. SCD Type 2 via dbt snapshots. dbt Core v2.0 (Rust Fusion engine) announced alpha at Snowflake Summit June 2026 — production deployments remain on stable 1.x.

ref() is the DAG

Use ref() for every model-to-model reference. Never hard-code table names. ref() is what gives dbt its dependency graph, slim CI, and lineage.

Materialization Rule

view for staging. table for intermediate + small dimensions. incremental for large facts. ephemeral for named CTEs you don't need to query directly.

Incremental Risk

is_incremental() filter must be correct. A wrong WHERE clause can truncate the model (too restrictive) or duplicate data (too permissive). Test incremental logic with dbt test after every change.
