Core libraries, pandas vs Polars, API patterns, and environment management — Python as a data engineering tool, not just a scripting language

3.3.1

Python's Place in the Data Pipeline

Python does not replace SQL in a data pipeline — it complements it. SQL is the right tool for set-based transformations on data already in a database or compute engine: the aggregations, joins, and window functions covered in Section 3.2. Python is the right tool for everything else: calling APIs that don’t have ADF connectors, processing files in non-standard formats, orchestrating multi-step workflows with complex branching logic, building custom data quality checks, and pre-processing data on a single machine before sending it to Spark.

The boundary matters. A common mistake is reaching for Python (pandas, loops, list comprehensions) to transform data that belongs in a SQL engine. A pandas loop that processes a 10-million-row table row by row on a single core will take hours. The same transformation in a Spark SELECT takes seconds. Use SQL for transformation. Use Python for orchestration, file handling, API calls, and anything Spark or SQL can’t express cleanly.

Python in a Medallion pipeline typically appears in three roles: (1) ADF calls a Python script to pre-process a non-standard file format before passing it to the main pipeline; (2) A Databricks notebook uses Python to call an external REST API and write the response to Bronze; (3) A dbt macro uses Python for logic that cannot be expressed in SQL Jinja. PySpark — covered in Section 3.5 — is a separate role: distributed transformation on very large datasets. This section covers single-machine Python. The two are not interchangeable.
3.3.2

pandas and Polars: The Two DataFrame Libraries

For single-machine tabular data work in Python, two libraries dominate: pandas and Polars. Understanding their differences — and knowing which to reach for — is increasingly essential.

pandas has been the default for over a decade. It has the richest API, the largest ecosystem of compatible libraries (scikit-learn, matplotlib, seaborn), and the most Stack Overflow answers. For datasets under roughly 500MB that fit comfortably in memory, pandas is a safe and well-supported choice. Its weaknesses become visible at scale: single-threaded execution by default, high memory overhead (often 5-10× the raw data size), and mutable operations that make debugging pipelines harder than necessary.

Polars launched in 2020 and reached production maturity in 2022-2023. By 2025-2026, it has become the default choice for new single-machine pipeline code in many engineering teams. Written in Rust with a Python API, Polars is multi-threaded by default, uses a columnar in-memory format based on Apache Arrow, and offers both eager (immediate) and lazy (deferred, optimised) execution modes. Benchmarks consistently show Polars running 5–20× faster than pandas on the same hardware for typical analytical workloads. New greenfield pipeline code written in 2025–2026 typically reaches for Polars first.

Python  ·  pandas vs Polars: same operation, different syntax
import pandas as pd
import polars as pl

# ── pandas ────────────────────────────────────────────────
df_pd = pd.read_parquet("sales.parquet")

result_pd = (
    df_pd[df_pd["region"] == "EMEA"]
    .groupby("product_category")
    .agg(
        total_revenue=("amount", "sum"),
        order_count=("order_id", "count"),
    )
    .sort_values("total_revenue", ascending=False)
)

# ── Polars (eager — immediate execution) ──────────────────
df_pl = pl.read_parquet("sales.parquet")

result_pl = (
    df_pl
    .filter(pl.col("region") == "EMEA")
    .group_by("product_category")
    .agg([
        pl.sum("amount").alias("total_revenue"),
        pl.count("order_id").alias("order_count"),
    ])
    .sort("total_revenue", descending=True)
)

# ── Polars (lazy — query plan optimised before execution) ─
result_lazy = (
    pl.scan_parquet("sales.parquet")   # LazyFrame — reads nothing yet
    .filter(pl.col("region") == "EMEA")
    .group_by("product_category")
    .agg(pl.sum("amount").alias("total_revenue"))
    .sort("total_revenue", descending=True)
    .collect()                          # Execute the optimised plan here
)

3.3.3

Polars on Databricks: The Critical Nuance

Polars is installed and usable on Databricks, which leads many practitioners to assume it benefits from the cluster. It does not. Polars on Databricks runs entirely on the driver node. It has no integration with Spark’s execution engine and cannot distribute work across worker nodes. A Polars operation on a 10-node Databricks cluster uses one core on the driver — exactly the same compute available on a single laptop. Cluster size does not help Polars.

This does not make Polars useless on Databricks. There are legitimate single-node tasks in a Databricks workflow: pre-processing a small lookup file before a Spark join, validating a configuration input, post-processing a small Spark result, or running a quick quality check on sampled data. For these tasks, Polars is genuinely faster than pandas. But any operation that requires the Spark cluster — reading a multi-terabyte Delta table, running a distributed join — requires PySpark, not Polars.

The decision rule: if the data fits comfortably on the driver node (typically <10GB for a standard Databricks driver), Polars is a valid and fast choice. If the data requires reading from a Delta table that does not fit in driver memory, use PySpark. Never assume that running Polars on a large cluster gives you distributed performance.
3.3.4

Core Python Patterns for Data Pipelines

Python  ·  REST API consumption pattern
import requests
import polars as pl
from pathlib import Path
import logging
from datetime import date

# Set up structured logging (not print statements)
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)s %(name)s %(message)s"
)
log = logging.getLogger(__name__)

def fetch_orders(api_url: str, api_key: str, target_date: date) -> pl.DataFrame:
    """Fetch orders from REST API and return as Polars DataFrame."""

    log.info("Fetching orders for %s", target_date)

    try:
        response = requests.get(
            f"{api_url}/orders",
            headers={"Authorization": f"Bearer {api_key}"},
            params={"date": target_date.isoformat()},
            timeout=30,          # always set a timeout
        )
        response.raise_for_status()   # raises on 4xx/5xx

    except requests.exceptions.Timeout:
        log.error("API request timed out after 30s")
        raise
    except requests.exceptions.HTTPError as e:
        log.error("API returned error: %s", e.response.status_code)
        raise

    records = response.json()["orders"]
    log.info("Received %d records from API", len(records))

    return pl.DataFrame(records)

def write_to_bronze(df: pl.DataFrame, output_path: Path) -> None:
    """Write DataFrame to Parquet in the Bronze landing zone."""
    output_path.parent.mkdir(parents=True, exist_ok=True)
    df.write_parquet(output_path)
    log.info("Written %d rows to %s", len(df), output_path)

# Entry point
if __name__ == "__main__":
    today = date.today()
    df = fetch_orders(
        api_url="https://api.example.com",
        api_key="...",           # in practice: load from environment or Key Vault
        target_date=today
    )
    write_to_bronze(df, Path(f"bronze/orders/date={today}/part-00001.parquet"))

🏢Field Note
A Data Engineering Team  ·  The pandas Memory Wall

A data engineering team at a logistics company processed daily shipment files using a pandas script. The files were CSV exports from their ERP — typically 200-300MB per day, which pandas handled comfortably. During peak season, the daily file grew to 2.4GB. The pandas script began failing with MemoryError on the team's standard 16GB CI runner, because pandas loaded the entire CSV into memory and then created multiple derived columns — pushing memory consumption to nearly 14GB before the failure.

The team migrated the script to Polars. The same pipeline in Polars used approximately 1.8GB of memory for the same 2.4GB file, thanks to Polars' Arrow columnar format and lazy filtering that reduced the working set before full materialisation. The script ran in 40 seconds instead of pandas' 4 minutes on normal-sized files. The peak-season file processed without errors. No cluster, no infrastructure changes — just a library switch and a one-day migration effort.

The Lesson

The Polars migration decision rule: if a pandas pipeline is hitting memory limits or taking more than a few minutes on single-machine data, try Polars before reaching for a Spark cluster. Cluster overhead is expensive in both time and money. Polars often eliminates the need.

Go DeeperPolars lazy execution — how the query optimiser works

Polars lazy mode works similarly to Spark: you describe a sequence of transformations, and the optimiser rewrites them into a more efficient physical plan before execution. The key optimisation is predicate pushdown: if you filter on a column early in your chain, the optimiser moves that filter to the earliest possible point — often before reading the file, so rows that don't match the filter are never loaded into memory.

Column pruning is a related optimisation: if your final select only uses 5 of a 50-column Parquet file's columns, the optimiser reads only those 5 columns off disk. In pandas, reading a Parquet file always reads all columns into memory first. In Polars lazy mode with scan_parquet(), only the columns used in the query are read.

To see the optimised plan, call df.lazy().filter(...).select(...).explain(). The output shows what the optimiser will actually execute, which is often significantly simpler than what you wrote. This is particularly useful for diagnosing why a lazy pipeline is fast or slow.

Common mistake: calling .collect() too early in a lazy chain forces execution before all optimisations can be applied. Build the full lazy chain first, then call .collect() once at the end. Intermediate collects defeat the purpose of lazy evaluation.

Go DeeperPython logging patterns for data pipeline scripts

Print statements work for interactive development but fail in production pipelines. Three problems: (1) print output is often lost when scripts run as background processes, (2) print gives no severity level — you cannot distinguish informational messages from errors, (3) print cannot be filtered or routed to different destinations (file, stdout, observability system).

Python's logging module solves all three. The pattern: call logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(name)s %(message)s") once at startup. Then log = logging.getLogger(__name__) in each module. Use log.info() for routine progress, log.warning() for recoverable issues, log.error() for failures, and log.exception() inside except blocks (it logs the full stack trace automatically).

In Databricks notebooks, the Databricks display methods and cluster logs handle output. But for Python scripts running as Databricks jobs or as external scripts called by ADF, structured logging is essential for diagnosing failures in the Azure Monitor log stream.

Row count logging is the minimum viable observability. Every pipeline step should log the row count of its input and output: log.info("Filter applied: %d rows in, %d rows out", len(df_in), len(df_out)). A sudden change in row count is almost always the first signal of a data quality problem.

+
Go Deeperpandas UDFs in PySpark — the bridge between Python libraries and Spark

Sometimes you need a Python library (scikit-learn, statsmodels, a custom business logic function) applied to data at Spark scale. Regular Python functions cannot run distributedly. The solution is a pandas UDF (User-Defined Function), which tells Spark: "take this partition of the DataFrame, convert it to a pandas DataFrame, run this Python function, convert the result back to a Spark DataFrame."

This is the architectural bridge: Spark handles distribution, Python handles the computation. The function runs on each executor in parallel — each executor processes its partition independently using the Python function. The Spark overhead (serialisation, inter-process communication) is real but manageable for compute-heavy per-partition operations.

Example: applying a scikit-learn model to score customer records at scale. The model is broadcast to all executors. A pandas UDF receives a batch of rows as a pandas DataFrame, calls model.predict(df), and returns the predictions as a new column. Spark handles sharding the 100-million-row customer table across executors; the UDF handles the actual scoring.

Pandas UDFs require that the Python environment on the Spark executors matches the driver (same library versions, same model file available). On Databricks, this is managed through cluster-level library installation and DBFS/Unity Catalog model storage.

🏢Field Note
A Platform Team  ·  Polars on Databricks for Driver Tasks

A platform team at a media company processed content metadata files as part of their Databricks ingestion pipelines. Each file was a 200-400MB JSON Lines export from their CMS, containing nested structures that required substantial parsing and flattening before the records could be written to Delta as Bronze. The original pipeline used pandas on the Databricks driver to parse and flatten each file, taking 6-8 minutes per file with 40-50 files per day.

A team member migrated the parsing code from pandas to Polars using Polars' native read_ndjson() and struct unnesting. The migration took one day. File processing time dropped from 6-8 minutes to 35-45 seconds per file. The total daily processing time for 40-50 files dropped from 5+ hours to under 30 minutes. Crucially, the team noted in their runbook that the Polars processing runs on the driver only — a reminder to future engineers not to expect cluster parallelism from the Polars sections of the notebook.

The Lesson

Polars on Databricks is a legitimate optimisation for driver-side processing tasks. The speed improvement is real. The key constraint — driver-only execution — must be documented clearly so future engineers don't mistakenly assume the cluster is being utilised.

A Databricks notebook processes a 500GB Delta table using Polars. The cluster has 10 worker nodes each with 32 cores. How much of the cluster's compute capacity does the Polars processing use?

A
All 320 worker cores — Polars distributes work across the cluster automatically
B
The cluster is used efficiently because Databricks optimises library execution
C
Only the driver node's compute — Polars has no integration with Spark and cannot distribute work across workers
D
32 cores on one worker node, plus the driver

C is correct. Polars runs on the driver node only. It has no integration with Spark's distributed execution engine. On Databricks, adding more workers or cores does not speed up Polars. For 500GB of Delta table data, Polars would likely fail with an out-of-memory error on a standard driver, because 500GB does not fit in driver memory. This workload requires PySpark (Section 3.5), which reads from Delta distributedly and processes it across all worker nodes.

Question 2 of 5

What is the primary advantage of Polars' lazy execution mode (pl.scan_parquet() vs pl.read_parquet())?

A
Lazy mode enables Polars to distribute work across a Spark cluster
B
Lazy mode builds a query plan that can be optimised before execution — for example, pushing filters before reads so fewer rows are loaded into memory
C
Lazy mode loads data one row at a time to save memory
D
Lazy mode is faster because it skips data validation

B is correct. Polars' lazy execution (scan_* functions returning a LazyFrame) builds a logical query plan without executing it. Calling .collect() triggers execution of the optimised plan. The optimiser can push filter predicates to reduce rows read, eliminate unused columns before processing, and reorder operations for efficiency. For file-based pipelines, lazy execution typically reduces memory usage significantly compared to eager mode — similar to how Spark's lazy evaluation model works.

Question 3 of 5

You are writing a Python script that calls an external REST API and needs to handle both network timeouts and HTTP error responses correctly. What is the correct approach?

A
Use a bare try/except Exception to catch all errors and log them
B
Use requests.get() without error handling — if the API is down, the script should fail with its default error
C
Use requests.get() with a timeout parameter, call response.raise_for_status() to convert HTTP errors to exceptions, and catch requests.exceptions.Timeout and requests.exceptions.HTTPError specifically
D
Check response.status_code manually after each request and use if/else branches

C is correct. Three elements are required: (1) timeout=N prevents the script hanging indefinitely if the API is unresponsive; (2) response.raise_for_status() converts 4xx/5xx responses into exceptions automatically; (3) catching specific exception types (requests.exceptions.Timeout, HTTPError) allows different logging and retry behaviour for network failures vs API errors. A bare except Exception (option A) catches too broadly and makes debugging harder by hiding the error type.

Question 4 of 5

A colleague says "pandas is always safer than Polars for production pipelines because it's been around for years and has more community support." What is the most accurate response?

A
Correct — pandas has more community support and is always the right choice
B
Partially correct — pandas has the larger ecosystem but Polars is production-ready, actively maintained, and the default for new pipeline code in 2025-2026
C
Incorrect — Polars has already replaced pandas entirely in the industry
D
Correct for data engineering, but Polars is better for data science

B is correct. Polars reached production maturity in 2022-2023 and by 2025-2026 is the default for new greenfield pipeline code in many engineering teams. It is actively maintained, has growing community resources, and is MIT-licensed. pandas still has advantages in ecosystem depth (scikit-learn compatibility, matplotlib, the vast number of existing examples), and existing pandas codebases don't need to be migrated without a clear performance reason. Neither is always right — the choice depends on data scale, ecosystem requirements, and existing code.

Question 5 of 5

What is the correct way to load a secret (API key) in a Python pipeline script intended for production use?

A
Hard-code it as a string constant at the top of the script for easy access
B
Store it in a comments section so it is not executed
C
Load it from an environment variable set by the deployment system, or from Azure Key Vault — never store it in the script file
D
Encrypt it in the script using base64 encoding

C is correct. Credentials in script files are a security incident waiting to happen — scripts are committed to Git repositories, shared with colleagues, stored in logs, and sometimes accidentally made public. The correct pattern: load secrets from environment variables (os.environ["API_KEY"]) set by the deployment system (a CI/CD secret, Databricks secret scope, or Azure Key Vault reference). In Databricks specifically, dbutils.secrets.get(scope="...", key="...") fetches from a Key Vault-backed secret scope. Base64 encoding (option D) is not encryption — it is trivially reversible.

✓ Section Summary

Python complements SQL and PySpark rather than replacing them. Use SQL for set-based transformation, Python for file I/O, APIs, and control flow, PySpark for distributed scale. For single-machine data: Polars is the modern default (5–20× faster than pandas, lower memory, multi-threaded). Polars on Databricks runs on the driver only — not distributed, regardless of cluster size. pandas remains valid for existing codebases and ML ecosystem compatibility. REST API patterns require explicit timeouts, raise_for_status() for HTTP errors, and specific exception types. Always load credentials from environment variables or Key Vault, never from script files.

Tool Boundary

SQL for transformation. Python for orchestration, file handling, and API calls. PySpark for distributed transformation. Use each for what it does best.

Polars on Databricks

Polars runs on the driver only — cluster size does not help. For data that exceeds driver memory, use PySpark. Document this clearly in notebooks.

New Pipelines

New single-machine pipeline code in 2025–2026 defaults to Polars. Start there. Switch to pandas only if you need scikit-learn compatibility or an existing pandas-native library.

Never In Code

Never hard-code credentials in Python scripts. Load from os.environ or Azure Key Vault. A key in a script file will eventually end up in Git.
