Delta Lake & File Formats

Why storage format choices matter, Parquet and its alternatives, Delta Lake's transaction log, and the 2026 state of the table format landscape

3.6.1

Why File Format Choice Matters

A file format is not just a container. It determines query speed, storage cost, schema flexibility, compression ratio, and compatibility with other tools. The wrong format for a workload can make queries 10–50× slower or consume 5–10× more storage than necessary. In a data lake serving terabytes of data to dozens of users, these differences are not academic.

The landscape has two layers that practitioners often conflate: file formats (how individual files store data — Parquet, Avro, ORC, CSV) and table formats (the metadata layer that organises collections of files into tables with ACID guarantees — Delta Lake, Apache Iceberg, Apache Hudi). A Delta Lake table, for example, stores data in Parquet files, with Delta providing the transaction log on top.

The most important file format fact for data engineers: columnar formats (Parquet, ORC) are dramatically faster than row-based formats (CSV, Avro, JSON) for analytical queries that read specific columns across many rows. A query reading three columns from a 200-column table reads 1.5% of the data stored in Parquet vs 100% in CSV. At petabyte scale, this difference is the gap between a 30-second query and a 30-minute one.

3.6.3

Delta Lake: ACID on Object Storage

Object storage (ADLS, S3, GCS) is cheap and scalable but has no concept of transactions. Writing ten Parquet files to ADLS is ten independent operations — if the process fails after five files, you have a partially written dataset with no rollback mechanism. Readers may see an inconsistent state mid-write. This is fine for a file archive; it is dangerous for a data pipeline where downstream consumers query the same table concurrently.

Delta Lake solves this by adding a transaction log — a _delta_log directory alongside the Parquet data files. Every write operation creates a new JSON file in the log (commit 000001.json, 000002.json, etc.) recording which files were added and which were removed. The log is the source of truth for the table’s current state; the Parquet files are immutable data. Reading a Delta table means consulting the log first to determine which files are valid, then reading only those files.

SQL (Databricks)  ·  Delta Lake core operations
-- ── ACID write ────────────────────────────────────────────
INSERT INTO prod.silver.orders
SELECT * FROM prod.bronze.orders WHERE ingestion_date = CURRENT_DATE();
-- All-or-nothing: either all rows are visible or none are.

-- ── Time travel ───────────────────────────────────────────
-- Query a previous version by version number
SELECT * FROM prod.silver.orders VERSION AS OF 42;

-- Query a previous version by timestamp
SELECT * FROM prod.silver.orders
TIMESTAMP AS OF '2024-01-15 08:00:00';

-- Find out what versions exist
DESCRIBE HISTORY prod.silver.orders;

-- Restore a table to a previous state (recovery from bad write)
RESTORE TABLE prod.silver.orders TO VERSION AS OF 41;

-- ── OPTIMIZE (compaction) ─────────────────────────────────
-- Merges many small Parquet files into fewer larger ones
-- Run after heavy incremental loads that produce small files
OPTIMIZE prod.silver.orders;

-- OPTIMIZE with Z-ordering on common filter columns
OPTIMIZE prod.silver.orders ZORDER BY (customer_id, order_date);

-- OPTIMIZE with Liquid Clustering (Databricks 13.3+, preferred)
-- Cluster columns defined at table creation:
-- CREATE TABLE ... CLUSTER BY (customer_id, order_date)
OPTIMIZE prod.silver.orders;

-- ── VACUUM (clean old files) ─────────────────────────────
-- Removes Parquet files no longer referenced by the Delta log
-- Default: retains files for 7 days (time travel window)
VACUUM prod.silver.orders;

-- Custom retention (CAUTION: reduces time travel window)
VACUUM prod.silver.orders RETAIN 168 HOURS;  -- 7 days

-- ⚠ DO NOT run VACUUM with 0 hours retention in production
-- It removes ALL historical versions, destroying time travel
Figure 3.11Delta Lake transaction log structure. The _delta_log directory contains JSON commit files recording which Parquet files were added or removed at each commit. Parquet files are immutable — "removed" files remain on disk until VACUUM cleans them, enabling time travel. Readers consult the log first to determine which files are currently active, then read only those files. Checkpoint files (Parquet format) periodically snapshot the log state to speed up log replay on large tables.
🏢Field Note
A Finance Team  ·  Time Travel Saving a Quarter-End Report

During a financial quarter-end close, a data engineering team ran a Gold layer rebuild that contained an incorrect filter condition — a bug introduced in the dbt model three days before the report deadline. The rebuild overwrote the Gold revenue table with incorrect data. The Finance team detected the discrepancy when the quarterly revenue figure was 12% lower than the prior run.

The data engineering team ran DESCRIBE HISTORY prod.gold.fact_revenue and identified the last known-good version from three days earlier. They ran RESTORE TABLE prod.gold.fact_revenue TO TIMESTAMP AS OF '2024-03-28 06:00:00' in under two minutes, instantly reverting the table to the pre-bug state. The Finance team's report ran from the restored table while the dbt bug was fixed and tested. Total time from incident detection to restored report: 35 minutes.

The Lesson

DESCRIBE HISTORY and RESTORE TABLE are among the most operationally valuable Delta Lake features. Configure VACUUM retention to at least 30 days on production Gold tables. The storage cost of retaining historical files is minimal compared to the recovery value in an incident.

🏢Field Note
A Platform Team  ·  The VACUUM Incident

A data platform team at a logistics company had a storage optimisation task on their quarterly review: reduce ADLS storage costs. A junior engineer ran VACUUM prod.bronze.shipments RETAIN 0 HOURS on the largest Bronze table, reasoning that the table had no "old" data that needed keeping — Bronze was only used as an input to Silver.

The VACUUM command deleted all Parquet files not referenced by the current Delta log version. Three days later, a bug was discovered in the Silver cleaning logic that had been running for the past two weeks. The standard remediation would have been to re-run the Silver transformation using the Bronze data from those two weeks via time travel. That Bronze data no longer existed. The team had to re-extract two weeks of data from the source ERP system, which took four days and required a maintenance window on the operational system.

The correct VACUUM command would have been RETAIN 720 HOURS (30 days). The storage saving from retaining historical Bronze files was less than $40 per month. The recovery effort cost significantly more in engineering time.

The Lesson

Never run VACUUM with RETAIN 0 HOURS. Configure a retention period that matches your recovery window requirements. For Bronze: at least 30 days. For Silver and Gold: at minimum 7 days (the Delta default). Add the retention period to your runbooks and CI/CD VACUUM scripts.

3.6.4

Delta vs Iceberg: The 2026 Landscape

The table format landscape has consolidated significantly. Two formats dominate: Delta Lake and Apache Iceberg. Apache Hudi, while technically capable, has largely lost the mindshare competition in 2025–2026 as both Delta and Iceberg absorbed its key capabilities.

The most significant development of the past 18 months: in mid-2024, Databricks acquired Tabular — the company founded by Iceberg’s original Netflix engineering team — for over $1 billion. Databricks now supports both Delta Lake and Apache Iceberg natively as first-class formats. This makes the historical framing of "Delta vs Iceberg as competing Databricks vs non-Databricks choices" increasingly obsolete.

The 2026 practical guidance: use Delta Lake if you are on Databricks and your data stays within the Databricks and Microsoft Fabric ecosystem (Fabric uses Delta natively). Use Iceberg if you need to share data across multiple compute engines without coupling to Databricks — Iceberg is supported natively by Snowflake, BigQuery, Trino, DuckDB, Athena, and Flink. When in doubt on a new Databricks deployment, Delta Lake with UniForm enabled exposes your tables as Iceberg for cross-engine access without migrating.

+
Go DeeperUniForm: Delta tables as Iceberg for cross-engine access

Databricks's UniForm (Universal Format) feature, available from Databricks Runtime 13.3, generates Iceberg-compatible metadata alongside the standard Delta transaction log. This means a Delta Lake table with UniForm enabled can be read by any engine that supports Iceberg (Snowflake, BigQuery, Trino, DuckDB, Athena) without copying or converting the data.

Enable UniForm at table creation: CREATE TABLE ... TBLPROPERTIES ('delta.universalFormat.enabledFormats' = 'iceberg'). Or enable on an existing table: ALTER TABLE ... SET TBLPROPERTIES ('delta.universalFormat.enabledFormats' = 'iceberg').

UniForm is significant because it dissolves the practical "Delta vs Iceberg" choice for most Azure-first organisations: you get Delta's mature Databricks integration and performance characteristics, while retaining cross-engine interoperability for any tool that queries via the Iceberg REST API. The trade-off: UniForm adds minor overhead to each write operation as it generates the Iceberg metadata alongside the Delta log.

+
Go DeeperChange Data Feed (CDF): reading only what changed

Delta Lake's Change Data Feed feature records row-level changes (INSERT, UPDATE, DELETE) in a separate change log alongside the standard data files. This enables downstream pipelines to read only the rows that changed since the last run, rather than reading the entire table.

Enable CDF on a table: ALTER TABLE table_name SET TBLPROPERTIES (delta.enableChangeDataFeed = true). Read changes from PySpark: spark.read.format("delta").option("readChangeFeed", "true").option("startingVersion", last_processed_version).table("table_name").

CDF is particularly valuable for propagating Silver changes to Gold without re-reading the entire Silver table, and for streaming Delta table changes to external systems (Kafka, event buses) without requiring a separate CDC capture mechanism at the source database. The performance benefit scales with table size: reading 100K changed rows from a 100M-row table via CDF is orders of magnitude faster than a full table scan.

+
Go DeeperOPTIMIZE and the small files problem

Every incremental write to a Delta table creates new Parquet files. A pipeline running every 15 minutes creates 96 files per day per table. After a year, a table may have 35,000+ small files. Reading such a table requires opening 35,000 file handles, parsing 35,000 file headers and footmasters, and filtering 35,000 column statistics — before reading a single row of data. Query planning time alone can exceed execution time.

OPTIMIZE solves this by compacting many small files into fewer, larger ones. It reads the current table state and rewrites files into the optimal size range (typically 128MB–1GB per file for analytical workloads). The Delta transaction log records the compaction as a series of file removals and additions — the data is not deleted, just reorganised.

OPTIMIZE with Z-ordering goes further: it clusters rows within files so that rows with similar values of the specified columns are stored near each other. This improves data skipping (the engine can skip entire files based on column statistics) for subsequent queries filtering on those columns. Liquid Clustering (Section 2.2) achieves a similar effect with better flexibility — the cluster columns can be changed without rewriting data, and incremental OPTIMIZE runs apply clustering only to new files.

Question 2 of 5

Why does running VACUUM with RETAIN 0 HOURS disable time travel entirely?

A
VACUUM 0 HOURS also deletes the _delta_log directory
B
VACUUM removes unreferenced Parquet files. With 0 hours retention, all files not referenced by the current log version are deleted — including the historical files that time travel reads from
C
VACUUM 0 HOURS corrupts the Delta transaction log
D
VACUUM only affects files older than the table creation date

B is correct. Delta time travel works by retaining the old Parquet files that previous log versions reference. The _delta_log still contains the historical commit JSON files, but if the Parquet files those commits point to have been deleted by VACUUM, reading the historical version fails with a FileNotFoundException. VACUUM RETAIN 0 HOURS removes all historical Parquet files immediately, leaving only the current version readable. Always set a meaningful retention period (RETAIN 720 HOURS for 30 days is a common production standard).

B is correct. Incremental pipelines create new Parquet files on every write. Over time, tables accumulate thousands of small files. ✓ Section Summary

File formats divide into columnar (Parquet, ORC — fast analytics, compressed) and row-based (Avro, CSV, JSON — fast writes, slow analytics). Parquet is the default for all Databricks work. Delta Lake adds a transaction log (_delta_log) to Parquet files, providing ACID transactions, time travel (VERSION/TIMESTAMP AS OF), and schema enforcement. OPTIMIZE compacts small files; always run after heavy incremental loads. VACUUM removes historical files — never run with RETAIN 0 HOURS as it destroys time travel. In 2026: Delta and Iceberg are the two dominant table formats; Hudi has lost the mindshare war. Enable UniForm on Delta tables for Iceberg compatibility with Snowflake, Trino, and BigQuery without data copying.
