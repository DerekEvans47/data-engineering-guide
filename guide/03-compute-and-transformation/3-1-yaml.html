YAML & Configuration Languages

The universal config syntax of the data engineering stack — and where it hides in every tool you use

✓

3.1.1

Why YAML Is Everywhere

YAML — which officially stands for YAML Ain’t Markup Language — is a human-readable data serialisation format. It was designed to be written and edited directly by humans, which makes it fundamentally different from XML or JSON, both of which are primarily consumed by machines. That human-readability made YAML the default choice for configuration files across almost every modern software and data tool.

In the data engineering stack, YAML appears in more places than most practitioners realise. A dbt project uses YAML for source definitions, test configurations, and documentation. GitHub Actions workflows are YAML files. Databricks Asset Bundles (DABs) use YAML to define jobs and pipelines. Azure DevOps CI/CD pipelines are YAML. Kubernetes manifests — which Databricks and many data tools run on — are YAML. Great Expectations configuration is YAML. Once you understand YAML, you can read and edit configuration across the entire modern stack without learning tool-specific syntax.

YAML is not a programming language. It has no logic, no loops, no conditionals. It is a way of expressing data as text — specifically, nested key-value pairs, lists, and scalars. Understanding YAML means understanding those three structures and the indentation rules that separate them.
3.1.2

YAML Syntax: The Rules That Matter

YAML has a small set of syntax rules but breaks in ways that are silent — an incorrect YAML file often parses without error, just not as the author intended. The rules below are the ones that cause real production failures.

Indentation is structural — and must use spaces

YAML uses indentation to represent nesting. Tabs are illegal. Using a tab character instead of spaces will either raise a parse error or — more dangerously — be silently consumed as a space by some parsers and misinterpreted. The convention is 2 spaces per indent level. The exact number does not matter (2 or 4 both work) but must be consistent within a file.

YAML  ·  Structure fundamentals
# YAML comment (JSON has no comments — YAML does)
# Three data structures: scalars, sequences, mappings

# Scalar (a single value)
name: sales_pipeline
version: 1.0
enabled: true

# Mapping (key: value pairs — the most common structure)
database:
  host: myserver.database.windows.net
  port: 1433
  name: gold_layer

# Sequence (a list — two equivalent notations)
models:
  - dim_customer
  - fact_orders
  - agg_revenue

tags: [finance, daily, gold]   # inline sequence notation

# Nested: mapping containing a sequence
sources:
  - name: crm
    tables:
      - name: customers
      - name: contacts

The boolean trap

YAML has an unusually large set of values it considers boolean. In YAML 1.1 (which many tools still use): true, yes, on, True, YES, ON are all boolean true. false, no, off, False, NO, OFF are all boolean false. This causes real problems when these words appear as string values.

YAML  ·  The boolean trap in practice
# Dangerous: "no" as a country code becomes boolean false
country: no         # Parsed as: country: false (WRONG)
country: "no"       # Correct: quoted string

# Dangerous: "on" as a column name
columns:
  - name: on        # Parsed as boolean true (WRONG)
  - name: "on"      # Correct: quoted

# dbt schema.yml real-world example of the trap:
models:
  - name: dim_customer
    columns:
      - name: is_active
        tests:
          - accepted_values:
              values: [yes, no]    # "yes" and "no" become true/false!
              # Correct: values: ["yes", "no"]  — always quote

YAML in the Data Engineering Stack

Figure 3.1 YAML syntax anatomy. Four building blocks: scalars (plain values), mappings (key-value pairs), sequences (lists, introduced by a dash), and multiline strings (| preserves newlines, > folds them). Anchors (&) define reusable blocks; aliases (*) reference them. Comments start with #. Indentation is structural — two spaces per level, no tabs.
Figure 3.2 YAML across the data engineering toolchain. Six tools, one syntax. The dbt schema.yml is the most commonly written YAML file in a data engineer's daily work. GitHub Actions and Azure DevOps workflows define CI/CD pipelines as YAML. Databricks Asset Bundles define jobs and clusters in YAML. Great Expectations uses YAML for quality suite configuration. Kubernetes — which runs many of these tools — expresses all infrastructure as YAML manifests.

The four most important YAML files a data engineer encounters regularly:

dbt schema.yml — documents models, sources, and columns; defines tests. Every dbt project has at least one, usually many. The most commonly written YAML file in a data engineering career.

GitHub Actions workflow (.github/workflows/*.yml) — defines CI/CD pipelines: what triggers them, what steps run, what environment they run in. Covered in Section 5.2.

Databricks bundle.yml — defines Databricks Asset Bundles (DABs): jobs, clusters, permissions, and deployment targets in code.

dbt_project.yml — the root configuration file for a dbt project: project name, version, model paths, materialization defaults, and variable definitions.

dbt schema.yml  ·  What a real data engineer writes daily
# models/schema.yml — documents a dim_customer model
version: 2

models:
  - name: dim_customer
    description: "One row per customer. SCD Type 2 — multiple rows for changed customers."
    columns:
      - name: customer_sk          # surrogate key
        description: "Surrogate key generated by the platform."
        tests:
          - not_null
          - unique

      - name: customer_id          # natural key from source CRM
        description: "CRM system customer identifier."
        tests:
          - not_null

      - name: region
        description: "Customer region."
        tests:
          - not_null
          - accepted_values:
              values: ["EMEA", "AMER", "APAC"]

      - name: is_current
        description: "True if this is the current version of the customer record."
        tests:
          - not_null

🏢Field Note
A Data Engineering Team  ·  The Tab Character Incident

A data engineering team migrated their dbt project to a new repository, copying schema.yml files from an older internal wiki where they had been stored as formatted text. The wiki had automatically converted leading spaces to tabs in some entries during copy-paste. The dbt YAML parser on most systems is tolerant of tabs in some positions but strict in others, and the specific combination of tab-indented test definitions that had been migrated caused some tests to be silently parsed as empty rather than throwing an error.

The missing tests were discovered three weeks later during a data quality review, when a null customer_id issue that the tests should have caught went unreported. The root cause was traced to a single tab character on line 14 of a schema.yml file. The fix took three minutes. Detecting it took three weeks.

The team's remediation: added yamllint as a pre-commit hook. Every committed YAML file is now validated for indentation consistency before reaching the repository.

The Lesson

YAML errors are silent until they are not. Add a YAML linter (yamllint, prettier, or your IDE's YAML extension) to your pre-commit workflow. A tab character discovered in a linter takes seconds to fix. A tab character discovered in a production incident takes weeks.

🏢Field Note
An Infrastructure Team  ·  YAML Anchors Saving 400 Lines

An infrastructure team managed Databricks cluster configurations across 40 environments: development clusters for 8 teams, 12 shared staging clusters, and 20 production clusters with varying sizes. Each cluster definition was nearly identical except for node type and count. Their initial bundle.yml was 480 lines of near-duplicate configuration, maintained manually across every cluster definition.

A senior engineer introduced YAML anchors and aliases. A single &base_config anchor defined the 15 fields shared by every cluster: runtime version, spark configuration, libraries, init scripts, and logging settings. Each environment definition then used <<: *base_config to inherit all shared fields and overrode only node_type and num_workers. The 480-line file became 80 lines. More importantly, when the runtime version needed updating, it was one change in one place rather than 40 parallel edits.

The team also added a CI check that validated all 40 cluster definitions against a JSON schema, so misconfigured overrides were caught in the pull request rather than at deploy time.

The Lesson

YAML anchors and aliases are the DRY principle applied to configuration. Whenever you find yourself copy-pasting a block of YAML and changing one or two values, an anchor is the correct tool. The syntax is unfamiliar at first but saves significant maintenance cost at scale.

Go DeeperMultiline strings: | vs > and when each is right

YAML has two multiline string notations with importantly different behaviour. The pipe character (|, literal block scalar) preserves newlines exactly as written. The greater-than character (>, folded block scalar) converts newlines within the content to spaces, folding the text into a single long line, with only blank lines preserved as paragraph breaks.

Use | (literal) when: the content is code, a shell command, a SQL query, or anything where newlines are meaningful. A GitHub Actions run: step with multiple shell commands must use | or each command will be concatenated into a single broken line.

Use > (folded) when: the content is prose text (a description, a README section) where the line breaks in the YAML file are just for readability and should not appear in the output string.

Both notations support a chomping indicator: |- strips the final newline, |+ preserves all trailing newlines. The default (|) strips trailing newlines but preserves a single one at the end. In practice, |- is the most commonly useful variant when embedding SQL in YAML — it ensures no trailing newline is appended to the query string.

+
Go DeeperYAML security: the deserialization attack surface

YAML parsers that support "full YAML" (including Python's default PyYAML with yaml.load()) can deserialize arbitrary Python objects — including calls to os.system(), subprocess.Popen(), and other code execution functions. A maliciously crafted YAML file loaded with an unsafe parser can execute arbitrary code on the machine running the parser.

The attack pattern: a user submits a YAML config file that contains a Python object tag (!!python/object/apply:os.system) followed by a shell command. The unsafe parser deserializes it, executes the code.

The fix is simple but not the default: always use yaml.safe_load() instead of yaml.load() in Python. safe_load() only deserializes basic YAML types (strings, ints, lists, dicts) and refuses object tags. This is the correct default for all data engineering use of YAML parsing in Python. dbt, which reads schema.yml internally, uses safe loading. The risk arises in custom scripts that parse YAML config files from external or user-supplied sources.

+
Go DeeperYAML schema validation with yamllint and json-schema

YAML is human-written, which means it has human errors. Two tools address this at different levels.

yamllint is a YAML syntax linter: it checks indentation consistency, trailing spaces, line length, and other style issues. It catches structural problems before a tool even tries to parse the file. Adding yamllint . as a pre-commit hook takes two minutes to configure and eliminates an entire class of frustrating CI failures.

JSON Schema validation goes deeper: after confirming the YAML is syntactically valid, it validates that the content matches an expected structure. dbt uses this to validate your schema.yml against a defined schema (unknown keys raise warnings). Kubernetes validators use JSON Schema to confirm your manifest defines required fields with the right types. For custom YAML configs in data pipelines, a JSON Schema validator run on every commit catches semantic errors — a required field missing, a value of the wrong type — that yamllint cannot see because they are structurally valid but logically wrong.

Question 1 of 5

A dbt schema.yml file defines tests for a column called "country". You write accepted_values: values: [no, yes] to validate the column only contains those two strings. What is the actual result?

A
The test validates correctly — YAML treats bare strings correctly in this context
B
The YAML file fails to parse and raises a syntax error
C
The values "no" and "yes" are parsed as boolean false and true, not as strings — the test validates against [false, true] rather than ["no", "yes"]
D
The test is silently skipped because boolean values are not valid for accepted_values

C is correct. In YAML 1.1 (which many tools including older dbt versions use), yes and no are boolean literals. The list [no, yes] is parsed as [false, true], not as strings. The fix is to quote them: ["no", "yes"]. This is one of the most common dbt schema.yml bugs encountered in practice, especially in projects that model data from countries whose ISO codes include "no" (Norway).

Question 2 of 5

You are writing a GitHub Actions step that runs three shell commands in sequence. Which YAML multiline string notation is correct?

A
Use > (folded) — it combines lines efficiently for shell commands
B
Use | (literal) — it preserves newlines so each command runs as a separate line
C
Either works — the shell ignores the difference
D
Inline single-quoted strings are required for shell commands

B is correct. The pipe character | (literal block scalar) preserves newlines. In a GitHub Actions run: step, each newline represents a separate shell command. Using > (folded) would collapse all commands into a single line, causing parse errors or unexpected concatenation. The | notation for shell commands in YAML is non-negotiable.

Question 3 of 5

Which of the following Python YAML calls is safe to use with user-supplied YAML files?

A
yaml.load(user_input)
B
yaml.load(user_input, Loader=yaml.FullLoader)
C
yaml.safe_load(user_input)
D
yaml.parse(user_input)

C is correct. yaml.safe_load() only deserializes basic YAML types and refuses Python object tags. yaml.load() without a Loader, and yaml.load() with FullLoader or UnsafeLoader, can deserialize arbitrary Python objects — a code execution vulnerability when the YAML source is not fully trusted. The PyYAML documentation has warned about this since 2017 and modern versions raise a warning when yaml.load() is called without a Loader argument.

Question 4 of 5

What does the << key in YAML do?

A
It opens a multiline string block
B
It creates a left-shift operator for integer arithmetic
C
It is a merge key: it imports all key-value pairs from the referenced anchor into the current mapping
D
It indicates a YAML comment that spans multiple lines

C is correct. The merge key << (spelled with two less-than signs) imports all key-value pairs from a referenced anchor (*alias_name) into the current mapping. It is the mechanism that makes YAML anchors useful for DRY configuration: define common settings once with &base, then reference them in each environment with <<: *base and override only what differs.

Question 5 of 5

A colleague says "tabs and spaces are equivalent in YAML — I use tabs because they're easier." Is this correct?

A
Yes — modern YAML parsers accept both tabs and spaces
B
Yes — the YAML specification allows tabs as an alternative to spaces
C
No — the YAML specification explicitly prohibits tab characters for indentation. They may be accepted silently by some parsers and cause hard-to-detect errors
D
Only partially — tabs are allowed at the root level but not in nested structures

C is correct. The YAML specification (all versions: 1.1 and 1.2) explicitly prohibits tab characters for indentation. The reason: different editors interpret tab width differently (2, 4, or 8 spaces), making tab-indented YAML visually correct in one editor but structurally wrong in another. Some parsers accept tabs silently and treat them as a fixed number of spaces, which introduces inconsistency. Always use spaces. Configure your editor to insert spaces on tab keypress for YAML files.

✓ Section Summary

YAML is the dominant configuration format across the modern data engineering stack: dbt, GitHub Actions, Databricks Asset Bundles, and Kubernetes all use it. Three data structures (scalars, mappings, sequences) plus indentation rules make up the entire format. Key gotchas: tabs are illegal, yes/no/on/off are booleans not strings, and | vs > for multiline strings controls newline preservation. YAML anchors and aliases implement DRY configuration at scale. Always use yaml.safe_load() in Python, never yaml.load() with user input.

Core Rule

YAML uses spaces only for indentation — tabs are illegal. Configure your editor to insert spaces on Tab for YAML files. This eliminates a whole class of silent parsing errors.

Boolean Trap

yes, no, on, off, true, false are all booleans. If you mean the string "yes" or "no", quote it: "yes".

DRY Config

YAML anchors (&name) define a reusable block. Aliases (*name) reference it. Merge keys (<<: *name) inherit all keys. Use these any time you copy-paste a YAML block and change one value.
