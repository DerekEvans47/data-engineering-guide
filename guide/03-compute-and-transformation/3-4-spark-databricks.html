3.4.1

What Spark Is and Why It Exists

Apache Spark is an open-source, distributed data processing engine. It splits a dataset into partitions and processes those partitions in parallel across a cluster of machines — each machine running a Spark process called an Executor. This is how Spark processes terabytes of data in minutes rather than hours: not by having one extremely fast processor, but by having hundreds of ordinary processors working on different pieces of the data simultaneously.

Spark emerged from academic research at UC Berkeley in 2009 as a faster alternative to Hadoop MapReduce. Where Hadoop wrote intermediate results to disk between every processing stage, Spark keeps data in memory across stages, dramatically reducing the I/O overhead of multi-step pipelines. Today, Spark is the dominant distributed compute engine for data engineering workloads on all three major cloud platforms.

The single most important thing to understand about Spark: it is lazy. When you write a transformation (a filter, a join, a window function), Spark does not execute it. It records the transformation in a directed acyclic graph (DAG) of planned operations. When you trigger an action (writing to a file, calling .count(), displaying results), Spark executes the entire DAG at once — optimised as a whole, not step by step. This lazy evaluation is the source of both Spark's performance and its most common debugging confusion.
3.4.2

Spark Architecture: Driver, Executors, and the DAG

Every Spark application has a Driver — the process that coordinates the work. The Driver runs the main program, builds the DAG, divides it into stages and tasks, and distributes those tasks to the Executors. Executors are JVM processes that run on worker nodes in the cluster. Each Executor has a fixed number of CPU cores and a fixed amount of memory. Tasks run on cores; data is stored in Executor memory. If data exceeds Executor memory, Spark spills to disk — a significant performance penalty.

The Cluster Manager allocates resources: it provisions worker nodes, manages their lifecycle, and communicates resource availability to the Driver. On Databricks, the Cluster Manager is the Databricks control plane — you do not interact with it directly. On AWS, it may be Apache YARN or Kubernetes.

When the Driver executes a DAG, it divides it into stages. Stage boundaries occur where data must be redistributed across Executors — a shuffle. Within a stage, tasks run in parallel across Executors. Between stages, all Executors must complete their current stage before the next begins. Shuffle is the most expensive Spark operation and the most important to minimise.

Figure 3.7Apache Spark architecture. The Driver runs your code and builds a DAG of planned transformations. Shuffle — data redistribution across Executors — occurs at stage boundaries during operations like GROUP BY and JOIN. Minimising shuffles is the primary Spark performance lever.
3.4.3

Databricks: Managed Spark with Enterprise Features

Databricks is a managed platform built on Apache Spark. It handles cluster provisioning, runtime versioning, library management, autoscaling, and infrastructure maintenance — abstracting away the operational complexity of self-managing a Spark cluster. For data engineering teams, Databricks provides the compute layer between ADLS (storage) and the transformation logic written in SQL or PySpark.

Beyond managed Spark, Databricks adds several features that have become core parts of the Azure data stack:

Photon Engine is a C++ vectorised query engine that replaces the JVM-based Spark execution for SQL and Delta Lake operations. Photon typically delivers 2–10× better query performance for analytical SQL workloads, particularly for scan-heavy operations on large Delta tables. It is enabled by default on Databricks-optimised clusters and is transparent to end users — you write the same SQL, Photon executes it faster.

Unity Catalog is Databricks\' unified governance layer. It provides a three-level namespace (catalog.schema.table), centralised access control (users and groups defined once, permissions applied across all workspaces), column-level security, row-level filters, automated data lineage, and an auditable access log. Unity Catalog replaced the legacy per-workspace Hive metastore and is now the recommended approach for any enterprise Databricks deployment.

Auto Loader is a streaming ingestion feature that monitors an ADLS path and automatically processes new files as they arrive. It handles schema inference, schema evolution, and exactly-once processing — making it the recommended Bronze ingestion pattern for file-based sources that land files continuously or on irregular schedules.

Databricks Asset Bundles (DABs) is the modern way to deploy Databricks resources as code: jobs, clusters, pipelines, and permissions defined in a bundle.yml file (covered in Section 3.1) and deployed via the Databricks CLI. DABs replaced the older Databricks CLI (v1) for infrastructure-as-code deployments.

3.4.4

Unity Catalog: Governance at Scale

Unity Catalog introduced a three-level namespace for all Databricks data assets: catalog.schema.table. A catalog is the top-level container — typically one per environment (prod, dev, staging) or one per business domain (finance, logistics, marketing). A schema (also called a database) groups related tables within a catalog. A table lives within a schema.

SQL  ·  Unity Catalog namespace in practice
-- Three-level namespace: catalog.schema.table
USE CATALOG prod;
USE SCHEMA gold;
SELECT * FROM dim_customer;   -- reads prod.gold.dim_customer

-- Explicit three-part reference (can be used anywhere)
SELECT c.*, o.total_spend
FROM prod.gold.dim_customer AS c
JOIN prod.gold.fact_orders   AS o ON c.customer_sk = o.customer_sk;

-- Grant access (Unity Catalog replaces per-cluster GRANT statements)
GRANT SELECT ON TABLE prod.gold.dim_customer TO GROUP analysts;
GRANT MODIFY  ON SCHEMA prod.silver          TO GROUP data_engineers;

-- Row filter (Unity Catalog managed — transparent to the querying user)
-- Users in the "finance" group only see their region's data:
ALTER TABLE prod.gold.fact_orders
SET ROW FILTER region_access_filter ON (region);

-- Column mask (Unity Catalog — masks email for non-PII roles)
ALTER TABLE prod.gold.dim_customer
ALTER COLUMN email
SET MASK email_mask;

Go DeeperPhoton engine — what it accelerates and what it does not

Photon is a C++ vectorised query engine that replaces Spark's JVM-based physical execution for specific operation types. Understanding what Photon accelerates — and what it does not — helps you set correct performance expectations.

What Photon accelerates: column-oriented scans of Parquet and Delta files, SQL aggregations (GROUP BY, SUM, COUNT, AVG), SQL joins (especially hash joins on numeric columns), Delta MERGE operations, window functions in SQL, and ORDER BY. These are the operations that appear in Silver-to-Gold transformation pipelines — which is where Photon delivers its most visible benefits.

What Photon does NOT accelerate: Python UDFs (these still run in the JVM/Python interpreter, not in Photon), custom Scala/Java code, operations on non-Delta file formats (CSV, unstructured files), and streaming operations using the older DStream API. Photon specifically targets the analytical SQL and Delta Lake workloads that dominate most production Databricks usage.

Practical guidance: enable Photon on all production clusters running SQL-heavy transformations (Silver-to-Gold, reporting queries). The performance improvement is 2–10× for typical workloads with no code changes required. For Python-heavy workloads running custom UDFs, Photon's impact is limited to the SQL portions of those workloads.

Go DeeperCluster types and cost strategy

Databricks has three cluster types with very different cost profiles and use cases.

All-Purpose Clusters (formerly Interactive Clusters): persistent clusters you start manually and leave running for interactive work (notebook development, ad-hoc analysis). They persist between jobs. Cost: full Databricks Units (DBUs) + VM cost for every minute running, whether you are actively using them or not. Idle all-purpose clusters are a major source of unexpected Databricks bills. Best practice: auto-terminate after 20-30 minutes of inactivity.

Job Clusters: clusters provisioned automatically when a Job runs and terminated immediately when it completes. Cost: only for the duration of the job. For production pipelines, job clusters are almost always the right choice. The start-up time (2-3 minutes) is acceptable for scheduled pipelines.

SQL Warehouses (Serverless): serverless compute for BI and SQL analytics. Spins up in seconds, scales automatically, and charges per query second rather than per cluster minute. Ideal for Power BI DirectQuery connections and ad-hoc SQL from business users.

Spot instances (AWS) or spot VMs (Azure) are interruptible cloud VMs available at 60-90% discount. Databricks handles interruption gracefully by retrying tasks. For batch jobs that can tolerate occasional retries, using spot workers for all worker nodes (while keeping the driver on on-demand) typically reduces cluster costs by 50-70%.

🏢Field Note
A Manufacturing Company  ·  Cluster Policies Saving 58%

A manufacturer running a Databricks platform for roughly 60 data engineers found that monthly Databricks compute costs had grown to over three times the original budget estimate. Investigation revealed that most of the overspend was on all-purpose cluster idle time: engineers were spinning up clusters for exploration, leaving them running overnight and over weekends. The default auto-termination setting was not being applied consistently, and cluster sizes were not being aligned to workload requirements.

The platform team implemented Databricks cluster policies: admin-defined configuration templates that engineers select rather than configure from scratch. Policies enforced maximum cluster sizes (8 workers for interactive, 32 for jobs), mandatory auto-termination of 20 minutes for all-purpose clusters, and required spot instances for all worker nodes. Engineers retained full flexibility in their notebooks but could not accidentally configure an oversized always-on cluster.

Monthly compute costs dropped 58% within two months. The primary driver was eliminating idle all-purpose cluster runtime. Secondary contributors were right-sizing (engineers had been requesting 10× more workers than jobs required) and spot instance adoption.

The Lesson

Cluster policies are the most effective single governance change for controlling Databricks costs in a multi-team environment. They do not restrict what engineers can do — they restrict how they can accidentally waste money while doing it.

🏢Field Note
A Retailer  ·  Unity Catalog Enabling Self-Service

A UK retailer had a Databricks environment where all data access was managed through a shared "data engineering" service account. Business analysts requested data by filing tickets with the data team, who manually granted access to specific files on ADLS. The process took 2-4 days per request and generated significant toil for the data engineering team.

After migrating to Unity Catalog, the retailer implemented role-based access control aligned to business domains: Finance analysts were granted SELECT on the finance catalog, Marketing analysts on the marketing catalog, and Data scientists on designated feature engineering schemas. Row-level security on Gold tables filtered data to each team's business region. Column masking automatically obscured PII fields for roles without explicit PII access grants.

Self-service access request time dropped from 2-4 days to under 4 hours (the time for a manager to approve a group membership change in Azure Active Directory, which Unity Catalog inherited automatically). The data engineering team eliminated approximately 40% of their ticket volume that had previously been data access requests.

The Lesson

Unity Catalog's integration with Azure Active Directory groups is its most operationally valuable feature: you manage access by managing group membership in the directory your organisation already uses, and Unity Catalog enforces the resulting permissions automatically across every workspace.

Go DeeperAdaptive Query Execution — how Spark optimises at runtime

Traditional Spark query optimisation happens before execution, based on table statistics (row counts, cardinality estimates). If those statistics are stale or absent, the plan may be suboptimal — the planner might choose a sort-merge join when a broadcast join would be faster, or allocate too many shuffle partitions for a small intermediate dataset.

Adaptive Query Execution (AQE), introduced in Spark 3.0 and enabled by default in Databricks Runtime 7.3+, allows Spark to revise the execution plan at runtime based on what it actually observes during execution. Three key AQE optimisations:

Dynamic coalescing of shuffle partitions: after a shuffle, AQE automatically merges small partitions into larger ones. A query that initially creates 200 shuffle partitions but produces a small intermediate dataset can be coalesced to 20 partitions, reducing task overhead without any configuration change.

Dynamic switch to broadcast join: if AQE determines at runtime that one side of a join is smaller than expected (after filtering), it can switch from a planned sort-merge join to a broadcast join, avoiding a full shuffle. This addresses a common pain point where filter statistics are not accurately reflected in the pre-execution plan.

Skew join optimisation: AQE detects skewed partitions (where one partition is significantly larger than others) at runtime and automatically splits them, distributing the work more evenly across executors. This directly addresses the data skew problem described in Section 2.2.

AQE is typically beneficial to leave enabled. The main exception: deterministic testing scenarios where you need predictable execution plans. spark.conf.set("spark.sql.adaptive.enabled", "true") to verify it is on.

Question 1 of 5

A Spark DataFrame transformation chain contains five filter, join, and select operations. When does Spark actually execute these transformations?

A
Each transformation executes immediately as it is called
B
Transformations are recorded in a DAG and executed only when an action (write, count, show) is triggered
C
Transformations are batched and executed every 30 seconds automatically
D
Transformations execute on the Driver; only writes are distributed to Executors

B is correct. Spark is lazy. Transformation methods (filter, join, select, groupBy, withColumn) do not execute when called — they add a node to the DAG. When an action is called (df.write, df.count(), df.show(), df.collect()), Spark optimises the full DAG and executes it. This laziness allows Spark to reorder and combine operations for efficiency — pushing filters before joins, eliminating unused columns, and batching reads. The most common consequence for practitioners: timing Spark operations is only meaningful at the action point, not at individual transformation calls.

Question 2 of 5

What is the correct Unity Catalog three-level namespace reference for a table named "dim_customer" in the "gold" schema of the "prod" catalog?

A
gold.dim_customer
B
prod/gold/dim_customer
C
prod.gold.dim_customer
D

C is correct. Unity Catalog uses dot-separated three-level notation: catalog.schema.table. In this case: prod.gold.dim_customer. This format is used in SQL queries, GRANT statements, and CLI references. The slash-separated path (option B) is the ADLS storage path format, not the Unity Catalog SQL reference format. Two-part references (option A) work only if a default catalog and schema are set for the session.

Question 3 of 5

Which Databricks cluster type is most cost-effective for a daily batch pipeline that runs for 45 minutes at 6am and then has no further usage?

A
An All-Purpose Cluster with auto-termination set to 45 minutes
B
A Job Cluster that is provisioned when the pipeline starts and terminated automatically when it finishes
C
A SQL Warehouse configured for batch compute mode
D
A shared cluster that multiple teams use throughout the day

B is correct. A Job Cluster is provisioned on demand when the job starts and terminated immediately when it completes — you pay only for 45 minutes of runtime. An All-Purpose Cluster with auto-termination (option A) could work but adds the risk of the cluster remaining running if the auto-termination is misconfigured, and costs slightly more in DBUs. SQL Warehouses (option C) are designed for SQL analytics, not batch pipeline compute. Shared clusters (option D) may have resource contention and retain state from other workloads.

Question 4 of 5

What does the Photon engine in Databricks accelerate most significantly?

A
Python UDFs applied row-by-row to DataFrames
B
SQL aggregations, joins, and Delta Lake scan operations — replacing JVM-based execution with a C++ vectorised engine
C

B is correct. Photon is a C++ vectorised query engine that replaces JVM-based physical execution for SQL aggregations, joins, Delta Lake scans, MERGE operations, and window functions. It typically delivers 2–10× improvement for these workloads. Photon does NOT accelerate Python UDFs (which run in the Python interpreter), custom Scala/Java code, or streaming DStream operations. For SQL-heavy Silver-to-Gold transformation pipelines, Photon is typically the single largest performance improvement available with zero code changes.

Question 5 of 5

A Databricks platform team wants to prevent data engineers from accidentally spinning up 50-node clusters for interactive notebook work. What is the correct governance mechanism?

A
Send a team email reminding engineers to use small clusters
B
Remove engineers' ability to create clusters entirely — only admins can create them
C
Implement Cluster Policies that engineers select when creating clusters, enforcing maximum node counts, auto-termination, and spot instance requirements
D
Create a shared all-purpose cluster that all engineers must use

C is correct. Databricks Cluster Policies are admin-defined configuration templates that enforce specific settings (max workers, auto-termination, instance types) while still allowing engineers to create their own clusters. Engineers select from policy templates rather than configuring clusters from scratch. This approach prevents wasteful configurations without restricting engineering autonomy. Removing cluster creation rights entirely (option B) would require engineers to file tickets for every cluster — creating toil and slowing development.

✓ Section Summary

Apache Spark distributes data across Executor processes on worker nodes and processes partitions in parallel. Spark is lazy: transformations build a DAG that only executes when an action is triggered. Shuffle (data redistribution across Executors) occurs at stage boundaries and is the most expensive operation. Databricks adds Photon (C++ vectorised engine for SQL/Delta, 2–10× faster), Unity Catalog (three-level namespace, centralised access control, lineage), Auto Loader (streaming file ingestion), and Databricks Asset Bundles (DABs, infrastructure-as-code). Cluster policies enforce cost governance. Job Clusters are the correct choice for production pipelines. Photon accelerates SQL and Delta operations but not Python UDFs or custom code.
