4.4.1

DAX vs M — Two Languages, One Tool

Power BI uses two distinct formula languages, and confusing them is the source of significant frustration for practitioners new to the platform. Understanding which language does what — and when each runs — is the essential foundation before writing a single formula.

M (Power Query) is the data preparation language. It runs during data refresh, before anything is loaded into the semantic model. M is responsible for connecting to sources, filtering, transforming, reshaping, and cleaning data — the equivalent of a SQL SELECT or a dbt model. M shapes the tables that land in the model. Once refresh completes, M has finished its work. It does not run at query time.

DAX (Data Analysis Expressions) is the analytical calculation language. It runs at query time — when a user interacts with a report, when a visual refreshes, when a measure is evaluated. DAX operates on the data that M has already loaded into the semantic model. It cannot change what data is in the model; it can only calculate against it. DAX writes measures (calculations evaluated in the current filter context) and calculated columns (values computed row by row at refresh time and stored in the model).

The rule of thumb: use M to clean and shape — anything that could be done in SQL or dbt belongs in M or upstream. Use DAX to calculate — ratios, running totals, period comparisons, and context-sensitive aggregations that must be evaluated dynamically against whatever filters a user has applied to the report.

4.4.2

Filter Context and Row Context

Everything in DAX ultimately comes down to understanding two evaluation contexts. These are the conceptual foundation that explains why DAX measures sometimes return unexpected values — and how to control what they return precisely.

Filter Context

Filter context is the set of filters currently applied to the data when a DAX measure is evaluated. It includes: slicer selections (Region = North), report page filters, visual-level filters, and the current row and column headers of a matrix visual. When a user selects "North" from a region slicer, the filter context for every measure on the page changes — the measure is now evaluated as if the data were already filtered to Region = North. Filters propagate through the model's relationships automatically: filtering dim_region to "North" also filters fact_sales to only the rows with North region FK values.

A measure receives its filter context from the report and cannot control it — unless CALCULATE is used to modify it deliberately.

Row Context

Row context is the current row being evaluated. It exists only inside calculated columns (which are evaluated row by row across an entire table) and inside iterator functions like SUMX and AVERAGEX. A calculated column expression [Revenue] = [Quantity] * [Unit Price] evaluates in row context — [Quantity] and [Unit Price] refer to the values in the current row. Measures do not have row context by default.

Context Transition

Context transition is the mechanism by which row context is converted into filter context when a measure is called from within a calculated column or an iterator function. When you call a measure inside SUMX (which iterates rows), CALCULATE implicitly converts the current row context into a filter context that filters the table to only the current row's values. This is one of the most commonly misunderstood behaviors in DAX — and the root cause of many unexpected results. The deep dive below covers it in detail.

Figure 4.7Filter context propagation in a Power BI semantic model. Page-level filters (Year = 2024, Business Unit = Advisory) establish the base context. Each matrix cell adds its row header (Region = North) and column header (Product = Product A) to the context. Filters propagate automatically through model relationships — filtering dim_region to "North" automatically restricts fact_sales to North rows only. 4.4.3

CALCULATE — The Most Important DAX Function

CALCULATE is what makes DAX fundamentally different from Excel formulas or SQL aggregations. It evaluates an expression in a modified filter context — allowing you to override, extend, or remove filters that the report has applied. Almost every non-trivial DAX measure uses CALCULATE directly or calls functions that use it internally.

The syntax: CALCULATE(<expression>, <filter1>, <filter2>, ...). The expression is what you want to calculate (usually a simple aggregation like SUM or COUNTROWS). The filter arguments modify the filter context before the expression is evaluated. CALCULATE does not change what data is in the model — it changes the lens through which the expression sees the data.

What CALCULATE's Filter Arguments Do

Each filter argument either adds a new filter, replaces an existing one, or removes filters entirely. CALCULATE([Revenue], dim_region[region] = "North") adds a region filter to whatever context exists — even if the report has already filtered to "South," this measure returns North revenue because the CALCULATE filter argument replaces the conflicting region filter. CALCULATE([Revenue], ALL(dim_region)) removes the region filter entirely, so the measure returns total revenue regardless of what region the user has selected.

Figure 4.8Three CALCULATE patterns. Left: adding a filter — DATESYTD extends the existing context with a year-to-date date filter, preserving other filters like Region=North. Centre: removing a filter — ALL(dim_region) strips the region filter entirely, enabling percentage-of-total measures that compare a selection against the unrestricted total. Right: USERELATIONSHIP — activates an inactive relationship for a single measure, enabling role-playing dimensions (introduced in section 4.1) where dim_date is joined to the fact table multiple times in different roles (order date, ship date, delivery date).
4.4.4

Context Modifiers: ALL, ALLEXCEPT, ALLSELECTED

Three functions remove filters from the evaluation context. Each removes a different scope of filters, and choosing the wrong one produces subtly wrong results that can pass visual inspection but fail under scrutiny.

ALL(table or column) removes all filters from the specified table or column, regardless of what the user has selected in the report. CALCULATE([Revenue], ALL(dim_region)) always returns total revenue across all regions, even if the user has selected North from a slicer. Used for: percentage of total, % share measures.

ALLEXCEPT(table, column1, column2, ...) removes all filters from a table except for the specified columns. CALCULATE([Revenue], ALLEXCEPT(dim_customer, dim_customer[segment])) removes all customer filters except segment — the measure evaluates across all customers within the currently selected segment. Used for: ranking within a group, share within a category.

ALLSELECTED(table or column) removes inner filters (from the visual itself) but preserves outer filters (from slicers and page-level filters). This is the correct function for "% of what's currently visible in this visual." If a user selects North from a slicer but the visual shows all regions, ALLSELECTED([Revenue]) returns the total for North only (respecting the slicer), not the total for all regions (which ALL would return). The most nuanced of the three and worth spending time with before using in production measures.

Go DeeperContext transition — the hardest concept in DAX, explained

Context transition is what happens when a measure is called from within a row context (inside a calculated column or an iterator function). CALCULATE implicitly converts the current row's values into a filter context that restricts the table to only that row's values.

The concrete example: You have a calculated column in dim_customer: Customer Revenue = [Total Revenue]. This seems like it should return the total revenue for that customer. But [Total Revenue] is a measure — and measures have no row context. Without context transition, [Total Revenue] would return the same grand total for every customer row.

With context transition (which CALCULATE triggers automatically when you reference a measure in a row context), the current row's dimension values — customer_sk = 1001, region = North, segment = Premium — are converted into filter context. The measure [Total Revenue] is then evaluated with those filters active, returning revenue for that specific customer. This is why referencing a measure inside a calculated column produces per-row values rather than a constant.

The dangerous implication: If you call [Total Revenue] inside SUMX without understanding context transition, you will get unexpected results. SUMX([customers], [Total Revenue]) does not sum Total Revenue across all customers — each row's measure call triggers context transition, filtering to that specific customer. The result is the same as SUM over per-customer revenue, which may or may not be what you intended.

Rule of thumb: When you reference a measure inside SUMX, AVERAGEX, or a calculated column, context transition is happening. Be explicit about what you want: if you want the total (not per-row), use the measure directly in CALCULATE with ALL(). If you want per-row, referencing the measure directly is correct — context transition will handle it.

4.4.5

Time Intelligence

Time intelligence is DAX's built-in set of functions for period comparisons, year-to-date calculations, and rolling windows. All time intelligence functions require a dedicated date dimension table (dim_date) with a contiguous date range — no gaps — marked as a date table in the Power BI model. This is why building a proper dim_date is required for any production Power BI model (referenced in section 4.1 — now you see why).

Core Time Intelligence Functions

SAMEPERIODLASTYEAR(dates) returns a set of dates one year earlier than the dates in the current filter context. Used in CALCULATE to compare this period to the equivalent period last year: Revenue LY = CALCULATE([Revenue], SAMEPERIODLASTYEAR(dim_date[date_actual])). Derive YoY % change as ([Revenue] - [Revenue LY]) / [Revenue LY].

DATEADD(dates, number, interval) shifts dates by a specified number of intervals (DAY, MONTH, QUARTER, YEAR). More flexible than SAMEPERIODLASTYEAR: CALCULATE([Revenue], DATEADD(dim_date[date_actual], -1, QUARTER)) returns prior quarter revenue.

TOTALYTD(expression, dates, [year_end_date]) calculates a year-to-date total. Revenue YTD = TOTALYTD([Revenue], dim_date[date_actual]). For fiscal years that don't end on 31 December, specify the year-end date: TOTALYTD([Revenue], dim_date[date_actual], "31-03") for a March fiscal year-end. TOTALQTD and TOTALMTD work identically for quarter-to-date and month-to-date.

🏢Field Note
A Manufacturing Company  ·  The Wrong Revenue Total

A manufacturing company's Power BI report showed a "Revenue by Product Category" matrix with a Grand Total row. The Grand Total consistently showed a higher number than the visible category rows added together — sometimes by 20%. Analysts assumed there was a calculation bug in the data warehouse and escalated to the data engineering team, who spent two days auditing Gold tables before finding no discrepancy. The data was correct. The measure was wrong.

The measure had been written as a calculated column that summed a revenue measure: Category Revenue = SUMX(fact_sales, [Unit Revenue Measure]). Because [Unit Revenue Measure] was a measure (not a column), calling it inside SUMX triggered context transition on every row of fact_sales — summing the revenue measure evaluated at the fact_sales row level, which included tax and shipping adjustments not visible at the category level. The Grand Total summed all adjusted values. The category rows, filtered by category, showed net category revenue before adjustments. Two different calculations, in the same report, called "Revenue."

The fix was replacing the calculated column with a direct measure using SUM over the correct column: Revenue = SUM(fact_sales[revenue_net]). Grand Total and category rows agreed immediately.

The Lesson

Context transition is silent. When a measure produces unexpected values — especially in Grand Total rows or when called from a calculated column — context transition is the first thing to audit. Never call a measure inside a calculated column or SUMX without explicitly understanding what context it will be evaluated in.

4.4.6

Variables and Iterators

VAR / RETURN

DAX variables (introduced to DAX in 2016) solve two problems at once: readability and performance. The pattern: declare named values with VAR, then use them in the RETURN expression. A variable is evaluated exactly once at the point of declaration and reused as needed — avoiding recalculation in complex measures.

Example — YoY revenue change percentage:

Revenue YoY % = VAR CurrentRevenue = [Revenue] VAR PriorRevenue = CALCULATE([Revenue], SAMEPERIODLASTYEAR(dim_date[date_actual])) RETURN DIVIDE(CurrentRevenue - PriorRevenue, PriorRevenue, BLANK())

Without variables, [Revenue] would be evaluated twice: once for CurrentRevenue and once inside SAMEPERIODLASTYEAR. With variables, it is evaluated once per RETURN call. At scale — a matrix with 200 cells — this halves the number of measure evaluations. Always use VAR/RETURN for any measure that references the same sub-expression more than once. Use DIVIDE() instead of the division operator (/) to handle division by zero gracefully — it returns BLANK() rather than an error when the denominator is zero.

Iterator Functions: SUMX, AVERAGEX, MAXX

Iterator functions evaluate an expression for each row in a table and aggregate the results. SUMX(table, expression) iterates each row, evaluates the expression in row context, and sums the results. This is powerful for row-level calculations that cannot be expressed as a simple column aggregation:

Weighted Revenue = SUMX(fact_sales, fact_sales[quantity_sold] * fact_sales[unit_price] * fact_sales[discount_factor])

This calculates the discounted revenue for each line and sums across all lines — not possible with a simple SUM if discount_factor is applied per-row. AVERAGEX, MINX, MAXX, and COUNTX follow the same pattern with different aggregation operations.

Go DeeperTime intelligence prerequisites — what breaks when dim_date is missing or incomplete

All DAX time intelligence functions have three requirements that must all be met before they work correctly:

1. A marked date table. In Power BI Desktop: select the date dimension table → Table tools → Mark as date table → specify the date column. Without this, time intelligence functions may work inconsistently or not at all. The marked column must be of type Date (not DateTime), must have no duplicate values, and must have no missing dates in the range you intend to use.

2. Contiguous date range with no gaps. If your dim_date has rows for 2020-2024 but is missing a bank holiday (because your source only included business days), TOTALYTD will give incorrect results for any period that includes the missing date. Your dim_date must include every calendar day in the range, regardless of whether any business activity occurred on that day. Generate dates programmatically: in Python or SQL, generate a row for every day from your start year to five years beyond your expected end year.

3. The relationship must be active. If your fact table has multiple date columns (order_date, ship_date, delivery_date) and only the order_date relationship is active, time intelligence functions will always filter by order_date. To apply time intelligence to ship_date, wrap the time intelligence function inside CALCULATE with USERELATIONSHIP — as shown in Figure 4.8.

Common symptom of a missing date table mark: SAMEPERIODLASTYEAR returns BLANK() for all periods. First check: has the table been marked as a date table? Second check: are there missing dates in the range?

🏢Field Note
An Insurance Company  ·  YoY Blank for Three Months

An insurance company launched a Power BI claims dashboard in October 2023 with a Year-on-Year claims comparison using SAMEPERIODLASTYEAR. The YoY measure worked correctly for all months except October, November, and December — for those three months it returned BLANK(). The data team confirmed that claims data existed for October–December 2022. The DAX formula was syntactically correct. The date table appeared complete.

The root cause: the dim_date in the Gold layer had been generated to cover 1 January 2023 through 31 December 2024. SAMEPERIODLASTYEAR attempts to find dates one year earlier than the current filter — for October 2023, it looks for October 2022 dates. Those dates did not exist in the dim_date table (which started at January 2023), so the function found no matching dates and returned BLANK(). The fix was regenerating dim_date to start from 1 January 2020, covering five years of historical comparison window.

The Lesson

Build dim_date to span a range significantly broader than your current data. A good rule: start at least five years before the earliest fact record and extend at least three years beyond the current date. The storage cost of 3,000 extra rows in dim_date is negligible. The debugging cost of a BLANK() YoY measure in production is not.

Go DeeperFILTER vs CALCULATE filter arguments — performance implications

There are two ways to apply a filter inside CALCULATE, and they have very different performance characteristics at scale:

Table filter (slow): CALCULATE([Revenue], FILTER(fact_sales, fact_sales[amount] > 1000)) — FILTER iterates every row of fact_sales, evaluates the condition for each row, and returns the matching rows. At 500 million rows, this is expensive.

Column filter (fast): CALCULATE([Revenue], dim_product[category] = "Electronics") — this uses the VertiPaq column filter directly, leveraging dictionary encoding and bitmap indexing. No row-by-row iteration. Dramatically faster at scale.

The rule: when filtering on a column value, use the column filter syntax directly in CALCULATE rather than wrapping it in FILTER(). Only use FILTER() when you need to filter based on a calculated measure value (FILTER returns rows where a measure is above a threshold) or when you need to filter on multiple columns of the same table in a way that the column filter syntax cannot express. When in doubt, try the column filter syntax first — if DAX accepts it, it's the faster option.

Practical example: North revenue share measure — recommended pattern using column filter syntax:

North Revenue Share = DIVIDE(CALCULATE([Revenue], dim_region[region] = "North"), CALCULATE([Revenue], ALL(dim_region)))

This uses two column filter arguments: one restricts to North, one removes all region filters. Both resolve at the storage engine level with no row iteration. A version using FILTER() for either argument would produce the same result but 10–50x more slowly at realistic fact table scales.

✓ Section Summary — Part 4 Complete

DAX is Power BI's query-time formula language (M handles refresh-time data preparation — never confuse the two). Everything in DAX operates within a filter context (the set of filters currently applied by the report) or a row context (the current row in a calculated column or iterator). CALCULATE is the function that modifies filter context: it can add filters, replace filters, remove filters (ALL/ALLEXCEPT/ALLSELECTED), and activate inactive relationships (USERELATIONSHIP — the role-playing dimension pattern from section 4.1). Time intelligence functions (SAMEPERIODLASTYEAR, DATEADD, TOTALYTD) require a marked date table with a contiguous date range extending beyond the data. VAR/RETURN patterns improve both readability and performance by evaluating shared sub-expressions once. Iterator functions (SUMX, AVERAGEX) evaluate expressions row by row and aggregate — correct for per-row calculations but dangerous without understanding context transition. Context transition is the silent conversion of row context to filter context when a measure is referenced inside a calculated column or iterator — the most common source of unexpected results in production DAX.

Question 1 of 5

A data engineer wants to calculate the percentage of total revenue that each product category contributes. The report has a region slicer. The denominator should always show total revenue for all regions regardless of what the user selects. Which DAX pattern is correct?

A
Revenue % = [Revenue] / TOTALYTD([Revenue], dim_date[date_actual])
B
Revenue % = DIVIDE([Revenue], CALCULATE([Revenue], ALL(dim_region))) — ALL removes the region filter so the denominator is always the grand total regardless of slicer selection
C
Revenue % = DIVIDE([Revenue], CALCULATE([Revenue], ALLSELECTED(dim_region)))
D
Revenue % = [Revenue] / SUM(fact_sales[revenue_net])

B is correct. CALCULATE([Revenue], ALL(dim_region)) removes all region filters from the evaluation context — CALCULATE overrides whatever the slicer has applied — so the denominator always returns total revenue across all regions. This is the standard "% of total" pattern. Option C (ALLSELECTED) would respect the slicer selection, making the denominator the total for the selected region(s) rather than the true grand total — correct for "% of what's visible" but wrong for "% of all revenue." Option A uses TOTALYTD which changes the date context, not the region context — not relevant here. Option D (SUM in a measure context) would behave identically to [Revenue] in this context and would also be affected by the slicer filter.

Question 2 of 5

What is the difference between ALLSELECTED and ALL when used as a CALCULATE filter argument?

A
ALLSELECTED removes all filters; ALL removes only the filters applied by slicers
B
They are functionally identical — ALLSELECTED is just an alias for ALL
C
ALL removes all filters regardless of origin (including slicers); ALLSELECTED removes filters from within the current visual but preserves filters from outer contexts like slicers and page-level filters
D
ALLSELECTED removes filters from the current column only; ALL removes filters from the entire table

C is correct. This distinction is critical for "% of visible total" vs "% of grand total." If a user selects "North" from a region slicer and the visual shows all products for North: ALL(dim_product) returns total revenue for North across all products (slicer still applies — only product filter removed). ALLSELECTED(dim_product) returns the same result in this case since no product filter is active within the visual. But if the visual itself also filters a subset of products, ALLSELECTED returns revenue for only those visible products while ALL returns revenue for all products. Use ALL for true grand total denominators. Use ALLSELECTED when you want the "% of what's currently showing in this visual" — a more nuanced and situational calculation.

Question 3 of 5

A SAMEPERIODLASTYEAR measure returns BLANK() for all months in Q4 of the current year. Q4 data exists in the fact table. What is the most likely cause?

A
The fact table does not have data for Q4 of the prior year
B
SAMEPERIODLASTYEAR only works for years that are complete — it cannot process the current year
C
The dim_date table does not include dates from Q4 of the prior year — SAMEPERIODLASTYEAR looks for dates one year earlier and finds no matching rows in dim_date
D
The dim_date table has not been marked as a date table in Power BI Desktop

C is correct. SAMEPERIODLASTYEAR(dim_date[date_actual]) returns the set of dates from dim_date that are exactly one year earlier than the dates in the current filter context. For Q4 2024, it looks for Q4 2023 dates in dim_date. If dim_date starts at 1 January 2024, those Q4 2023 dates do not exist in the table — the function returns an empty set and CALCULATE evaluates over no dates, returning BLANK(). The fix: extend dim_date's start date to cover at least as far back as the earliest comparison period you will ever need. Option D (not marked as date table) would cause issues across all time intelligence functions, not specifically Q4.

Question 4 of 5

Why should VAR/RETURN be used in measures that reference the same sub-expression more than once?

A
Variables allow the measure to be edited without republishing the semantic model
B
Variables make the measure available as a calculated column without additional code
C
Variables are evaluated once at declaration and reused without recalculation — improving performance by avoiding redundant measure evaluations, and improving readability by naming complex sub-expressions
D
Variables prevent context transition when a measure is used inside an iterator function

C is correct. A DAX variable is evaluated exactly once when the RETURN statement is reached, and its value is stored and reused anywhere it is referenced in the RETURN expression. A measure like YoY % = ([Revenue] - CALCULATE([Revenue], SAMEPERIODLASTYEAR(...))) / CALCULATE([Revenue], SAMEPERIODLASTYEAR(...)) evaluates the SAMEPERIODLASTYEAR expression twice. Rewriting with VAR evaluates it once. In a matrix with 500 cells, this halves the formula engine work for that calculation. Additionally, named variables make complex expressions self-documenting: VAR PriorYearRevenue = ... is clearer than an inline nested expression. Variables do not prevent context transition (D is incorrect — context transition is a CALCULATE behavior, not a variable behavior).

Question 5 of 5

A Power BI model has fact_sales with three date foreign keys: order_date_sk, ship_date_sk, and delivery_date_sk — all referencing dim_date. Only the order_date relationship is active. How do you write a measure that calculates revenue filtered by ship date while the report is filtered by order date?

A
Create three separate dim_date tables — one for each date role — and make all relationships active
B
It is not possible to filter by ship date and order date simultaneously in a single Power BI model
C
Use RELATED(dim_date[date_actual]) inside a calculated column on fact_sales to create a ship date column, then filter on that column
D
Use CALCULATE with USERELATIONSHIP(fact_sales[ship_date_sk], dim_date[date_sk]) to activate the inactive ship date relationship for that specific measure only, without affecting other measures

D is correct. USERELATIONSHIP is CALCULATE's mechanism for activating an inactive relationship within the scope of a single measure evaluation. The model maintains one physical dim_date table with three relationships to fact_sales — only one can be active at a time (the order_date relationship). USERELATIONSHIP temporarily activates the ship_date_sk → date_sk relationship for the duration of that specific CALCULATE expression, without deactivating the default relationship for any other measure. This is the role-playing dimension pattern introduced in section 4.1. Creating three separate dim_date tables (A) works but dramatically increases model size and creates maintenance overhead. The pattern is not technically impossible (B is wrong) — USERELATIONSHIP is the designed solution for exactly this scenario.
