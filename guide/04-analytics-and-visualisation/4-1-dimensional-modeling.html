4.1

Dimensional Modeling

Designing data for how analysts actually query it — star schemas, grain, fact and dimension tables, surrogate keys, and the patterns Power BI expects to find at the Gold layer

✓

  2. Define grain precisely and explain why it is always the first decision in fact table design
  3. Distinguish between fact tables and dimension tables, and select the correct fact table type for a given scenario
  4. Explain the purpose of surrogate keys and why they are architecturally required for Slowly Changing Dimensions

4.1.1

Why Dimensional Modeling Exists

Every operational system that stores business data — an order management platform, a banking ledger, a hospital patient record — is almost certainly built on a normalized schema. Normalization, specifically Third Normal Form (3NF), is a set of design rules that eliminate redundancy: each fact is stored in exactly one place, related data is split into separate tables, and nothing is duplicated. It is the right model for systems that write data thousands of times per second and cannot afford inconsistency between related records.

The problem is that what makes a schema good for writing makes it bad for reading at scale. A normalized e-commerce schema might record a single sale across seven or more tables — order header, order line, customer, address, product, category, promotion. To answer "what was total revenue by product category and customer region last quarter?" a query has to join all seven tables, apply aggregations across millions of rows, and resolve foreign keys through multiple indexes. At small data volumes this is merely slow. At the volumes typical of a Gold layer — tens of millions to billions of rows — it becomes unusable.

In the early 1990s, Ralph Kimball formalized a different approach: design the data model not for how data is written, but for how it will be read. His insight was that analytical queries follow a predictable pattern — they measure something (revenue, claims paid, units sold, call duration) across dimensions of analysis (time, geography, product, customer). If you model your data around that pattern rather than fighting against it, queries become fast, simple, and legible to business users who do not write SQL.

The critical distinction: 3NF optimizes for write integrity. Dimensional modeling optimizes for read performance and analytical clarity. The Gold layer of a Medallion architecture should almost always be dimensional — not because 3NF is wrong, but because it is wrong for this job.

This is not historical trivia. Every Power BI report, every Databricks Gold table queried by analysts, every Azure Synapse Analytics (a Microsoft analytics service integrating SQL and Spark workloads in one workspace) query against a data warehouse performs better — often dramatically better — when the underlying model is dimensional. Understanding dimensional modeling is not optional for a senior data engineer. It is the design language of the consumption layer.

🏢Field Note
A UK Retailer  ·  The Cost of 3NF in the Gold Layer

A mid-sized UK retailer migrated their reporting from an on-premise SQL Server data warehouse to Azure. The data engineering team faithfully reproduced the normalized structure of their source systems in the Gold layer of ADLS — 24 tables in 3NF, matching the OLTP schema. Power BI reports were connected via DirectQuery (a Power BI connection mode that queries the data source live rather than importing data into memory). Performance was catastrophic: simple sales-by-region reports timed out after 30 seconds. Analysts began building their own Excel extracts rather than using the dashboards, defeating the purpose of the migration entirely.

An architectural review identified the root cause immediately: DirectQuery against 3NF Gold required 8-to-12-table joins at query time, resolved by Power BI's engine on every page refresh. Power BI's VertiPaq columnar engine — designed to operate against pre-joined star schemas in Import mode — could not compensate. The fix required rebuilding the Gold layer as a star schema: one central fact_sales table and five dimension tables. After the rebuild, the same reports that had been timing out ran in under three seconds.

The Lesson

Migrating a 3NF schema to the cloud is not a data warehouse modernization. It is moving the wrong design to a new location. The Gold layer must be designed for consumption. Dimensional modeling is that design.

4.1.2

The Star Schema: Grain, Facts, and Dimensions

A star schema has two types of tables: a fact table at the centre and dimension tables surrounding it. When drawn on a whiteboard the fact table is the hub and the dimension tables are the spokes — which is where the name comes from. The model is deliberately simple. That simplicity is the feature, not a limitation.

Grain: The First and Most Important Decision

Before deciding what columns your fact table has, you must decide what one row represents. This is the grain. Not "sales data" — too vague. Not "one row per order" — closer, but still ambiguous if orders contain multiple products. A correct grain declaration for a retail sales fact looks like this: one row per individual order line item, representing the sale of a single product within a single customer order.

Grain determines everything downstream. Which measures are valid (you can sum line-level revenue across any dimension; you cannot sum order-header totals without double-counting when a customer has multiple lines). Which dimensions can be attached (a line-level fact can join to dim_product because each line has one product; an order-header fact cannot without losing granularity). How large the table grows (finer grain = more rows = larger table, potentially by orders of magnitude). The most common dimensional modeling failure is not choosing the wrong grain — it is failing to choose one explicitly at all, and discovering the ambiguity six months into production.

State the grain in one sentence before writing a single column name. Then validate it: "Can every row be described by this sentence? Does every measure I want to capture exist at this level of detail?" If yes to both — proceed. If not — restate the grain before continuing.

Fact Tables: Measurable Events

A fact table stores quantitative measurements of events — the things a business measures, aggregates, and reports on. Revenue, quantity sold, claim amount, call duration, page views. These are the measures: numeric values that are mathematically meaningful to sum, average, min, max, or count. Alongside the measures, a fact table contains foreign keys (FK) pointing to each surrounding dimension table, providing the context for each event: who, what, when, where, and how.

What a fact table does not contain: descriptive text, long string columns, or information about the entities involved beyond their keys. If you find yourself storing customer names or product descriptions directly in the fact table, you have either violated the grain or denormalized into the fact table attributes that belong in a dimension. The fact table is intentionally lean — foreign keys and numeric measures, nothing else.

Dimension Tables: Descriptive Context

Dimension tables provide the context for the numbers in the fact table. When a business user asks "what was revenue by customer region and product category?" — customer region lives in dim_customer, product category lives in dim_product, and revenue lives in fact_sales. Dimension tables are deliberately denormalized: rather than normalizing product into separate product, category, sub-category, and department tables (which would be 3NF), all that information is collapsed into a single wide dim_product table. This denormalization is what enables the one-or-two-join query pattern that makes star schemas fast in Power BI and Synapse.

Figure 4.1 The star schema for a retail sales subject area. The central fact_sales table stores one row per order line item (the grain), with five foreign keys pointing to surrounding dimensions. Each dimension is denormalized — all product hierarchy, all customer attributes, and all date properties collapsed into single wide tables. Arrows indicate FK relationships: dimension primary keys (surrogate keys) are referenced as fact foreign keys. A query joining any two of these tables requires exactly two joins — fact to one dimension. Any combination of all five requires five joins, not the 12+ joins of an equivalent 3NF schema.
4.1.3

Keys: Surrogate, Natural, and Business

One of the most consequential decisions in dimensional modeling is how you identify rows. Three types of keys appear in every dimensional model, and they serve very different purposes.

Surrogate Keys

A surrogate key is a system-generated integer with no business meaning: customer_sk = 1, 2, 3, 4…. The warehouse creates these keys; they mean nothing outside the warehouse. Every dimension table has a surrogate key as its primary key (PK), and the fact table stores surrogate keys as its foreign keys (FK). This is not a convention — it is the architectural contract of dimensional modeling, required for the approach to work correctly.

Why not use the natural key from the source system? Four reasons. First, natural keys can be recycled — a product code reassigned to a different product breaks historical reporting if you have not preserved the old version. Second, source systems change — when a CRM migration renumbers all customer IDs, a surrogate key insulates your warehouse from that upstream change completely. Third, Slowly Changing Dimensions (SCDs) — covered fully in section 4.2 — require surrogate keys. When a customer moves region and you want to preserve history, you insert a new row with a new surrogate key; without surrogate keys, you cannot have two coexisting versions of the same customer entity. Fourth, integer joins are faster than string joins at the scale of Gold layer fact tables.

Natural Keys and Business Keys

A natural key is the identifier from the source system (customer_id from the CRM, product_code from the ERP). A business key is an identifier the business recognizes and uses — an account number, a policy number, an ISIN (International Securities Identification Number) in financial services. These are often the same field. Both should be stored in the dimension table alongside the surrogate key — never discarded. They are the bridge back to source systems for debugging, lineage auditing, and reconciliation when numbers don't match.

Go DeeperGenerating surrogate keys in Databricks and dbt — two approaches

Two common approaches are used in the Azure stack, and both are valid for different reasons:

Hash-based surrogate keys (dbt + Databricks — recommended for most cases): The dbt_utils package provides dbt_utils.generate_surrogate_key(['natural_key_column']), which produces a deterministic MD5 or SHA-256 hash of the natural key. The same natural key always produces the same surrogate key — making pipelines idempotent (re-running never creates duplicates, because the key is derived rather than generated). The downside: hash strings are wider than integers (16–32 bytes vs 4–8 bytes) and are marginally slower on joins at extreme scale. At most Gold layer volumes, this is not a meaningful difference.

Sequence-based surrogate keys (Azure Synapse Analytics / SQL): Use an IDENTITY column or ROW_NUMBER() OVER (ORDER BY natural_key) for compact integers. Integer joins are faster and compress better in columnar storage. The complication: sequence-based keys are not deterministic — if you rerun the load with a new natural key, you must look up whether that key already has a surrogate assigned, which requires a lookup table or upsert pattern. More moving parts, but worth it for tables where join performance at billions of rows matters.

Pattern recommendation: Use hash-based keys for tables managed by dbt (idempotent by default, no lookup table needed). Use sequence-based identity keys for tables managed directly in Synapse Analytics or SQL Database where columnar compression and join performance justify the extra engineering. In both cases: always store the natural key alongside the surrogate key in the dimension table — never throw it away.

4.1.4

Four Types of Fact Tables

Not every measurable event fits the same fact table pattern. Kimball identified four distinct fact table types, each suited to a different analytical scenario. Choosing the wrong type is a common design mistake — usually discovered when someone needs to answer a question the design cannot support.

1. Transaction Fact Table (Most Common)

Records individual events at the moment they occur. One row per event: one sale, one insurance claim, one page view, one call. Grain is always fine — each row is a discrete, atomic event. Measures are the properties of that event: revenue, quantity, duration, amount. Transaction fact tables grow continuously and are almost always the largest tables in the warehouse. When someone says "fact table" without qualification, they almost certainly mean this type.

2. Periodic Snapshot Fact Table

Records the state of something at regular intervals, regardless of whether anything happened. One row per period per entity: daily inventory level per SKU (Stock Keeping Unit — the unique identifier for each product-variant-location combination) per warehouse, monthly account balance per customer, weekly headcount per department. Unlike transaction facts, periodic snapshots are populated even if nothing changed — the inventory level at end of Tuesday is recorded even if no stock moved. This allows analysts to answer "what was the state of X on date Y?" without needing to reconstruct it from a transaction log, which is both complex and error-prone.

3. Accumulating Snapshot Fact Table

Records one row per process instance, updated as the process moves through defined stages. Used for workflows with a known sequence of milestones: order placed → picked → packed → shipped → delivered → returned. The row is created when the process starts and updated in-place as each stage completes. Contains multiple date foreign keys — one per stage — many of which are NULL until their milestone is reached. Enables analysis of process duration and bottleneck identification: "on average, how long does the picked-to-shipped stage take?" is answerable directly from the accumulated row.

4. Factless Fact Table

Records that an event occurred — but carries no numeric measures. Used when the event itself is the meaningful data: a student enrolled in a course, an employee attended a training session, a promotion was active on a given day in a given store. The table contains only foreign keys. Measures are derived by counting rows, or by joining to other facts. The name is counterintuitive — a fact table with no facts — but the pattern is the only correct one when what you need to answer is "did event X happen?" rather than "how much of X happened?"

Go DeeperThe grain trap — why "one row per order" reliably breaks in production

The most common dimensional modeling mistake: stating a grain imprecisely, discovering the ambiguity after the table is live and the business has started trusting the numbers.

How it typically unfolds: A team designs fact_orders with the stated grain "one row per order." Six months later, an analyst reports that revenue figures don't match the source system. Investigation reveals: multi-product orders were being counted at order-header level, so an order for £100 (two items × £50 each) had one fact row with revenue = £100. The finance team had always understood "one row per order" to mean "one row per line" — a distinction the data team hadn't surfaced at design time. The source system matches. The warehouse is wrong.

Why a coarser grain cannot be patched later: A coarser grain cannot be drilled to a finer level without rebuilding the table from scratch. If your fact is at order-header level and the business later needs product-level revenue analysis, you cannot derive it — you must rebuild. You cannot reconstruct line-level data from header-level aggregates. The cost of getting the grain wrong is not a hotfix; it is a rebuild during a period when the business already trusts the wrong numbers.

The grain statement template: "One row per [event] for each [entity] at [level of detail]." Example: "One row per order line item for each individual product sold in each customer order." This precision eliminates ambiguity before a single column is named. Run this statement past at least two business stakeholders before designing the fact table. Their reactions will tell you whether the grain is unambiguous — or whether four different departments have four different interpretations of the same word.

4.1.5

Dimension Patterns

Beyond the basic dimension table, four patterns appear repeatedly in real dimensional models. Each solves a design problem that the basic star schema description does not address directly.

Conformed Dimensions

A conformed dimension is defined once and shared across multiple fact tables — using identical column names, identical surrogate key values, and identical business definitions. The canonical example is dim_date: one authoritative date dimension shared by fact_sales, fact_inventory, fact_claims, and every other subject area in the warehouse. Because all facts reference the same dim_date with the same date_sk values, a Power BI report can compare sales volume against inventory levels on the same calendar date, with no reconciliation needed — the shared key is the join. This is what Kimball called the "enterprise bus": the shared dimensional infrastructure that makes cross-subject-area analysis possible without custom integration work for every new report.

Role-Playing Dimensions

A role-playing dimension is a single physical table that is joined to a fact table multiple times in the same query, each time in a different logical role. The most common example: an order fact table with order_date_sk, ship_date_sk, and delivery_date_sk — all three reference dim_date, but represent different points in the order lifecycle. In SQL, handled with aliased joins: JOIN dim_date AS order_date ON f.order_date_sk = order_date.date_sk. In Power BI: one physical dim_date table with multiple relationships to the fact — all but one inactive by default, activated in DAX (Data Analysis Expressions — the formula language of Power BI) using USERELATIONSHIP() when needed.

Junk Dimensions

Low-cardinality flags and codes that do not naturally belong to a main dimension create a design problem: adding them directly to the fact table produces a wide, poorly documented mess of boolean columns. A junk dimension consolidates these attributes into a single purpose-built dimension table. Example: is_online (Y/N), order_type (Standard/Rush/Return), payment_method (Card/Cash/BNPL), is_gift_order (Y/N). These four attributes have 2 × 3 × 3 × 2 = 36 possible combinations. The junk dimension has 36 rows — one per unique combination — and the fact table stores a single FK. Cleaner fact table, fully queryable in Power BI, no change to the fact schema when a new flag is added.

Degenerate Dimensions

A degenerate dimension is a dimension identifier stored directly in the fact table — with no associated dimension table. The classic example is invoice_number, transaction_id, or ticket_number: highly granular identifiers that carry no attributes worth dimensioning separately. Storing the invoice number directly in the fact avoids a dimension table with a single column and no rows joining to anything useful. Degenerate dimensions are also used when you need to group or filter on a transaction identifier: "show me all line items from order INV-20240815-00142" — the order number lives in the fact, not in a dimension.

🏢Field Note
A Financial Services Firm  ·  The Grain Conversation

A financial services data team was designing a new fact table to support regulatory reporting and analytics across three business areas: trading, settlements, and risk. The initial requirements document stated the grain as "one row per transaction." The data engineer ran a structured grain validation session with a representative from each area.

Trading defined "transaction" as one row per trade execution — a single filled order, even if executed across multiple parts in the market. Settlements defined it as one row per settlement instruction — which could map one-to-many with trade executions. Risk defined it as one row per end-of-day position — an entirely different concept that aggregated across all activity in a security. All three had used the word "transaction" in the requirements document and meant three completely different things.

The outcome was three separate fact tables at three different grains — fact_executions, fact_settlement_instructions, and fact_eod_positions — all linked by conformed dimensions. This was the correct answer. It required one additional two-hour scoping session to reach. Without that session, the team would have built a single fact table that satisfied none of the three use cases correctly, and the error would have been discovered only after the tables were live and the regulatory reporting cycle had begun.

The Lesson

The grain conversation is not a technical exercise — it is a stakeholder alignment exercise. Different business areas use the same words to mean different things. The data engineer's job is to surface that ambiguity before it becomes a production defect.

4.1.6

Schema Variants: Star, Snowflake, and Data Vault

Dimensional modeling is not a single rigid pattern — it is a family of related approaches. The star schema is the baseline. Two significant variants exist: the snowflake schema and Data Vault. Each makes different trade-offs between query performance, storage efficiency, auditability, and implementation complexity.

Star Schema (Recommended Default)

Dimension tables are fully denormalized. All attributes related to a dimension — including hierarchical relationships between categories, sub-categories, and departments — are collapsed into a single wide table. A dim_product with 40 columns beats a 3NF product schema with eight tables for analytical workloads every time. Maximum of two joins for any query. The Microsoft-recommended pattern for Power BI: VertiPaq, Power BI's in-memory columnar engine, compresses repeated string values in denormalized dimension tables exceptionally efficiently. The counterintuitive result is that denormalized star schemas often consume less memory in Power BI than equivalent snowflake schemas, despite the apparent data redundancy.

Snowflake Schema

Dimension tables are partially or fully normalized. A dim_product in a snowflake schema might join to a dim_category, which joins to a dim_department. This reduces storage (no repeated category name for every product row) but adds joins and complexity. Power BI handles snowflake schemas but with query performance penalties that worsen as nesting depth increases. Use snowflake schema only when a taxonomy is genuinely deep and volatile enough that maintaining a denormalized copy is impractical to keep synchronized. In most Azure warehouse scenarios, the storage savings do not justify the query performance cost or the increased model complexity in Power BI Desktop.

Galaxy Schema (Enterprise Bus)

Not a distinct schema type — the natural result of a mature dimensional model. Multiple fact tables at different grains sharing conformed dimensions. Sales, inventory, and HR each have their own fact table; all share dim_date, dim_store, and dim_employee. The "enterprise bus matrix" is a planning tool: a grid mapping which conformed dimensions each fact table uses. Useful for communicating the intended scope of a warehouse design to stakeholders, and for identifying which dimensions need to be conformed before the next subject area can be built.

Figure 4.2 Three schema patterns compared at a glance. 3NF (left): correct for transactional OLTP systems, wrong for analytical Gold layers — too many joins, too slow, Power BI hostile. Star schema (centre): the Microsoft-recommended pattern; denormalized dimensions, 1–2 joins maximum, VertiPaq-optimized for excellent Power BI performance. Snowflake (right): normalized sub-dimensions reduce storage but add join depth and degrade Power BI query performance. Use the star schema by default. Adopt snowflake only when a taxonomy is deep and genuinely volatile — not simply because it looks more "normalized."

Go DeeperData Vault — the alternative architecture for regulated enterprise environments

Data Vault is a modeling methodology developed by Dan Linstedt, designed for environments where auditability, parallel loading, and full historical accuracy are non-negotiable — financial institutions, insurance companies, healthcare organizations. It is genuinely different from Kimball dimensional modeling; not an extension or a variant, but an alternative approach to raw data storage.

Data Vault has three component types:

Hubs store business keys — the core identities (Customer, Product, Policy). One row per unique business key, with a load timestamp and a record source column. No descriptive attributes in the Hub itself — just the key and its provenance.

Links record relationships between Hubs — the equivalent of a many-to-many association table. A sale linking a Customer to a Product at a Store on a Date would be a Link table, with FK references to each Hub's surrogate key.

Satellites store descriptive attributes and their complete history. When a customer's email changes, a new Satellite row is inserted with a new load timestamp — the old row is never updated or deleted. All versions are preserved permanently. This is full historical tracking by design, not a special case handled by SCD logic.

When Data Vault is the right choice: Organizations with regulatory obligations to maintain an immutable audit trail of every change (Solvency II, Basel III, HIPAA). Environments with many source systems feeding the same entities — a customer may exist in the CRM, the ERP, and the billing platform, each with slightly different attributes. Teams that need to load from multiple sources in parallel without coordinating load order (Data Vault's Hub-Link-Satellite structure makes parallel loading safe by design).

The critical caveat: Raw Vault tables are not directly consumable by Power BI or business analysts. They are a storage and governance layer. You must build a Business Vault (applying business rules and data quality) and then a Gold mart (a star schema) on top before anything is queryable. Data Vault changes how you store and audit data — you still end up with a star schema at the consumption layer. It is a more expensive path to the same destination, justified only when the auditability and parallel loading properties are required.

✓ Section Summary

Dimensional modeling exists because 3NF schemas optimized for write integrity perform poorly at analytical read workloads. The star schema — a central fact table plus surrounding dimension tables — is the standard Gold layer pattern. Grain (what one row means precisely) is always the first design decision. Every dimension table has a surrogate key as its PK; the fact table stores surrogate keys as FKs. Natural keys are preserved in the dimension for lineage and debugging. Four fact table types: transaction (most common — discrete events), periodic snapshot (state at regular intervals), accumulating snapshot (process pipeline milestones), factless (events with no numeric measure). Four dimension patterns: conformed (shared across subject areas), role-playing (same table in multiple join roles), junk (low-cardinality flags consolidated), degenerate (key stored in fact, no dimension table). Star schema is the Power BI default — VertiPaq compresses denormalized columns efficiently. Snowflake adds joins and degrades performance. Data Vault is the enterprise alternative for regulated environments but always requires a Gold star mart on top for consumption.

Question 2 of 5

Why are surrogate keys architecturally required for SCD (Slowly Changing Dimension) Type 2, rather than natural keys from source systems?

A
Natural keys contain PII (Personally Identifiable Information) that cannot be stored in a dimension table
B
SCD Type 2 creates multiple dimension rows for the same entity (one per historical version) — surrogate keys give each version a unique PK while the natural key remains the same across all versions
C
Surrogate keys are required because Power BI cannot create relationships on string columns
D
Natural keys cannot serve as primary keys in Azure Data Lake Storage

B is correct. SCD Type 2 preserves history by inserting a new row for each version of a dimension record. A customer who moves from London to Manchester gets two rows in dim_customer — one for each address version. Both rows share the same natural key (customer_id = 12345) but must have different surrogate keys (customer_sk = 1001 and 1002). The fact table FK references the surrogate key, so historical sales facts correctly point to the "London version" of the customer, while new sales point to the "Manchester version." Without surrogate keys, two rows for the same customer_id would create a non-unique primary key — which makes the join semantics undefined and the design architecturally invalid.

Question 3 of 5

A supermarket needs to track the stock level of every SKU in every store at the end of each day — including days when no stock movement occurred. Which fact table type is correct?

A
Transaction fact table — one row is inserted per stock movement event
B
Periodic snapshot fact table — one row per day per SKU per store, regardless of whether any movement occurred
C
Accumulating snapshot fact table — one row per SKU, updated daily with the latest stock level
D
Factless fact table — inventory presence is an event, not a measure

B is correct. The key phrase is "including days when no stock movement occurred." A transaction fact records only events — it cannot represent the absence of activity. If no stock moved on Tuesday, a transaction fact has no Tuesday rows for that SKU, making it impossible to answer "what was the stock level on Tuesday?" without reconstructing it from all prior transactions. A periodic snapshot is designed exactly for this: it photographs the state of the entity (stock level) at a regular interval (end of day), always creating a row whether or not anything changed. This makes point-in-time queries trivially simple — SELECT * WHERE date_sk = 'Tuesday' AND sku_sk = X.

Question 4 of 5

A fact table has three date columns: order_date_sk, ship_date_sk, and delivery_date_sk — all three referencing dim_date. What dimension design pattern does this represent?

A
A conformed dimension — dim_date is being shared across multiple fact tables
B
A junk dimension — date-related flags are consolidated into a single table
C
A role-playing dimension — the same physical dim_date table is joined multiple times to the same fact, each in a different logical role
D
A degenerate dimension — date keys are stored in the fact with no associated dimension table

C is correct. A role-playing dimension is a single physical table joined to a fact multiple times, each time serving a distinct analytical purpose. In SQL: three separate JOINs to dim_date, each with a different alias (AS order_date, AS ship_date, AS delivery_date). In Power BI: one physical dim_date table with multiple relationships to the fact — only one active by default, additional roles activated using DAX USERELATIONSHIP(). A conformed dimension (A) refers specifically to sharing the same dim_date across multiple subject areas (fact_sales AND fact_inventory), which is a related but different concept.

Question 5 of 5

Microsoft recommends star schemas over snowflake schemas for Power BI semantic models. What is the primary technical reason for this recommendation?

A
Star schemas consume less storage than snowflake schemas, reducing Azure data lake costs
B
Snowflake schemas are incompatible with Power BI's row-level security (RLS) feature
C
Star schemas support more simultaneous users because Power BI caches dimension joins at load time
D
Power BI's VertiPaq in-memory engine compresses repeated string values in denormalized dimension columns extremely efficiently, and fewer join relationships means the query engine can fully leverage its columnar storage advantage without cross-table resolution overhead

D is correct. Power BI uses the VertiPaq columnar in-memory engine (the same engine as Analysis Services). VertiPaq applies value encoding and run-length compression to columns — it compresses repeated string values (product category, customer region, month name) in denormalized dimension tables to a small fraction of their logical size. A wide dim_product with 40 columns and 50,000 rows may compress to a few megabytes in memory. Snowflake schemas force the engine to resolve additional join relationships across tables at query time, adding overhead that negates the columnar advantage. The counterintuitive result: in Power BI's Import mode, denormalized star schemas typically consume less memory and return results faster than equivalent snowflake schemas — despite the apparent redundancy in the data.

Grain First, Always

State the grain in one sentence before designing any column. Every downstream decision — which measures are valid, which dimensions attach, how large the table grows — flows from the grain statement.

Surrogate Key Rule

Every dimension table has a surrogate key as its PK. The fact table stores surrogate keys as FKs. Keep the natural key in the dimension for lineage — never discard it, and never put it in the fact table.

Snowflake Penalty

Each additional join depth in a snowflake schema degrades VertiPaq performance in Power BI. Denormalize dimensions into a single wide table unless there is a specific, justified reason not to.

Interview Insight

"Design a data model for X" is a standard senior data engineering interview question. Start by stating the grain. Then name the fact table type. Then list the dimensions. Grain-first thinking is the signal interviewers are watching for.
