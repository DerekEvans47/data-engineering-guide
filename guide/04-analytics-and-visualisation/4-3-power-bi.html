Power BI

Microsoft's business intelligence platform — semantic models, storage modes, row-level security, deployment pipelines, and where Power BI fits in the evolution toward Microsoft Fabric

4.3.1

What Power BI Is and Where It Fits

Power BI is Microsoft's business intelligence (BI) platform — the consumption layer of the Azure data stack. It connects to data sources, models data using a semantic layer (a business-friendly abstraction over physical tables), and renders interactive reports and dashboards for business users. It sits at the top of the four-layer architecture: everything built in Parts 1 through 3 of this guide — the ingestion, storage, transformation, and Gold-layer dimensional models — exists so that Power BI can answer questions quickly and accurately.

Power BI has three distinct components that practitioners regularly confuse. Power BI Desktop is the free Windows application used by data engineers and analysts to build semantic models and reports. Power BI Service is the cloud-hosted platform (app.powerbi.com) where reports are published, shared, scheduled for refresh, and governed. Power BI Mobile is the iOS/Android app for consuming published reports. A data engineer's primary concern is Power BI Desktop (model design and DAX) and Power BI Service (deployment, refresh, governance, and security).

The semantic model is the product. Reports are windows into the model. Building a report without a well-designed semantic model is like building a house without foundations — it works until it doesn't, and fixing it after the fact is far more expensive than designing correctly from the start.
4.3.2

The Semantic Model

A Power BI semantic model (previously called a dataset — Microsoft renamed it in 2023) is a structured, calculated, and governed version of the data that reports query against. It contains: the data tables (either imported into memory or referenced via DirectQuery), the relationships between those tables, calculated measures written in DAX (Data Analysis Expressions — Power BI's formula language, covered fully in section 4.4), calculated columns, and row-level security rules. Every Power BI report connects to a semantic model, either one embedded within the same .pbix file or a published shared model in the Service.

The semantic model is where the star schema from section 4.1 becomes directly relevant. Power BI's VertiPaq (a high-performance in-memory columnar storage engine) is optimized for the star schema pattern: a central fact table joined to surrounding dimension tables. The engine compresses repeated string values in denormalized dimension columns with exceptional efficiency, which is why star schemas perform better in Power BI than equivalent snowflake or 3NF schemas — as established in section 4.1.

Measures vs Calculated Columns

Two ways to create new fields in a Power BI model, and choosing incorrectly has significant performance consequences. A calculated column is evaluated row by row at refresh time and stored in memory for every row of the table — it increases model size. Use calculated columns only when you need the value to be available as a slicer/filter or as a sort order that cannot be derived at query time. A measure is evaluated at query time, in the current filter context, and stored nowhere — it is computed fresh for each visual. Measures are almost always the correct choice for anything that aggregates (total revenue, average order value, customer count). The rule: if it aggregates, it's a measure. If it must exist as a property of each row, it's a calculated column.

4.3.3

Storage Modes: The Most Important Technical Decision

How a Power BI model connects to its data source is the single most consequential technical decision in any Power BI implementation. The four storage modes make different trade-offs between query speed, data freshness, model size, and feature completeness. Choosing the wrong mode early in a project is one of the most common and costly BI architecture mistakes.

Import Mode

Data is extracted from the source, compressed by VertiPaq, and stored in the Power BI model's in-memory engine. Every query against the report runs against this in-memory copy — no network round-trips to the source. Result: exceptionally fast query performance regardless of source system load. The limitations: the dataset must be refreshed on a schedule (the maximum is eight times per day on Pro, 48 on Premium) and there is a dataset size limit (1 GB for Pro workspaces, 25 GB for Premium per user, larger for Premium capacity). Import is the default recommendation. Start here unless a specific constraint forces you elsewhere.

DirectQuery Mode

No data is stored in the model. Every report visual generates a SQL query sent to the source system at the moment a user interacts with the report. Data is always live — the report shows whatever is in the source right now. The trade-off: every interaction requires a network round-trip and a query execution at the source. Performance depends entirely on source system speed and the efficiency of the generated SQL. DAX functionality is also reduced in DirectQuery: time intelligence functions, calculated columns, and some filter operations either don't work or work differently. DirectQuery is appropriate when: data must be real-time (financial positions, live operational metrics), the dataset is too large to import, or data residency requirements prevent copying data into Power BI.

Composite Mode

Individual tables within a single model can have different storage modes — some imported, some DirectQuery. The canonical pattern: large fact tables on DirectQuery (live, no size limit), dimension tables imported (fast, cached). When a report filters by a dimension (customer region, product category), the filter resolves against the in-memory dimension and is then pushed down to DirectQuery for the fact. This hybrid approach captures most of the performance benefit of Import for filtering while keeping fact data live. Composite mode adds complexity — table interactions, aggregations, and RLS behavior all require careful consideration. Only adopt it when the trade-off is clearly justified.

Live Connection

Connects to an existing published Power BI semantic model, an Azure Analysis Services model, or a SQL Server Analysis Services (SSAS) model. The report has no model of its own — it reads exclusively from the connected model. No modifications to the data model are possible from the report file. This is the pattern for enterprise BI governance: a central team owns and publishes the semantic model; report authors across the business connect to it without modifying the underlying definitions. Measures defined in the central model are available to all reports, ensuring consistent metric definitions.

4.3.4

Row-Level Security

Row-Level Security (RLS) restricts which rows of data each user can see, without creating separate reports or models per audience. A sales manager in the North region sees only Northern sales data; a national director sees everything — from the same published report. RLS is enforced at the semantic model level, not the report level, meaning it cannot be bypassed by a user exporting data to Excel or connecting a new report to the same model.

Static RLS

Static RLS defines fixed filter rules in Power BI Desktop. Each role has a DAX filter expression applied to one or more tables: [Region] = "North". Users are assigned to roles in the Power BI Service. Straightforward to implement but requires manual maintenance as regions, territories, or user assignments change. Suitable for small, stable audience segmentations.

Dynamic RLS

Dynamic RLS uses the USERNAME() or USERPRINCIPALNAME() DAX function to filter data based on the identity of the signed-in user, matched against a security table in the model. The DAX filter expression: [email] = USERPRINCIPALNAME() applied to a security mapping table (user_email → region), which then filters the dimension and propagates through the model via table relationships. No role assignment is required for each individual user — just ensure the user_email exists in the security table. This is the scalable pattern for enterprise deployments: onboard a new regional manager by adding one row to the security table, not by modifying the Power BI model.

Go DeeperDynamic RLS implementation — security table design and testing

Security table design: Create a dim_user_security table in your Gold layer with columns: user_email (TEXT), region (TEXT), or whatever dimension attributes the user is permitted to see. Maintain it as a regular Gold dimension — loaded from HR or identity management systems, refreshed on the same schedule as the rest of the model.

DAX filter for dynamic RLS: In Power BI Desktop, create a role (e.g., "Regional Users"). In the role's table filter for dim_user_security: [user_email] = USERPRINCIPALNAME(). The relationship from dim_user_security to dim_region (or directly to the fact via region_code) propagates the filter automatically — users only see data for regions where their email appears in the security table.

Testing RLS before publishing: In Power BI Desktop → Modeling tab → View as → select a role and enter a test email. The report switches to that user's perspective. This is the only way to verify RLS logic before it reaches production users. Always test with: (1) a user with one region assignment, (2) a user with multiple regions, (3) an admin user who should see everything (usually exempt from RLS via role assignment in the Service).

The admin exemption: Members of the workspace Admin and Member roles in Power BI Service bypass RLS by default. Viewers (the role for end-user consumers) are subject to RLS. Verify that report consumers are assigned the Viewer role, not Member or Admin, or RLS will silently have no effect for those users.

🏢Field Note
A Logistics Company  ·  DirectQuery at Scale

A logistics company built their Power BI reporting stack on DirectQuery against their Azure SQL Database Gold layer, reasoning that live data was essential for operational decision-making. Reports were delivered to 240 concurrent users across three shifts. Within two weeks of go-live, the Azure SQL Database was running at 95% CPU during shift handovers — when all three shifts were simultaneously active and all 240 users were refreshing their dashboards.

Each Power BI visual generates between one and five SQL queries. With 240 users, 15 visuals per report page, and three pages open at handover, the database was receiving upward of 5,000 simultaneous queries. The Gold layer, designed for analytical batch queries, was not built to handle this OLTP-style request pattern. Report load times degraded from under two seconds to over 45 seconds.

The fix was a hybrid approach: Import mode for all dimension tables and the previous-day summary fact table (covering 95% of report use cases), DirectQuery only for the live-position table that genuinely required real-time data. Report load times returned to sub-two-second. The Azure SQL Database CPU dropped to under 20% at peak. The "real-time" requirement had been real for one table — not for all 18 tables in the model.

The Lesson

Before choosing DirectQuery, audit which tables genuinely need to be live versus which can tolerate a scheduled refresh. The answer is almost always "fewer than you think." Composite mode gives you real-time where it matters and Import performance everywhere else.

4.3.5

Power BI Service — Workspaces, Deployment Pipelines, and Governance

Power BI Service is where models are published, governed, and consumed. Understanding its organizational structure is essential for any engineer responsible for delivering and maintaining BI solutions at scale.

Workspaces

Workspaces are containers for Power BI content — semantic models, reports, dashboards, dataflows. Every workspace has access control (Admin, Member, Contributor, Viewer roles). The My Workspace is personal and cannot be shared. All production content should live in a shared workspace, not My Workspace. Workspaces map to teams or subject areas: one workspace per business domain (Finance, Operations, HR) or per project, depending on organizational scale.

Deployment Pipelines

Power BI's built-in Dev/Test/Prod promotion mechanism. Three linked workspaces (Development, Test, Production) where content is promoted stage by stage — with the ability to configure stage-specific data source connections (Development points to a test database; Production points to the live Gold layer). Report authors work in Development, validate in Test, and promote to Production without modifying the report itself. This mirrors the software engineering CI/CD pattern and prevents the common problem of developers editing production reports directly.

Dataflows

Dataflows are Power Query transformations running in the Power BI Service rather than in Power BI Desktop. They produce reusable, cloud-executed datasets that multiple semantic models can connect to — avoiding the pattern where the same transformation logic is duplicated across ten different .pbix files. Think of dataflows as the Power BI equivalent of dbt models: centralized, reusable transformation logic. For teams without a mature Azure data platform, dataflows can serve as a lightweight Silver layer. For teams with a proper Gold layer in ADLS/Databricks, dataflows are largely unnecessary — your Gold tables are already the reusable, curated datasets.

+
Go DeeperMicrosoft Fabric — what it is and what it means for your current stack

Microsoft Fabric (announced 2023, GA since late 2023) is Microsoft's attempt to unify the entire data stack into a single platform: data engineering (Spark notebooks, Delta Lake), data warehousing (Fabric Warehouse), data integration (Data Factory pipelines), real-time analytics (KQL databases, Event Streams), data science (ML experiments), and Power BI — all under one licence, one governance model (Microsoft Purview), and one storage layer (OneLake).

What changes for Power BI users: Power BI is now embedded natively inside Fabric. The Power BI experience in the Fabric portal is identical to the current Service. Semantic models, reports, and workspaces continue to work exactly as before. The benefit: direct connectivity from Power BI to Fabric lakehouses and warehouses without leaving the platform, and shared governance between the data and analytics layers.

What changes for data engineers: Fabric's unified compute uses Spark (via Fabric notebooks) on top of Delta/Parquet files in OneLake. If you already run Databricks + ADLS, you have 80% of what Fabric provides — but as two separately licensed, separately governed products. Fabric consolidates this into one. Microsoft is positioning Fabric as the successor to the separate Azure-stack-plus-Power-BI architecture.

Migration urgency: Low for most organizations in 2025–2026. Existing Azure + Power BI stacks continue to be fully supported. Fabric is the direction of travel — not an immediate migration requirement. Begin evaluating Fabric for net-new projects and greenfield builds. Do not migrate a working production stack without a clear business case for the consolidation benefits.

Go DeeperDirectQuery performance optimization — query folding, aggregations, and dual storage

When DirectQuery is unavoidable, three techniques materially improve performance before resorting to Composite mode:

Query folding: Power BI should be able to translate every step of a Power Query transformation into a single SQL query sent to the source. When folding works, the source database applies filters and aggregations before returning data — reducing network transfer dramatically. When folding breaks (usually from a step Power Query cannot translate to SQL, such as a custom function or a complex merge), Power BI pulls all rows and processes them in memory. Check folding status in Power Query Editor: right-click a step → "View Native Query." If this option is greyed out, folding has broken at that step. Reorder or rewrite steps to restore folding.

Pre-built aggregation tables: Power BI supports user-defined aggregation tables — pre-aggregated versions of large fact tables stored in Import mode. When a report visual can be answered by the aggregation table (the query grain is at or above the aggregation grain), Power BI automatically uses the Import copy instead of querying the DirectQuery source. Users see Import-speed performance for common queries and DirectQuery for detail-level drillthrough. Configure aggregation tables in Power BI Desktop → Manage aggregations on the aggregation table.

Dual storage mode: Individual dimension tables in a Composite model can be set to Dual storage mode — they exist both as an Import copy and as a DirectQuery reference. When Power BI queries fact data via DirectQuery and filters via a dimension, it selects whichever dimension source (Import or DirectQuery) produces the most efficient execution plan. Dual mode tables are cached for fast local filtering but can still participate in DirectQuery query generation when needed. Set via column storage properties in the Model view.

🏢Field Note
A Professional Services Firm  ·  Deploying RLS for 12 Business Units

A professional services firm needed a single Power BI report covering project performance across 12 business units. Each unit head should see only their own unit's data. Partners should see all 12. The initial design used static RLS — 12 roles, one per business unit, each with a hardcoded filter like [business_unit] = "Advisory". When a new business unit launched six months later, the data engineer had to open Power BI Desktop, add a 13th role, republish the dataset, and manually assign every new unit manager in the Service.

The team rebuilt using dynamic RLS with a user_security table maintained in Databricks Gold. The security table had one row per user per business unit (some users had rows for multiple units). The DAX filter used USERPRINCIPALNAME() to look up the current user's permitted units. Adding a new business unit required inserting rows into the security table — no Power BI Desktop changes, no republish. When the firm added four new units over the following year, each was live in the BI platform within 20 minutes of the data engineer updating the security table.

The Lesson

Static RLS is a prototype pattern. For any production deployment with more than five audience segments or any likelihood of organizational change, implement dynamic RLS from day one. The security table belongs in your Gold layer alongside every other governed dimension.

✓ Section Summary

Power BI is the consumption layer of the Azure data stack, built around a semantic model (VertiPaq-powered) that sits atop the Gold-layer dimensional model built in sections 4.1 and 4.2. Four storage modes: Import (data in memory — fastest, scheduled refresh, size-limited — default choice), DirectQuery (live queries to source — always fresh, slow, reduced DAX), Composite (hybrid — Import for dims, DirectQuery for facts), Live Connection (read-only connection to existing published model). Row-level security is enforced at model level: static RLS uses hardcoded role filters; dynamic RLS uses USERPRINCIPALNAME() matched to a security mapping table — the scalable pattern for enterprise deployments. Power BI Service organizes content in workspaces and promotes between Dev/Test/Prod via deployment pipelines. Dataflows provide reusable cloud-executed Power Query transformations — less relevant when a mature Gold layer already exists. Microsoft Fabric is the strategic direction: unified data + BI platform on OneLake — evaluate for new projects, do not rush migration of working production stacks.

Question 1 of 5

A fact table containing 2.5 billion rows cannot be imported into a Power BI Pro workspace due to the 1 GB dataset size limit. The business requires data no older than 15 minutes. Which storage mode is most appropriate?

A
Import mode with a scheduled refresh every 15 minutes
B
DirectQuery for all tables in the model
C
Composite mode — dimension tables imported for fast filtering, fact table on DirectQuery for near-real-time data without size constraints
D
Live Connection to an Azure Analysis Services model

C is correct. Composite mode addresses both constraints simultaneously. The fact table's 2.5 billion rows exceed the Import size limit, ruling out full Import. Pure DirectQuery would work but would make every report interaction slow — 2.5 billion row queries against any source are expensive. Composite mode imports dimension tables (they are small — typically thousands to hundreds of thousands of rows) so filter operations are fast and cached, while the fact table remains on DirectQuery so its size isn't a constraint and data is near-real-time. Import with 15-minute refresh (A) doesn't help with the size limit and Power BI Pro only allows 8 refreshes per day. DirectQuery for all tables (B) would work but performs poorly at this scale without optimization.

Question 2 of 5

What is the correct DAX function to use in dynamic RLS to retrieve the identity of the currently signed-in Power BI user?

A
CURRENTUSER()
B
LOOKUPVALUE()
C
USERPRINCIPALNAME() — returns the UPN (user principal name / email) of the authenticated user, used to filter the security mapping table
D
USERNAME() — returns the domain and username in domain\username format

C is correct. USERPRINCIPALNAME() returns the email address of the currently signed-in Power BI user (e.g., j.smith@company.com), which matches the format stored in a security mapping table keyed by email. This is the standard function for dynamic RLS in Power BI Service, where users authenticate via Azure Active Directory (Microsoft Entra ID) and their UPN is the email address. USERNAME() (option D) exists but returns a domain\username format that typically doesn't match the email addresses stored in security tables, making it less reliable. CURRENTUSER() is not a DAX function — it does not exist. LOOKUPVALUE() is a DAX lookup function used in calculated columns, not in RLS filter expressions.

Question 3 of 5

A Power BI workspace member (not Admin) has been given access to a report that uses Row-Level Security. During testing, the member reports seeing all data rather than just their permitted subset. What is the most likely cause?

A
The security table in the semantic model is missing the user's email
B
The RLS role was defined using static filters instead of dynamic
C
Members of the workspace Admin and Member roles bypass RLS by default — the user should be assigned the Viewer role to be subject to RLS restrictions
D
RLS only works in Import mode and the model is using DirectQuery

C is correct. In Power BI Service, workspace Admins and Members bypass RLS — they always see all data regardless of what RLS roles are defined. Only users with the Viewer workspace role are subject to RLS restrictions. This is a very common production oversight: a developer tests the RLS logic correctly in Power BI Desktop, publishes to the Service, assigns the user as a Member to let them "view" the report — and the RLS appears to do nothing. The fix: assign end-user consumers as Viewers, not Members. Members are for content contributors and developers. RLS does work with DirectQuery (option D is incorrect).

Question 4 of 5

What is the primary purpose of Power BI deployment pipelines?

A
To automatically refresh datasets on a scheduled basis across multiple workspaces
B
To promote content (reports, semantic models) through Dev, Test, and Production workspaces in a governed, repeatable way — with stage-specific data source configurations
C
To deploy Power BI Gateway configurations to new servers
D
To migrate reports from Tableau or Qlik into Power BI format

B is correct. Deployment pipelines are Power BI's built-in CI/CD mechanism for BI content. Three linked workspaces (Development, Test, Production) allow content to be promoted stage by stage with a single click. Crucially, each stage can have different data source configurations: Development connects to a dev database, Production connects to the live Gold layer — the same report works correctly in both environments because the data source is swapped at promotion time. This prevents the common anti-pattern of editing production reports directly. Deployment pipelines require Power BI Premium or Premium Per User (PPU) licensing.

Question 5 of 5

A financial services organisation is evaluating Power BI vs Looker for their new analytics platform. Their data team is highly technical (strong SQL engineers), their primary data warehouse is Google BigQuery, and they want strict centralised control over how metrics like "Revenue" and "Churn Rate" are defined across the business. Which tool is the stronger fit, and why?

A
Power BI — it has better business user adoption and Microsoft 365 integration
B
Looker — its code-first LookML semantic layer provides version-controlled, centralised metric governance; it integrates natively with BigQuery; and the SQL-first approach suits the technical team
C
Power BI — it is cheaper than Looker and has a larger market share
D
Looker — it is always the better tool for financial services organisations regardless of technical context

B is correct — in this specific context. Three factors align with Looker: the technical team is SQL-strong (LookML is SQL-based and version-controlled in Git, which technical teams manage well); the primary warehouse is BigQuery (Looker is Google-native and integrates with BigQuery without an intermediate layer); and the strict metric governance requirement matches Looker's core value proposition — LookML defines "Revenue" once in a central repository and every report that references it is guaranteed to use the same definition. Power BI has metric management features (Power BI Goals) but they are less mature than Looker's approach. This question illustrates the importance of context: Power BI is the better choice for Microsoft-ecosystem organizations with broad non-technical user bases; Looker is better for technically sophisticated teams with BigQuery and strict metric governance needs.

Default to Import

Start every Power BI project in Import mode. Move to DirectQuery or Composite only when a specific constraint — size limit or real-time requirement — forces the change. Don't pre-optimize for problems you haven't hit yet.

RLS Bypassed by Members

Workspace Admins and Members always see all data regardless of RLS. End-user consumers must be assigned the Viewer role. Test RLS by signing in as a Viewer — not as the developer who built it.

Measures, Not Columns

If the calculation aggregates (sum, count, average), it's a measure. Calculated columns increase model size for every row — only use them when a value must exist as a per-row attribute for slicing or sorting.

Interview Insight

"When would you use DirectQuery over Import?" is a standard Power BI interview question. The answer: real-time requirement OR dataset too large to import. Always add: "I'd evaluate Composite as the first option — Import dims, DirectQuery fact."
