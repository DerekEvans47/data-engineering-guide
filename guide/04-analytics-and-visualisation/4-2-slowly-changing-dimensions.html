4.2

Slowly Changing Dimensions

When dimension data changes over time — Types 0 through 7, the trade-offs of each, and Azure implementation patterns using dbt snapshots and Delta Lake MERGE

✓

Expected Learning Outcomes

  1. Explain what a Slowly Changing Dimension is and why the choice of SCD type has lasting consequences for analytical accuracy
  2. Apply SCD Types 0, 1, 2, and 3 correctly, and explain when each is the appropriate choice
  3. Describe the structural requirements of an SCD Type 2 dimension table: surrogate key, effective/expiry dates, and current-row flag
  4. Recognize SCD Types 4, 5, 6, and 7 and identify the specific problems they solve

4.2.1

The Problem: Dimensions Are Not Static

Section 4.1 introduced dimensions as the descriptive context surrounding a fact — a customer's region, a product's category, a store's type. The star schema presented there treated those dimensions as stable: dim_customer has a region column, and the value in that column is the region. But what happens when a customer moves? When a product is recategorized? When a store is reclassified from Standard to Premium?

These changes happen constantly in production systems. The term "Slowly Changing Dimension" (SCD) — coined by Ralph Kimball — acknowledges this reality. "Slowly" is relative: an SCD can change once a year (a company's legal name) or once a week (a customer's loyalty tier). What matters is that the change is infrequent enough to be modeled as a dimension attribute rather than as a fact event. The critical question is not whether dimensions change. They do. The question is: when they change, what happens to the historical facts that point to the old value?

The choice of SCD type is not a technical preference — it is a business accuracy decision. Apply the wrong type and your historical reports are quietly wrong: last year's regional sales totals reflect where customers live today, not where they lived when they bought.

Kimball defined seven SCD types (0 through 6, plus a hybrid Type 7). Types 0, 1, 2, and 3 are the foundational patterns. Types 4, 5, 6, and 7 are composites that solve specific edge cases. Every data engineer should be fluent in all of them — not because all are commonly used, but because interviewers expect command of the full vocabulary, and architects need to recognize when a hybrid approach is warranted.

4.2.2

Type 0 — Fixed (Retain Original)

SCD Type 0 is the simplest case: no change is permitted. The dimension attribute is set at initial load and never updated, regardless of what happens in the source system. This is appropriate for attributes that are definitionally immutable in the context of the business model: a customer's original acquisition channel (how they first came to the business), a contract's inception date, a financial instrument's (a security, bond, or derivative) country of issuance. These are facts about the entity at a specific moment in time — updating them would destroy information.

Type 0 is not a design decision to make by default or by laziness. It is a deliberate statement that this attribute's original value has permanent analytical meaning. The implementation is trivial: on updates from the source, ignore the changed column. In dbt, this means not including the column in the snapshot definition. In an ADF pipeline, filter out or discard updates to this field.

4.2.3

Type 1 — Overwrite (No History)

SCD Type 1 overwrites the existing value when a change is detected. The dimension table always reflects the current state. No additional rows are created. No historical values are preserved. A customer who moves from London to Manchester has their region column updated to "Manchester" — and every historical fact that joins to this customer via the surrogate key now shows "Manchester" regardless of when the sale occurred.

When Type 1 Is Correct

Type 1 is the right choice when either the change is a correction (a misspelled name, a wrong postcode entered at point of sale) or when historical context for that attribute genuinely does not matter to any analytical use case. If no stakeholder has ever asked "where did this customer live at the time of the purchase?" — and no regulatory or audit requirement exists — Type 1 is simpler and cheaper to implement and maintain.

The Risk of Defaulting to Type 1

Type 1 is the most commonly misapplied SCD type precisely because it is the easiest to implement. Teams apply it by default — often because they didn't know there was an alternative — and discover the problem only when someone asks a historical question that can no longer be answered. Once history is overwritten in a Type 1 dimension, it cannot be recovered from the warehouse. It is gone unless it exists in an audit log, a raw Bronze table, or the source system's own change history. The Bronze layer's role as the immutable raw record becomes critical here: if you ever realize you needed Type 2 but applied Type 1, you can rebuild the dimension from Bronze if that layer has been preserved.

Type 1 implementation is straightforward: a MERGE (also written as UPSERT — insert if new, update if exists) on the dimension table's natural key, overwriting changed columns. In Delta Lake: MERGE INTO dim_customer USING updates ON dim_customer.customer_id = updates.customer_id WHEN MATCHED THEN UPDATE SET ...

🏢Field Note
A Retail Group  ·  Two Years of Regional Data, Silently Wrong

A national retail group had built regional sales reporting over two years. Customers were assigned to regions based on their delivery postcode. The dim_customer table used SCD Type 1 — when a customer updated their address, the region was overwritten. Nobody had thought to question this when the model was built.

Two years later, the commercial director noticed that the Southern region appeared to consistently outperform Northern for the same product categories — a gap that didn't match the on-the-ground experience of Northern regional managers. Investigation revealed that the South-East of England had the highest postcode mobility (young professional renters moving into the area from other regions), meaning thousands of customers who had historically bought as Northern customers were now classified as Southern. The two-year trailing figures had been quietly reattributed as customers moved.

Rebuilding with Type 2 required going back to Bronze ingestion logs to reconstruct the customer-region timeline. That was only possible because the Bronze layer had been designed as append-only. The rebuild took three weeks. The prior two years of regional reporting were re-issued to the board with a correction notice.

The Lesson

Type 1 is never "safe by default." The question to ask before applying it is: could any business user, in any future analytical context, need to know what this attribute's value was at a specific point in time? If the honest answer is "possibly yes," the correct choice is Type 2.

4.2.4

Type 2 — Add Row (Full History)

SCD Type 2 is the most important and most widely applicable SCD pattern. When a dimension attribute changes, a new row is inserted into the dimension table with a new surrogate key. The original row is marked as expired. Historical facts continue to reference the original surrogate key — and therefore the original attribute values. New facts reference the new surrogate key and see the updated values. The dimension table maintains a complete, auditable history of every version of every entity.

The Required Columns

Beyond the regular dimension attributes, a Type 2 dimension table requires three additional columns that collectively allow users to identify which version of a record was current at any given point in time:

row_effective_date: the date (or datetime) from which this row version became the active truth. Set to the load date when a new version is inserted.

row_expiry_date: the date from which this row version ceased to be the active truth. Set to a sentinel value (9999-12-31 is the standard) when the row is current, and updated to the day before the new version's effective date when the row is expired.

is_current: a boolean flag (TRUE/FALSE or 1/0) indicating whether this row is the currently active version. Redundant with date logic but dramatically simplifies queries: WHERE is_current = TRUE rather than WHERE row_expiry_date = '9999-12-31'.

Figure 4.3SCD Type 2 before and after a customer moves regions. The expired row (sk=1001) retains its surrogate key and historical facts continue to join to it — correctly reporting "London" for 2022 and 2023 sales. The new row (sk=1002) carries the updated region and all new facts from 2024 onwards point here. The natural key (C-001) is the same in both rows; only the surrogate key distinguishes the versions. The effective_date, expiry_date, and is_current columns allow any query to reconstruct the dimension state at any point in time.

Go DeeperImplementing SCD Type 2 with dbt snapshots — the production pattern

dbt snapshots automate SCD Type 2 without requiring you to write the comparison, expiry-date update, and insert logic yourself. You define a snapshot once; dbt handles the rest on each run.

Snapshot definition (in snapshots/snap_customers.sql):

{% snapshot snap_dim_customer %}
{{ config(target_schema='gold', unique_key='customer_id', strategy='timestamp', updated_at='updated_at') }}
SELECT customer_id, name, email, region, segment FROM {{ source('silver', 'customers') }}
{% endsnapshot %}

Two snapshot strategies: timestamp (recommended) compares the updated_at column — if it's newer than the last seen value, dbt treats the row as changed and creates a new version. check strategy compares specified column values directly — slower but works when source systems don't provide reliable updated_at timestamps.

What dbt snapshots add automatically: dbt_scd_id (hash surrogate key for the version), dbt_updated_at, dbt_valid_from (equivalent to row_effective_date), and dbt_valid_to (equivalent to row_expiry_date, NULL for current rows). Run dbt snapshot to execute; combine with dbt run in your pipeline to keep Gold marts up to date.

Important limitation: dbt snapshot tables grow indefinitely. Monitor row counts and consider archiving very old expired rows (pre-2015, say) to a cold storage table if the snapshot table becomes unwieldy. In practice, most dimension tables are small enough that this never becomes an issue — customer tables with 5 million active customers and 3 years of SCD history rarely exceed 20–30 million rows.

4.2.5

Type 3 — Add Column (Limited History)

SCD Type 3 stores a single previous value alongside the current value, using an additional column per tracked attribute. A dim_customer with Type 3 for the region attribute gains a previous_region column. When a customer moves from London to Manchester, the current_region updates to "Manchester" and the previous_region stores "London."

Type 3 is occasionally the right choice when: you need before/after comparison but only one level of history, the attribute changes infrequently (once in the entity's lifetime), and the "before" value has specific analytical meaning. A common example: tracking the initial onboarding channel and the current primary channel for a customer — "how many customers who originally came through organic search are now classified as loyalty-tier members?" requires exactly two states, no more.

The limitation is severe: Type 3 handles exactly one historical change per attribute. A customer who moves three times has their first address lost permanently after the second move. More than one change makes Type 3 inadequate. For most slowly-changing attributes in production systems, Type 2 is the appropriate choice and Type 3 is a compromise made for implementation simplicity that usually creates regret later.

4.2.6

Types 4, 5, 6, and 7 — Advanced Patterns

The higher-numbered SCD types are composites — combinations of the foundational types (0, 1, 2, 3) designed to solve specific performance or modelling problems that the basics cannot handle cleanly. In interviews, knowing these exists and when they apply is the signal of senior-level modeling knowledge. In production, you will encounter Types 4 and 6 most frequently.

Type 4 — Mini-Dimension

Type 4 addresses a specific performance problem: when a subset of dimension attributes changes frequently (demographic band, price tier, credit rating) while the rest of the dimension is stable. Including the rapidly-changing attributes in the main dimension table causes Type 2 to generate enormous numbers of expired rows. Type 4 separates the volatile attributes into a small separate "mini-dimension" table with its own surrogate key. The main dimension references the mini-dimension by FK. When the mini-dimension attribute changes, only the small table is updated — the main dimension is untouched.

Type 5 — Mini-Dimension + Type 1 Outrigger

Type 5 extends Type 4 by adding a Type 1 (overwrite) reference to the current mini-dimension row directly in the main dimension. This allows users to query the current state of the volatile attributes without joining to the mini-dimension, while historical facts still join through the mini-dimension FK preserved in the fact table. Rarely implemented in modern cloud lakehouses — columnar storage makes joins cheap enough that the optimization is usually unnecessary.

Type 6 — Hybrid (1 + 2 + 3)

Type 6 is the most practically useful of the advanced types. It combines: Type 2 (new row on change, with effective/expiry dates), Type 1 (overwrite a "current value" column on all rows of that entity), and Type 3 (carry the previous value in a separate column). The result: a dimension table where historical rows carry both their historical value AND the current value. This means you can query "what is this customer's current region?" without filtering on is_current — and still see historical context on older rows. Often described as 1+2+3=6. Used in environments where analysts frequently need current state alongside historical context in the same query.

Type 7 — Dual Type 1 and Type 2

Type 7 maintains a single physical dimension table but exposes two views from it: a current-state view (equivalent to Type 1, showing only is_current rows) and a full-history view (equivalent to Type 2, showing all rows). The fact table stores a reference to both: the durable surrogate key (points to the Type 2 historical row) and the natural key (used by the Type 1 current-state view). Analysts choose which view to join based on their query intent. Elegant in theory; complex in implementation and often replaced by simply creating two views in the semantic layer.

Figure 4.4SCD type selection flowchart. Start at the top with any dimension attribute that changes. Three binary decisions determine the correct type. Type 2 (add row, full history) is the default for any attribute where history matters and changes happen more than once. Types 4–7 layer on top of Type 2 when performance or query-pattern requirements demand a composite approach — not as alternatives to it.
4.2.7

Implementation in the Azure Stack

The Azure stack offers multiple implementation paths for SCD logic. The choice depends on where your transformation layer lives and which type you are implementing.

SCD Type 2: dbt Snapshots (Recommended for dbt-managed Gold)

dbt snapshots are the cleanest implementation of Type 2 in a dbt-managed pipeline. Define the snapshot once in the snapshots/ folder using the {% snapshot %} block, specify the unique_key (natural key) and strategy (timestamp or check), and run dbt snapshot. dbt compares the current source data against the snapshot table, expires changed rows (sets dbt_valid_to), and inserts new rows for the updated values. The snapshot table grows as history accumulates. Mart models built on the snapshot always filter to dbt_valid_to IS NULL for current state, or apply date-range filters for historical queries.

SCD Type 1: Delta Lake MERGE (Recommended for Databricks Gold)

When transformations are managed directly in Databricks PySpark rather than dbt, Delta Lake's MERGE command implements Type 1 efficiently. MERGE evaluates a source dataset against the target Delta table, updating matched rows (WHEN MATCHED THEN UPDATE) and inserting new rows (WHEN NOT MATCHED THEN INSERT). Delta Lake's transaction log ensures the operation is atomic — no partial updates if the job fails.

+
Go DeeperDelta Lake MERGE implementation for Type 1 — annotated example

A full Delta Lake MERGE for SCD Type 1 in PySpark:

from delta.tables import DeltaTable
dim_table = DeltaTable.forPath(spark, "abfss://gold@storage.dfs.core.windows.net/dim_customer")
updates_df = spark.table("silver.customers_latest")

dim_table.alias("target").merge(
  updates_df.alias("source"),
  "target.customer_id = source.customer_id"
).whenMatchedUpdate(set={
  "name": "source.name",
  "email": "source.email",
  "region": "source.region",
  "updated_at": "source.updated_at"
}).whenNotMatchedInsertAll().execute()

Key points: The merge condition is the natural key (customer_id). WHEN MATCHED updates all Type 1 columns; the surrogate key (customer_sk) is deliberately excluded from the update — it is set once on INSERT and never changed. WHEN NOT MATCHED handles new customers. The operation is atomic in Delta Lake — if it fails at any point, the table rolls back to its state before the MERGE began.

Adding Type 2 to Delta Lake: Type 2 in pure PySpark is more complex than dbt snapshots. The pattern requires: (1) identify changed rows by comparing source to current dim; (2) expire old rows by updating their expiry_date and is_current flag; (3) insert new rows with new surrogate keys and effective_date set to today. This is multi-step logic. For Type 2 in Databricks, dbt snapshots running against Databricks SQL warehouses is the lower-maintenance path; only build custom PySpark Type 2 logic if dbt is not part of your stack.

🏢Field Note
A B2B SaaS Company  ·  The Price Column Mistake

A B2B SaaS company built a dim_product table that used SCD Type 2 for all columns, including unit_price. The price changed whenever the commercial team updated their pricing model — which happened three to four times per year. SCD Type 2 was technically correct: historical invoices needed to show the price the product had at the time of sale, and the team wanted to preserve that history.

Six months later, the team added list_price_daily — the price at which the product was listed to new prospects on any given day. This changed with every promotion, every campaign, and every A/B test. Within two months, dim_product had 340,000 rows for 800 active products. Power BI report load times on the semantic model quadrupled. The query WHERE is_current = 1 was no longer sufficient — current state queries had to filter across a dimension that had become effectively a fact table.

The fix was to separate list_price_daily into a Type 4 mini-dimension (dim_product_pricing), updated daily, with the main dim_product retaining only the attributes that changed quarterly. The mini-dimension had 800 rows per day — a manageable 584,000 rows per two-year period, but stored separately and joined only when pricing history was needed.

The Lesson

Applying Type 2 to a high-frequency attribute is a common mistake. If an attribute changes daily, weekly, or more than quarterly, consider whether it belongs in a dimension at all — or whether it is better modeled as a Type 4 mini-dimension or even as a fact.

Go DeeperLate-arriving changes — what happens when SCD updates arrive out of order

SCD Type 2 implementations assume updates arrive in chronological order. In practice, they sometimes don't — source systems batch updates, CDC (Change Data Capture) streams are occasionally delayed, or a historical correction is applied retroactively.

The problem scenario: Your snapshot runs on 16 March and correctly captures the change from London to Manchester. On 20 March, a correction arrives saying the actual change date was 12 March (not 16). The snapshot now has an incorrect effective_date on the Manchester row.

How dbt snapshots handle this: By default, they don't — dbt sets dbt_valid_from to the timestamp of the snapshot run, not the business date of the change. If business-date accuracy matters, you must either (a) include the business change date as a source attribute and apply custom logic to set effective_date from it, or (b) accept snapshot-run-date as the effective_date and rely on the transaction log in the Bronze layer for exact-date reconstruction.

The Bronze layer as the backstop: This is another reason the Bronze layer must be append-only and never overwritten. If a late-arriving correction makes a SCD rebuild necessary, you can replay Bronze records in chronological order to reconstruct a correct Type 2 history. Without Bronze, that reconstruction is impossible.

For regulatory environments: where the exact change date is auditable, consider adding a source_change_date column to your snapshot that captures the business date of the change from the source system, and use that in reporting rather than dbt_valid_from.

✓ Section Summary

SCDs handle the reality that dimension data changes over time. The choice of type determines whether historical analytical reports are accurate. Type 0 (fixed) is for immutable attributes — never update. Type 1 (overwrite) is for corrections and attributes with no historical requirement — always shows current state, destroys history. Type 2 (add row) is the standard default for any attribute where history matters — inserts a new row with a new surrogate key, expires the old row with an expiry date, preserves full history through FK relationships in fact tables. Requires surrogate keys, effective/expiry dates, and is_current flag. Implemented in Azure via dbt snapshots or custom Delta Lake MERGE logic. Type 3 (add column) stores one previous value — limited to a single historical change. Types 4, 5, 6, 7 are composites: Type 4 separates volatile attributes into a mini-dimension; Type 6 combines 1+2+3 for current-state accessibility without filtering. The most common interview mistake: not knowing that Type 1 destroys history, or applying Type 2 to high-frequency columns (daily price changes) that will cause table explosion.

A customer's email address was entered incorrectly at registration. The correct email is now available from the source system. Which SCD type should be applied?

A
Type 0 — the original value should be retained as it was the email at time of registration
B
Type 1 — overwrite the incorrect value; this is a data quality correction, not a genuine attribute change, so preserving the wrong email has no analytical value
C
Type 2 — insert a new row with a new surrogate key preserving the typo as history
D
Type 3 — store the old incorrect email in a previous_email column

B is correct. SCD Type 1 (overwrite) is the correct choice for data quality corrections. A misspelled email is not a genuine attribute change that an analyst would ever need to preserve for historical reporting. The old value has no analytical meaning — preserving it via Type 2 would create a dimension row that accurately records an error, which is neither useful nor accurate. Type 1 corrects the record in place. This is the clearest use case for Type 1 and a standard interview question to test whether the candidate can distinguish between corrections and genuine attribute evolution.

Question 2 of 5

A retailer needs to track which region each customer belonged to at the time of every historical sale — including customers who have moved regions since those sales occurred. Which SCD type is required?

A
Type 1 — overwrite the region; current region is what the business reports against
B
Type 3 — add a previous_region column so the old region is available
C
Type 2 — insert a new row with a new surrogate key when the region changes; historical fact rows continue to join to the old surrogate key, preserving the correct regional attribution
D
Type 0 — retain the original region regardless of changes

C is correct. This scenario requires full historical accuracy: the fact that a sale occurred when the customer was in region A must be preserved even after the customer moves to region B. Only SCD Type 2 achieves this correctly. When the customer moves, a new dim_customer row is created with a new surrogate key. All new sales reference the new SK (region B). All historical sales continue to reference the original SK (region A) — because that was the FK value at the time the fact was inserted. Type 1 would overwrite the region, causing all historical sales to silently reattribute to the new region. Type 3 stores one previous value — insufficient if the customer moves more than once.

Question 3 of 5

Which three additional columns does an SCD Type 2 dimension table require, beyond its standard business attributes?

A
version_number, change_reason, modified_by
B
row_effective_date (when this version became active), row_expiry_date (when this version was superseded), and is_current (boolean flag for the active row)
C
created_date, deleted_date, is_deleted
D
source_system, batch_id, load_timestamp

B is correct. SCD Type 2 requires exactly these three control columns to support historical queries. row_effective_date marks when this version became the active truth (typically the load date or the business change date). row_expiry_date marks when this version was superseded — set to 9999-12-31 for current rows, updated to one day before the new row's effective date when expired. is_current (TRUE/FALSE) is technically redundant with the date logic but makes current-state queries dramatically simpler: WHERE is_current = TRUE rather than WHERE row_expiry_date = '9999-12-31'. In dbt snapshots these are named dbt_valid_from, dbt_valid_to, and must be inferred (dbt_valid_to IS NULL means current).

Question 4 of 5

A dim_product table uses SCD Type 2. A product's list price changes three to four times per day due to dynamic pricing. What problem does this cause, and what is the correct solution?

A
No problem — Type 2 handles high-frequency changes by design and will simply create more rows
B
Switch to Type 1 — overwrite the price on each change, accepting the loss of pricing history
C
The main dimension table will grow explosively (hundreds of thousands of rows per year per product), degrading query performance. Separate the price into a Type 4 mini-dimension table, updated frequently, keeping the main dim_product lean
D
Switch to Type 3 — store current_price and previous_price columns; this handles frequent changes more efficiently

C is correct. Type 2 applied to a high-frequency attribute creates table explosion: for 800 products with price changes 3x per day, you generate 2,400 new rows per day — 876,000 new rows per year. After two years, dim_product has ~1.8 million rows for 800 products. Power BI semantic model load times, is_current queries, and join performance all degrade. The solution is SCD Type 4: separate the volatile attribute (list_price) into a dedicated mini-dimension (dim_product_pricing) that has its own lifecycle and is updated independently. The main dim_product retains only the attributes that change slowly (category, brand, product name) and stays compact. Fact tables reference both the main dim_product SK and the dim_product_pricing SK.

Question 5 of 5

In a dbt snapshot using the timestamp strategy, what does dbt use to detect that a dimension row has changed and requires a new version?

A
A hash comparison of all columns in the source row against all columns in the current snapshot row
B
The updated_at timestamp column — if the source row's updated_at is more recent than the last time dbt saw it, the row is treated as changed and a new version is inserted
C
A row count comparison between the source table and the snapshot table
D
The natural key — any row whose natural key appears in the source is treated as changed on every snapshot run

B is correct. The dbt snapshot timestamp strategy compares the updated_at column (or whichever timestamp column you specify in the config) between the source and the current snapshot row. If source.updated_at > snapshot.dbt_updated_at for a matching natural key, dbt treats that row as changed: it expires the existing snapshot row (sets dbt_valid_to) and inserts a new row. The alternative strategy is check, which compares specific column values directly — useful when source systems don't maintain reliable updated_at timestamps, but slower because it requires column-by-column comparison. The hash comparison approach described in option A is actually similar to dbt's internal implementation of check strategy, but it is not what the timestamp strategy does.
