# Map Art Pipeline — Post-Processing Generated Maps

Every battle/region map generated by Nano Banana (Gemini) goes through this
pipeline before it can be committed as a game asset. None of these steps are
optional — the generator reliably produces the defects they fix.

## 1. Gemini sparkle watermark — check & removal

### The problem

Gemini stamps a **semi-transparent, light-gray, 4-point-star ("sparkle")
watermark** near a corner of generated images — most often bottom-right,
sometimes bottom-left (and it mirrors along if you flip the image). It:

- survives explicit "no watermark, no logo, no sparkle marks" prompt language
  (tested repeatedly — do not expect prompting to prevent it);
- appears on most but not all generations, so every export must be checked;
- is translucent, so over dark texture (forest) it reads as a subtle gray
  ghost rather than a bright mark — easy to miss at a glance.

### The insight: you do not need generative inpainting

A common belief (including among AI assistants) is that watermark removal
requires a diffusion/inpainting model, and that a text-model session
"can't do it." **False for this watermark.** The sparkle is small (~50px on a
1520px-wide export) and always lands on *repeating organic texture* — forest
canopy or grass. Replacing it with a **feathered clone-stamp of a nearby patch
of the same texture** is visually seamless at game zoom. Plain PIL + numpy,
no ML anywhere, fully deterministic, done in-session.

Removal was validated on the fortified-frontier map
(`Gemini_Generated_Image_i7e101i7e101i7e1.png`, sparkle at (1398, 585) over
dense pines): the patched area is indistinguishable from surrounding forest at
1:1 zoom.

### The process

```bash
# 1. Propose candidates (bright + desaturated blobs vs. local neighborhood)
python3 scripts/remove_gemini_sparkle.py map.png
#   candidates (confirm visually, then re-run with --center):
#     (1399, 583)  size ~47px  783px blob     <- corner-most listed first
#     (1164, 145)  size ~19px  155px blob

# 2. Confirm each candidate by eye (gray rocks/stone false-positive!)
#    Crop ~100px around the coordinate and look:
python3 -c "from PIL import Image; Image.open('map.png').crop((1340,525,1460,645)).resize((480,480)).save('/tmp/cand.png')"

# 3. Pick a clean clone source: same texture, offset dx,dy from the sparkle.
#    CROP AND CHECK THE SOURCE FIRST — it must contain no boulders, buildings,
#    clearing edges, or any distinctive feature, or you'll paste a visible twin.

# 4. Remove (feathered clone-stamp), then re-run detection on the output:
python3 scripts/remove_gemini_sparkle.py map.png --center 1399,583 --size 100 --source 45,-120 --out map-clean.png
python3 scripts/remove_gemini_sparkle.py map-clean.png   # sparkle gone; rocks may still list
```

### Detection notes (why it's semi-automatic, not automatic)

- The sparkle is translucent, so absolute-brightness thresholds fail over dark
  forest. Detection compares each pixel against its **local 120px-tile
  median**: sparkle pixels are brighter *and* grayer than their neighborhood.
- Gray **rocks and stone walls false-positive** — they're also bright+gray
  local blobs. Shape filters (compactness, aspect, size) cut most, but a human
  look at each candidate is required before removal. Candidates are ranked
  corner-most first, since that's where Gemini stamps.
- Flipping a map flips the watermark with it — check the *mirrored* corner on
  any image that has been `transpose`d.

## 2. Road edge contact — check & crop

Generations consistently refuse to run the road off the frame edge (vignette
bias: image models keep subjects inside the border). Even with crop-framing
prompt language, verify with pixel detection and fix by cropping:

- Detect path-colored pixels (tan: `r > 140`, `r > b + 30`, `g > b + 10`,
  `r >= g`) per column; the road's column span vs. the canvas span tells you
  how much dead margin each side has.
- Crop the entry side (and exit side for through-maps) to the road's actual
  bounds so the path touches the border by construction. Landmark-terminus
  maps (road ends at a town gate): **never crop the landmark side based on
  road pixels** — it would cut into the settlement.
- The game's fit code scales any aspect ratio, so post-crop dimensions don't
  need to hit 2.16:1 exactly — just record the new size for the map's config.

## 3. Content checklist (from the master prompt criteria)

Per generation, before accepting:

- [ ] No people (frozen villagers break combat scenes)
- [ ] Clearings: enough, organic-shaped, empty, mixed near-road and fringe
      (near = mortar/bastion/barracks-viable, fringe = ranger-only —
      the mix is what keeps tower strategy open, don't accept a map that
      railroads one build)
- [ ] Road enters/exits per the map's spec (through-map vs. landmark terminus)
- [ ] Scale sane: largest building ~8–20% of canvas width per the map's spec
- [ ] Sparkle watermark checked (step 1) — all four corners
- [ ] No text/labels/icons anywhere

## 4. Integration reminder

A cleaned map is scenery only. Before it's playable, hand-place against the
final pixels: waypoint path, build-slot positions (in clearings), and any
region-map node coordinates. Budget real time for this — it is not a config
tweak.
